# 四阶段不可解案例深度诊断

## 1. 结论先行

本次审查覆盖：

- `data/truth2`：113 个案例；
- `data/truth3`：34 个案例；
- 合计：147 个案例。

综合最近一次全量产物、后续定点修复产物，以及对关键案例使用当前代码进行的复跑，持续阻塞全流程严格完成的根因案例共有 16 个。它们不能统一称为“数学不可解”，而应分为四类：

| 类型 | 案例 | 数量 | 是否证明全局物理不可解 |
|---|---|---:|---|
| 唯一目标线硬容量不足 | `0424Z`，以及 Stage4 的 10 个容量案例 | 11 | 是，在目标和容量不变时不可解 |
| 阶段边界或冻结规则导致的模型不可行 | `0103W`、`0327Z`、`0406W` | 3 | 否，只证明当前阶段契约不可行 |
| 搜索策略形成拓扑死锁 | `0415W` | 1 | 否，已构造并验证可行反例 |
| 上下阶段交接状态不可达 | `0211Z` | 1 | 否，改变交接机车位置后 Stage2 可完成 |

最重要的结论有五条：

1. **真正由物理长度直接证明不可解的，是 11 个唯一目标线容量案例。**增加搜索深度、候选数量或运行时间均无效。
2. **`0103W` 不是全流程不可解，而是“Stage1 要求所有大库车同时停进四条编组线”这一阶段边界不可行。**26 辆车总长 411.4m，而四条编组线只有 333.2m。
3. **`0415W` 是当前 Stage1 策略失败，不是问题本身不可解。**在第一个 planlet 后先执行一组合法的“清空机走棚、放入机南”动作，当前求解器随后可以严格完成。
4. **`0211Z` 是 Stage1 和 Stage2 的接口契约缺项。**Stage1 债务为零，但把调机留在被三条占用编组线围住的 `机走棚`；Stage2 不允许移动这些编组线车辆，因此 0 勾即失败。
5. **`0327Z`、`0406W` 的 Hall 不可行证书是严格的，但严格范围只是“已到位库内车冻结不动”的当前 Stage3 模型。**若允许临时取出并恢复已到位车辆，可用槽位集合会改变。

因此，下一步不应继续对每个案例增加候选补丁。首先需要修改阶段契约：

- 把阶段终态从“本阶段债务为零”扩展为“本阶段债务为零且下一阶段可进入”；
- 把编组线、机走线同时建模为存车资源和拓扑通道资源；
- 对超过瞬时容量的作业线采用分批/队列模型；
- 对确实无解的输入在搜索前给出可核验的不可行证书。

#### A. 物理/业务硬不可行

存在独立于搜索过程的矛盾，例如：

```text
唯一目标车辆总长 > 目标股道有效长度 + 0.5m 容差
```

在当前阶段附加约束下无解，但放宽阶段边界后可能有全局解，例如：

- 所有大库车必须在 Stage1 结束时同时进入四条编组线；
- 已到位库内车辆在 Stage3 中完全冻结；
- Stage2 只能处理大库/卸轮来源，不能清理 Stage1 遗留的通道车。

#### C. 搜索未找到解

没有不可行证书，且可以构造合法替代动作。这属于动作生成、状态评价或前瞻能力不足。

#### D. 上游不可用

例如 Stage3 报 `stage2_not_complete`。这不是新的 Stage3 无解，只是上游根因的传播。

## 4. Stage1：`0103W` 的本质是阶段容量不可行

### 4.1 精确容量账

Stage1 将去大库车辆的合法编组线限定为：

| 编组线 | 有效长度 |
|---|---:|
| 机南 | 90.1m |
| 洗油北 | 62.9m |
| 机走棚 | 111.1m |
| 机走北 | 69.1m |
| 合计 | 333.2m |

`0103W` 中需要进入这些编组线的车辆为 26 辆，总长 411.4m：

| 初始来源 | 辆数 | 总长 |
|---|---:|---:|
| 存1线 | 4 | 56.1m |
| 存2线 | 7 | 118.8m |
| 存3线 | 8 | 127.6m |
| 洗罐站 | 1 | 12.1m |
| 洗罐线北 | 6 | 96.8m |
| 合计 | 26 | 411.4m |

按照代码中的 0.5m 容差：

```text
Stage1 编组容量缺口
= 411.4 - 333.2 - 0.5
= 77.7m
```

此外还有 2 辆去卸轮线的车需要进入存4线，所以初始 `stage1_debt` 是 28，而不是 26。

### 4.2 最少必须暂缓多少辆

对 26 辆大库车做最小留置组合计算，至少要暂缓 5 辆；存在一组 5 辆总长 78.1m，刚好覆盖 77.7m 缺口。因此：

```text
当前 Stage1 严格完成：不可能
Stage1 可行动完成：最多先编入 21 辆大库车，至少暂缓 5 辆
全流程分批完成：仍有可能
```

### 4.3 当前求解器不仅遇到容量矛盾，还浪费了搜索预算

现有产物显示：

- 只执行了 2 个 planlet、4 个业务勾；
- 债务从 28 降到 21，只完成了 7 辆；
- 第 3 个决策点检查 133 个候选后耗时 301.573 秒；
- 最终原因为 `solve_time_budget_exhausted`，同时报告 `assembly_capacity_deficit_m:77.7`。

这说明当前行为有两层问题：

1. **结构上严格无解。**即使搜索无限快，也不能把全部 26 辆同时放入四条线。
2. **实现上没有在开局立即切换到容量受限模式。**`depot_assembly_capacity_deficit_m()` 是静态可计算的，但主循环要等无进展次数或勾数达到阈值才因容量退出；本例先被候选评估耗尽时间。

相关代码：

- `scripts/stage1_simple/solve.py::depot_assembly_capacity_deficit_m`
- `scripts/stage1_simple/solve.py::stage1_car_complete`
- 主循环中的 `assembly_capacity_impossible` 判定

### 4.4 正确处理方式

不应把 `0103W` 当作普通搜索失败。应在搜索前返回：

```json
{
  "status": "contract_capacity_infeasible",
  "required_m": 411.4,
  "capacity_m": 333.2,
  "tolerance_m": 0.5,
  "overflow_m": 77.7,
  "minimum_holdout_count": 5
}
```

业务上有两种可选合同：

1. **Stage1 actionable complete**：先编入最大可行子集，明确留下最少 5 辆；
2. **Stage1/Stage3 滚动波次**：Stage1 编一批，Stage3 送库释放编组线，再回到 Stage1 编下一批。

第二种才符合“如果一次调不完，可以多次”的业务语义。当前严格串行的四阶段结构无法表达这种波次。

## 5. Stage1：`0415W` 是可解的拓扑死锁

### 5.1 当前版本复跑结果

使用当前代码、60 秒预算复跑 `0415W`：

```text
status                  = partial
business_hooks          = 14
pending_stage1_cars     = 4
pending                 = 1578843,1785553,4897699,5495955
blocking                = put/get route blocked + dead_end_after_candidate
service_satisfied_gain  = -1
```

这不是容量不足：`机南`、`洗油北`仍为空，`机走北`也有余量。

### 5.2 真正的阻塞结构

所有股道只能从北端作业。对从预修线取出的车辆，去南侧编组线的关键通道可抽象为：

```text
北场 -> 机走北 -> 机走棚 -> 机南
                         -> 洗油北
```

本例开局时，`机走棚`已有 3 辆大库车：

| 车辆 | 长度 |
|---|---:|
| 3428729 | 16.5m |
| 3837065 | 17.6m |
| 3463012 | 16.5m |
| 合计 | 50.6m |

求解器没有先清空这条通道，反而执行：

1. 从调梁棚取 4 辆，继续放入机走棚；
2. 后续把 3 辆放入机走北；
3. 再尝试把预修线剩余车辆放入机南/洗油北。

到最后：

- `机走棚`有 7 辆、101.2m；
- `机走北`有 3 辆、40.7m；
- `机南`、`洗油北`虽为空，但去往它们必须穿过已占用的机走北/机走棚；
- 对预修线车辆放机南，阻塞集合为 `{机走北, 机走棚}`；
- 放洗油北，阻塞集合为 `{机走棚}`；
- 放机走棚容量不足；
- 放机走北后，剩余车辆进入不可恢复死局，因此被 `dead_end_after_candidate` 拒绝。

### 5.3 可行反例已经通过物理校验

在第一个 planlet 完成后，不执行“调梁棚 4 辆 -> 机走棚”，而是先执行：

```text
Get  机走棚: 3428729,3837065,3463012
Put  机南:   3428729,3837065,3463012
```

当前 `physical.validate_planlet` 的结果：

```text
accepted = true
Get path = 机北1 -> 机北2 -> 渡5 -> 机走北 -> 机走棚
Put path = 机走棚 -> 机南
```

将这一合法“通道打开动作”注入后，再让当前 Stage1 求解器继续运行，得到：

```text
status              = complete
business_hooks      = 23
pending_stage1_cars = 0
```

因此 `0415W` 已经被反例严格证明为：

> 不是数学不可解，而是当前搜索没有在正确时机生成并选择通道打开动作。

### 5.4 缺失的能力

当前评价主要看车辆债务和局部延续性，缺少“拓扑割容量”概念。应增加：

1. **通道股道识别**：`机走北`、`机走棚`不仅是编组线，也是去 `机南/洗油北` 的割集。
2. **远端优先填充**：先填必须穿过割集才能到达的远端线，再填靠近北场的线。
3. **主动清通道动作**：即使车辆已经 Stage1 到位，只要它占用了关键割集，也允许有目的地迁移到另一编组线。
4. **终态连通势函数**：评价放车后“剩余待办源到剩余目标”的可达关系，而不只是 depth=1 是否还有候选。

一个可计算的指标是：

```text
R(S) = 当前状态下，所有 pending 车辆到任一合法目标的可达对数量
```

候选若让 `R(S') < R(S)`，且减少的是不可替代路径，应作为硬风险，而不是普通 tie-break。

## 6. Stage2：`0211Z` 是交接机位错误

### 6.1 Stage1 看起来完成，但 Stage2 一勾也走不了

`0211Z` 的 Stage1 状态为 complete，Stage2 当前复跑结果为：

```text
operations         = 0
global expansions  = 1
episode expansions = 20
status             = partial
```

主要阻塞为：

```text
get_route_blocked:机走棚->修2库内
get_route_blocked:机走棚->修3库内
get_route_blocked:机走棚->修4库内
get_route_blocked:机走棚->卸轮线
```

### 6.2 Stage1 终态把调机封在了机走棚

Stage1 最后一勾 Put 到机南后，调机站位为 `机走棚`。它周围三条可离开线路全部被占用：

| 相邻方向 | 车辆数 | 已占长度/容量 |
|---|---:|---:|
| 机走北 | 4 | 56.1/69.1m |
| 洗油北 | 4 | 57.2/62.9m |
| 机南 | 5 | 72.6/90.1m |

这些线路在容量上并未满，但物理模型禁止调机穿越停有车辆的普通股道。结果是调机所在连通分量只剩 `机走棚` 本身。

### 6.3 Stage2 无法自救

Stage2 的动作源集合定义在：

```python
SOURCE_LINES = (
    修1-修4库内,
    修1-修4库外,
    卸轮线,
    存4线,
)
```

它不允许把 `机走北/机走棚/机南/洗油北` 上的大库编组车作为清障源。因此：

- Stage1 交给 Stage2 一个不可达起点；
- Stage2 的单调 episode 模型又禁止清理造成不可达的线路；
- 扩大 Stage2 搜索宽度不会产生任何第一步。

### 6.4 反事实实验说明车辆任务本身可解

保持 Stage1 最终车辆布局完全不变，只把 Stage2 初始机车位置从 `机走棚` 改为 `渡5`，当前 Stage2 优化器可以生成：

```text

pending_stage2_nos     = 0
standalone replay      = physical ok
model state consistent = true
```

组合回放失败的唯一原因是原 Stage1 实际记录的终点仍是 `机走棚`，与实验机位 `渡5` 不一致。

这证明：

> `0211Z` 的核心不是车辆顺序或 Stage2 搜索能力，而是 Stage1 没有保证下一阶段可进入。

## 7. Stage3：Hall 不可行与真实容量不可行

### 7.1 Hall 条件是什么

Stage3 的统一布局器把每辆强制入库车看作左侧节点，把可使用的库内位置看作右侧槽位。每辆车只连接到满足目标库线、强制台位、厂段修顺序和冻结前缀的槽位。

若存在车辆子集 `C'`，其可用槽位并集 `N(C')` 满足：

```text
|N(C')| < |C'|
```

则不存在完美匹配。代码在 `stage3_simple/placement.py::_root_inner_hall_witness` 中构造该证书。

### 7.2 `0327Z`：12 辆车竞争 9 个可达槽位

使用当前统一布局器复跑：

```text
cars    = 12
slots   = 9
deficit = 3
```

可达槽位为：

```text
修1库内 #1,#2,#3
修3库内 #1
修4库内 #1,#2,#3,#4,#5
```

为什么总容量 20 位，却只有 9 位可达：

| 库线 | 既有车位置 | 当前冻结模型下可插入位置 |
|---|---|---|
| 修1库内 | #4、#5 | #1-#3 |
| 修2库内 | #1、#5 | 无；#1 的冻结车封住后方空位 |
| 修3库内 | #2-#5 | 只有 #1 |
| 修4库内 | #4、#5 是将出库车辆 | 预计清空后 #1-#5 |

这是一份严格的模型不可行证书，搜索节点数为 0 即可判定。

但它不是完整物理不可行证明：若允许临时取出修2库内 #1 的已到位车、放入待入库车，再恢复原车，则槽位域会变化。当前模型把这些车列为 fixed，未枚举这种重排。

### 7.3 `0406W`：14 辆车竞争 13 个可达槽位

最新证书：

```text
cars    = 14
slots   = 13
deficit = 1
```

可达槽位：

```text
修1库内 #1-#5
修2库内 #1-#3
修3库内 #1-#5
```

修4库内已有 4 辆正确车辆占据 #1-#4。虽然 #5 在名义上为空，但它位于已冻结前缀之后；北端作业无法直接把新车放到 #5，所以修4库内对当前新车提供 0 个可达槽位。

同样，这证明的是“冻结既有正确车辆”合同下不可行，不代表允许临时倒库时仍不可行。

### 7.4 `0424Z`：修3库外真实长度不足

该案例与前两个不同。四辆车的唯一目标都是 `修3库外`：

| 车辆 | 长度 | 初始位置 |
|---|---:|---|
| 1573612 | 14.3m | 修3库外 |
| 5310275 | 13.2m | 机南 |
| 5317394 | 13.2m | 机南 |
| 6608952 | 12.1m | 机南 |
| 合计 | 52.8m |  |

`修3库外` 有效长度为 49.3m：

```text
52.8 - 49.3 = 3.5m
```

四辆车均无其他合法目标线。无论如何重排、增加中转、改变机车位置，都不可能让它们同时停入该线。

这是 Stage3 中真正的物理硬不可行案例。解决方式只能是：

- 修正输入目标；
- 允许其他修库外线作为替代目标；
- 或把业务终态改为分批进入，而不是同时终到。

## 8. Stage4：10 个严格不完整案例全部是硬容量问题

### 8.1 全量结果

`stage4_block_flow_final` 的结果：

| 数据集 | 进入 Stage4 | strict complete | actionable complete | 容量受限 |
|---|---:|---:|---:|---:|
| truth2 | 110 | 103 | 110 | 7 |
| truth3 | 30 | 27 | 30 | 3 |

这些 partial 案例没有 active residual：求解器已经完成所有可行动作，只留下容量证明要求的最小 holdout。因此它们应称为 **capacity-limited actionable complete**，而不是普通搜索失败。

### 8.2 容量判定公式

Stage4 对唯一目标车辆集合使用：

```text
overflow
= 所有唯一目标车辆总长
  - 目标股道有效长度
  - 0.5m 容差
```

然后枚举最小数量的 holdout，使其总长覆盖 overflow。对应代码：

- `scripts/stage4_simple/scope.py::build_scope`
- `scripts/stage4_simple/scope.py::minimum_holdout`

### 8.4 三种容量矛盾不能用同一策略处理

#### 存5线南

这是存车线容量矛盾。若所有车辆确实必须同时终到存5线南，则输入本身不可行，应在全流程开始前返回容量证书。

#### 洗罐站

这是作业线。`0401W` 要求 171.6m 车辆同时进入 88.7m 的洗罐站，几乎是容量的两倍。业务上更合理的模型通常是：

```text
洗罐线北排队 -> 洗罐站分批对位 -> 完成后释放
```

如果真实业务允许波次加工，则“所有目标车最终同时停在洗罐站”本身就是错误的终态合同，应改成时间展开的服务完成约束。

#### 调梁棚

`0203W` 还带有强制位置 `6/7/8/9`。既要满足对位窗口，又要满足总长度。当前长度缺口 4.5m，单纯优化顺序无法解决。

### 8.5 `label_budget_exhausted` 不是这些案例的不可解原因

`0109W`、`0128W` 等个别案例的搜索停止原因为 `label_budget_exhausted`。但容量证书在搜索前已经成立，并且所有 active 车辆均已完成。因此：

- 标签预算只影响是否继续优化勾数；
- 不影响“严格终态无法容纳全部车辆”的结论；
- 不应把预算耗尽误判为容量 partial 的根因。

## 10. 共同结构矛盾

### 10.1 阶段串行化与实际波次作业冲突

当前结构要求：

```text
Stage1 全部编组完成
-> Stage2 全部出库完成
-> Stage3 全部入库完成
-> Stage4 全部收尾完成
```

但 `0103W`、`0401W` 表明现场问题天然需要：

```text
编一批 -> 送一批 -> 释放容量 -> 再编下一批
```

阶段完全串行会人为制造瞬时容量不可行。

### 10.2 股道同时是“容器”和“通道”

`机走北`、`机走棚`、`机南`、`洗油北`既用于存车，也决定图连通性。当前目标函数主要计算剩余容量和车辆债务，却没有为“占用后切断多少 OD 对”定价。

`0415W` 和 `0211Z` 都是同一矛盾的不同表现：

- `0415W`：本阶段尚未完成时封死后续合法目标；
- `0211Z`：本阶段完成后封死下一阶段调机出口。

### 10.3 局部 complete 与全局可继续不等价

当前 Stage1 的 complete 主要由 `stage1_car_complete()` 决定：车辆是否到了存4或四条编组线。它不包含：

- 调机最终位置；
- 下一阶段强制源线是否可达；
- 编组线是否仍保留必要通道；
- 下一阶段是否至少有一个合法首动作。

因此 `0211Z` 可以 Stage1 complete、Stage2 0 勾失败。

### 10.4 冻结正确车辆提高了稳定性，但缩小了可行域

Stage3 的 Hall 证书非常有价值，但当前槽位域基于 fixed 前缀。其含义是：

```text
不动已经正确的车辆时无解
```

不能直接外推为：

```text
允许临时取出并恢复正确车辆时仍无解
```

稳定性与可解性之间需要显式成本，而不是把“已到位不动”永久硬编码为单调规则。

### 10.5 strict complete 与 actionable complete 被混在一个成功口径中

Stage4 已经正确计算 actionable complete，但全流程/API 如果只看 strict complete，会把“所有可行动作已完成、仅剩物理容量 holdout”与“搜索失败”混为一类。

建议状态机至少区分：

```text
complete
actionable_complete
contract_infeasible
search_incomplete
upstream_unavailable
error
```

## 11. 建议的重构顺序

### P0：统一可行性预检和证书

在任何搜索前运行：

1. 唯一目标线长度容量检查；
2. Stage1 编组总容量与最小 holdout；
3. Stage3 内库 Hall 匹配和外线加权容量；
4. 当前占用下的调机连通分量；
5. 阶段终态到下一阶段强制源集合的可达性。

预检结果必须区分：

- `physical_infeasible`；
- `stage_contract_infeasible`；
- `feasible_but_search_required`。

### P1：增加阶段交接合同

每阶段结果除车辆债务外，增加：

```text
handoff_loco_component
next_stage_reachable_sources
reserved_corridor_lines
next_stage_first_action_exists
```

Stage1 候选不得接受会让下一阶段首动作集合为空的终态。

### P2：把关键通道占用纳入状态势函数

对每条线预计算其作为图割时覆盖的 OD 对数量。候选评价增加：

```text
cut_damage = 放车后新增不可达的 pending/source-target 对数量
cut_release = 取车后恢复可达的 pending/source-target 对数量
```

对于不可替代通道，`cut_damage > 0` 应进入硬门槛；对于有替代路径的通道，可作为加权代价。

### P3：增加受控的“打开通道”宏动作

只在以下条件触发：

- 某条已到位编组线是多个 pending 目标的共同 blocker；
- 移动其可取前缀能严格增加可达对数量；
- 新落点仍属于合法编组线或稳定缓存线；
- 动作完成后不会形成新的更严重割集。

`0415W` 的 `机走棚 -> 机南` 就是标准样例。

### P4：为 Stage3 引入“有代价解冻”

先保留当前 Hall 预检。若 Hall 失败：

1. 找出 witness 涉及的首个 frozen blocker；
2. 将其临时取出/恢复作为显式决策；
3. 每解冻一辆增加较高成本；
4. 重新计算槽位域和 Hall 条件。

只有在允许解冻后仍存在 Hall 缺口，才能进一步接近全局物理不可行证明。

### P5：把作业线改成时间展开服务模型

对洗罐站、油漆线、抛丸线、调梁棚等，不应始终要求所有目标车同时终到。更合理的状态是：

```text
queued -> spotted -> processed -> released
```

目标函数应约束完成加工的车辆集合，而不是把作业线当成无限期终点存车线。

此外应增加两个跨案例不变量：

1. 任一阶段 complete 时，下一阶段强制源至少有一条合法首动作；
2. 任一 `hard_capacity_infeasible` 必须附带可独立复算的需求、容量、容差和最小 holdout。

## 13. 最终判断

当前剩余不可解案例并非说明搜索器整体能力不足，也不能通过统一增加搜索预算解决。它们暴露的是三种不同层级的问题：

1. **输入/终态确实违反物理容量**：应快速证明并停止搜索；
2. **阶段分解或冻结规则人为缩小可行域**：应修改合同，而不是调权重；
3. **求解器忽略拓扑通道价值**：应增加割集、交接可达性和主动开通道能力。

其中最优先的不是继续增加多摘多挂模板，而是建立统一的可行性证书和阶段交接合同。否则复合动作只会更快地把车辆放进会封死后续路径的位置。
