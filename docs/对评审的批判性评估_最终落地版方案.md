# 对评审的批判性评估 + 第一阶段落地版方案

## 0. 定位

这份文档只覆盖一个目标：**把第一阶段做成低勾数、高可求解、物理和业务都正确的调车器**。

不做这些事：

- 不再分 V1 / V2 / V3。
- 不做跨阶段债务继承。
- 不做阶段交接协议。
- 不做全站通用优化器。

保留这些原则：

- 不用 fallback 掩盖真实阻塞。
- replay 是最终裁判。
- 中间层保留，但只作为候选和债务，不作为承诺对象。

---

## 1. 这次审查该接受什么

这份审查里，真正该接受的是四件事：

1. **量化必须补齐。** 名词不能只停在概念。
2. **业务规则必须对齐。** 不是“能走就行”，而是要和调车业务一致。
3. **数据契约必须明确。** trace、partial、config、replay 都要有边界。
4. **验收必须可判定。** 不能只说“看起来不错”。

但有两类内容要删掉：

- 阶段分期。
- 阶段交接。

因为现在的目标已经收窄了，只看第一阶段是否能稳定落地。

---

## 2. 第一阶段目标

第一阶段只做一件事：**把去大库相关车辆编成可执行、低勾数、可 replay 的计划**。

对应的调车骨架是：

- 先处理洗南、洗北、油相关阻塞。
- 优先清掉 `预修 / 调梁 / 存5` 这类高频阻塞源。
- 编组落位按 `机南 -> 洗油北 -> 机棚 -> 机北3` 的优先级收束。
- 去卸轮线的车单独编到 `存4`。
- 第一阶段结束时，`存4` 和编组线不能残留非大库目标车辆。

这里的“低勾数”是主目标，“高可求解”是硬要求。

### 2.1 四个结构为什么还要保留

这版不再追求抽象统一，但四个结构仍然要保留，因为它们正好分别负责四件事：

1. **偏序 + 单调细化**：决定什么必须先做，什么可以后做。
2. **势函数 / Lyapunov 下降**：决定往哪个方向走更稳、更省勾。
3. **列生成 / 对偶分解**：决定当前只生成哪些候选片段，不把全空间摊开。
4. **债务台账（Debt Ledger）**：决定哪些临时让步必须在截止期前还掉。

它们不是四套求解器，而是一条执行链上的四个职责。

#### 偏序 + 单调细化

P 只表达“必须先于”的关系，不表达“合法/不合法”。

- 只加边，不删边。
- 依赖满足后只做状态标记，不回改历史。
- 目标是把复杂调车过程变成一个不断收缩的可行前缀。

这比“任务对象”更稳，因为它不会把现实塞不进去的动作硬包装成一个任务。

#### 势函数 / Lyapunov 下降

Φ 不是评分总表，而是“当前走这步会不会更接近收口”的方向场。

- 摘头、抽连续 G、清掉依赖，应该让 Φ 下降。
- 暂存、倒车、绕行可以允许，但要把它们限制在可控窗口内。
- 如果一个动作短期升势，但能带来后续明显下降，可以保留；否则就拒绝。

这就是第一阶段里“低勾数优先”的数学表达。

#### 列生成 / 对偶分解

候选不是一次性枚举全部计划，而是按需生成。

- 主问题只看当前状态下最值得做的少量片段。
- 子问题负责从 P、Φ、D 里读出可执行候选。
- 资源紧张的线靠 λ 计价，贵的片段自然少出现。

这样做的目的很直接：**让 5 分钟预算只花在少量真正可能成交的候选上**，不在全空间里空转。

#### 债务台账（Debt Ledger）

D 记录的是“本来不该长期存在，但为了当前片段临时留下来的东西”。

- 例如暂存占用。
- 例如终态还没收口的边界车。
- 例如称重尾序还没补齐。

它不是承诺协议，只是账本。

- 账本里每笔债都要有最晚清偿时间。
- 关键债务要有 witness。
- 临期前必须检查，不能拖到 replay 才发现还不上。

### 2.2 第一阶段的目标顺序表

第一阶段不做全局摆位优化器，只做**最小顺序约束集**。

目标顺序的来源不是“拍脑袋排出来”，而是来自第一阶段真实业务优先级：

1. **先清前段阻塞。** `洗南 / 洗北 / 油` 相关阻塞先处理。
2. **先解锁高频依赖源。** `预修 / 调梁 / 存5` 优先清。
3. **先收敛编组线。** `机南 -> 洗油北 -> 机棚 -> 机北3` 按这个顺序收束。
4. **卸轮单独处理。** 去卸轮线的车先编到 `存4`。
5. **同类等价项不强制细排。** 同一目标库、同一业务等级、同一台位区间内部不额外做全局排序。

这个“目标顺序表”不是第二个求解器，只是 `Φ` 计算失序度的参照轴。

### 2.3 缓存线选择判据

`X` 暂存不能凭感觉放，必须先过一个轻量选择器。

候选缓存线按下面的顺序打分：

```text
score(ℓ, x) =
  w1 * 长度适配
  + w2 * 往返代价
  + w3 * 主进路干涉
  + w4 * 可回收性
  + w5 * 新增阻塞惩罚
```

含义分别是：

- `长度适配`：剩余长度扣掉债务占用后是否还能放下。
- `往返代价`：把 `x` 暂存过去，再从那里取回来的路径成本。
- `主进路干涉`：这条线会不会挡住后面更关键的整拉或清债。
- `可回收性`：未来能不能顺手把这笔暂存债还回来。
- `新增阻塞惩罚`：放进去后会不会把还没处理的 `G` 线前段堵死。

优先级建议是：

1. 临停线。
2. 一般存车余量。
3. 借编组线短暂缓存。

硬规则只有一条：

> 不能把 `X` 暂存到还没处理完的 `G` 前面，也不能把关键机动线同时塞死。

### 2.4 机动余量约束

这不是“完成性证明”，只是一个安全守卫。

每轮执行后，要尽量保住至少一条可回退机动线，关键折返通道也不能同时全满。  
如果候选会把剩余机动空间压到太低，就降权或者直接删掉。

这条约束的作用是：

- 防止前期暂存把后面完全卡死。
- 防止每一步都合法，合起来却没路可走。
- 让阻塞尽量在早期暴露，而不是拖到最后。

## 3. 运行时状态

状态只保留四个分量：

```text
State = (P, Φ, λ, D)
```

### 3.1 Kernel

Kernel 只管可行性边界，不管排序。

| 状态 | 含义 | 例子 |
|---|---|---|
| `hard_forbidden` | 永不可修，直接过滤 | 第一阶段大库相关线路 |
| `state_infeasible` | 现在不通，等外部状态变化 | 线路上有车、路径被占、容量不足 |
| `terminal_invariant` | 终态必须修掉 | 第一阶段结束时的纯净边界、称重尾序 |

说明：

- `terminal_invariant` 不是全程禁令。
- Kernel 不打分，只决定候选能不能进池。
- 运行时只保留 replay 能证实的状态，不搞额外哲学分类。

### 3.2 P

P 只记必要先后，不记合法性。

| P 负责 | 不负责 |
|---|---|
| X 挡 G，先摘 X 再拉 G | 路径通不通 |
| 接收线先抽空，再做归位 | 容量够不够 |
| 北端栈序派生的顺序约束 | 目标台位是否合规 |
| 依赖实例 open -> satisfied | 线性扩展计数 |

建议拆成两层：

- `P_static`：静态必要先后，单调存在。
- `P_active`：当前实例状态，只标记 `open / satisfied`。

### 3.3 Φ

Φ 只管方向与意图，不管硬禁令。

第一阶段建议只保留四项：

| 项 | 定义 |
|---|---|
| 被挡 G 数 | 当前还被其他车挡住的去大库车数量 |
| 未满足依赖数 | 还没解锁的前置依赖数量 |
| 编组失序度 | 按第一阶段目标顺序计算的逆序对数 |
| 债务临期压力 | 距最晚清偿钩位的紧迫度 |

建议用字典序，而不是让“少几勾”抵消一次明显违规。

### 3.4 λ

λ 只做软资源价格。

建议定价对象：

- `渡10 / 联7`
- `机南 / 洗油北 / 机棚 / 机北3`
- `预修 / 调梁 / 存5`
- 暂存线紧张度
- 编组线北端可取性
- 称重位
- 机车换端和走行代价
- 债务清偿通道占用

一个足够简单的更新式就够了：

```text
λ_r <- clip((1 - γ) * λ_r + γ * usage_r / cap_r, 0, λ_max)
```

其中 `usage_r` 取最近窗口的资源占用，`γ` 和 `λ_max` 放配置里。

### 3.5 D

D 是债务台账，不是承诺协议。

第一阶段只记四类债务：

1. 暂存占用。
2. 终态纯净欠账。
3. 称重尾序欠账。
4. 目标线未腾空欠账。

每笔债务至少带：

- 阶段标签
- 最晚清偿钩位
- 占长
- 阻塞影响集
- 清偿 witness

规则很简单：

- **容量账不能懒**，占长实时扣。
- **witness 可以懒，但要有安全视界兜底**，临期前 `h=2~3` 钩必须强制检查。
- 关键通道上的债务保留 witness，普通债务可延迟重找，但不能没有兜底。

## 4. 量化定义

下面这些量，必须落成公式或默认规则。

| 概念 | 定义 |
|---|---|
| 编组失序度 | 按第 2.2 节的目标顺序表计算逆序对数；同线内部按当前候选片段的落位顺序计算 |
| 债务截止期 | 最晚清偿钩位 `deadline_hook`，用钩序号表示，不用自然时间 |
| 债务临期压力 | `max(0, h - slack_hooks) / h`，`h` 默认取 2~3 |
| witness | 当前状态下，经过 sound + replay 验证的一条清偿路径 |
| 切换成本 | `a * route_change + b * loco_end_change + c * fragment_change` |
| top-k | 每类片段默认 2~3 个，决赛圈不超过 4 个 |

这些值不是“哲学真理”，只是默认实现口径。

## 5. 第一阶段的业务规则覆盖

这部分要和真实调车规则对齐，不能只看最终结果。

| 规则 | 组件归属 |
|---|---|
| 只能从北端操作，不能中途加塞 | Kernel + P + replay |
| 线路有车时，线路不可当作可达目标 | Kernel |
| 调车机自身 15m 要计入倒车 / 折返长度 | Kernel |
| 1 辆重车按 4 辆空车折算，纯空车最多 20 | sound 粗滤 |
| 关门车顺位限制 | Kernel + replay |
| 机库称重只在机库发生 | Candidate + replay |
| 单钩挂多辆称重车时，只标记最后一辆完成 | Candidate + replay |
| 临停线只能做过程缓冲，不能当最终终点 | Kernel |
| 第一阶段大库相关线路禁用 | Kernel hard_forbidden |

### 关门车口径

- 不去 `存4北` 时，机后总辆数超过 10，关门车不能位于第一顺位。
- 不去 `存4北` 时，若本钩含重车，第一辆不能是关门车。
- 去 `存4北` 时，关门车最终摆位不能在第 1、2、3 顺位。

### 称重口径

- 只有 `IsWeigh=true` 的车需要称重。
- `Action=Weigh` 时，`TrainCars` 是整钩经过机库的车组。
- 如果一钩带多辆称重车，只有最后一辆在本次 `Weigh` 中完成。

## 6. 候选片段

第一阶段只保留真实调车片段，不做通用语法搜索。

| 片段 | 用途 |
|---|---|
| 抽连续 G | 处理干净型线路，做整拉 |
| 摘头阻塞 | 先清头部 X，再拉后面的 G |
| X 暂存 / 归位 | 处理依赖没解锁时的临时落点 |
| 称重顺路 | 顺路经过机库完成称重 |
| 清阶段边界 | 收口第一阶段终态纯净约束 |

说明：

- 之前讨论过的“6 类片段”里，`大库进出` 在第一阶段属于 `hard_forbidden`，**不进入候选池**。
- 候选片段不是任务模板，而是可 replay 的短动作串。

## 7. 执行循环

每轮只做一件事：选一个最值得做的真实调车片段。

```text
1. 从热点先验和 P/D/Φ 结构里生成候选
2. sound 粗滤：容量、北端栈序、P 前置
3. heuristic 只加价，不删除
4. 粗排：ΔΦ + λ + 切换成本
5. 决赛圈 replay 精判
6. 选中一个片段执行
7. 更新 P / D / λ
8. 输出解释 trace
```

### 没有 fallback

如果没有候选通过：

- 不静默换逻辑。
- 不偷偷跳到另一套策略。
- 直接输出结构化阻塞原因。

## 8. 数据契约

### 8.1 trace

每轮至少记录：

- 候选 ID
- 片段类型
- 满足的前置
- `ΔΦ` 分解
- 支付的资源价格
- 打开的债务
- 被拒绝候选及原因

### 8.2 partial 报告

如果本轮不能继续，应输出：

- 卡在哪个 Kernel 类别
- 卡在哪笔债务
- 哪个资源被占满
- 下一步最短可行修复路径

### 8.3 config

配置表至少包含：

- 热点先验
- `P_static` 规则顺序
- `hard_forbidden` 集合
- `k` 和 `h`
- `λ` 初值
- 资源价格权重

### 8.4 replay API

replay 的边界要清楚：

- 输入：当前状态 + 原子勾操作序列
e- 输出：通过 / 不通过 + 违规明细 + 路径明细
- replay 的判定粒度必须落到单勾；片段只是上层编译出来的短序列，最后还是要拆成勾计划来校验
- 不要只返回一个布尔值

---

## 9. 验收标准

第一阶段的通过线应当是：

1. **物理违规 = 0**
2. **业务违规 = 0**
3. **同一输入的输出确定**
4. **replay 通过率 = 100%**
5. **在基准集上，勾数不差于人工计划基线**
6. **5 分钟内能给出 complete 或结构化 partial**

如果达不到，就返回阻塞原因，不做静默兜底。

---

## 10. 风险控制

最容易把方案做坏的地方只有四个：

1. 把终态要求写成全程禁令。
2. 把 witness 写成第二套预约系统。
3. 把候选片段扩成无限多。
4. 把热点写死在代码分支里。

所以最终纪律是：

> **Kernel 只管可行性边界，P 只管必要先后，Φ 只管方向，λ 只管软资源，D 只管临时债务，候选只保留真实调车片段。**

## 11. 实施步骤与验收标准

这部分只定义第一阶段的开工顺序和每一步的验收线。原则是：

- 每步都能单独验证。
- 物理违规优先暴露。
- replay 先作为地基，再作为裁判。
- 先跑通合法计划，再追低勾数。
- 实现要简洁优雅，代码最简化


### 第 0 步：replay 现状核实

目的：先确认现有 replay 到底能不能复用。

要核实三件事：

1. replay 是否已经存在。
2. replay 的输入输出粒度是否够用。
3. replay 的耗时是否能放进 5 分钟预算。

验收标准：

- 若 replay 已存在且可调用，能对单个候选片段返回通过 / 不通过 + 违规明细。
- replay 能判出物理违规和业务违规，而不是只返回一个布尔值。
- replay 至少能作为“去大库全部编组、编组线上没有去其他地方的车辆、且不违背物理和业务规则”的最低裁判。

结论分支：

- 若 replay 可直接复用，进入第 1 步。
- 若 replay 不存在，第 1 个开发任务就是 replay 本身，优先级最高。

### 第 1 步：数据地基与 L0 资产

目的：把静态资产固化成配置和查表结构。

内容：

- 拓扑不变量加载。
- 初始 `P_static` 离线计算。
- 配置骨架补默认值。

验收标准：

- 能从配置表加载全部主数据。
- 能吐出初始 `State`。
- 能用单元测试验证拓扑、断点、依赖弧都能正确读入。
- 数据缺失时能明确报错，而不是静默 fallback。

### 第 2 步：Kernel + sound 粗滤

目的：先把世界边界立起来。

内容：

- 三类状态判定。
- sound 粗滤的确定性规则：容量、15m 机车、栈序、重车折算、P 前置。

验收标准：

- 给定 `State + 候选片段`，能稳定判定为合法 / 不合法 / 终态债。
- 核心规则纯函数化，可独立单测。
- 任意明显违规候选都能在 replay 前被挡掉。

### 第 3 步：五类片段生成器 + 冷启动

目的：让候选先长出来。

内容：

- 五类片段各自生成函数。
- 从 `P` 结构和热点先验读出候选。
- 先不用 `Φ / λ`，只用固定优先级。

验收标准：

- 每类片段都能生成至少一批真实候选。
- 不需要靠通用语法搜索也能跑出最小闭环。
- 候选总数可控，不会爆成无限分支。

### 第 4 步：执行循环骨架 + trace

目的：把第 2、3 步串起来。

内容：

- 执行循环。
- trace 输出。
- partial 报告输出。

验收标准：

- 输入真实 case，能输出带 trace 的第一阶段计划。
- 至少在最简单的几个 case 上做到 `物理违规 = 0`。
- trace 能说清楚为什么选它、为什么拒绝别的候选。

### 第 5 步：D 债务台账

目的：让暂存和收口有账可算。

内容：

- 四类债务。
- 容量账实时扣减。
- 截止期按钩序号记录。
- witness 建立、维护、临期兜底。

验收标准：

- `摘 X 暂存 -> 拉 G -> 归位` 这类跨钩片段能正确记账和清偿。
- 临期债务能提前暴露，不拖到最后。
- 交错断点 case 能专项测试通过。

### 第 6 步：Φ + λ 上线

目的：把“能跑”变成“更省勾”。

内容：

- `Φ` 四项 + 字典序。
- `λ` 拥堵分 + 更新式。
- 切换成本。

验收标准：

- 计划能从“合法”进化到“低勾数”。
- 不出现明显偏科：疯狂摘 X、边界收不了口、一阶段顺成三阶段返工。
- 基准集上能稳定优于固定优先级版本。

### 第 7 步：预算控制 + 降级阶梯 + 验收固化

目的：保证 5 分钟内可交付、可诊断、可复现。

内容：

- 求解预算控制。
- 降级阶梯。
- 自动化回归套件。
- 监控仪表盘。

验收标准：

- 5 分钟内能给出 complete 或结构化 partial。
- 同一输入输出确定。
- 回归套件能稳定复现物理 / 业务 / 求解失败。
- 仪表盘能看见候选数、Kernel 拒绝分布、witness 重找次数、`Φ` 窗口净降。

### 时间线小结

| 步骤 | 内容 | 估时 | 验证的关键假设 |
|---|---|---:|---|
| 0 | replay 现状核实 | 0.5d | 地基是否存在 |
| 1 | L0 资产 + config | 2-3d | 数据够不够 |
| 2 | Kernel + sound 粗滤 | 2-3d | 边界能否立住 |
| 3 | 五类片段 + 冷启动 | 3-4d | 候选能否生成 |
| 4 | 执行循环 + trace | 2d | 骨架能否跑通 |
| 5 | D 债务台账 | 2-3d | 跨钩摊还能否闭合 |
| 6 | Φ + λ + 反例标定 | 3-4d | 能否低勾数、不偏科 |
| 7 | 预算 / 降级 / 验收 | 2-3d | 能否交付、可复现 |

里程碑：

- 第 4 步结束：先验收“合法计划”。
- 第 6 步结束：再验收“低勾数计划”。
- 第 7 步结束：验收“可交付”。

---

## 12. 一句话总结

这份方案现在的目标很明确：**只做第一阶段，不再分期；只保留真实调车片段，不再造任务模板；只看低勾数和可求解，不再追求完美优化器。**

实现上就是：Kernel 复用 replay 当裁判，P 只记必要先后，D 只记四类债务，Φ 和 λ 只负责方向与拥堵排序，候选只保留五类真实片段。这样就能把第一阶段做成一条短、稳、能调试的主线。
