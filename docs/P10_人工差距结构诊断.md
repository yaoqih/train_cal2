# P10 与人工计划差距的结构诊断

日期：2026-06-27

## 0. 结论

当前应该先诊断，再优化。

最新 P10 运行时已经证明了一件重要的事：物理路径、线路占用、长度、台位、牵引和基本称重/关门车规则可以进入自动校验，并且当前已接受动作的硬物理违规为 0。这是必要进展。

但它还没有证明方案已经达到人工计划，更不能证明已经超越人工计划。核心原因不是单个排序权重不够好，而是当前代码只实现了一个 P10 物理运行时骨架和若干启发式策略，没有真正实现方案文档要求的 `FlowGraph -> FlowEdge -> EdgeContract -> TargetContractSelector -> ResourceRequest -> ContractDelta -> ResourceDelta -> LocalTieBreakSearch -> Trace` 闭环。

所以，之前的判断需要精确拆开：

| 问题 | 当前结论 |
|---|---|
| 如果文档中的 P0-P10 结构和 H1-H5 阶段门全部达到量化验收，是否足以达到甚至超越人工？ | 是。结构上是充分的，因为合同、资源、阶段、候选和少钩搜索形成闭环。 |
| 当前 P10 运行时是否已经说明这些结构全部达标？ | 否。当前主要是 source-target 候选、物理 validator、静态 depot assignment、启发式排序。 |
| 当前是否已经达到人工计划勾数水平？ | 否。84 个可与人工计划匹配且求解完成的案例中，算法平均比人工多 22.99 个业务挂摘勾，只有 2 个不劣于人工。 |
| 当前最主要问题是什么？ | H1-H5 阶段门没有真实落地，远端修库/卸轮区域过早过频参与，P4/P7/P8 缺少合同收益、资源收益和同结构少钩搜索。 |

## 1. 诊断口径

### 1.1 勾数口径

人工单中一勾是一次挂或摘动作。

当前 P10 artifact 中可比字段是：

| 字段 | 含义 | 是否用于和人工勾数对比 |
|---|---|---|
| `business_get_put_hook_count` | `Get + Put` 的业务挂摘动作数 | 是 |
| `internal_move_batch_count` | 一个内部 source-target 移动批次，通常包含一次 `Get` 和一次 `Put` | 否 |
| `interface_operation_count` | API 操作数，含 `Weigh` | 否 |

因此本文使用 `business_get_put_hook_count` 比较人工计划勾数。

### 1.2 数据来源

| 类型 | 路径 |
|---|---|
| 当前 P10 artifact | `artifacts/p10_early_base_full` |
| 人工案例 | `data/人工调车数据/*人工调车作业单/*.xlsx` |
| 当前运行时代码 | `scripts/generate_physical_runtime_trace.py` |
| 结构方案标准 | `docs/flow_edge_structure_model_design.md` |
| 人工基准研究 | `docs/人工调车案例基准研究.md` |

人工单线路名可能不是最新，人工勾数也可能有少量省略勾。但当前差距远大于“少量省略勾”能解释的范围。

## 5. H1-H5 与人工阶段的偏差

### 5.1 人工阶段位置

人工基准显示，标准链常见阶段是：

```text
H1 前段全站组织与服务处理
H2 存4释放口成形与保护
H3 存4大释放-机接硬边界
H4 修库消化、库位和出入库冲突处理
H5 尾项收束与库回
```

典型人工案例：

| 案例 | 人工勾数 | 人工 variant | H1 | H2 | H3 | H4 | H5 |
|---|---:|---|---|---|---|---|---|
| `0104W` | 41 | `FULL_CHAIN_REPAIR` | 1-28 | 29-33 | 34-35 | 36-38 | 41 |
| `0210W` | 39 | `FULL_CHAIN_REPAIR` | 1-23 | 24-31 | 32-33 | 34-36 | 39 |
| `0304W` | 46 | `FULL_CHAIN_REPAIR` | 1-30 | 31-37 | 38-39 | 40-43 | 46 |
| `0306W` | 10 | `MIXED_SIGNAL_REPAIR` | 无标准 H1 | 2-6 | 无标准 H3 | 3 | 10 |
| `0327W` | 11 | `DEPOT_DIGEST_ONLY` | 1-3 | 无 | 无 | 4 | 11 |

人工并不是永远不先做库侧动作。短链、库内消化、低信号案例会很早触碰修库。但标准链的大多数案例确实先完成外场/功能线/调棚/存车区组织，再进入修库/卸轮远端区域。

### 5.2 当前求解远端触碰过早

这里把修1-修4库内/外和卸轮线视为远端区域。人工口径用 `修1/修2/修3/修4/卸轮` 前缀近似，当前求解口径用 `修1-修4库内/外 + 卸轮线`。

匹配且求解完成的 84 案：

| 指标 | 人工 | 当前求解 |
|---|---:|---:|
| 首次触碰远端的绝对位置 P50 | 第 20 钩 | 第 1 个业务动作 |
| 首次触碰远端的相对位置 P50 | 全案 62.5% | 全案 2.1% |
| 远端相关挂摘动作 P50 | 7 | 15.5 |
| 远端相关内部批次 P50 | 不适用 | 14 |
| 远端 session P50 | 2 | 3 |
| 远端跨入/跨出批次 P50 | 不适用 | 12.5 |

这个结果非常关键：当前求解不是“偶尔提前进大库”，而是在绝大多数匹配完成案例中第一个或第二个业务动作就触碰修库/卸轮远端区域。人工计划的标准链通常在中后段才集中处理。

## 6. 当前代码和方案结构的差距

方案文档 `20A.2B` 明确写了 P10 不能替代 P0-P9：一个案例最终跑通，但中间结构指标失败，仍然不能证明方案可行。

当前运行时代码的实际结构如下：

```text
输入 JSON
  -> normalized_car
  -> build_depot_assignment / build_short_direct_depot_assignment
  -> build_candidates
  -> PhysicalValidator.validate
  -> phase_candidate_sort_key / candidate_sort_key
  -> apply_candidate
  -> operation_trace / response JSON
```

这和方案要求的结构闭环不同：

```text
FlowGraph
  -> FlowEdge
  -> EdgeContract
  -> TargetContractSelector
  -> StructuralIntent
  -> WorkPatternTemplate
  -> ResourceRequest
  -> StationResourceGraph
  -> ContractDelta
  -> ResourceDelta
  -> AcceptRejectGate
  -> LocalTieBreakSearch
  -> State Update / FlowGraph Rebuild / Trace
```

### 6.1 P0-P10 对照

| 过程 | 方案要求 | 当前实现状态 | 影响 |
|---|---|---|---|
| P0 在线证据边界 | 在线 feature、status source、无人工泄漏 | 只有输入字段校验和 normalize，没有 status source trace | 不能证明低信号和阶段判断来自在线状态 |
| P1 车辆分类 | `FlowClassify / NoMoveVehicle / ResidualItem` | 以目标线和满足状态粗分，无合同覆盖 trace | 不能区分“无需动”和“阻塞关键资源但目标已满足” |
| P2 建边与身份稳定 | `FlowGraph / FlowEdge / EdgeKey` | 没有 active edge、edge identity、subject overlap trace | 主业务流不能稳定表达，阶段判断无 owner |
| P3 状态与合同生成 | `FlowEdgeStatus / EdgeContract / RepairInboundVariant` | 只有 action family 和少量 case_id 短链硬编码 | 变体名存在，但没有合同条款和置信度 |
| P4 目标合同与结构意图 | `TargetContractSelector / StructuralIntentBuilder` | `planned_target_for_car` + sort key | 不知道当前动作服务哪个主合同、释放多少下游债务 |
| P5 候选生成 | 边界化候选、人工模板、resource request | source line 第一个可达未满足车的 prefix 候选 | 候选要么缺短链模板，要么拆成大量 source-target 批次 |
| P6 资源仲裁 | `StationResourceGraph / GlobalGate / DepotSlotGraph / LocoCarryState` | 有物理路由、占用避让、静态台位分配、机车位置 | 能过滤部分非法动作，但不能做资源等待、swap、session 和 ordered detach |
| P7 Delta 与硬门 | `ContractDelta / ResourceDelta / AcceptRejectGate` | 物理 validator 通过即候选可进入排序 | 物理合法但结构上劣质的动作会被接受 |
| P8 同结构少钩 | `ContractOptimizer / LocalTieBreakSearch` | 贪心排序，无 planlet search，无 dominated accepted 审计 | 总勾数系统性高于人工 |
| P9 状态更新与 trace | rebuild、车辆守恒、failure bucket、结构 trace | 有车辆位置更新、CSV trace、物理原因 | 缺少 FlowGraph diff、contract delta、resource delta、phase transition |
| P10 全量压测 | 113 案完成、硬违规 0、勾数通过 P8 | 89/113 完成，硬违规 0，但勾数失败 | 当前只能证明 P10 骨架局部有效 |

## 7. 主要根因

### 7.1 PhaseGate 是软启发式，不是 HumanPhaseContract

当前 `phase_state_for_case` 只有三个状态：

```text
H1_H2_LINK7_CLOSED
H3_H4_DEPOT_ALLOWED
H3_SHORT_DIRECT_DEPOT
```

它不是 H1-H5。它没有 H1/H2/H3/H4/H5 的进入条件、退出条件、跳过条件、失败条件和阶段债务。

更关键的是，`run_case` 默认会尝试：

```text
phase_gate_base
early_depot_base
phase_gate_cluster
early_depot_cluster
short_direct_cluster
```

其中 `early_depot_base` 和 `early_depot_cluster` 会把 `short_direct_override=True` 传入 phase state。这样很多标准链案例会进入 `SHORT_DIRECT_DEPOT` 的排序逻辑，直接优先 `REPAIR_INBOUND / DEPOT_SLOT / DEPOT_OUTBOUND`。

这解释了为什么大量标准链案例第 1 或第 2 个业务动作就触碰修库/卸轮。当前的 `early_depot` 不是人工计划式的“等待联7开放后集中处理”，而是“提前把远端目标放进最高优先级”。

另一个问题是 `phase_reject_reason` 只是在联7关闭时 defer depot 候选；如果没有前段候选，代码会走 `phase_link7_forced_open_no_front_candidate`，也就是软打开。统计中出现了多次 front_service_progress 仍为 0.00、0.31、0.58 时被迫放行远端候选。

诊断结论：

```text
当前 PhaseGate 不能证明 H1-H5 完成。
它只能证明某些联7/远端候选曾被延迟或被迫放行。
```

### 7.2 当前目标选择是“车到目标线”，不是“合同债务选择”

当前 `planned_target_for_car` 基本按：

```text
depot slot assignment
else target line with lower load
```

当前 `candidate_sort_key` 基本按：

```text
candidate_kind priority
line_priority
batch size
candidate_id
```

这和方案中的 `TargetContractSelector` 不同。方案要求先问：

```text
当前最重要的合同债务是什么？
这个动作能释放多少资源？
会不会破坏更高优先级合同？
是否处于 H1-H5 允许的阶段？
```

当前代码更多是在问：

```text
这辆车最终该去哪条线？
从当前线直接送过去是否合法？
哪个候选排序更靠前？
```

所以它会把 `修库入库`、`修库出库到存4`、`洗罐到修库` 等物理合法动作提前做掉，即使人工计划结构上要先完成 H1/H2。

### 7.3 候选生成粒度过细，缺少人工计划的“带车列连续动作”

人工计划的少钩能力来自当前车列的连续挂摘：

```text
挂 A
挂 B
摘 C
挂 D
去存4北摘
机接
按尾端可摘顺序分库
```

当前运行时的基本动作是一个内部 source-target 批次：

```text
Get(source_line, batch)
Put(target_line, batch)
```

每个批次通常对应 2 个业务挂摘勾。它可以批量搬同一 source、同一 target、同一 prefix 的车，但很难表达人工计划中的混合携带、一路挂摘、多目的地分解和尾端可摘策略。

这直接导致：

| 表现 | 根因 |
|---|---|
| `0306W` 短链 10 勾变 70 勾 | 没有短链 planlet，只能逐 source-target 搬运 |
| `0327W` 远端 session 集中但仍 58 勾 | 远端内部没有 ordered detach 和 depot digest 压缩 |
| `0210W` H1 来回整理到 112 勾 | 外场/调棚/存车区没有 co-carry batching |

### 7.4 DepotSlotGraph 目前是静态分配，不是动态 swap

当前有 `build_depot_assignment` 和 `build_short_direct_depot_assignment`，能给修库目标车分配台位，并检查长度、厂修/段修规则、强制位置等。

但这不是方案中的 `DepotSlotGraph / DepotSwapDelta`。缺少：

```text
slot occupied_by
blocked_by
released_slot
reserved_slot
swap dependency
stayer lock
库内/库外衔接位
一次远端 session 内的出库-入库-摘解压缩
```

结果就是：台位规则能过滤非法位置，但不能告诉求解器“先出哪一组、一次进去处理哪几组、哪些入库可以和哪些出库合并”。这会让修1-修4库内外与其他线路交互次数偏多。

### 7.5 PhysicalValidator 只能判合法，不能判结构收益

当前 `PhysicalValidator` 做得比较实用：

```text
路径是否存在
路径是否被占用阻断
走行线禁止停
牵引折算
倒车长度
目标线长度
台位规则
称重规则
关门车规则
```

这解释了为什么 accepted hard violation 为 0。

但它不能替代 `ResourceRequest / StationResourceGraph / ResourceDelta`。它不能回答：

```text
这个动作释放了哪个关键口？
是否占用了 H2/H3 需要保护的存4北？
是否造成后续必须多一次远端跨区？
是否让修库 slot swap 更难？
是否让当前机车位置远离下一个主合同？
```

因此当前会出现“每一步都物理合法，但全案结构很差”的情况。

### 7.6 P7/P8 缺失导致“合法但劣质”的动作无法被拒绝

方案文档中 P7/P8 的关键标准是：

```text
ContractDelta / ResourceDelta 必须解释动作收益和破坏。
LocalTieBreakSearch 只能在同 contract、同 intent、同结构等级里少钩比较。
标准链钩数 <= manual_hook_count * 1.05。
短链/直接入库 <= manual_hook_count + 1。
```

当前没有这些对象和指标。被接受的候选只说明：

```text
物理 validator 通过
没有 state signature loop
排序靠前
```

它不说明：

```text
这个动作是否是同阶段最优
是否被另一个合法候选支配
是否破坏 H1/H2/H3/H4 的人工结构价值
是否会增加后续远端跨区
```

这正是当前勾数系统性高于人工的直接原因。

### 7.7 blocked 案例说明可解性也没有闭环

24 个 blocked 案例中：

| blocked reason | 数量 | 诊断 |
|---|---:|---|
| `all_runtime_candidates_rejected` | 15 | 候选生成和资源预判不足 |
| `target_final_capacity_infeasible:*` | 9 | 目标输出或线路容量数据与物理长度不一致，需要输入/目标一致性审计 |

物理拒绝原因中：

| 拒绝原因 | 次数 | 诊断 |
|---|---:|---|
| `get_route_blocked_by_occupied_line` | 612 | 候选生成前没有足够的路线占用资源计划 |
| `put_route_blocked_by_occupied_line` | 168 | 目标送入路径缺少提前清路/让步动作 |
| `depot_slot_rule_violation` | 63 | 台位候选仍会生成非法位置，靠 validator 拦截 |
| `weigh_car_not_last_in_carry_order` | 13 | 缺少 ordered carry segment |
| `source_front_blocked_by_satisfied_or_lower_position_cars` | 20 | 源头阻塞 relocation 搜索不足 |

这说明可解性也不是“已经完整证明”。当前完成 89/113，距离接口输入后稳定可用还有差距。

## 8. 结构方案本身的问题和当前实现的问题要分开

### 8.1 方案结构方向仍然成立

从业务和人工案例看，方案文档中的主结构是合理的：

```text
FlowClassify
FlowGraph / FlowEdge / EdgeContract
TargetContractSelector / StructuralIntent
StationResourceGraph / DepotSlotGraph / GlobalGate / LocoCarryState
ContractDelta / ResourceDelta
LocalTieBreakSearch
HumanPhaseContract / PhaseGate
Trace
```

如果这些结构真的逐项达到文档标准，尤其是：

```text
H1-H5 阶段门可审计
每个动作都有 ContractDelta + ResourceDelta
DepotSlotGraph 能处理动态 swap
ordered carry/drop 可验证
P8 同结构少钩通过人工计划基准
P10 全量完成
```

那么整体达到人工是有结构充分性的。

### 8.2 当前实现没有达到这些结构标准

当前实现更像：

```text
目标线分配
source-target prefix 候选
物理合法性过滤
启发式排序
状态更新
```

它不是完整 P0-P10。尤其缺：

| 缺口 | 直接后果 |
|---|---|
| 没有 `HumanPhaseContract` | 标准链远端动作过早 |
| 没有 `TargetContractSelector` | 目标选择不能解释合同收益 |
| 没有 `ContractDelta` | 不能拒绝结构上返工的动作 |
| 没有 `ResourceDelta` | 不能度量释放/占用关键资源 |
| 没有真实 `DepotSlotGraph` | 修库出入和 swap 无法压缩 |
| 没有 `ordered carry segments` | 人工式连续挂摘不能表达 |
| 没有 `LocalTieBreakSearch` | 同结构少钩无法保证 |
| 没有 phase/contract/resource trace | 失败无法定位到方案结构 |

因此，不能把“P10 skeleton 跑出 response JSON”理解为“P0-P10 全部完成”。

## 9. 下一步不应直接优化排序，而应先补诊断闭环

建议下一步先产出以下诊断 artifact，再改策略：

| 诊断产物 | 目的 |
|---|---|
| `manual_vs_solver_case_compare.csv` | 每案人工勾、算法勾、差值、策略、variant、首次远端位置、远端批次、blocked reason |
| `runtime_phase_trace.csv` | 每个 accepted candidate 的 expected H phase、actual phase、phase permission、进入/退出/跳过原因 |
| `contract_trace.csv` | 每个候选绑定的 target contract、owner edge、obligation、protection、suppressed contracts |
| `candidate_dominance_audit.csv` | 判断 accepted candidate 是否被同阶段、同合同、物理合法的候选或 planlet 支配 |
| `resource_delta_trace.csv` | 每个候选申请/释放/阻塞的资源，尤其存4北、联7、修库 slot、机车位置 |
| `depot_session_audit.csv` | 修1-修4库内外、卸轮线 session、跨入跨出、slot/swap、一次远端作业压缩程度 |
| `route_blocker_audit.csv` | 612 次 get route blocked 和 168 次 put route blocked 的清路需求归因 |
| `capacity_consistency_audit.csv` | 9 个 `target_final_capacity_infeasible` 是否是输入目标、线路长度或业务口径问题 |

这些诊断不是重复文档，而是为了把每一次优化绑定到 P0-P10/H1-H5 的具体失败点。

## 10. 优先修复方向

### 10.1 先把 `early_depot` 降级为诊断策略

当前 `early_depot_base/cluster` 会把许多标准链案例当成 `SHORT_DIRECT_DEPOT` 排序，导致远端动作过早。它可以保留用于对照，但不能作为“人工计划阶段达标”的候选策略。

生产口径应区分：

```text
STANDARD_FULL_CHAIN
DIRECT_REPAIR_ENTRY
DEPOT_DIGEST_ONLY
MIXED_SIGNAL_REPAIR
```

只有后 3 类有证据时，才允许早期触碰修库。

### 10.2 先实现可审计的 H1-H5 PhaseTrace

不要先调一堆排序权重。应该先让运行时每个 accepted candidate 输出：

```text
current_phase
phase_entry_reason
phase_exit_condition
phase_permission
phase_forbidden_reason
manual_variant_baseline
front_service_progress
depot_progress
release_accept_gap
remote_session_id
```

验收目标：

```text
accepted_without_phase_permission_count = 0
phase_transition_trace_coverage = 100%
标准链首次远端位置接近人工计划分布
短链/库内消化有显式 skip reason
```

### 10.3 把 P4 从目标线排序改为合同债务选择

`planned_target_for_car` 仍可保留，但不能直接等于当前目标合同。需要显式输出：

```text
selected_contract
owner vehicles
hard obligations
protected resources
downstream unlock count
suppressed contracts and reason
```

否则算法会继续选择物理合法但结构价值低的 source-target 动作。

### 10.4 补 P7/P8，先拒绝 dominated 动作，再谈少钩

短期可以先做旁路审计，不一定马上完全替换求解器：

```text
对每个 accepted candidate 计算 ContractDelta / ResourceDelta
标记 dominated_accepted
统计每个 case 的 dominated_accepted_rate
```

只要 dominated accepted 大量存在，就不要继续调排序，因为排序会在错误候选集合里做局部修补。

### 10.5 对远端区域做 session 级求解

修1-修4库内外、卸轮线与其他线路距离远，不适合频繁跨区。当前已有远端指标，但还没有用于结构决策。

应把远端作业视为 session：

```text
enter_remote
collect outbound
release slots
inbound placement
depot digest
exit_remote
```

验收目标：

```text
remote_interaction_batch_count <= manual_remote_hook_count + tolerance
remote_interaction_cross_count <= manual_remote_hook_count + tolerance
standard chain 不得在 H1 初期触碰远端
短链案例必须在远端 session 内压缩完成
```

### 10.6 对 `0306W / 0327W` 建短链专项诊断

这两个案例不应靠全局排序解决。

| 案例 | 当前问题 | 必须补的结构 |
|---|---|---|
| `0306W` | `short_direct_cluster` 仍 70 勾，远端 session 6 | `MIXED_SIGNAL_REPAIR` 保守模板、短链 planlet、低置信边界 |
| `0327W` | 远端 session 1 但仍 58 勾 | `DEPOT_DIGEST_ONLY` 库内消化压缩、ordered detach、slot swap |

短链验收不应使用标准链平均指标，而应使用：

```text
solver_hook_count <= manual_hook_count + 1
new_outer_pickup_obligation_count = 0
unnecessary_remote_cross_count = 0
```

## 11. 当前是否可以继续实施性验证

可以继续，但验证对象要改。

不应该继续验证“当前 P10 是否已经达到人工”，因为结论已经是否定的。

应该验证：

```text
每个 accepted action 是否有 phase permission
每个 accepted action 是否有 contract/resource delta
每个远端 session 是否有 owner 和压缩目标
每个高差案例的多出来的勾来自哪个 P/H 失败桶
```

只有这些诊断 trace 建起来后，后续优化才不会变成“越改越复杂，但不知道修了什么”。

## 12. 最终判断

当前版本距离“接口文档输入后直接求解、物理条件符合、可用并达到甚至超越人工计划”还有明显差距。

更精确地说：

```text
物理硬门：已有较好基础，但还不完整。
可解性：89/113，未达可用要求。
勾数：整体显著高于人工，P8 未通过。
H1-H5：当前不是完整 H1-H5，只是联7/远端软门控。
P0-P9：大部分结构没有在 runtime 中输出可验收 trace。
P10：能生成 response，但不能替代 P0-P9，也不能证明达到人工。
```

所以当前不应该声称“P0-P10 已经足够达到甚至超越人工”。正确表述是：

```text
方案文档中的结构充分性仍然成立；
当前代码实现尚未达到该结构充分性；
当前差距主要来自阶段门、合同债务、资源 delta、远端 session、短链模板和同结构少钩搜索没有真实闭环。
```

下一步应先补诊断 trace，再基于诊断逐项优化。
