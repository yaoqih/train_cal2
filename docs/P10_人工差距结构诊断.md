## 5. 典型案例

### 5.1 短链失败

| 案例 | 人工勾数 | 求解勾数 | 人工切换 | 求解切换 | 问题 |

人工短链不是标准 H1-H5 的缩短版，而是一种专门模板：

```text

存4北头摘解
机接或尾项

```

当前 `short_chain_planlet` 只是允许提前做远端动作，并没有阻止后续标准链尾巴继续展开。

最重要的发现：

```text
DEPOT_DIGEST_PLANLET 是当前最大收益结构，
也是当前最大远端切换来源。

它把远端和非远端源/目标混在一个候选里，

```

这不是排序问题。排序只能在候选之间选择，无法修复候选本体已经混合远端和非远端作业的问题。

## 7. 代码层实际问题

### 7.1 RemoteSessionState 不是业务 session

当前 `RemoteSessionState` 主要由这些状态决定：

```text
last_hook_touched_remote
remote_interaction_debt > 0
remote_streak_count < max
```

这只能说明上一候选是否触碰远端，不能说明业务勾序列是否连续。

结果是：

```text
一个候选只要 touches_remote=True，就会被视为远端 session 的一部分；
但这个候选内部可能包含多个非远端 Get/Put。
```

因此 session 数看起来不高，远端业务切换却很高。

### 7.2 candidate 的 remote profile 缺失

当前有 `candidate_touches_remote_interaction()`、`planlet_remote_cross_count()` 和 `candidate_remote_business_transition_count()`，但它们更多用于审计和弱排序。

缺少一个硬结构字段：

```text
FRONT_ONLY
REMOTE_ONLY
FRONT_TO_REMOTE
REMOTE_TO_FRONT
MIXED_REMOTE_FRONT
```

没有这个 profile，`DEPOT_DIGEST_PLANLET`、`REMOTE_EXCHANGE_PLANLET`、`MULTI_DROP_PLANLET`、`target_move` 都可能在不合适的阶段跨区域混排。

### 7.3 DEPOT_DIGEST 没有拆成有方向的模板

人工远端作业一般是有方向的：

```text
前段去修库：F -> R
修库出来去存4/机接：R -> F
修库内部整理：R -> R
```

当前 `depot_digest_planlets()` 为了追求一次消化更多车，会混合：

```text
outbound
inbound
internal
mixed_outbound_then_inbound
```

但没有保证一个候选只表达一个清晰方向。

结果是：

```text
候选收益很高，
但远端/非远端切换爆炸。
```

### 7.4 H1-H5 仍然是软许可，不是债务完成条件

当前阶段门会按前段进度、hook index、short direct 等条件打开 `link7`。如果没有 phase permitted 的前段候选，还会 forced open。

当前失败信号：

| 信号 | 数量 |
|---|---:|
| `phase_action_mismatch` | 262 |
| `over_manual_soft_hook_bound` | 116 |
| `remote_entered_before_front_service_debt_closed` | 95 |
| `remote_business_transition_exceeds_manual_plus_2` | 59 |

这说明 H1/H2 没有真正生成足够强的前段替代候选，只是在候选不足时放行远端。

### 7.5 Recovery 清理还没有完成

本轮已经抑制了一部分 ownerless recovery，但仍有负收益结构进入主流程：

| 结构 | 净收益 | 非正收益业务勾 | 判断 |
|---|---:|---:|---|
| `ROUTE_CLEAR_SESSION_PLANLET` | -36 | 96 | 应降级 |
| `SOURCE_FRONT_BLOCKER_RELOCATION` | -59 | 24 | 应 owner-bound |
| `SOURCE_EXIT_SESSION_PLANLET` | -14 | 8 | 应降级 |
| `TARGET_CAPACITY_RELEASE_TO_STAGING` | 0 | 10 | 应并入容量合同 |
| `FRONT_CAPACITY_RELEASE_CONTRACT` | 2 | 16 | 需要被 digest 替代 |

Recovery 不能作为独立主结构继续参与全局排序。它只能作为某个 owner candidate 的内部步骤。

## 8. 哪些结构保留，哪些降级，哪些重写

### 8.1 保留并强化

| 结构 | 原因 |
|---|---|
| `PhysicalValidator` | 硬物理违规为 0，是当前最稳定资产 |
| `FRONT_CAPACITY_DIGEST_PLANLET` | 前段收益高，远端切换低 |
| `H1_CARRY_PLANLET` | 能表达人工计划前段连续挂摘 |
| `SOURCE_LINE_DIGEST_PLANLET` | 当前净收益高，远端切换低 |
| `DEPOT_SAME_LINE_REPACK` | 远端内部整理，不产生内部切换 |
| `MULTI_DROP_PLANLET / MULTI_PICK_PLANLET` | 能减少勾数，但必须加区域 profile |

### 8.2 降级为 fallback 或 owner-bound 内部步骤

| 结构 | 处理 |
|---|---|
| `ROUTE_CLEAR_SESSION_PLANLET` | 不再作为主候选，必须绑定 remote/corridor owner |
| `ROUTE_CLEAR_RESTORE_PLANLET` | 只允许服务一个已声明 owner 的候选 |
| `SOURCE_EXIT_CLEAR_RESTORE_PLANLET` | 并入 ordered detach，不再独立排序 |
| `SOURCE_CLEAR_RESTORE_PLANLET` | 只允许正收益且不增加远端切换 |
| `SOURCE_FRONT_BLOCKER_RELOCATION` | 必须有 owner 和后续收益 |
| `CAPACITY_RELEASE_TO_STAGING` | 并入 CapacityReleaseContract |
| `SPOT_RELEASE_TO_STAGING` | 并入目标台位/容量合同 |

### 8.3 必须重写

| 结构 | 重写方向 |
|---|---|
| `DEPOT_DIGEST_PLANLET` | 拆成 `FRONT_TO_REMOTE_INBOUND`、`REMOTE_TO_FRONT_OUTBOUND`、`REMOTE_INTERNAL_REPACK` 三类方向模板 |
| `REMOTE_SESSION_CONTRACT` | 从 touches_remote 改为真实 session 状态，持有 corridor、remote debt set、允许退出点 |
| `SHORT_CHAIN_PLANLET` | 从提前许可改成完整短链模板，不再追加标准链尾巴 |
| `TargetMove` 远端动作 | 远端相关单车 target move 只能在 remote session 内作为尾项，不应反复跨区 |

## 9. 下一步不是补丁，而是六个结构工作

### 9.1 先修指标和 trace

目的不是优化结果，而是让后续验证不会误判。

要做：

| 项 | 要求 |
|---|---|
| 人工远端口径 | 明确 artifact 继续用原始口径，另输出 normalized 口径作为校验 |
| candidate remote profile | 每个 accepted candidate 输出 `remote_profile` 和 `internal_remote_transition_count` |
| transition attribution | 每个案例输出远端切换由哪个结构产生 |

验收：

```text
不改变求解结果；
能直接看出每个案例的切换来源。
```

### 9.2 增加 Candidate Remote Profile

所有候选必须声明区域形态：

| profile | 含义 |
|---|---|
| `FRONT_ONLY` | 不触碰修1-修4/卸轮 |
| `REMOTE_ONLY` | 全部业务勾都在修1-修4/卸轮 |
| `FRONT_TO_REMOTE` | 一次进入远端 |
| `REMOTE_TO_FRONT` | 一次退出远端 |
| `MIXED_REMOTE_FRONT` | 发生两次及以上远端/非远端切换 |

验收：

```text
普通候选不得是 MIXED_REMOTE_FRONT。
只有显式 session template 可以持有 MIXED，但必须受人工切换预算约束。
```

### 9.3 重写 DepotDigest

不要再让一个 digest 同时做入库、出库、前段收尾。

新的 digest 应拆为：

| 模板 | 允许的远端形态 |
|---|---|
| `DepotInboundDigest` | `FRONT_TO_REMOTE` |
| `DepotOutboundDigest` | `REMOTE_TO_FRONT` |
| `DepotInternalRepack` | `REMOTE_ONLY` |

验收：

| 指标 | 目标 |
|---|---:|
| `DEPOT_DIGEST_PLANLET` 内部切换 | 接近 0 |
| `0306W` | 不再首候选 F/R/F/R 混排 |
| `0327W` | 不再 short_chain 后继续多次跨区 |
| 全量 completed | 不低于当前 102，容量不可行另算 |

### 9.4 实现真实 RemoteSessionState

Remote session 不应由 `last_hook_touched_remote` 推断，而应持有明确状态：

```text
session_open
corridor_owner
remote_debt_set
allowed_next_profile
transition_budget
exit_allowed
```

规则：

```text
进入远端后，优先消化 remote-only 和 session 内 outbound/inbound。
未到 exit_allowed，不允许穿插普通 FRONT_ONLY 候选。
普通 target_move 不能打断 session。
```

验收：

| 指标 | 当前 | 目标 |
|---|---:|---:|
| 远端切换 P50 | 8 | <= 6 |
| 远端切换 P90 | 12 | <= 8 |
| `solver transition > 8` | 41/96 | <= 10/96 |
| 物理硬违规 | 0 | 0 |

### 9.5 Recovery 全部 owner-bound

不再允许无 owner 的 clear/release/recovery 独立参与主流程。

验收：

| 指标 | 目标 |
|---|---:|
| `ROUTE_CLEAR_SESSION_PLANLET` 非正收益业务勾 | 0 |
| `SOURCE_FRONT_BLOCKER_RELOCATION` 非正收益业务勾 | 0 |
| `CAPACITY_RELEASE_TO_STAGING` 独立候选 | 0 |
| completed | 不低于当前有效可行案 |

短链案例不能走标准长链。

## 11. 当前最终判断

P0-P10 / H1-H5 的理论方向仍然成立，但当前实现没有完全达到理论验收。

已经起效的部分：

```text
P0-P3 物理基础能力稳定。
P4/P7 部分抑制了无效 recovery。
P8/P10 的 planlet 已经能显著降低总勾数。
H1 carry 和 front capacity digest 有明确收益。
```

没有真正起效的部分：

```text
R4 RemoteSession 只是 touches_remote 连续，不是人工式远端连续作业。
R5 ShortChain 只是提前允许远端，不是短链业务模板。
P4 TargetContractSelector 没有把远端切换预算作为硬合同。
P6 ResourceGraph 没有持久 corridor/session owner。
P7 RejectGate 仍允许部分非正收益 recovery。
P8 LocalSearch 没有以区域 profile 约束候选本体。
```

下一步的主线应该是：

```text
先让候选结构本身不再混合远端和非远端动作，
再让 RemoteSessionState 约束连续远端作业，
最后才做排序和权重优化。
```

如果不先做这一步，继续调排序只会让代码越来越复杂，也无法从根上把远端切换降到人工计划的 4 到 6 次水平。

## 13. 2026-06-28 远端会话结构合同实施验证

本轮不是继续调排序权重，而是把人工计划中的“去远端区域一次尽量多办事，少做远端/非远端往返”落成结构合同。

### 13.1 合同定义

新增结构：`remote_session_bundle_planlet`。

它只表达一种业务意图：

```text
一次远端会话内同时处理远端出车和远端入车。
```

合同边界：

| 约束 | 标准 |
|---|---|
| outbound side | 远端线路 -> 非远端线路，至少 3 辆 |
| inbound side | 非远端线路 -> 远端线路，至少 3 辆 |
| 会话形态 | `outbound_then_inbound` 或 `inbound_then_outbound` |
| remote profile | 必须为 `MIXED_REMOTE_FRONT` |
| session 内远端/非远端切换 | 必须等于 2 |
| 最大候选数 | 每轮最多 2 个 |
| 拉车能力 | 不超过 `PULL_LIMIT_EQUIVALENT` |
| owner | 全部车辆写入 `remote_session_bundle_owner` |
| hard obligation | `single_entry_exit_remote_contract` + 远端 owner 约束 |
| protection | `protect_remote_session_contract` + `protect_single_remote_entry_exit` |

这不是允许普通 planlet 混合远端和非远端动作。普通候选仍然受 `p8_reject_mixed_remote_front_candidate` 限制；只有明确带 owner 的远端会话合同可以合法产生 2 次 session 内切换。

### 13.2 代码结构调整

本轮保留了稳定主线，只增加一个结构本体和一个策略变体：

2. bundle 只能处理“双向都有足够车辆”的远端会话，不能覆盖单向 digest、短链直接送修、尾部清场等所有人工会话。
3. `RemoteSessionState` 仍偏统计，不是强状态机；还不能强制“进入远端后必须完成本会话 owner 再离开”。

### 13.8 下一步

下一步不应该继续加零散候选，而应沿着这个结构合同继续做三个结构本体：

1. `RemoteSessionPlanner`：把单向 digest、双向 bundle、短链直送、尾部 closeout 统一成一个远端 session owner。
2. `RemoteSessionStateMachine`：进入远端、远端内部作业、退出远端、禁止无 owner 重入，变成运行时硬约束。
3. `RemoteTransitionBudget`：从人工计划推导每案预算，默认目标 `manual + 2`，严格目标 `manual + 1`，超过预算只能由容量不可行或物理不可达解释。

这三个结构完成后，才适合继续压勾数；否则继续优化排序会重新把代码推向补丁化。

### 14.3 当前实现中真正没有起效的结构

#### 14.3.1 `RemoteSessionState` 只是统计，不是状态机

当前 session 是否 active 主要由这些条件决定：

```text
last_hook_touched_remote
remote_interaction_debt > 0
remote_streak_count < REMOTE_CONTINUITY_MAX_STREAK
```

这只能表达：

```text
上一候选碰过远端。
```

但人工会话需要表达：

```text
我这次进远端要完成哪些 owner；
哪些远端债务必须在离开远端前完成；
允许在哪个口退出；
退出后是否允许再次重入；
重入的业务理由是什么。
```

所以当前出现一个假象：

```text
remote session 数看起来不高，
但 remote business transition 很高。
```

原因是 session 按候选统计，远端业务切换按 Get/Put 业务行统计。一个候选内部仍可能出现 F/R/F/R。

| `REMOTE_SESSION_BUNDLE_CONTRACT -> REMOTE_SESSION_BUNDLE_CONTRACT` | 68 |

这说明问题不是“结构之间连接不好”这么简单，而是：

```text
同一个结构自身没有稳定方向。
同一个结构可以连续多次跨远端边界。
```

所以继续调结构之间的排序，只能有限改善，不能把远端切换压到人工水平。

#### 14.3.4 H1-H5 仍是软阶段，不是债务边界

当前阶段信号：

| 信号 | 数值 |
|---|---:|
| `phase_link7_forced_open_no_front_candidate` | 166 |
| expected H1/H2 中 accepted remote candidate | 293 |
| `phase_action_mismatch` 仍大量存在 | 是 |

当前 `PhaseState` 的核心开口条件：

```text
front_service_progress >= PRE_DEPOT_PROGRESS_OPEN_RATIO
or hook_index > PRE_DEPOT_MAX_HOOKS
or short_direct
```

这不是人工阶段。人工计划的 H1/H2 边界不是“前段进度到 80%”，而是：

```text
前场必须形成一个可承接的远端会话入口。
存4/机接/修库/卸轮相关债务必须被一个 owner 接住。
```

所以 H1-H5 当前只是候选许可，不是合同债务。

#### 14.3.5 `short_chain_planlet` 不是短链合同

当前短链仍失败：

| 指标 | 当前 |
|---|---:|
| short chain cases | 11 |
| failed | 4 |
| 代表失败 | `0306W`、`0327W` |

原因：

```text
short_chain_planlet 只生成一个早期候选；
它没有声明“本案是短链，后续禁止展开标准 H1-H5 尾巴”。
```

人工计划短链的核心是：

```text
进入远端一次，完成短链主债务，然后前场收尾。
```

当前实现是：

```text
先做一个短链/远端候选，然后普通结构继续接管。
```

### 14.4 为什么 P0-P10 局部达标仍不能推出整体达标

之前说 P0-P10 / H1-H5 完备后理论上可达到人工，这个判断的前提是：

```text
每个结构不仅能局部工作，
还必须共同服从同一个 FlowEdge / EdgeContract 主控。
```

当前没有满足这个前提。

具体断点：

| 层 | 当前实现 | 问题 |
|---|---|---|
| P0-P3 物理 | 稳定 | 只能保证动作能走，不能保证组织像人工 |
| P4 TargetContract | 仍偏候选收益 | 没有先选远端 session owner |
| P5 Candidate | 候选多且局部有效 | 候选没有被 FlowEdge 状态约束 |
| P6 Resource | 只有可用/占用 | 没有 corridor/session resource lease |
| P7 Reject | 能挡硬违规 | 没有挡住“业务会话破碎” |
| P8 LocalSearch | 能减少部分勾数 | 不能修复候选本体方向错误 |
| P9 Phase | 软许可 | 不是人工债务完成条件 |
| P10 Runtime | 能输出 trace | 还不是会话级可解释求解 |

因此下一阶段的目标不是“让 P10 再少几勾”，而是：

```text
把 FlowEdge / EdgeContract / RemoteSession 变成主控。
候选只是履约动作，不再自己决定业务方向。
```

### 14.6 下一阶段结构方案

下一阶段只做六个结构本体，禁止继续加散乱补丁。

#### 1. `RemoteSessionPlanner`

作用：

```text
在候选生成之前，先把远端债务分成 1-3 个远端会话。
每个会话有 owner、入口、出口、允许处理的远端/前场交互、预算。
```

输入：

```text
FlowEdge / EdgeContract
StartStatus
DepotAssignment
车辆目标
线路容量
当前机车位置
人工计划远端预算，仅作为验收/弱参考，不作为动作泄漏
```

输出：

```text
RemoteSessionPlan:
  session_id
  session_type
  owner_vehicles
  entry_area
  exit_area
  allowed_profiles
  max_internal_transition
  must_finish_before_exit
  allowed_tail_after_exit
```

session 类型：

| session type | 含义 | 允许 profile |
|---|---|---|
| `REMOTE_PRE_DIGEST` | 存4/机接前的远端预处理 | `FRONT_TO_REMOTE`, `REMOTE_ONLY`, `REMOTE_TO_FRONT` |
| `CUN4_MACHINE_ACCEPT` | 存4北头摘 / 机接桥接 | `FRONT_ONLY`，禁止混远端 |
| `REMOTE_FINAL_DIGEST` | 机接后的修库摘解 | `FRONT_TO_REMOTE`, `REMOTE_ONLY`, `REMOTE_TO_FRONT` |
| `SHORT_DIRECT_REMOTE` | 短链直接远端 | 最多一次进入、一次退出 |
| `REMOTE_INTERNAL_REPACK` | 修库内部整理 | `REMOTE_ONLY` |

验收：

```text
每个 remote accepted candidate 必须绑定 session_id。
同一 session 离开远端前，must_finish_before_exit 完成率 = 100% 或有容量/物理不可达解释。
```

#### 2. `RemoteSessionStateMachine`

状态：

```text
NOT_STARTED
ENTERING_REMOTE
IN_REMOTE
EXITING_REMOTE
CLOSED
REOPEN_FOR_EXPLAINED_EXCEPTION
```

硬规则：

| 规则 | 标准 |
|---|---|
| 无 session 的远端动作 | reject |
| session 内普通候选 `MIXED_REMOTE_FRONT` | reject |
| session 已退出后重入 | 必须消耗新的 session budget |
| `ROUTE_CLEAR_RESTORE` 跨远端 | 只能作为 active session 内部步骤 |
| `TargetMove` 跨远端 | 只能作为 session tail，不得独立反复出现 |
| 短链 session | 关闭后禁止标准链远端尾巴继续展开 |

验收：

```text
remote_cross_without_owner = 0
non_session_mixed_remote_front_accepted = 0
session_reopen_without_reason = 0
```

#### 3. `RemoteTransitionBudget`

预算不是单纯排序惩罚，而是硬合同。

预算来源：

| case family | budget |
|---|---:|
| `FULL_CHAIN_REPAIR` | 默认 `manual + 2`，目标 `manual + 1` |
| `DIRECT_REPAIR_ENTRY` | 默认 `manual + 1` |
| `DEPOT_DIGEST_ONLY` | 默认 `manual + 1` |
| `MIXED_SIGNAL_REPAIR` | 默认 `manual + 2`，必须解释异常 |
| 无人工 baseline | 用 family prior：标准链 6，短链 4，异常 8 |

硬规则：

```text
如果下一候选会导致超过 budget：
  只有容量不可行、物理不可达、必须清 blocker 三类原因可以越过；
  越过必须写入 trace。
```

验收：

```text
solver <= manual + 2 的 completed 可比案例 >= 90%。
solver > manual + 2 的每案必须有 exception reason。
```

#### 4. `FlowEdge-owned DepotDigest`

重写当前 `DEPOT_DIGEST_PLANLET`。

当前问题：

```text
DEPOT_DIGEST 有收益，但它只是每轮局部看 unsatisfied。
它不知道自己属于哪个远端会话，也不知道本轮是入远端、出远端还是远端内部整理。
```

新结构：

| 新模板 | 方向 | 说明 |
|---|---|---|
| `DEPOT_INBOUND_DIGEST` | `FRONT_TO_REMOTE` | 前场车辆集中送修库 |
| `DEPOT_OUTBOUND_RELEASE` | `REMOTE_TO_FRONT` | 修库/卸轮车辆集中出远端 |
| `DEPOT_INTERNAL_REPACK` | `REMOTE_ONLY` | 修库内部同区域整理 |
| `DEPOT_SWAP_IN_SESSION` | `REMOTE_ONLY` 或一次 `F/R` | 只能在 active session 中做 |

硬规则：

```text
同一个 depot digest 候选不得自行决定会话方向。
方向来自 RemoteSessionPlan。
```

验收：

```text
DEPOT_DIGEST_PLANLET 自身 internal transition 总量下降 80%。
DEPOT_DIGEST -> DEPOT_DIGEST 的重复跨区切换从 120 降到 <= 20。
```

#### 5. `ShortChainContract`

短链必须是 case-level 合同，不是一个 planlet。

适用：

```text
0306W / 0327W / 0213W 等短链或近远端直接处理案例。
```

合同：

```text
短链主 session 关闭后：
  禁止再展开标准 FULL_CHAIN 的 H3/H4 远端尾巴；
  只允许 FRONT_CLOSEOUT 或 REMOTE_INTERNAL_REPACK。
```

验收：

| 指标 | 标准 |
|---|---:|
| short chain completed | 100% |
| short chain hook | `<= manual + 1`，过渡期 `<= manual + 5` |
| short chain remote transition | `<= manual + 1` |
| short chain 后标准链尾巴 | 0 |

#### 6. `Owner-bound Recovery`

恢复结构不再是主结构。

处理方式：

| 当前结构 | 下一阶段处理 |
|---|---|
| `ROUTE_CLEAR_SESSION_PLANLET` | 只能作为 active session 的 corridor step |
| `ROUTE_CLEAR_RESTORE_PLANLET` | 只能有 owner，且不得增加远端切换预算 |
| `SOURCE_EXIT_CLEAR_RESTORE_PLANLET` | 并入 session exit |
| `SOURCE_CLEAR_RESTORE_PLANLET` | 只允许 front-only 或 remote-only |
| `blocker_relocation` | 必须绑定 owner candidate |

验收：

```text
recovery 结构 accepted 后 unsatisfied_reduction < 0 的数量 = 0。
recovery 结构产生的 remote transition <= 全量 5%。
```

### 15.5 剩余主因

全量剩余高差距案例中，主要不是可解性问题，也不是物理问题，而是远端会话还没有真正成为单一主控。

高差距案例中常见结构：

| 残余结构 | 现象 | 下一步 |
|---|---|---|
| `DEPOT_DIGEST_PLANLET FRONT_TO_REMOTE` | 预算后仍大量出现，贡献 over +2 切换 | digest 必须归属于 planned session，而不是独立候选反复入远端 |
| `MOVE_TO_PLANNED_TARGET FRONT_TO_REMOTE` | 单车或小批量入远端仍存在 | 预算后只能作为 session tail，不能独立开新远端动作 |
| `DEPOT_SAME_LINE_REPACK REMOTE_ONLY` | 本身不跨远端，但经常夹在前后跨远端动作之间 | remote-only repack 要并入同一 session 的内部步骤 |
| `ROUTE_CLEAR_SESSION_PLANLET` 负收益 | 少数尾部会用负收益打开路线，制造额外切换 | 只能由 session owner 调用，且必须证明后续收益 |
| `REMOTE_SESSION_CONTRACT MIXED` | 有改善，但仍会产生内部多次切换 | 需要 FlowEdgeOwnedSessionTemplate 统一 entry/exit |

### 15.6 下一步结构工作

下一步不应该继续新增宽泛策略，而要把 `remote_session_only_contract` 提升为真正的会话主控：

1. `RemoteSessionPlanner`

在每案 H3/H4 前生成 1-3 个 planned session：

```text
session_id
session_type
entry_side
exit_side
allowed_source_lines
allowed_target_lines
expected_transition_budget
owned_candidate_kinds
```

2. `SessionOwnedDepotDigest`

`DEPOT_DIGEST_PLANLET` 必须绑定 session：

```text
同一 session 内允许多次 Get/Put；
跨远端 entry/exit 只能按 session 模板发生；
预算后禁止独立 front_to_remote digest。
```

3. `SessionTailPolicy`

尾部 `MOVE_TO_PLANNED_TARGET`、`ROUTE_CLEAR_SESSION_PLANLET`、`SOURCE_EXIT_CLEAR_RESTORE_PLANLET` 必须满足：

```text
有 session owner；
后续收益可证明；
不能为了单车制造新的远端来回。
```

4. `RemoteTransitionBudget` 从软排序变硬合同

本轮 strict gate 单独验证会阻塞，说明现在还不能硬开。

硬开前必须满足：

```text
预算内候选池能覆盖剩余远端债务；
预算后候选只能是 session tail 或物理必要恢复；
代表集不阻塞。
```

5. 下一轮验收目标

```text
completed >= 103
hard physical = 0
unknown route = 0
remote transition P50 <= 6
remote transition P90 <= 8
remote transition max <= 10
solver <= manual + 2 >= 70/97
solver > manual + 2 <= 27/97
hook delta P50 <= +1
hook delta P90 <= +8
```

达到这一档后，才进入“接近人工”的最终验收。

## 16. 2026-06-28 SessionOwnedDepotDigest 实施验证

本轮继续沿第 15 节的结论推进，不再增加尾部硬拒绝。

先验证过一个失败方向：

| 实验 | 结果 | 结论 |
|---|---|---|
| 远端预算后硬拒绝低收益 tail | 代表集 `0309W` 从完成变阻塞 | 撤回，不能用硬拒绝替代会话承接能力 |

保留的结构改动只有三类：

| 结构 | 作用 |
|---|---|
| 多策略选优先比较远端业务切换 | 完成、物理合法之后，不再用少量勾数优势掩盖远端往返过高 |
| 远端会话未清债务时惩罚提前退出远端 | 仍是软排序，不阻断必要恢复 |
| `remote_session_contract_planlets` 纳入同线远端未满足车 | 把运行时产生的 `DEPOT_SAME_LINE_REPACK` 债务并入远端 session owner |

关键结构判断：

```text
`DEPOT_SAME_LINE_REPACK` 不是初始输入债务，
而是远端入库/出库过程中产生的运行时同线位置债务。

如果 `remote_session_contract_planlets` 不接住这类同线车，
求解器只能在会话结束后再单独进远端重排，
从而制造人工里很少出现的远端/非远端反复切换。
```
