## 8. 优化前根因诊断

本轮扩大样本后，不能把问题简单归因成“远端太早”或“排序不好”。当前主要有几类差距：

| 类型 | 代表案例 | 现象 | 根因 |
|---|---|---|---|
| 高勾数 + 高远端切换 | `0210W`、`0204W`、`0312W`、`0112W`、`0212W` | `remote_business_transition_count` 远高于人工 | 每次远端动作前后都清路/恢复，没有持久 corridor/session |
| 高勾数但切换一般 | `0127W`、`0203Z`、`0126Z`、`0324Z`、`0302W` | 远端切换不是最高，但总勾数仍高 | H1/H2 前段服务和容量释放被拆成大量单车 target move/release |
| 短链失败 | `0306W`、`0327W`、`0206W`、`0130Z` | 人工 10-37 勾，求解仍展开成长链 | short-chain 只是早远端许可，不是短链动作模板 |
| 接近人工 | `0117Z`、`0205W`、`0225W`、`0311Z`、`0321Z` | 勾数不高于或接近人工 | recovery 动作少，能直接生成远端 session 或 H1 carry |
| 阻塞案 | `0106Z`、`0120Z`、`0311W`、`0317Z` | 勾数已经很多仍 blocked | recovery/release 没有形成可恢复资源图，最后候选断路 |

### 8.1 人工和当前求解的关键差别

人工不是每次进远端都先把路清空再恢复原状。典型人工计划是：

```text
H1/H2：先做功能线、调棚、存5/存2/存3、存4北头成形。
H3/H4：进入修1-修4/卸轮后，连续完成接入、存4北头摘、机接、修库摘解。
H5：再做库回或少量尾项。
```

例如：

| 案例 | 人工 | 当前求解 |
|---|---|---|
| `0210W` | 39 勾；远端切换 4；远端勾 6 | 136 勾；远端切换 32；其中 95 个业务勾属于 clear/release/recovery 类动作 |
| `0206Z` | 40 勾；远端切换 6；远端勾 6 | 116 勾；远端切换 16；远端 corridor 多次以 `预修线/机走棚 -> 远端 -> restore` 形式重复 |
| `0312W` | 34 勾；远端切换 4；远端勾 11 | 70 勾；远端切换 20；远端和调梁棚/存5线北多次交替 |
| `0306W` | 10 勾；短链直接修库接入、存4摘、机接 | 40 勾；虽然用了 `short_chain_planlet`，但后续又追加 H1 carry、corridor、reorder |
| `0104W` | 41 勾；前 29 勾完成前段，30-38 集中修库 | 60 勾；第 7 勾已触碰远端，远端动作分散到多个 session |

人工勾数可能有少量省略，但这些差异不是“少量省略勾”能解释的。

### 8.2 最大可量化根因：recovery 类动作泛滥

把以下结构视为 recovery/clear/restore 类动作：

```text

BLOCKER_CLEARANCE
SOURCE_CLEAR_RESTORE_PLANLET
SOURCE_EXIT_CLEAR_RESTORE_PLANLET
ROUTE_CLEAR_RESTORE_PLANLET
same_line_stage_out / spot_release_to_staging
```

当前统计：

| 范围 | 业务勾 | recovery 业务勾 | 非正收益 recovery 勾 | recovery 远端切换 |
|---|---:|---:|---:|---:|
| 全量 113 案 | 6426 | 1630 | 957 | 227 |

人工可比完成案的总勾数差距是 `+1638`，其中 recovery 业务勾就有 `1003`。相关性也很强：

这说明当前最大的结构失效点不是远端 planlet 本身，而是求解器大量接受“先清路、做一件事、再恢复”的补丁动作。

### 8.3 代码层实际发生了什么

当前代码不是一个稳定的 P0-P10 闭环，而是：

```text
target_move 先生成很多单车/小批候选；
如果路径或源线被挡，再生成 clear_and_restore / route_clear_restore；
如果目标容量不够，再生成 capacity_release_to_staging；
排序会按 reduction/hook 接受这些动作；
动作完成后恢复原状，但没有把 corridor、释放位、远端 session 当成持久资源；
下一轮又重新清同一条路或释放同一类容量。
```

问题最大的结构：

| 当前结构 | 当前作用 | 诊断 |
|---|---|---|
| `route_clear_restore_planlet` | 为单个候选临时清线路，做完恢复 | 应降级为最后 fallback；不能作为主求解结构 |
| `source_exit_clear_restore_planlet` | 为从源线拿目标车临时带出非目标车再放回 | 应被 ordered detach / carry chain 替代 |
| `source_clear_restore_planlet` | 同线清障再恢复 | 当前容易制造非正收益动作，应硬约束 |
| `capacity_release_to_staging` | 为容量腾位 | 393 批、786 勾，但全量收益合计为 -4，说明它不是完整 release contract |
| `blocker_relocation` | 搬前端阻挡车 | 83 批全是非正收益，必须只有明确 owner/recovery 时才允许 |

应该保留并强化的结构：

| 结构 | 证据 |
|---|---|
| `h1_carry_planlet` | 好案例中常见，能减少前段服务碎片 |
| `remote_session_contract_planlet` | 有收益，但还缺 session 内 ordered detach |
| `remote_exchange_planlet` | 能一次搬多车，但需要受远端 owner/corridor 限制 |
| `depot_same_line_repack` | 远端内部整理不产生切换，方向正确 |
| `multi_drop_planlet / multi_pick_planlet` | 能表达人工计划的连续挂摘，但需要更强合同归属 |

### 8.4 方案层哪些结构没有起效

| 方案结构 | 现在的问题 |
|---|---|
| P4 `TargetContractSelector` | 仍允许非正收益 recovery 动作，没有 owner 也能进主序列 |
| P6 `StationResourceGraph` | 没有持久 corridor、释放位、存4北口、远端 session 资源；每轮都重新清路 |
| P7 `ResourceDeltaRejectGate` | 只审计 dominated，不拒绝大量非正收益 recovery |
| P8 `LocalTieBreakSearch` | 没有把 2-6 步内的 clear/release/remote/digest 作为一个计划单元评估 |
| H1-H5 | 阶段像软许可，不是“阶段债务完成条件”；超过人工软上界后仍继续搬 |
| R4/R5 | 远端 session 和短链只是在生成候选，未形成人工计划式模板 |

因此，结构理论仍成立，但当前真正起效的只有物理 validator 和部分 planlet；P4/P6/P7/P8 没有形成闭环。

## 9. 接下来做什么：先去除无用结构，再优化

不要继续叠加排序权重。下一步应按以下顺序做结构本体。

| 顺序 | 工作 | 具体要求 | 验收 |
|---:|---|---|---|
| 1 | Recovery 动作准入重构 | `capacity_release/blocker/clear_restore` 必须有 owner、后续收益和闭环恢复计划；无 owner 的非正收益直接拒绝 | `nonpositive_recovery_hooks` 从 957 降到接近 0；可解率不下降 |
| 2 | 持久 Corridor / Session Resource | 一旦为远端打开 `机走棚/预修线/存4线` corridor，不能每个候选做完就恢复；session 内复用 | `remote_business_transition_count` P50 从 10 降到 <=5，P90 <=8 |
| 3 | FrontServiceCarryChain | H1/H2 用 receiver-aware carry 合并功能线、调棚、预修、存车整理 | 高勾数低切换案的 recovery 勾下降 50% |
| 4 | DepotDigest/OrderedDetach | 修1-修4/卸轮/存4摘/机接必须是一组远端 digest 模板 | `remote_hook_delta_p50 <= 1`，`remote_session_not_compressed=0` |
| 5 | ShortChainPlanlet 重写 | `MIXED_SIGNAL_REPAIR/DEPOT_DIGEST_ONLY` 直接走短链模板，不再追加标准链 H1/H4 尾巴 | 11 个短链全部 `completed` 且 `hook <= manual+1` |
| 6 | 容量不可行前置门 | 最终目标线长度不可行的案例先标输入不可行，不进入优化比较 | 9 个 capacity infeasible 从求解失败中拆出 |

应降级或删除的结构口径：

| 结构 | 处理 |
|---|---|
| `route_clear_restore_planlet` | 降级为 fallback，只允许服务一个已声明 session/corridor owner |
| `source_exit_clear_restore_planlet` | 降级为 ordered detach 的内部步骤，不再作为独立候选参与全局排序 |
| `source_clear_restore_planlet` | 只允许净收益为正且不增加阶段债务；否则拒绝 |
| `capacity_release_to_staging` | 不能独立存在，必须并入 `CapacityReleaseContract`，包含释放对象、占用窗口和回收动作 |
| `blocker_relocation` | 必须绑定 owner candidate；无 owner 非正收益拒绝 |

### 9.1 本轮结构清理和验证结果

本轮已经不再只做审计字段，而是开始改结构本体。结论是：

```text
勾数和可解性明显改善；
远端业务勾切换仍未达到人工；
不能验收为“达到甚至超过人工”。
```

已完成的结构改动：

| 项 | 本轮完成 | 作用 |
|---:|---|---|
| 1 | `route_clear_restore_planlet / source_exit_clear_restore_planlet` 降级为 fallback，并新增 `route_clear_session_planlet / source_exit_session_planlet` | 清路动作不再优先作为主流程，开始区分“临时恢复”和“会话打开” |
| 2 | 重写 `depot_digest_planlets()`，支持 `outbound_only / inbound_only / remote_internal / mixed_outbound_then_inbound` | 修1-修4、卸轮、存4线相关车可以被远端 digest 成组处理，不再要求入库和出库同时满足旧阈值 |
| 3 | 新增 `corridor_closeout_planlet` | 会话打开后可以批量把前端 corridor / 功能线尾债送回目标，减少散落收尾 |
| 4 | 新增 `front_capacity_digest_planlet`，并把 front capacity contract 接入默认 contract 策略 | 调梁棚、预修线、洗罐站等前端容量释放开始 owner-bound 化，减少“腾一辆、送一辆” |
| 5 | `line_has_length_capacity()` 修正多步 planlet 中同一批车已被拿走时的长度扣减 | 允许“先腾位、再放入”的同一 planlet 被正确判定 |
| 6 | 默认策略从 7 个近似变体收敛为 base / contract / remote_session_contract | truth2 默认全量可以跑完，避免多策略组合爆炸 |

验证 artifact：
