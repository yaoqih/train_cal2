# P10 人工差距结构诊断：当前先压远端切换

日期：2026-06-30

本文只保留当前仍成立的结论。当前基线以 `artifacts/roundFM_depot_no_compact_clean_full` 为准。

## 1. 当前判断

现在不应该继续把“总勾数”作为主优化目标。

原因有三点：

1. 人工勾数存在省略勾和口径差异，solver 输出的是接口级 `Get/Put`，两者不是完全同一个计数口径。
2. 当前远端切换已经接近人工计划分布，但仍有少数明确超标案例，这些更适合作为结构优化目标。
3. 勾数高的案例里，很多远端切换已经在 `manual + 2` 内，继续直接压勾数容易误伤可解性和物理约束。

所以当前阶段目标应收敛为：

```text
主目标：在物理不放松、可解性不下降的前提下，降低远端业务切换。
辅目标：勾数只作为诊断信号，重点看“勾数高且远端切换也高”的案例。
```

- 人工计划省略了部分取放动作。
- solver 输出了完整接口 `Get/Put`。

因此短链下一步应先作为“勾数诊断专项”，不要和远端切换主线混在一起改。

## 6. 当前真正要修的结构

### 6.1 Session Closeout

目标：远端 session 关闭前，必须检查剩余远端债务。

规则：

- 如果剩余远端 debt <= 3，优先纳入当前 session closeout。
- closeout 后只允许 front-only 收尾。
- 如果 closeout 后仍需要再次进入远端，必须写入 exception trace，不能静默由 `target_move`、`depot_digest` 或 `source_exit_session` 接管。

对应案例：

- `0104Z` 的尾部 `卸轮线 -> 存4线` 单车重入。
- `0228W` 的尾部 `存4线 -> 修1/修4库内` 单车重入。
- `0121Z` 的尾部 `DEPOT_DIGEST_PLANLET`。

### 6.2 FlowEdgeOwnedSessionTemplate

目标：`REMOTE_SESSION_BUNDLE_CONTRACT` 内部不要反复 F/R/F/R。

当前超标案例的大部分远端切换来自：

```text
REMOTE_SESSION_BUNDLE_CONTRACT -> REMOTE_SESSION_BUNDLE_CONTRACT
```

说明 bundle 虽然叫 session，但内部仍是多次跨远端。下一步应把 bundle 改成 session-owned template：

- 先明确本 session 的 outbound、inbound、internal、same-line repack、weigh debt。
- 再按一个 session 的 entry/body/exit 生成候选。
- 不允许 bundle 之外再由 `depot_digest` 或 `target_move` 处理同一批远端尾债。

### 6.3 Route Clear / Source Exit 降级

`ROUTE_CLEAR_SESSION_PLANLET` 和 `SOURCE_EXIT_SESSION_PLANLET` 不能作为独立远端主控。

它们只能是 active session 内的清障步骤。否则会出现 `0120Z` 这种情况：

```text
ROUTE_CLEAR_SESSION -> REMOTE_SESSION_BUNDLE -> SOURCE_EXIT_SESSION -> MOVE_TO_PLANNED_TARGET
```

这不是人工式两段远端作业，而是远端段被多个结构拆碎。

## 7. 当前不做什么

下一轮暂时不做以下优化：

1. 不以总勾数下降作为主目标。
2. 不为了压勾数放松路径、占用、容量、台位、关门车、对位规则。
3. 不继续增加全局排序项。
4. 不把短链勾数强行压到人工计划，因为人工计划可能存在省略勾。
5. 不引入新的 legacy fallback。

勾数只用于发现问题：

```text
如果一个案例勾数高且远端切换也高，优先修远端结构。
如果一个案例勾数高但远端已达标，先记录为局部批量或人工口径问题。
```

## 9. 下一步实施顺序

1. 先修 `Session Closeout`：把尾部 1 到 3 个远端 debt 纳入当前远端 session。
2. 再修 `FlowEdgeOwnedSessionTemplate`：减少 bundle 内部 F/R/F/R。
3. 把 `depot_digest`、`source_exit_session`、远端 `target_move` 降级为 session 内动作或 exception。
4. 跑 5 个远端超标案例 probe。
5. 再跑全量 113 案。
6. 只有当远端切换稳定后，再开短链勾数专项。

一句话结论：

```text
现在不用纠结全局勾数。
当前最清楚、最可验证、最接近人工计划的优化方向，是先把 5 个远端切换超标案例压回 manual + 2 内。
```
