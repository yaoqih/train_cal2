# P10 与人工计划差距的结构诊断

日期：2026-06-27

## 0. 结论

当前应该先诊断，再优化。

最新 P10 运行时已经证明了一件重要的事：物理路径、线路占用、长度、台位、牵引和基本称重/关门车规则可以进入自动校验，并且当前已接受动作的硬物理违规为 0。这是必要进展。

但它还没有证明方案已经达到人工计划，更不能证明已经超越人工计划。核心原因不是单个排序权重不够好，而是当前代码只实现了一个 P10 物理运行时骨架和若干启发式策略，没有真正实现方案文档要求的 `FlowGraph -> FlowEdge -> EdgeContract -> TargetContractSelector -> ResourceRequest -> ContractDelta -> ResourceDelta -> LocalTieBreakSearch -> Trace` 闭环。

所以，之前的判断需要精确拆开：

| 问题 | 当前结论 |
|---|---|
| 如果文档中的 P0-P10 结构和 H1-H5 阶段门全部达到量化验收，是否足以达到甚至超越人工？ | 是。结构上是充分的，因为合同、资源、阶段、候选和少钩搜索形成闭环。 |
| 当前 P10 运行时是否已经说明这些结构全部达标？ | 否。当前主要是 source-target 候选、物理 validator、静态 depot assignment、启发式排序。 |
| 当前是否已经达到人工计划勾数水平？ | 否。按标准验收口径，80 个可与人工计划匹配且求解完成的案例中，算法平均比人工多 46.23 个业务挂摘勾，只有 1 个不劣于人工。 |
| 当前最主要问题是什么？ | H1-H5 阶段门没有真实落地，远端修库/卸轮区域过早过频参与，P4/P7/P8 缺少合同收益、资源收益和同结构少钩搜索。 |

## 1. 诊断口径

### 1.1 勾数口径

人工单中一勾是一次挂或摘动作。

当前 P10 artifact 中可比字段是：

| 字段 | 含义 | 是否用于和人工勾数对比 |
|---|---|---|
| `business_get_put_hook_count` | `Get + Put` 的业务挂摘动作数 | 是 |
| `internal_move_batch_count` | 一个内部 source-target 移动批次，通常包含一次 `Get` 和一次 `Put` | 否 |
| `interface_operation_count` | API 操作数，含 `Weigh` | 否 |

因此本文使用 `business_get_put_hook_count` 比较人工计划勾数。

### 1.2 数据来源

| 类型 | 路径 |
|---|---|
| 当前 P10 标准 artifact | `artifacts/p10_10_1_10_6_full_final` |
| 当前 P10 诊断 artifact | `artifacts/p10_10_1_10_6_diagnostic_full_final` |
| 人工案例 | `data/人工调车数据/*人工调车作业单/*.xlsx` |
| 当前运行时代码 | `scripts/generate_physical_runtime_trace.py` |
| 结构方案标准 | `docs/flow_edge_structure_model_design.md` |
| 人工基准研究 | `docs/人工调车案例基准研究.md` |

人工单线路名可能不是最新，人工勾数也可能有少量省略勾。但当前差距远大于“少量省略勾”能解释的范围。

## 2. 当前 P10 运行结果

最新标准 artifact：`artifacts/p10_10_1_10_6_full_final`。

这里的“标准口径”已经把 `early_depot_base/cluster` 从默认选中策略中移除，只保留 `phase_gate_base/cluster` 和有人工变体证据的 `short_direct_cluster`。因此它比旧的 `p10_early_base_full` 更适合做 H1-H5/P0-P10 验收口径，但指标会更差。

|---|---:|---|
| truth case 数 | 113 | 全量 truth2 |
| completed case 数 | 85 | 可解率 75.2%，没有达到全量可用 |
| blocked case 数 | 28 | 仍有明显可解性缺口 |
| total_final_unsatisfied_vehicle_count | 585 | 未完成车辆仍多 |
| hard_physical_violation_accepted_count | 0 | 物理硬门表现好 |
| unknown_route_count | 0 | 路径图覆盖当前输出 |
| depot_slot_failure_count | 0 | 静态台位分配没有失败，但不等于动态 swap 已解决 |
| business_get_put_hook_count | 8654 | 总业务挂摘勾 |
| internal_move_batch_count | 4327 | 内部移动批次 |

作为对照，诊断模式 artifact `artifacts/p10_10_1_10_6_diagnostic_full_final` 允许 `early_depot_base/cluster` 参与选中，结果是 88/113 完成、6470 个业务挂摘勾。它比标准口径少勾、完成更多，但大量标准链案例会过早触碰修库/卸轮远端区域，因此只能作为证明 `early_depot` 效果和风险的诊断对照，不能作为 H1-H5 验收结果。

## 5. H1-H5 与人工阶段的偏差

### 5.1 人工阶段位置

人工基准显示，标准链常见阶段是：

```text
H1 前段全站组织与服务处理
H2 存4释放口成形与保护
H3 存4大释放-机接硬边界
H4 修库消化、库位和出入库冲突处理
H5 尾项收束与库回
```

典型人工案例：

| 案例 | 人工勾数 | 人工 variant | H1 | H2 | H3 | H4 | H5 |
|---|---:|---|---|---|---|---|---|
| `0104W` | 41 | `FULL_CHAIN_REPAIR` | 1-28 | 29-33 | 34-35 | 36-38 | 41 |
| `0210W` | 39 | `FULL_CHAIN_REPAIR` | 1-23 | 24-31 | 32-33 | 34-36 | 39 |
| `0304W` | 46 | `FULL_CHAIN_REPAIR` | 1-30 | 31-37 | 38-39 | 40-43 | 46 |
| `0306W` | 10 | `MIXED_SIGNAL_REPAIR` | 无标准 H1 | 2-6 | 无标准 H3 | 3 | 10 |
| `0327W` | 11 | `DEPOT_DIGEST_ONLY` | 1-3 | 无 | 无 | 4 | 11 |

人工并不是永远不先做库侧动作。短链、库内消化、低信号案例会很早触碰修库。但标准链的大多数案例确实先完成外场/功能线/调棚/存车区组织，再进入修库/卸轮远端区域。

### 5.2 当前标准求解仍没有学会远端压缩

这里把修1-修4库内/外和卸轮线视为远端区域。人工口径用 `修1/修2/修3/修4/卸轮` 前缀近似，当前求解口径用 `修1-修4库内/外 + 卸轮线`。

匹配且求解完成的 80 案：

| 指标 | 人工 | 当前标准求解 |
|---|---:|---:|
| 首次触碰远端的绝对位置 P50 | 第 20 钩 | 第 29 个业务动作 |
| 首次触碰远端的相对位置 P50 | 全案 62.5% | 全案 37.8% |
| 远端相关挂摘动作 P50 | 7 | 17 |
| 远端相关内部批次 P50 | 不适用 | 15.5 |
| 远端 session P50 | 2 | 2 |
| 远端跨入/跨出批次 P50 | 不适用 | 14 |

这个结果非常关键：`early_depot` 降级后，远端首次出现位置已经不像旧口径那样第一个或第二个动作就触碰修库/卸轮；但它相对人工计划仍偏早，而且远端相关挂摘、批次和跨区没有压缩。当前问题已经从“远端过早”进一步暴露为“远端 session 只是连续发生，内部没有人工计划式消化”。

## 6. 当前代码和方案结构的差距

方案文档 `20A.2B` 明确写了 P10 不能替代 P0-P9：一个案例最终跑通，但中间结构指标失败，仍然不能证明方案可行。

当前运行时代码的实际结构如下：

```text
输入 JSON
  -> normalized_car
  -> build_depot_assignment / build_short_direct_depot_assignment
  -> build_candidates
  -> PhysicalValidator.validate
  -> phase_candidate_sort_key / candidate_sort_key
  -> apply_candidate
  -> operation_trace / response JSON
```

这和方案要求的结构闭环不同：

```text
FlowGraph
  -> FlowEdge
  -> EdgeContract
  -> TargetContractSelector
  -> StructuralIntent
  -> WorkPatternTemplate
  -> ResourceRequest
  -> StationResourceGraph
  -> ContractDelta
  -> ResourceDelta
  -> AcceptRejectGate
  -> LocalTieBreakSearch
  -> State Update / FlowGraph Rebuild / Trace
```

### 6.1 P0-P10 对照

| 过程 | 方案要求 | 当前实现状态 | 影响 |
|---|---|---|---|
| P0 在线证据边界 | 在线 feature、status source、无人工泄漏 | 只有输入字段校验和 normalize，没有 status source trace | 不能证明低信号和阶段判断来自在线状态 |
| P1 车辆分类 | `FlowClassify / NoMoveVehicle / ResidualItem` | 以目标线和满足状态粗分，无合同覆盖 trace | 不能区分“无需动”和“阻塞关键资源但目标已满足” |
| P2 建边与身份稳定 | `FlowGraph / FlowEdge / EdgeKey` | 没有 active edge、edge identity、subject overlap trace | 主业务流不能稳定表达，阶段判断无 owner |
| P3 状态与合同生成 | `FlowEdgeStatus / EdgeContract / RepairInboundVariant` | 只有 action family 和少量 case_id 短链硬编码 | 变体名存在，但没有合同条款和置信度 |
| P4 目标合同与结构意图 | `TargetContractSelector / StructuralIntentBuilder` | `planned_target_for_car` + sort key | 不知道当前动作服务哪个主合同、释放多少下游债务 |
| P5 候选生成 | 边界化候选、人工模板、resource request | source line 第一个可达未满足车的 prefix 候选 | 候选要么缺短链模板，要么拆成大量 source-target 批次 |
| P6 资源仲裁 | `StationResourceGraph / GlobalGate / DepotSlotGraph / LocoCarryState` | 有物理路由、占用避让、静态台位分配、机车位置 | 能过滤部分非法动作，但不能做资源等待、swap、session 和 ordered detach |
| P7 Delta 与硬门 | `ContractDelta / ResourceDelta / AcceptRejectGate` | 物理 validator 通过即候选可进入排序 | 物理合法但结构上劣质的动作会被接受 |
| P8 同结构少钩 | `ContractOptimizer / LocalTieBreakSearch` | 贪心排序，有 dominance audit，但无 planlet search | 总勾数系统性高于人工 |
| P9 状态更新与 trace | rebuild、车辆守恒、failure bucket、结构 trace | 有车辆位置更新、CSV trace、物理原因、phase/contract/resource trace | 仍缺少 FlowGraph diff 和真实 contract/resource 决策闭环 |
| P10 全量压测 | 113 案完成、硬违规 0、勾数通过 P8 | 标准口径 85/113 完成，硬违规 0，但勾数失败 | 当前只能证明 P10 骨架局部有效 |

## 7. 主要根因

### 7.1 PhaseGate 是软启发式，不是 HumanPhaseContract

当前 `phase_state_for_case` 只有三个状态：

```text
H1_H2_LINK7_CLOSED
H3_H4_DEPOT_ALLOWED
H3_SHORT_DIRECT_DEPOT
```

它不是 H1-H5。它没有 H1/H2/H3/H4/H5 的进入条件、退出条件、跳过条件、失败条件和阶段债务。

旧口径中更关键的是，`run_case` 默认会尝试：

```text
phase_gate_base
early_depot_base
phase_gate_cluster
early_depot_cluster
short_direct_cluster
```

其中 `early_depot_base` 和 `early_depot_cluster` 会把 `short_direct_override=True` 传入 phase state。这样很多标准链案例会进入 `SHORT_DIRECT_DEPOT` 的排序逻辑，直接优先 `REPAIR_INBOUND / DEPOT_SLOT / DEPOT_OUTBOUND`。

这解释了为什么旧 `early_depot` 口径里大量标准链案例第 1 或第 2 个业务动作就触碰修库/卸轮。当前标准口径已经把 `early_depot` 降级为诊断策略；但 `phase_reject_reason` 仍只是软门，不能构成完整 HumanPhaseContract。

另一个问题是 `phase_reject_reason` 只是在联7关闭时 defer depot 候选；如果没有前段候选，代码会走 `phase_link7_forced_open_no_front_candidate`，也就是软打开。统计中出现了多次 front_service_progress 仍为 0.00、0.31、0.58 时被迫放行远端候选。

诊断结论：

```text
当前 PhaseGate 不能证明 H1-H5 完成。
它只能证明某些联7/远端候选曾被延迟或被迫放行。
```

### 7.2 当前目标选择是“车到目标线”，不是“合同债务选择”

当前 `planned_target_for_car` 基本按：

```text
depot slot assignment
else target line with lower load
```

当前 `candidate_sort_key` 基本按：

```text
candidate_kind priority
line_priority
batch size
candidate_id
```

这和方案中的 `TargetContractSelector` 不同。方案要求先问：

```text
当前最重要的合同债务是什么？
这个动作能释放多少资源？
会不会破坏更高优先级合同？
是否处于 H1-H5 允许的阶段？
```

当前代码更多是在问：

```text
这辆车最终该去哪条线？
从当前线直接送过去是否合法？
哪个候选排序更靠前？
```

所以它会把 `修库入库`、`修库出库到存4`、`洗罐到修库` 等物理合法动作提前做掉，即使人工计划结构上要先完成 H1/H2。

### 7.3 候选生成粒度过细，缺少人工计划的“带车列连续动作”

人工计划的少钩能力来自当前车列的连续挂摘：

```text
挂 A
挂 B
摘 C
挂 D
去存4北摘
机接
按尾端可摘顺序分库
```

当前运行时的基本动作是一个内部 source-target 批次：

```text
Get(source_line, batch)
Put(target_line, batch)
```

每个批次通常对应 2 个业务挂摘勾。它可以批量搬同一 source、同一 target、同一 prefix 的车，但很难表达人工计划中的混合携带、一路挂摘、多目的地分解和尾端可摘策略。

这直接导致：

| 表现 | 根因 |
|---|---|
| `0306W` 短链 10 勾变 70 勾 | 没有短链 planlet，只能逐 source-target 搬运 |
| `0327W` 远端 session 集中但仍 58 勾 | 远端内部没有 ordered detach 和 depot digest 压缩 |
| `0210W` H1 来回整理到 112 勾 | 外场/调棚/存车区没有 co-carry batching |

### 7.4 DepotSlotGraph 目前是静态分配，不是动态 swap

当前有 `build_depot_assignment` 和 `build_short_direct_depot_assignment`，能给修库目标车分配台位，并检查长度、厂修/段修规则、强制位置等。

但这不是方案中的 `DepotSlotGraph / DepotSwapDelta`。缺少：

```text
slot occupied_by
blocked_by
released_slot
reserved_slot
swap dependency
stayer lock
库内/库外衔接位
一次远端 session 内的出库-入库-摘解压缩
```

结果就是：台位规则能过滤非法位置，但不能告诉求解器“先出哪一组、一次进去处理哪几组、哪些入库可以和哪些出库合并”。这会让修1-修4库内外与其他线路交互次数偏多。

### 7.5 PhysicalValidator 只能判合法，不能判结构收益

当前 `PhysicalValidator` 做得比较实用：

```text
路径是否存在
路径是否被占用阻断
走行线禁止停
牵引折算
倒车长度
目标线长度
台位规则
称重规则
关门车规则
```

这解释了为什么 accepted hard violation 为 0。

但它不能替代 `ResourceRequest / StationResourceGraph / ResourceDelta`。它不能回答：

```text
这个动作释放了哪个关键口？
是否占用了 H2/H3 需要保护的存4北？
是否造成后续必须多一次远端跨区？
是否让修库 slot swap 更难？
是否让当前机车位置远离下一个主合同？
```

因此当前会出现“每一步都物理合法，但全案结构很差”的情况。

### 7.6 P7/P8 缺失导致“合法但劣质”的动作无法被拒绝

方案文档中 P7/P8 的关键标准是：

```text
ContractDelta / ResourceDelta 必须解释动作收益和破坏。
LocalTieBreakSearch 只能在同 contract、同 intent、同结构等级里少钩比较。
标准链钩数 <= manual_hook_count * 1.05。
短链/直接入库 <= manual_hook_count + 1。
```

当前没有这些对象和指标。被接受的候选只说明：

```text
物理 validator 通过
没有 state signature loop
排序靠前
```

它不说明：

```text
这个动作是否是同阶段最优
是否被另一个合法候选支配
是否破坏 H1/H2/H3/H4 的人工结构价值
是否会增加后续远端跨区
```

这正是当前勾数系统性高于人工的直接原因。

### 7.7 blocked 案例说明可解性也没有闭环

24 个 blocked 案例中：

| blocked reason | 数量 | 诊断 |
|---|---:|---|
| `all_runtime_candidates_rejected` | 19 | 候选生成和资源预判不足 |
| `target_final_capacity_infeasible:*` | 9 | 目标输出或线路容量数据与物理长度不一致，需要输入/目标一致性审计 |

物理拒绝原因中：

| 拒绝原因 | 次数 | 诊断 |
|---|---:|---|
| `get_route_blocked_by_occupied_line` | 1239 | 候选生成前没有足够的路线占用资源计划 |
| `put_route_blocked_by_occupied_line` | 186 | 目标送入路径缺少提前清路/让步动作 |

这说明可解性也不是“已经完整证明”。当前标准口径完成 85/113，距离接口输入后稳定可用还有差距。

## 8. 结构方案本身的问题和当前实现的问题要分开

### 8.1 方案结构方向仍然成立

从业务和人工案例看，方案文档中的主结构是合理的：

```text
FlowClassify
FlowGraph / FlowEdge / EdgeContract
TargetContractSelector / StructuralIntent
StationResourceGraph / DepotSlotGraph / GlobalGate / LocoCarryState
ContractDelta / ResourceDelta
LocalTieBreakSearch
HumanPhaseContract / PhaseGate
Trace
```

如果这些结构真的逐项达到文档标准，尤其是：

```text
H1-H5 阶段门可审计
每个动作都有 ContractDelta + ResourceDelta
DepotSlotGraph 能处理动态 swap
ordered carry/drop 可验证
P8 同结构少钩通过人工计划基准
P10 全量完成
```

那么整体达到人工是有结构充分性的。

### 8.2 当前实现没有达到这些结构标准

当前实现更像：

```text
目标线分配
source-target prefix 候选
物理合法性过滤
启发式排序
状态更新
```

它不是完整 P0-P10。尤其缺：

| 缺口 | 直接后果 |
|---|---|
| 没有 `HumanPhaseContract` | 标准链远端动作过早 |
| 没有 `TargetContractSelector` | 目标选择不能解释合同收益 |
| 没有 `ContractDelta` | 不能拒绝结构上返工的动作 |
| 没有 `ResourceDelta` | 不能度量释放/占用关键资源 |
| 没有真实 `DepotSlotGraph` | 修库出入和 swap 无法压缩 |
| 没有 `ordered carry segments` | 人工式连续挂摘不能表达 |
| 没有 `LocalTieBreakSearch` | 同结构少钩无法保证 |
| 没有 phase/contract/resource trace | 失败无法定位到方案结构 |

因此，不能把“P10 skeleton 跑出 response JSON”理解为“P0-P10 全部完成”。

## 10. 10.1-10.6 实施结果与下一步边界

### 10.1 先把 `early_depot` 降级为诊断策略

当前 `early_depot_base/cluster` 会把许多标准链案例当成 `SHORT_DIRECT_DEPOT` 排序，导致远端动作过早。它可以保留用于对照，但不能作为“人工计划阶段达标”的候选策略。

生产口径应区分：

```text
STANDARD_FULL_CHAIN
DIRECT_REPAIR_ENTRY
DEPOT_DIGEST_ONLY
MIXED_SIGNAL_REPAIR
```

只有后 3 类有证据时，才允许早期触碰修库。

已实施：

| 项 | 结果 |
|---|---|
| 默认 `--strategy-mode standard` | 不再把 `early_depot_base/cluster` 放入可选最优策略，只跑 `phase_gate_base/cluster`；短链有证据时仍允许 `short_direct_cluster` |
| `--strategy-mode diagnostic` | 保留 `early_depot_base/cluster` 作为对照策略，用来判断早开远端对完成数和勾数的影响 |
| `run_case(...)` 兼容性 | 保持原四元组返回，供 `app.py` 等可视化调用；新增 `run_case_with_diagnostics(...)` 给 CLI 输出 trace |
| 输出 artifact | 标准口径：`artifacts/p10_10_1_10_6_full_final`；诊断口径：`artifacts/p10_10_1_10_6_diagnostic_full_final` |

实施后结果：

| 口径 | completed | blocked | 业务挂摘勾 | 匹配完成平均差值 | 诊断意义 |
|---|---:|---:|---:|---:|---|
| standard | 85/113 | 28 | 8654 | +46.23 | 可审计性更强，但勾数和可解性更差 |
| diagnostic | 88/113 | 25 | 6470 | +22.67 | `early_depot` 确实帮助少勾/完成，但不是标准链验收证据 |

当前判断：10.1 已完成。它没有优化求解器，反而故意暴露出标准 phase discipline 下的真实缺口。这个结果说明以前 `early_depot` 把一部分结构问题遮住了。

不要先调一堆排序权重。应该先让运行时每个 accepted candidate 输出：

```text
current_phase
phase_entry_reason
phase_exit_condition
phase_permission
phase_forbidden_reason
manual_variant_baseline
front_service_progress
depot_progress
release_accept_gap

```

验收目标：

```text
accepted_without_phase_permission_count = 0
phase_transition_trace_coverage = 100%
标准链首次远端位置接近人工计划分布
短链/库内消化有显式 skip reason
```

已实施输出：`runtime_phase_trace.csv`，每个 accepted candidate 一行。

字段覆盖：

### 10.3 把 P4 从目标线排序改为合同债务选择

`planned_target_for_car` 仍可保留，但不能直接等于当前目标合同。需要显式输出：

```text

```

否则算法会继续选择物理合法但结构价值低的 source-target 动作。

但这里必须明确：当前 10.3 只是“合同审计 trace”，不是完整 `TargetContractSelector`。`selected_contract` 仍然由 `candidate_kind/action_family` 映射出来，不能真正替代文档中的合同债务选择器。

当前判断：10.3 审计层已完成，决策层未完成。下一步不能继续只调 `candidate_sort_key`，而应把 `contract_trace.csv` 里低收益或破坏保护资源的动作，反推成 P4 的显式拒绝/降级规则。

### 10.4 补 P7/P8，先拒绝 dominated 动作，再谈少钩

短期可以先做旁路审计，不一定马上完全替换求解器：

```text
对每个 accepted candidate 计算 ContractDelta / ResourceDelta
标记 dominated_accepted

```

只要 dominated accepted 大量存在，就不要继续调排序，因为排序会在错误候选集合里做局部修补。

已实施输出：

| `resource_delta_trace.csv` | 记录 accepted candidate 的资源申请、释放、联7/远端/机车位置变化 |

标准口径全量结果：

| 指标 | 数量 |
|---|---:|
| accepted candidate 总数 | 4327 |
| dominated accepted | 1156 |
| dominated rate | 26.7% |
| `better_unsatisfied_reduction_same_round` | 993 |
| `better_non_depot_reduction_same_round` | 81 |
| `same_progress_without_remote_cross` | 82 |

当前判断：10.4 审计已完成，拒绝门未启用。因为之前尝试直接把 dominance 变成默认 selector 会让 smoke 勾数变差，所以现在保持 audit-only 是对的。下一步要按 contract/phase/resource 分桶，而不是把所有 dominance 一刀切拒绝。

### 10.5 对远端区域做 session 级求解

修1-修4库内外、卸轮线与其他线路距离远，不适合频繁跨区。当前已有远端指标，但还没有用于结构决策。

应把远端作业视为 session：

```text
enter_remote
collect outbound
release slots
inbound placement
depot digest
exit_remote
```

验收目标：

```text
remote_interaction_batch_count <= manual_remote_hook_count + tolerance
remote_interaction_cross_count <= manual_remote_hook_count + tolerance
standard chain 不得在 H1 初期触碰远端
短链案例必须在远端 session 内压缩完成
```

已实施输出：

| 文件 | 作用 |
|---|---|
| `manual_vs_solver_case_compare.csv` | 每案人工/算法远端首次位置、远端业务勾、远端 session、远端跨区对比 |
| `depot_session_audit.csv` | 每个远端 session 内的批次序号、动作类型、enter/exit/internal、跨区标记 |
| `resource_delta_trace.csv` | 每个动作的 `remote_session_id`、`remote_cross`、机车起止位置 |

标准口径匹配完成 80 案：

| 指标 | 人工 | 当前标准求解 |
|---|---:|---:|
| 首次触碰远端绝对位置 P50 | 第 20 钩 | 第 29 个业务动作 |
| 首次触碰远端相对位置 P50 | 62.5% | 37.8% |
| 远端相关挂摘动作 P50 | 7 | 17 |
| 远端内部批次 P50 | 不适用 | 15.5 |
| 远端 session P50 | 2 | 2 |
| 远端跨入/跨出批次 P50 | 不适用 | 14 |

这说明 10.1 后远端“首次出现位置”明显比旧 `early_depot` 口径更接近人工，但远端批次和跨区次数仍远高于人工。也就是说，后移远端不等于远端作业压缩成功。

当前判断：10.5 session 审计已完成，session 级求解未完成。下一步应围绕 `depot_session_audit.csv` 设计 `enter_remote -> collect outbound -> release slots -> inbound placement -> depot digest -> exit_remote` 的短 planlet，而不是继续让每辆车按 source-target 拆批。

### 10.6 对 `0306W / 0327W` 建短链专项诊断

这两个案例不应靠全局排序解决。

| 案例 | 当前问题 | 必须补的结构 |
|---|---|---|
| `0306W` | `short_direct_cluster` 仍 70 勾，远端 session 6 | `MIXED_SIGNAL_REPAIR` 保守模板、短链 planlet、低置信边界 |
| `0327W` | 远端 session 1 但仍 58 勾 | `DEPOT_DIGEST_ONLY` 库内消化压缩、ordered detach、slot swap |

## 12. 所有诊断完成后的修复结论

### 12.1 总修复结论

当前差距不能靠单一排序权重修复。所有 trace 指向同一个结论：

```text
当前求解器每一步大多物理合法，
但它没有把人工的阶段、合同、资源、远端 session、短链模板和少钩局部搜索变成硬结构。
```

因此后续修复必须按以下顺序推进：

| 顺序 | 必修结构 | 为什么排在这里 | 主要验收指标 |
|---:|---|---|---|
| 1 | HumanPhaseContract | 不先限定 H1-H5，后续合同和资源会在错误阶段被执行 | phase mismatch、forced open、remote too early |
| 2 | TargetContractSelector | 不先知道动作服务哪个合同，就无法判断动作是否值得做 | zero/negative contract delta、suppressed contract |
| 3 | ResourceDelta + RejectGate | 不拒绝结构劣质动作，P8 搜索会在错误候选集合里浪费 | dominated accepted、remote cross with same progress |
| 4 | RemoteSessionPlanlet | 修1-修4/卸轮线距离远，必须一次 session 内集中消化 | remote batch、remote cross、session compression |
| 5 | ShortChainPlanlet | `0306W/0327W` 不是标准链，不能用全局排序慢慢搬 | short-chain hook <= manual + 1 |
| 6 | Blocker/Capacity Feasibility | 解决剩余 28 个 blocked 案例的可解性 | completed 113/113、final unsatisfied = 0 |

### 12.2 修复结论一：PhaseGate 必须升级为 `HumanPhaseContract`

诊断证据：

| 指标 | 当前值 |
|---|---:|
| `deferred_by_runtime_phase_gate` | 582 |
| `phase_action_mismatch` | 465 |
| `remote_too_early_against_manual_phase` | 9 |
| `over_manual_soft_hook_bound` | 1591 |
| forced open 中 `REPAIR_INBOUND` | 272 |
| forced open 中 `DEPOT_OUTBOUND` | 177 |

修复结论：

```text
当前 PhaseGate 只能软延迟联7/远端动作，不能表达 H1-H5 的合同边界。
必须新增 HumanPhaseContract，把 H1/H2/H3/H4/H5 变成每个候选的硬许可上下文。
```

必须实现：

| 项 | 要求 |
|---|---|
| phase entry | 每个阶段必须有进入条件，例如 H2 需要存4释放口债务出现，H4 需要修库 session 可执行 |
| phase exit | 每个阶段必须有退出条件，例如 H1 前段债务完成比例、H2 存4口保护完成 |
| phase skip | 短链必须显式写 `skip_reason`，不能靠 case_id 或 `short_direct_override` 暗开远端 |
| phase hard deny | 标准链 H1/H2 禁止 `REPAIR_INBOUND/DEPOT_OUTBOUND/DEPOT_SLOT`，除非有短链证据 |
| forced open bucket | 如果被迫打开，必须归入 failure bucket，而不是当作正常动作 |

禁止事项：

```text
禁止用提高/降低 action_family 排序权重替代阶段许可。
禁止让 early_depot 重新进入 standard 口径。
禁止在没有 skip reason 的情况下让短链提前触碰修库。
```

验收标准：

| 指标 | 必须达到 |
|---|---:|
| `accepted_without_phase_permission_count` | 0 |
| `remote_too_early_against_manual_phase` | 0 |
| `phase_action_mismatch` | 接近 0，且每个剩余项有 skip/forced reason |
| `deferred_by_runtime_phase_gate` | 不作为 accepted 正常路径出现 |
| 标准链首次远端相对位置 P50 | 接近人工计划 62.5%，不能低于 50% |

### 12.3 修复结论二：P4 必须从目标线排序升级为合同债务选择

诊断证据：

| 指标 | 当前值 |
|---|---:|
| `contract_trace.csv` accepted rows | 4327 |
| `unsatisfied_reduction <= 0` | 1075 |
| `CAPACITY_RELEASE` 零/负收益 | 443 |
| `TEMPORARY_STAGING` 零/负收益 | 255 |
| `BLOCKER_CLEARANCE` 零/负收益 | 228 |
| `REPAIR_INBOUND` 零/负收益 | 80 |

修复结论：

```text
当前 selected_contract 只是从 action_family/candidate_kind 映射出来的标签。
它不能回答“当前最该还哪笔债、这个动作释放什么资源、是否压低后续勾数”。
P4 必须成为真实的 TargetContractSelector，而不是 candidate_sort_key 的别名。
```

必须实现：

| 项 | 要求 |
|---|---|
| contract owner | 每个动作绑定 owner edge / owner vehicles / owner phase |
| debt score | 计算当前合同债务：目标未满足、阻塞关键口、释放修库 slot、释放存4口等 |
| downstream unlock | 计算动作完成后释放的后续候选数量或关键资源 |
| protection cost | 计算动作是否破坏存4北、联7、远端 session、临时 staging 回收 |
| suppression reason | 被压制的合同必须有 reason，便于回放审计 |

禁止事项：

```text
禁止把 planned_target_for_car 直接等同于 selected_contract。
禁止接受 unsatisfied_reduction <= 0 且没有明确 unlock/protection 收益的动作。
禁止只因为物理可达就把车从 source 搬到 target。
```

验收标准：

| 指标 | 必须达到 |
|---|---:|
| accepted `selected_contract` coverage | 100% |
| zero/negative delta accepted | 接近 0，除非有明确 unlock reason |
| 每个高差案例 top 多余动作 | 能归属到具体合同失败桶 |
| P4 输出 | 能解释为什么当前动作优于同轮其他物理合法候选 |

### 12.4 修复结论三：P7/P8 必须先有拒绝门，再做少钩搜索

诊断证据：

| 指标 | 当前值 |
|---|---:|
| accepted candidate 总数 | 4327 |
| dominated accepted | 1156 |
| dominated rate | 26.7% |
| `better_unsatisfied_reduction_same_round` | 993 |
| `better_non_depot_reduction_same_round` | 81 |
| `same_progress_without_remote_cross` | 82 |

修复结论：

```text
当前不是缺少一个更精细的排序权重，而是缺少 P7 的 AcceptRejectGate。
如果一个动作被同轮合法候选支配，它通常不应该进入 P8。
P8 只能在同 phase、同 contract、同 structural intent 的候选集合里做少钩搜索。
```

必须实现：

| 项 | 要求 |
|---|---|
| dominance scope | 只在同 phase、同 contract、同 owner overlap、同 resource class 内比较 |
| hard reject | 对 `same_progress_without_remote_cross` 这类明显劣质动作先硬拒绝 |
| soft reject | 对收益更低但可能服务其他合同的动作降级，不一刀切 |
| planlet search | 在通过 P7 的小候选集内搜索 2-4 步局部动作 |
| rollback trace | 被拒绝动作必须记录被谁支配、哪个指标支配 |

禁止事项：

```text
禁止把所有 dominated 动作一刀切拒绝。
禁止在没有 contract/resource 分桶时直接改全局 candidate_sort_key。
禁止用更大的 max_hooks 掩盖 P8 少钩失败。
```

验收标准：

| 指标 | 必须达到 |
|---|---:|
| dominated accepted rate | 从 26.7% 降到 < 5% |
| `same_progress_without_remote_cross` accepted | 0 |
| smoke 勾数 | 不能因为拒绝门变差 |
| 匹配完成案例平均差值 | 应从 +46.23 明显下降 |

### 12.5 修复结论四：远端区域必须 session 化，不允许反复跨区

诊断证据：

| 指标 | 人工 | 当前标准求解 |
|---|---:|---:|
| 远端相关挂摘动作 P50 | 7 | 17 |
| 远端内部批次 P50 | 不适用 | 15.5 |
| 远端跨入/跨出批次 P50 | 不适用 | 14 |
| 最高远端跨区案例 `0120W` | 不适用 | 24 |
| `0306W` 远端 session | 1 | 6 |
| `0327W` 远端 session | 1 | 1，但内部 15 批 |

修复结论：

```text
修1-修4库内外、卸轮线不能继续按普通 source-target 批次处理。
一旦进入远端，必须把出库、清 slot、入库、库内消化、退出作为一个 session planlet。
```

必须实现：

| 项 | 要求 |
|---|---|
| remote entry decision | 只有当 session 内债务足够、路径可达、slot/swap 已准备时才进入远端 |
| collect outbound | 先一次性识别需要从修库/卸轮带出的车 |
| release slot | 出库动作必须服务后续入库或存4/机库交接 |
| inbound placement | 入库动作必须按 depot slot、尾端可摘顺序、库内/库外衔接位生成 |
| depot digest | session 内消化能在远端内部完成的短动作，减少跨区 |
| remote exit | 退出远端后除非新合同出现，否则不应马上回远端 |

禁止事项：

```text
禁止把修库/卸轮动作和普通 YARD_REBALANCE 混在同一个全局排序池。
禁止上一勾远端、下一勾外场、再下一勾远端的无合同往返。
禁止只用 remote_streak_count 控制 session；它只能保持连续，不能压缩勾数。
```

验收标准：

| 指标 | 必须达到 |
|---|---:|
| 标准链远端相关挂摘 P50 | 接近人工 7，允许小幅高于人工 |
| 标准链远端跨区 P50 | 明显低于当前 14 |
| 远端 session 内平均批次 | 大幅下降，且每批有 slot/swap/contract owner |
| `0120W` 远端跨区 | 从 24 显著下降 |
| `0306W` 远端 session | 从 6 降到 1 |

### 12.6 修复结论五：短链必须有专项模板，不能走标准链排序

诊断证据：

| 案例 | variant | 人工勾 | 当前勾 | 差值 | 远端批次 | 远端跨区 | session |
|---|---|---:|---:|---:|---:|---:|---:|
| `0306W` | `MIXED_SIGNAL_REPAIR` | 10 | 70 | +60 | 15 | 14 | 6 |
| `0327W` | `DEPOT_DIGEST_ONLY` | 11 | 58 | +47 | 15 | 13 | 1 |
| 全量短链 | 11 案 | 不适用 | 不适用 | 只有 1 案勾数在 `manual + 1` 内，但该案 blocked | 不适用 | 不适用 | 不通过 |

修复结论：

```text
短链不是“标准链提早打开远端”。
短链必须有独立 WorkPatternTemplate：低信号保守边界、库内消化、ordered detach、slot swap、少量外场补勾。
```

必须实现：

| variant | 必须补的模板 |
|---|---|
| `MIXED_SIGNAL_REPAIR` | 保守短链模板，先识别少量关键外场/远端互换，不展开全量修库整理 |
| `DEPOT_DIGEST_ONLY` | 库内消化模板，优先在修库 session 内完成出入库/摘解 |
| `DIRECT_REPAIR_ENTRY` | 直接入库模板，只允许证据充分的车辆进入修库，不扩散到普通外场整理 |

禁止事项：

```text
禁止用 case_id 白名单替代短链识别。
禁止把 short_direct_cluster 当成短链模板；它现在只是 depot assignment 变体。
禁止短链案例跑出几十勾后仍标记为正常 completed。
```

验收标准：

| 指标 | 必须达到 |
|---|---:|
| `0306W` | <= 11 勾 |
| `0327W` | <= 12 勾 |
| 短链 completed 案例 | `solver_hook_count <= manual_hook_count + 1` |
| `new_outer_pickup_obligation_count` | 0 |
| `unnecessary_remote_cross_count` | 0 |

### 12.7 修复结论六：可解性缺口要拆成输入一致性和候选生成两类

诊断证据：

| blocked reason | 当前数量 | 修复方向 |
|---|---:|---|
| `all_runtime_candidates_rejected` | 19 | 候选生成、路线占用预判、清路动作 |
| `target_final_capacity_infeasible:*` | 9 | 输入目标/线路长度/业务口径一致性 |
| `get_route_blocked_by_occupied_line` | 1239 | 路径占用资源计划 |
| `put_route_blocked_by_occupied_line` | 186 | 目标送入前清路/让步动作 |
| `source_front_blocked_by_satisfied_or_lower_position_cars` | 28 | 源头 blocker relocation |

修复结论：

```text
blocked 案例不能混在勾数优化里处理。
9 个 final capacity infeasible 先做输入一致性审计；
19 个 all_runtime_candidates_rejected 再进入候选生成和资源让步修复。
```

必须实现：

| 类别 | 要求 |
|---|---|
| capacity consistency | 判断是目标输出错误、线路长度表错误、还是业务允许压线/外溢 |
| route blocker | 候选生成前预判 get/put route 会被哪条占用线阻断 |
| source blocker | 为源头前端已满足车辆生成可恢复 relocation |
| target release | 目标线容量不足时生成 release/split/staging 动作 |
| failure bucket | blocked 案例必须落到唯一主因，不能只报 `all_runtime_candidates_rejected` |

验收标准：

| 指标 | 必须达到 |
|---|---:|
| completed cases | 113/113，或输入一致性问题有明确剔除清单 |
| final unsatisfied | 0 |
| unknown route | 0 |
| hard physical violation accepted | 0 |
| `all_runtime_candidates_rejected` | 0 或有明确不可解证明 |

### 12.8 修复优先级和停止条件

下一轮实现不应一次性重写所有结构。推荐按以下顺序做，每一步都以 artifact 指标为停止条件：

| 轮次 | 修复内容 | 停止条件 |
|---:|---|---|
| R1 | HumanPhaseContract + forced open bucket | phase mismatch/remote too early 明显下降，standard 不退回 early_depot |
| R2 | P4 ContractSelector 审计转决策 | zero/negative delta accepted 大幅下降，top delta 案例能解释合同选择 |
| R3 | P7 RejectGate + 小范围 P8 planlet | dominated accepted < 5%，smoke 勾数不反弹 |
| R4 | RemoteSessionPlanlet | 远端跨区 P50 明显低于 14，`0120W/0104W/0210W` 明显下降 |
| R5 | ShortChainPlanlet | `0306W/0327W` 达到 `manual + 1` |
| R6 | Blocker/Capacity feasibility | completed 接近或达到 113/113 |

只有当这些停止条件逐项达成后，才可以重新讨论“是否达到甚至超越人工计划”。在此之前，正确目标是：

```text
把每个失败 trace 转成对应结构的硬决策；
把每个硬决策转成可量化验收；
让勾数下降来自结构收益，而不是来自不可解释的排序偶然性。
```

## 13. 二次综合诊断：为什么现在差距仍然巨大

本节回答一个更根本的问题：

```text
如果 P0-P10 和 H1-H5 的结构标准理论上足够，
为什么当前代码和人工计划仍差这么多？
```

结论是：当前差距不是证明结构方案不可行，而是证明当前实现还没有进入结构方案定义的求解层。现在的运行时更接近：

```text
逐股道找第一个未满足车辆
  -> 按 planned target 生成 source-target 直达候选
  -> 物理 validator 过滤
  -> 启发式排序选一个候选
  -> 输出 Get + Put
```

人工计划实际更接近：

```text
围绕一个阶段合同组织一列车
  -> 连续挂/摘/代位/顶送/称重/北头靠口
  -> 保持调车机携带状态
  -> 在同一次结构作业中完成多个合同债务
  -> 再进入下一阶段
```

这两者的求解对象不同，所以即使每一步物理合法，勾数也会系统性高于人工。

### 13.1 勾计划层：当前输出不是人工计划式挂摘序列

当前 `operation_rows()` 对每个内部候选固定输出：

```text
Get(source_line)
Put(target_line)
```

也就是说，一个内部 source-target 批次天然变成 2 个业务挂摘动作。人工计划的一钩则是一个挂或摘动作，并且调车机可以在连续多钩中保持携带状态，先挂若干车，再沿途摘、代、顶、称、回。

全量默认审计口径中：

| 指标 | 数值 | 诊断 |
|---|---:|---|
| 内部移动批次 | 4327 | 每批基本都是一次完整 source-target move |
| 业务挂摘动作 | 8654 | 几乎等于内部批次乘 2 |
| 全量批次车辆数 P50 | 1 | 中位数是单车搬运 |
| 全量批次平均车辆数 | 2.07 | 合批能力弱 |
| 单车批次占比 | 54.7% | 大量动作是单车 source-target |
| 不超过 2 车批次占比 | 71.6% | 不能形成人工计划式主列 |
| 匹配完成案例 hook delta 与内部批次数相关系数 | 0.963 | 勾数差距几乎直接由批次数决定 |

所以当前最大问题不是“某个排序优先级差一点”，而是求解原子太小。只要求解原子仍是 source-target 批次，而不是带有 `LocoCarryState / ordered carry segments` 的连续挂摘 planlet，算法就很难接近人工勾数。

代表案例：

| 案例 | 人工 | 当前求解 | 结构诊断 |
|---|---|---|---|
| `0104W` | 41 勾；前 18 勾围绕机库、抛、存5、调、洗、油连续挂摘塑形 | 104 勾；前 22 批多为 `A -> B` 直搬，且穿插 `BLOCKER_CLEARANCE/CAPACITY_RELEASE` 负收益动作 | 缺少持续携带和 H1 co-carry batching |
| `0210W` | 39 勾；前段大量 `存2/洗/抛/油/机/调/存5` 连续组织 | 112 勾；后 10 批仍在 `存5/存4/调梁棚/预修` 间补目标 | H1/H2 没有形成阶段完成边界，尾段还在补前段债 |
| `0306W` | 10 勾；短链低信号，人工直接围绕存4、修库和机库收束 | 70 勾；先远端展开，再回外场补功能线/调棚/预修 | 识别了短链变体，但没有短链模板 |
| `0327W` | 11 勾；库内消化为主 | 58 勾；15 个远端批次后又回外场处理 | `DEPOT_DIGEST_ONLY` 没有转成库内消化 planlet |

### 13.2 人工动作的“中间价值”没有被当前实现表达

人工很多动作不是直接把车辆送到最终目标，而是在服务中间结构：

| 人工动作类型 | 结构价值 | 当前实现的问题 |
|---|---|---|
| `北头` | 靠口、端别、释放条件准备 | 只作为人工审计信号，运行时没有 port state |
| `叉线` | 用缓冲线重排，减少后续返工 | 当前只有反应式 staging，不知道它服务哪个合同 |
| `机 + 接` | 存4释放后的主列承接硬边界 | 没有 `MACHINE_ACCEPT` planlet |
| 修库 `摘/接/库外` | H4 内部消化和库位/库外衔接 | 被拆成多次 `REPAIR_INBOUND/DEPOT_OUTBOUND` 直搬 |
| `顶/代/称/回` | 端别、台位、称重和尾项收束 | 只有少量 validator 规则，没有合同 owner |

默认审计口径中，已完成案例仍有 818 个 accepted 动作满足：

```text
unsatisfied_reduction <= 0
且 non_depot_unsatisfied_reduction <= 0
```

其中主要来自：

| 合同标签 | 零/负收益 accepted 数 |
|---|---:|
| `CAPACITY_RELEASE` | 312 |
| `TEMPORARY_STAGING` | 221 |
| `BLOCKER_CLEARANCE` | 166 |
| `REPAIR_INBOUND` | 68 |
| `DEPOT_OUTBOUND` | 36 |

这些动作未必都错。人工计划也会做临时让步动作。但当前问题是：这些动作没有真实 `owner edge / unlock reason / protection cost`，所以无法区分“必要让步”和“返工噪声”。

### 13.3 代码实现层：P0-P10 大多仍是标签或审计，不是决策闭环

对照 `scripts/generate_physical_runtime_trace.py`，当前实现和方案结构的关系如下：

| 结构 | 方案要求 | 当前代码实际 | 诊断 |
|---|---|---|---|
| P0 `OnlineObservable / FlowGraphBuilder` | 从输入构建在线可用的全站状态和证据边界 | 没有 `FlowGraph`、`OnlineObservable`、`FlowGraphBuilder` | 缺少结构图，后续合同无根 |
| P1 `FlowClassify / StationFlowContract` | 每辆需动车归入合同族，no-move 不污染资源 | 主要依赖 `car_is_satisfied()` 和 per-car planned target | 车辆只有“满足/未满足”，没有业务流身份 |
| P2 `FlowEdge / EdgeKey` | 稳定表达人工结构边 | 无 `FlowEdge/EdgeKey/FlowSubject` | 不能表达一条主流从形成到完成 |
| P3 `EdgeContract / RepairInboundVariant` | 为每条边生成合同条款和变体 | `ManualBaseline` 只用于审计，运行时没有真实 `EdgeContract` | 变体不能约束候选生成 |
| P4 `TargetContractSelector` | 在 active contracts 中选当前合同 | `selected_contract()` 只是 `candidate_kind/action_family` 映射 | 先生成候选再贴标签，不是先选合同 |
| P5 `WorkPatternTemplateSelector` | 按合同生成有限 planlet 候选 | `build_candidates()` 逐股道取第一个未满足前缀 | 没有人工计划式工作模板 |
| P6 `StationResourceGraph` | 资源占用、等待、释放、死锁 break | `PhysicalValidator` 有路径/长度/台位硬校验 | 物理合法性较好，但资源仲裁不完整 |
| P7 `ContractDelta / ResourceDelta / Gate` | 合同和资源 delta 决定准入 | 默认 `audit` 只记录；`aligned` 初版拒绝门会降可解率 | 有 trace，未形成稳定决策闭环 |
| P8 `LocalTieBreakSearch` | 同结构小候选集内做少钩搜索 | `remote_continuity_key/aligned_selection_key` 贪心排序 | 没有 2-4 步 planlet search |
| P9 `Trace / failure_bucket` | 每步 rebuild 和失败主因定位 | CSV trace 已有，但多为 post-hoc | 可诊断性提升，但不是状态图重建 |
| P10 `PhysicalRuntime` | 全量物理运行时压测 | 已实现路径/占用/长度/台位等硬门 | P10 不能替代 P0-P9 |

这解释了为什么“P0-P10 达标则足够”的判断和“当前差距很大”并不矛盾：当前只是 P10 骨架较强，P0-P8 的核心结构还没有真正成为求解决策。

### 13.4 H1-H5 层：阶段不是开关，而是合同完成条件

当前 `phase_state_for_case()` 的核心逻辑是：

```text
non_depot_progress 达到阈值
或 hook_index 超过阈值
或 short_direct
  => link7_open
```

这不是完整 `HumanPhaseContract`。它没有回答：

```text
H1 哪些合同已经完成？
H2 存4释放口是否真的成形？
H3 存4大摘和机接是否连续完成？
H4 修库/卸轮 session 是否消化完？
H5 是否只在主体债务完成后收束？
```

默认审计口径中的阶段失败信号：

| 指标 | 数值 | 含义 |
|---|---:|---|
| `over_manual_soft_hook_bound` | 1591 | 大量动作发生在人工计划软上界之后 |
| `deferred_by_runtime_phase_gate` | 582 | 运行时阶段门只能延迟，无法完成合同 |
| `phase_action_mismatch` | 285 | 当前动作族和人工计划阶段不一致 |
| `remote_too_early_against_manual_phase` | 10 | 早远端仍存在 |
| forced open | 582 | 没有前段候选后被迫打开远端/联7 |

按动作族看，`over_manual_soft_hook_bound` 主要是：

| 动作族 | 数量 |
|---|---:|
| `REPAIR_INBOUND` | 740 |
| `YARD_REBALANCE` | 433 |
| `DISPATCH_SHED_QUEUE` | 337 |
| `DEPOT_OUTBOUND` | 209 |
| `FUNCTION_LINE_SERVICE` | 104 |

这说明阶段失败不是单纯“远端早了”。更严重的是：算法到人工计划应当收束的时候，还在补 H1/H2/H4 债务。

### 13.5 R1-R3 的 `aligned` 结果证明：只加门会牺牲可解性

本轮 `aligned` 做了初版 R1/R2/R3 结构门，结果：

| 指标 | 默认审计 | aligned |
|---|---:|---:|
| completed | 85/113 | 66/113 |
| `all_runtime_candidates_rejected` | 19 | 37 |
| final unsatisfied | 585 | 1179 |
| dominated rate | 26.72% | 4.93% |
| remote cross | 1397 | 1146 |

这说明 R3 的 dominated 动作可以被压住，但如果 P5 候选生成、R4 session planlet、R5 短链模板、R6 feasibility 没补齐，拒绝门会直接把案例推向无解。

因此正确顺序不是：

```text
继续收紧 R1/R2/R3
```

而是：

```text
先补能替代坏动作的结构候选和 planlet，
再逐步收紧 R1/R2/R3。
```

### 13.6 方案逻辑层判断：方案充分，当前实现不足

方案文档中的核心链条仍然成立：

```text
FlowEdge
  -> EdgeContract
  -> TargetContractSelector
  -> WorkPatternTemplate
  -> ResourceRequest
  -> ContractDelta + ResourceDelta
  -> AcceptRejectGate
  -> LocalTieBreakSearch
```

它足以表达人工计划的结构价值。当前差距来自实现没有把这条链落成 runtime 决策，而不是这条链本身无效。

但方案后续实施必须把两点提到最高优先级：

| 必须强化的点 | 原因 |
|---|---|
| `LocoCarryState / ordered carry segments` 不是附属资源，而是少钩的核心状态 | 没有持续携带，就只能 Get+Put 直搬，勾数天然偏高 |
| `WorkPatternTemplate / planlet` 不是优化器细节，而是候选生成的基本单位 | 没有连续挂摘 planlet，P8 无法把批次压到人工计划水平 |

换句话说，当前最关键的缺口不是“如何给候选打更好的分”，而是“候选本身就不是人工计划的动作单位”。

### 13.7 当前问题的优先级排序

后续在优化前，应按以下根因顺序处理：

| 优先级 | 根因 | 证据 | 对应修复 |
|---:|---|---|---|
| 1 | 求解原子错误：source-target 批次代替连续挂摘 | 54.7% 单车批次；hook delta 与内部批次数相关系数 0.963 | `LocoCarryState + ordered carry segments + planlet` |
| 2 | P4 不是真合同选择器 | `selected_contract()` 只是动作族映射；零/负收益 accepted 1075 | `TargetContractSelector + owner edge + unlock/protection cost` |
| 3 | P5 缺少人工工作模板 | `0306W` 70 vs 10，`0327W` 58 vs 11 | `SHORT_CHAIN / DEPOT_DIGEST / H1 co-carry` 模板 |
| 4 | R4 远端没有 session 化 | 远端 cross P50 14，remote hook P50 17 | `RemoteSessionPlanlet` |
| 5 | P6 资源图只有 validator，没有仲裁 | aligned 后 `all_runtime_candidates_rejected` 从 19 到 37 | `StationResourceGraph + feasibility candidates` |
| 6 | H1-H5 阶段门是软进度阈值 | forced open 582，phase mismatch 285 | `HumanPhaseContract` 绑定合同完成条件 |
| 7 | P8 没有同结构少钩搜索 | dominated accepted 26.72%，大量贪心选择 | `LocalTieBreakSearch` 只在同合同/同意图内搜索 |

因此优化前的诊断结论是：

```text
不要先调排序；
不要把 aligned 设为默认；
不要把 P10 物理合法当成 P0-P10 验收；
不要用人工计划阶段标签直接驱动线上求解。

先把当前 source-target move generator 升级为合同约束下的连续挂摘 planlet generator，
再让 R1/R2/R3/R4/R5/R6 的门控逐步变硬。
```

## 14. 最终判断

当前版本距离“接口文档输入后直接求解、物理条件符合、可用并达到甚至超越人工计划”还有明显差距。

更精确地说：

```text
物理硬门：当前全量 accepted hard violation = 0，unknown route = 0，已有较好基础。
可解性：本轮提升到 95/113，但仍未达到 113/113 或明确不可解剔除清单。
勾数：completed hook P50 从 82 降到 68，但整体仍显著高于人工，P8 未通过。
H1-H5：当前不是完整 H1-H5，只是联7/远端软门控。
P0-P9：已有 phase/contract/resource/session/dominance trace，并有 opt-in aligned 结构门；但多数仍不是可用级决策闭环。
P10：能生成 response，但不能替代 P0-P9，也不能证明达到人工。
```

所以当前不应该声称“P0-P10 已经足够达到甚至超越人工”。正确表述是：

```text
方案文档中的结构充分性仍然成立；
当前代码实现尚未达到该结构充分性；
当前差距主要来自阶段门、合同债务、资源 delta、远端 session、短链模板和同结构少钩搜索没有真实闭环。
```

下一步应基于已有诊断 trace 逐项做结构优化，并且每一项优化都必须回写到对应的 trace 指标上。

当前最合理的下一步不是继续补清路规则，也不是继续调排序权重，而是：

```text
先做一个小而清晰的连续挂摘 planlet 内核：
  LocoCarryState + ordered carry segments + contract-owned WorkPatternTemplate。

再用这个内核优先落 R4/R5：
  远端 session planlet + 短链模板 + H1 前段 co-carry 模板。

最后再做 R7/P8：
  同合同、同阶段、同资源类内的 2-4 步 LocalTieBreakSearch。

等替代候选足够强以后，再回头收紧 R1/R2/R3，
让阶段门、合同门和拒绝门在不降可解率的前提下变硬。
```

## 15. 优化前锁定诊断：当前到底哪里没起效

本节用于回答优化前的最终问题：

```text
为什么方案结构理论上足够，但当前实现和人工计划差距仍很大？
接下来到底应该补哪些结构，而不是继续打补丁？
```

结论很明确：

```text
当前差距不是 P10 物理路径校验的问题，
也不是某个候选权重的问题。

当前差距来自求解对象错误：
算法仍主要在解 source-target 批次搬运；
人工计划是在解持续携带状态下的阶段合同 planlet。
```

阶段门现在只是“什么时候允许远端”的开关，还不是“每个阶段合同是否完成”的证明。

这些动作不一定都错。人工也会做让步、代位和临时整理。

问题是当前 trace 不能稳定证明：

因此 P4 仍不是 `TargetContractSelector`。它只是候选生成后贴上的 `selected_contract` 标签。

这说明现在的问题已经不是简单“远端太早”。更深的问题是：


当前的 `remote_session_planlet` 数量太少，且多数远端动作仍是普通 `target_move` 或 `multi_drop/multi_pick`。

### 15.6 关键证据五：短链识别有了，短链模板没有

最新短链结果：

全量 11 个短链中，仍有 7 个完成案超过 `manual + 1`，另有 `0103W` blocked。

```text
短链作业模板不存在。
```

`short_direct_cluster` 不能再被当成短链完成。它最多说明“允许早触碰远端”，不说明“按人工计划短链方式完成”。

### 15.7 为什么不能继续打补丁

现在最危险的做法是继续加：

```text
某线路优先级
某动作降权
某 case 特例
某条清路规则
```

原因有三点。

第一，`aligned` 已经证明只加硬门会牺牲可解性：

| 指标 | audit | aligned |
|---|---:|---:|
| completed | 85/113 | 66/113 |
| final unsatisfied | 585 | 1179 |
| dominated rate | 26.72% | 4.93% |

拒绝门压住了劣质动作，但没有替代候选时，案例直接无解。

第二，清路/恢复 planlet 已经证明可以提高可解性，但会推高部分勾数：

| 案例 | 人工 | 最新求解 | 诊断 |
|---|---:|---:|---|
| `0109Z` | 34 | 139 | 清路/恢复救活案例，但不是人工计划结构 |
| `0212W` | 38 | 133 | 可解但勾数失控 |
| `0210W` | 39 | 128 | 前段反复补债 |

第三，继续调排序会遮住结构问题：

```text
排序只能在已有候选里选。
如果候选不是人工计划式连续挂摘 planlet，
排序无法凭空产生少钩结构。
```

### 15.8 下一步应该做什么

下一步不是继续“优化排序”，而是把当前实现从 source-target move generator 升级为 contract-owned planlet solver。

推荐顺序：

| 顺序 | 要做的结构 | 目标 | 停止条件 |
|---:|---|---|---|
| T0 | 冻结当前诊断基线 | 防止继续用单案补丁漂移 | `root_cause_fix_full` 指标作为回归基线 |
| T1 | `LocoCarryState + ordered carry segments` | 让调车机能持续携带、沿途挂摘 | operation 不再以 2 步 Get/Put 为主体 |
| T2 | `WorkPatternTemplate` 作为 P5 基本候选 | 候选从“车到目标线”变成“人工结构模板” | H1 co-carry、H2 存4口、H4 远端、短链都有模板 |
| T3 | `TargetContractSelector` 前置到候选生成前 | 先选合同，再生成模板候选 | 零/负收益动作必须有 owner/unlock/recovery |
| T4 | `RemoteSessionPlanlet` | 远端一次 session 内处理出库、slot、入库、消化、退出 | 远端挂摘 P50 接近 7，远端跨区 P50 明显低于 9 |
| T5 | `ShortChainPlanlet` | `MIXED_SIGNAL_REPAIR/DEPOT_DIGEST_ONLY` 不走标准链 | `0306W <= 11`，`0327W <= 12` |
| T6 | `StationResourceGraph + DepotSwapDelta` | 让资源仲裁产生替代候选，而不是只靠 validator 拦截 | `all_runtime_candidates_rejected` 接近 0 |
| T7 | `LocalTieBreakSearch` | 在同合同、同阶段、同资源类候选里做 2-4 步少钩搜索 | dominated accepted < 5%，且勾数不反弹 |
| T8 | 收紧 R1/R2/R3 | 替代候选足够强后再硬化门控 | 可解率不下降，phase mismatch/forced open 下降 |

这套顺序的核心是：

```text
先补“好候选”，再加“硬拒绝”。
先让算法具备人工计划的动作单位，再讨论是否超越人工计划。
```

### 15.9 重新定义下一轮验收

下一轮不能只看 completed 数，也不能只看总勾数。必须同时看四组指标：

| 组 | 指标 | 目标 |
|---|---|---|
| 可解性 | completed / final unsatisfied / hard violation | 不低于当前 95/113，物理违规继续 0 |
| 勾数 | 匹配完成 P50、mean delta、不劣于人工数 | P50 从 66 向 35 收敛，mean delta 明显低于 +34 |
| 结构 | 单车/双车批次占比、2 操作候选占比、planlet 占比 | `target_move` 不再是主体，连续挂摘 planlet 成为主体 |
| 人工阶段 | 首次远端位置、远端挂摘、短链上界、phase mismatch | 远端挂摘 P50 接近 7，短链进入 `manual + 1` |

只要这四组指标没有同时改善，就不能说优化有效。

### 15.10 最终诊断结论

当前应该先诊断再优化，这个判断是正确的。

诊断已经把问题收敛到一条主线：

```text
当前 P10 物理运行时能保证很多动作合法，
但 P0-P8 的结构求解还没有真正落地。

当前实现的主控仍是：
  目标线 + 小批次候选 + validator + 排序。

人工计划的主控是：
  阶段合同 + 持续携带 + 资源口保护 + 远端/短链 planlet + 少钩搜索。
```

因此接下来最重要的不是再问“某个案例怎么少几勾”，而是：

```text
把求解器的基本动作单位从 source-target 批次，
改成有 LocoCarryState 的合同 planlet。
```

只有这个变化完成后，P4/P5/P7/P8/H1-H5 才会真正起效。否则继续加规则会让代码越来越复杂，但勾数仍然靠偶然排序改善，无法稳定达到甚至超越人工计划。
