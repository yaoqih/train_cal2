# 第二阶段：卸轮翻库与大库出库编组设计实施方案

## 0. 本轮评审继承结论

本轮评审里有几条意见必须吸收，但也有几条与现场口径或已统计数据不一致，不能原样接受。

### 0.1 明确拒绝或改写的点

| 评审点 | 处理 |
|---|---|
| “必须建模完整留库车挖出再放回” | **不作为主模型接受**。当前统计显示出库车基本位于留库车北侧，第二阶段主模型仍按“出库前缀”处理。实现中增加前缀不变量审计；若发现留库车压住出库车，输出结构化 partial 或进入专项分支，不把全量模型复杂化。 |
| “用存4南中转 OFF” | **拒绝**。本阶段不能把存4南作为通用中转线。遇到 `C4...OFF` / `OFF...C4` 栈型时，只能用机后携带、修库内/库外缓存重排、尾甩解决。 |
| “脱轨器时序必须进入模型” | **本阶段不考虑**。当前第二阶段求解只处理卸轮翻库和大库出库编组，脱轨器约束不进模型。 |
| “勾数按 move batch 计” | **拒绝**。第二阶段验收勾数按 API 的 `Get/Put/Weigh` 行数计，和第一阶段 `business_hooks` 口径统一。 |
| “台位方向待澄清” | 已澄清：**1 号位靠北**，栈顺序按位置 1 -> 5。 |

### 0.2 接受并修正方案的点

| 评审点 | 继承方式 |
|---|---|
| 修库外既有占用会影响出库 | 接受。修库外作为入口/出口阻塞线，必须纳入源栈、容量和路径校验。 |
| 临挂到修库外不是零成本 | 接受。临挂车后续被取走时也要计入 Get 行数、当量、长度、关门车过程约束。 |
| A* 启发函数可能不可采纳 | 接受。MVP 默认用 Dijkstra 或字典序代价，不依赖有风险的断点启发。 |
| `OFF*C4*` 的钩序语义需写全 | 接受。需要显式建模 C4 段、边界钩、OFF 段。 |
| Phase A 不能硬贪心 | 接受。Phase A 产生候选初始状态，交给 Phase B 规划比较；当前数据无触发时可直接跳过。 |
| 关门车是过程约束，不只是终态约束 | 接受。每条 Put/相关操作行都要过过程约束，终态再检查 C4 前 3 位。 |
| 第 0 步应升级为接口 spike | 接受。先手写多 Get、缓存 Put、尾甩、最终单次 Put 存4线等代表性 planlet 跑通 validator。 |
| `OFF...C4` 需要机后携带 | 接受。禁用存4南后，机后车列是唯一不增加临停线的 OFF 缓冲。核心状态加入 `held_consist`。 |
| 存4北只能放一次 | 接受。`Put(存4线)` 只能作为最后一条操作行提交，且只能出现一次；搜索期间不能用存4线做中间栈。 |
| 进路长度上限会改变下界 | 接受。第 0 步必须按真实口径实测最终 `存4南 -> 存4线` 进路及大库侧相关进路的可行长度，并纳入枚举约束和下界。 |
| 存4北初始占用边界未定义 | 接受。`OFF*C4*` 只作用于第二阶段新增到存4北北端的段，trace 记录新旧分界。 |
| OFF 段内部顺序是第三阶段接口 | 接受。Stage2 summary 必须输出 OFF 段序列；同目的地相邻作为字典序次级目标。 |
| 称重断言缺失 | 接受。第 1 步审计 `IsWeigh => _Weighed`，否则归为阶段交接债。 |

### 0.3 数学核心的修正

第二阶段不应退化成规则流水线。修正后的核心是一个**操作行级最短路问题**：

```text
节点 = 真实可区分状态
边   = 一条合法 API 操作行（Get / Put / Weigh）
代价 = (操作行数, 联7穿越, 路径代价, 第三阶段友好度)
```

其中状态必须包含：

```python
State = (
    source_stacks,          # 修库内/卸轮线/库外等仍在原线上的车序
    buffer_line_state,      # 修库内/库外缓存线实时占用
    held_consist,           # 当前机后车列，允许跨操作携带
    store4_committed,       # 最终 Put(存4线) 是否已经提交
    store4_base_boundary,   # 第二阶段开始时存4既有段的分界
    loco_line,              # 精确机车位置
)
```

这样 `C4...OFF`、`OFF...C4`、合并翻库、尾甩、库外/库内缓存重排、库外占用，都不是额外规则，而是同一个状态图上的不同路径。Dijkstra 在这个状态图上做字典序最短路，才能保留数学上的最优性。唯一例外是 `Put(存4线)`：它不是普通中间转移，只能作为终态提交边。

## 1. 阶段边界

第二阶段只处理两件事：

1. **卸轮翻库**：如果存4北存在目标为卸轮线的车，则先保证卸轮线可接收，再把这些车翻到卸轮线。
2. **大库出库编组**：把起始在大库区域、目标要出库的车辆从大库拉出，并按第三阶段需要编到存4北。

这里的大库区域按当前数据口径定义为：

```text
修1库内 / 修2库内 / 修3库内 / 修4库内 / 卸轮线
```

出库目标分两类：

```text
C4  = 目标为存4线
OFF = 目标为油漆线 / 修X库外 / 机走棚等非存4线目标
```

当前 113 个 case 的数据规模：

| 项 | 数量 |
|---|---:|
| 出库车总量 | 1527 |
| C4：目标存4线 | 1395 |
| OFF：非存4目标 | 132 |
| 单 case OFF 最大数 | 5 |
| 单 case 出库车最大数 | 18 |
| 当量超过 20 的 case | 2 |
| 卸轮线已有出库车最大数 | 2 |

结论：第二阶段状态空间很小，适合做精确搜索，不需要 MILP 或大型优化器。

补充口径：

- 勾数按 `Get/Put/Weigh` 操作行数计。
- 位置 `1` 靠北，库内栈顺序为 `1 -> 5`。
- 存4北在代码中对应 `存4线`。
- 存4南不作为本阶段通用中转线。

## 2. 设计原则

### 2.1 线路是栈

修1-4库内、卸轮线、临挂线都只能从北端摘挂，因此每条线都按北端可取顺序建模为一个栈：

```text
Get(line, k) = 取该线北端连续 k 辆
```

### 2.2 存4北只能终态一次提交

存4北仍然是反序堆叠线，但第二阶段口径要求 `Put(存4线)` 只能出现一次，且必须是最后一条操作行。  
因此搜索过程中不能把存4线当作中间缓存，也不能用多次压入来修正顺序。最终存4北北到南顺序就是最后一次 `Put(存4线)` 的车列顺序：

```text
final_held_consist
```

第二阶段要求“非存4目标车在北边、存4目标车在南边”，所以最终存4北序列必须满足：

```text
OFF* C4*
```

注意：这个正则只约束**最终一次进入存4北的车**。如果 OFF 车在本阶段顺路尾甩到修X库外，它不进入这个序列。

### 2.3 搜索粒度是操作行，验证可按 planlet 批量

第二阶段的最优目标按 `Get/Put/Weigh` 行数计，因此求解图的边应对应一条操作行。这样可以自然表达“机后携带”和“缓存重排”：

```text
Get(修2库内 C4前缀)          # held = C4
Put(安全缓存线, C4前缀)       # 存4线不参与中转
Get(修2库内 OFF后缀)         # held = OFF
Get(安全缓存线 C4前缀)        # held = OFF+C4
Put(存4线, OFF+C4)           # 唯一一次，且最后提交
```

工程上可以把若干操作行批量打包成 planlet 调 `physical.validate_candidate`，但这是校验优化，不是数学模型。第 0 步必须确认 validator 是否支持“非空 held_consist”状态；若不支持，就需要补一个 stage2 专用 operation simulator，或者只在闭合片段上批量校验、逐行用 replay 语义维护状态。

## 3. 状态图模型

### 3.1 状态定义

第二阶段求解状态不再以“任务片段”作为本体，而是以真实可区分的调车状态作为节点：

```python
State = (
    source_stacks,          # 修库内、卸轮线、库外等线上的剩余车序，北端在前
    buffer_line_state,      # 修库内/库外可用缓存线实时占用，含临挂车、尾甩车
    held_consist,           # 当前机后车列，北端/贴机车在前，尾部在后
    store4_committed,       # 是否已经执行最终 Put(存4线)，一旦为 true 搜索只能终止
    store4_base_boundary,   # 第二阶段开始时存4既有段的分界，只作容量和 trace
    loco_line,              # 精确机车位置
)
```

`held_consist` 和 `buffer_line_state` 是本版方案的关键。禁用存4南中转、且存4北只能最终提交后：

- `OFF...C4` 可以直接由机后携带形成 `OFF+C4`，等待最终一次 `Put(存4线)`；
- `C4...OFF` 必须先把北端 C4 临挂到安全缓存线，再取出 OFF，再把 C4 挂回机后，最后一次提交存4线；
- `store4_committed=true` 的状态必须是终态，不能继续 Get/Put。

### 3.2 操作行转移

状态图的边是一条 API 操作行，代价主项为 1：

| 转移 | 状态变化 | 核心约束 |
|---|---|---|
| `Get(line, prefix)` | 从线北端取前缀，追加到 `held_consist` 尾部 | 前缀可取、当量不超限、路径可达、不会非法穿越占用线 |
| `Put(缓存线, suffix)` | 从 `held_consist` 尾部摘连续后缀，压入修库内/库外缓存线北端 | suffix 必须为同质段；缓存线安全、容量足、不会阻断后续源栈 |
| `Put(存4线, all)` | 把完整 `held_consist` 一次性提交到存4北，并置 `store4_committed=true` | 必须是最后一行；所有 C4/OFF 已在机后或已尾甩；`held_consist ∈ OFF*C4*`；容量/关门车合规 |
| `Put(卸轮线, suffix)` | 从尾部摘目标卸轮线车辆到卸轮线 | 卸轮线容量足、目标匹配、路径可达 |

车列顺序约定：多次 Get 时，先 Get 的车贴机车，后 Get 的车在尾部；Put 只能摘尾部连续后缀。这与现有 replay/validator 的 `TrainCars` 语义一致。

缓存线包括已经可安全使用的修库外线，以及已清空且不会再作为源栈的修库内线。是否安全不靠人工白名单硬猜，而由两层条件共同决定：

1. 抽象条件：该线无 STAY 车，不是仍有未处理出库车的必经出口，放入后不会把剩余源栈锁死；
2. 物理条件：从当前机车位置到该线的 Put 路径、后续从该线 Get 回来的路径都能通过 validator/simulator。

### 3.3 合法性 Oracle

每条转移必须通过两层校验：

1. 抽象层快速校验：栈前缀、尾部后缀、当量、长度、`OFF*C4*`、库外容量、关门车过程约束；
2. 物理层校验：用 `physical.validate_candidate` 或 stage2 专用 operation simulator 校验路径、折返、联6/联7长度、北端进路、占用绕行。

第 0 步 spike 决定具体调用方式：如果现有 validator 不能从非空 `held_consist` 状态校验单行操作，就在 Stage2Solver 内实现一个轻量 simulator，并用 replay validator 做端到端兜底。

### 3.4 缓存重排的最优性与复杂度控制

重排不能退化成“看到 `C4...OFF` 就写一个模板”。正确做法是把缓存动作放进同一个状态图，并用 Dijkstra 统一比较。但为了避免任意甩车导致状态爆炸，枚举必须采用**可证明不丢最优解的规范动作集**。

#### 3.4.1 规范缓存动作

只枚举以下缓存 Put：

```text
Put(buffer, suffix)
```

其中 `suffix` 必须满足：

1. 是 `held_consist` 的尾部连续后缀；
2. 是同质段：全 C4、全 OFF、全 U，或同一目标线的 OFF 段；
3. 该动作至少满足一个目的：
   - 暴露某条源栈中被 C4 压住的 OFF；
   - 释放卸轮翻库的目标车；
   - 将 OFF 尾甩到其真实目标修库外；
   - 降低最终 `held_consist` 对 `OFF*C4*` 的违反。

不枚举混合 `C4+OFF` 后缀缓存。理由是混合缓存不会比先分段缓存更少操作行，且会增加未来取回时的顺序约束；在操作行数主目标下被分段方案支配。

#### 3.4.2 缓存线选择

候选缓存线来自：

```text
修1-4库外
已清空且不再作为源栈的修1-4库内
```

每个候选缓存线都必须通过“安全缓存”判断：

```text
safe_buffer(line):
    线内无 STAY 车；
    容量足够；
    若是修i库外，则修i库内无未处理出库车，或本次缓存不会挡住其后续出库；
    从当前状态 Put 进去可达；
    后续 Get 回来可达。
```

如果没有安全缓存线，求解器不能伪造可行性；应继续尝试机后携带、尾甩，最终仍无解则输出结构化 partial。

#### 3.4.3 支配剪枝

Dijkstra 的最优性来自“完整状态 + 非负字典序代价”。以下剪枝只删除被支配状态，不改变最优解：

1. 同一状态签名只保留字典序代价最小者；
2. `Put(buffer)` 后若没有任何中间动作改变源栈/卸轮/终态可行性，立刻 `Get(buffer)` 回来，该往返被删除；
3. 同一同质段可放入多个等价空缓存线时，只保留路径代价最小、且不阻断后续源栈的缓存线；
4. 缓存中的同质 C4 段只保留车号顺序和关门车/重车属性，不为等价 C4 做全排列；
5. 已经执行最终 `Put(存4线)` 的状态只允许判终态，不再扩展任何后继。

#### 3.4.4 状态规模上界

当前数据的关键规模是：

```text
单 case 出库车 <= 18
OFF <= 5
源栈 <= 9
可用缓存线 <= 8
```

需要缓存的对象不是所有车，而是“打破 `OFF*C4*` 的边界块”。一个 OFF 块最多需要一个北侧 C4 块临挂，因此缓存块数上界近似为：

```text
buffer_blocks <= OFF_blocks + 卸轮临挂块 <= 6
```

邻接生成复杂度控制在：

```text
Get 候选: O(源栈数 * 每栈可切点)
Put 缓存候选: O(安全缓存线数 * held尾部同质段数)
Put 存4候选: O(1)  # 只在终态候选时生成
```

因此 MVP 可设置硬预算：

```text
max_states_per_case = 100000
max_buffer_blocks = 6
```

这不是启发式替代搜索，而是基于当前数据规模和块结构的实现预算。若触发预算，应输出 `stage2_reorder_complexity_budget_exhausted`，并把 case 加入反例集，而不是静默降级成贪心。

## 4. Phase A：卸轮翻库作为状态图的一部分

Phase A 不再作为前置贪心流程。它只是在初始状态附近开放的一组转移，由 Dijkstra 与后续出库转移一起比较。

### 4.1 触发条件

```text
if 存4北新增/既有北端段中存在目标=卸轮线的车:
    开放卸轮翻库转移
else:
    跳过这些转移
```

当前 113 个 case 中没有真实触发，但必须用人工小 case 固化回归。

### 4.2 卸轮线被 C4 占用时的两类候选

若卸轮线上已有目标存4线的出库车，不能只写死“先挂到修库外”。候选至少有两类：

**候选 A：临挂库外**

```text
Get(卸轮线 C4前缀)
Put(某修库外)
后续从该库或库外取走，进入 C4 出库编组
```

这不是零成本。后续取走临挂车仍要计入 Get 行数、当量、长度和关门车过程约束。

**候选 B：合并翻库，利用 held_consist**

```text
Get(卸轮线 C4车)              # held = C4
Get(存4北 卸轮目标前缀)        # held = C4 + 卸轮目标
Put(卸轮线, 尾部=卸轮目标)     # held = C4
继续携带 C4，或 Put(安全缓存线, C4)，等待最终一次 Put(存4线)
```

这个候选不占修库外容量，通常比临挂少一条回收 Get，必须进入统一搜索。它依赖 `held_consist`，也是本版引入机后携带状态的主要收益之一。

### 4.3 存4北卸轮目标被压住

若目标卸轮线的车不是存4北北端可取前缀：

```text
若压车本身属于第二阶段 C4/OFF 出库车:
    先按普通出库转移取走压车，再翻卸轮目标车
否则:
    输出 stage2_unwheel_blocked_by_store4_prefix
```

不使用存4南中转，不挖非出库车再回放。

## 5. Phase B：大库出库编组

### 5.1 输入线与前缀审计

基础输入线：

```text
修4库内 / 修3库内 / 修2库内 / 修1库内 / 卸轮线 / 修1-4库外
```

修库内先审计出库前缀：

```text
位置 1 靠北。
若出库车不是北端连续前缀，则普通模型不强行挖留库车，输出 stay_car_blocks_outbound。
```

当前数据中此前缀不变量几乎成立，但存在极少例外，因此必须审计并落 trace。

修库外先审计占用类型：

```text
outside_occupied_by_outbound      # 可作为源栈或随库内出库带走
outside_occupied_by_non_outbound  # 普通模型不处理，输出结构化 partial 或后续专项分支
```

### 5.2 `C4...OFF` 与 `OFF...C4` 都由同一状态机制处理

不再分别写一堆特例规则。统一原则如下：

- 搜索期间不允许 `Put(存4线)`；
- 最终 `held_consist` 必须满足 `OFF*C4*`，才能执行唯一一次 `Put(存4线)`；
- 若源栈是 `OFF...C4`，通常直接 Get 全段并机后携带即可；
- 若源栈是 `C4...OFF`，必须把北侧 C4 临挂到安全缓存线，再取 OFF，再挂回 C4；
- 若 OFF 可尾甩到修库外，则 Dijkstra 会在“尾甩”与“带到存4北”之间按字典序代价选择。

例子：`OFF...C4` 的合法路径不需要存4南：

```text
Get(OFF+C4)              # held = OFF+C4
...                      # 可继续 Get 其他 C4/OFF，只要最终仍能形成 OFF*C4*
Put(存4线, OFF+C4)        # 唯一一次，最后提交
```

例子：`C4...OFF` 的合法路径需要缓存重排：

```text
Get(C4前缀)               # held = C4
Put(安全缓存线, C4前缀)    # held = 空，C4 临挂
Get(OFF后缀)              # held = OFF
Get(安全缓存线, C4前缀)    # held = OFF+C4
Put(存4线, OFF+C4)        # 唯一一次，最后提交
```

这样 `OFF*C4*` 是终态提交条件，缓存重排是状态图上的普通路径，不是若干手工模板。

### 5.3 存4北终态边界

第二阶段开始时，存4线可能已有车。终态校验只作用于**第二阶段最后一次压入北端的新增段**：

```text
final_store4_segment ∈ OFF* C4*
```

既有段作为 `store4_base_boundary` 保留在 trace 中，只参与容量、路径和状态一致性校验，不参与本阶段 `OFF*C4*` 判定。

C4 段前 3 位关门车约束是终态约束：

```text
取 final_store4_segment 中 C4 子序列的前 3 辆，不能是关门车。
```

不要在普通候选枚举期提前过滤“当前机后 C4 是否在前 3 位”，因为缓存重排可能改变最终提交段。只在唯一最终 `Put(存4线)` 前检查。

过程约束仍然每条 Put 都要查：

- 机后超过 10 辆时，第一顺位不能是关门车；
- 所挂车辆含重车时，第一辆不能是关门车；
- 存4相关关门车业务规则由 validator 或 stage2 simulator 统一执行。

### 5.4 OFF 段内部顺序

进入存4北的 OFF 段是第三阶段输入接口。第二阶段不做第三阶段联合优化，但必须输出稳定契约：

```text
stage2_summary.off_segment = [车号...]
stage2_summary.off_segment_groups = 按目标线分组统计
```

字典序代价的最后一项加入第三阶段友好度：同目标 OFF 相邻优先，目标顺序建议为：

```text
油漆线 -> 修库外 -> 机走棚
```

这只是 tie-break，不得牺牲操作行数和物理可行性。

## 6. 搜索算法

### 6.1 Dijkstra 字典序最短路

默认使用 Dijkstra，不依赖启发函数。优先队列 key：

```python
cost = (
    operation_rows,       # Get/Put/Weigh 行数，主目标
    lian7_crossings,      # 联7穿越/往返次数
    route_cost,           # 路径长度、折返、道岔等
    off_group_penalty,    # OFF 段同目标不相邻惩罚
    tie_breaker,
)
```

使用字典序比较，不使用加权和。这样能保证：任何路径代价、OFF 分组偏好都不会扰动最小操作行数。

### 6.2 终态

终态条件：

```text
所有第二阶段出库车都不在源栈中；
所有临挂缓存线清偿完毕，或仅保留合法尾甩到目标修库外的 OFF；
held_consist 为空；
Put(存4线) 恰好执行一次，且是最后一条操作行；
final_store4_segment 满足 OFF*C4*；
C4 子序列前 3 位关门车合规；
所有尾甩到修库外的 OFF 位于其目标线；
replay/validator 物理状态一致。
```

### 6.3 下界与剪枝

Dijkstra 不需要启发也能出最优，但仍可做安全剪枝：

```text
remaining_put_lower_bound = max(
    ceil(剩余当量 / 20),
    ceil(剩余总长 / actual_store4_approach_limit),
    ceil(剩余入存4长度 / store4_remaining_length),
)
```

其中 `actual_store4_approach_limit` 必须按第二阶段真实口径实测：最终 `Put(存4线)` 走 `存4南 -> 存4线`，不能误用联6上限。若不确定，宁可不剪枝，交给 validator/simulator 拒绝。

禁止使用会高估的启发项，例如“所有强制断点数”，因为尾甩或 held_consist 可能消除断点成本。

## 7. 工程实施路径

### 第 0 步：接口 spike 与口径确认（1-1.5 天，最高优先）

产出：

- 明确 `存4北` 在代码里对应 `存4线`；
- 明确 `卸轮线` 作为第二阶段源栈；
- 明确位置 `1` 靠北；
- 明确勾数按 `Get/Put/Weigh` 行数计；
- 明确 `OFF*C4*` 只作用于第二阶段新增存4北段；
- 明确 `Put(存4线)` 只能出现一次，且必须是最后一条操作行；
- 明确最终 `Put(存4线)` 的进路口径为 `存4南 -> 存4线`；
- 明确哪些修库内/库外线可作为安全缓存线；
- 确认 validator 是否支持非空 `held_consist` 的单行校验；
- 实测大库到存4北真实进路在典型第一阶段占用下的最大可行车列长度；
- 确认卸轮线北侧操作是否有额外强制对位要求。

验收：

- 手写并校验多 Get + 唯一最终 Put 存4线；
- 手写并校验 `OFF...C4`：Get 全段、最终一次 Put 存4线；
- 手写并校验 `C4...OFF`：Get C4、Put 安全缓存线、Get OFF、Get 缓存 C4、最终一次 Put 存4线；
- 手写并校验合并翻库：Get 卸轮 C4、Get 存4卸轮目标、Put 卸轮，随后继续携带或缓存 C4，最终唯一一次 Put 存4；
- 手写并校验尾甩修库外；
- 输出 `actual_store4_approach_limit` 的实测结果或“不启用该剪枝”的结论。

### 第 1 步：Stage2 状态抽取与审计（1 天）

实现：

- 抽取修库内、卸轮线、修库外源栈；
- 给车打标签：`C4 / OFF / STAY / IGNORE`；
- 审计出库前缀；
- 审计库外占用：`outside_occupied_by_outbound / outside_occupied_by_non_outbound`；
- 审计 `IsWeigh => _Weighed`；
- 记录 `store4_base_boundary`；
- 输出 stage2 初始 trace。

验收：

- C4 总量约 1395；
- OFF 总量约 132；
- 单 case 出库车最大 18；
- `OFF...C4`、`C4...OFF` 模式统计落 trace；
- 安全缓存线清单、每条缓存线阻塞的源栈关系落 trace；
- 非前缀库线、库外非出库占用、未称重遗留都能输出结构化原因。

### 第 2 步：操作行生成器（2 天）

实现：

- `Get`：枚举可取源线北端前缀，追加到 `held_consist` 尾部；
- `Put 安全缓存线`：枚举 `held_consist` 尾部同质后缀，更新 `buffer_line_state`；
- `Get 安全缓存线`：枚举缓存线北端前缀，追加到 `held_consist` 尾部；
- `Put 存4线`：只在终态候选时枚举一次，move 必须等于完整 `held_consist`；
- `Put 修库外尾甩`：枚举目标匹配 OFF 尾部后缀；
- `Put 卸轮线`：枚举目标卸轮线尾部后缀；
- 每条操作行做抽象校验 + 物理校验缓存。

验收：

- 双栈 `OFF...C4` 反例能找到合法路径或给出真实物理阻塞；
- `C4...OFF` 且 OFF 不可尾甩时，不使用存4南、不多次 Put 存4线，能通过安全缓存重排保证终态；
- complete case 的 `Put(存4线)` 次数恒为 1，且 index 等于操作行总数；
- 过程关门车两条规则分别有单元测试；
- 库外占用不会被非法穿越。

### 第 3 步：Dijkstra 求解器（2 天）

实现：

- 状态签名与去重；
- 字典序优先队列；
- 支配剪枝与缓存块预算；
- validator/simulator 缓存；
- `time_budget_seconds`；
- partial 阻塞原因；
- 终态校验。

验收：

- 简单 case 输出搜索空间内最优；
- 当量 >20 自动拆分；
- 真实存4进路超过实测上限时提前拆分或输出 `store4_approach_length_exceeded`；
- `max_buffer_blocks` 触发时输出 `stage2_reorder_complexity_budget_exhausted`，不静默降级；
- 同一输入多次输出一致。

### 第 4 步：Phase A 候选并入搜索（1 天）

实现：

- 临挂库外候选；
- 合并翻库候选；
- 卸轮线容量校验；
- 存4北压车处理。

验收：

- 当前 113 case 无触发时无副作用；
- 人工构造 case 覆盖：卸轮线被 C4 占用、翻入超容量、存4卸轮目标被压住、合并翻库优于临挂。

### 第 5 步：输出、trace 与 replay（1 天）

实现：

- 输出 API operations；
- 输出 `stage2_summary.off_segment`；
- 输出每条操作行的状态变化、路径、校验结果；

验收：

- physical/state consistency 100%；
- complete case 的 `pending_outbound_nos = 0`；
- `final_store4_segment ∈ OFF*C4*`；
- `Put(存4线)` 恰好一次且为最后一条操作行；
- C4 子序列前 3 位关门车合规。

### 第 6 步：全量回归与人工基准（1-2 天）

验收标准：

1. 113 case 无 solver error；
2. complete case 全部 replay 通过；
3. partial 必须输出以下结构化原因之一：
   - `stay_car_blocks_outbound`；
   - `outside_line_occupied_by_non_outbound`；
   - `store4_approach_length_exceeded`；
   - `store4_capacity_insufficient`；
   - `unwheel_capacity_insufficient`；
   - `closed_door_unsatisfied`；
   - `stage_handoff_unweighed_car`；
   - `stage2_reorder_complexity_budget_exhausted`；
   - `solve_time_budget_exhausted`；
4. 若有人工阶段二基准，complete case 操作行数应不高于人工对应段落，或给出可解释差异。

## 8. 风险与处理

| 风险 | 处理 |
|---|---|
| validator 不支持非空机后车列中间态 | 第 0 步 spike 判定；必要时写 stage2 simulator，最终仍用 replay 兜底。 |
| `OFF...C4` 多栈导致状态膨胀 | `held_consist` 车数受当量 20 和线长限制，状态仍小；必要时压缩等价 C4。 |
| 缓存线安全判断过松/过紧 | 过松会制造假解，过紧会漏解；第 0-1 步必须把缓存线安全条件落 trace，并用 `C4...OFF` 反例校验。 |
| 关门车终态被提前过滤 | 只在终态或最后 C4 Put 判断；过程规则每条 Put 判断。 |
| 进路长度下界失真 | 第 0 步按真实存4南入口实测 `actual_store4_approach_limit`；不确定则不做该剪枝。 |
| 修库外非出库车高频 | 第 1 步统计频率；高频则必须做清库外子模型，不能靠 partial。 |
| OFF 段第三阶段不好用 | 同目的地相邻作为 tie-break，并在 summary 输出 OFF 段序列。 |

## 9. 最小可交付版本

MVP 仍保持简单，但必须保留数学核心：

1. 操作行级 Dijkstra；
2. `held_consist`；
3. `final_store4_segment` 只校验第二阶段最终一次新增段；
4. 支持 `OFF...C4` 和 `C4...OFF`；
5. 支持安全缓存线重排；
6. 支持库外占用审计；
7. 支持修库外简单尾甩；
8. 支持 Phase A 无压车直接翻库和合并翻库人工用例；
9. 全量 replay。

MVP 明确不做：

- 不使用存4南中转；
- 不挖留库车再放回；
- 不考虑脱轨器时序；
- 不做第三阶段联合优化；
- 不放开“同一操作片段内反复重访同一源线”的高级枚举，除非反例证明必须。
