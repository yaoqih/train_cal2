# 调车计划生成 WebAPI 接口文档


---

## 1. 生成调车计划

* **功能描述**：根据当前站内车辆的初始位置、目标线路及各物理股道的配置，返回调车机车的详细作业指导步骤（勾计划）及计算后的实际对位结果。
* **请求路径**：`/api/plan/generate`
* **请求方式**：`POST`
* **Content-Type**：`application/json`

当前实现由 `plan_api.server:app` 提供，并统一串联 `stage1_simple` 至 `stage4_simple`。默认请求同步响应；生产环境可使用 `?async=true` 提交后台任务，也可通过 `/api/plan/generate/batch` 并行提交多个案例。部署、鉴权、任务查询和批量请求详见 [四阶段并行API部署.md](四阶段并行API部署.md)。

### 1.1 请求参数 (Request Body)

请求体统一使用 `PlanSubmission`，不接受把业务 request 直接作为根对象的旧格式。

#### 根对象 (`PlanSubmission`)

| 字段名 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| `case_id` | string | 是 | 4 位数字加 `W/Z`，例如 `0104W`。 |
| `request` | `PlanCreatePostData` | 是 | 车辆、库线和调机的业务输入。 |
| `options` | `PipelineOptions` | 否 | 四阶段资源预算；不能选择其它求解策略。 |

#### 业务对象 (`PlanCreatePostData`)

| 字段名 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| `StartStatus` | `List<StartCarDto>` | 是 | 车辆的初始状态列表。 |
| `TerminalLines` | `List<TerminalLineDto>` | 是 | 终点规划器可用的目标股道配置信息。 |
| `locoNode` | `LocoNodeDto` | 是 | 调车机当前位置，包含股道全称和南北端。 |

#### `LocoNodeDto` (调车机位置)

| 字段名 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| `Line` | string | 是 | 调车机所在股道全称，例如：`"机库线"`。 |
| `End` | string | 是 | 调车机所在端：`"North"` 或 `"South"`。 |

#### `StartCarDto` (初始状态)

| 字段名 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| `Line` | string | 是 | 起点股道全称，例如：`"修1库外"`。 |
| `Position` | int | 是 | 起点连续台位号，非负整数。 |
| `RepairProcess` | string | 是 | 非空修程字符串，按上游业务值传递，例如 `"段修"`、`"厂修"`、`"临修"`。 |
| `Type` | string | 是 | 车型信息。 |
| `No` | string | 是 | 车号。 |
| `Length` | decimal | 是 | 车辆长度，单位m |
| `IsHeavy` | bool | 否 | 是否为重车，默认 `false`。 |
| `IsWeigh` | bool | 否 | 是否需要称重，默认 `false`。 |
| `IsClosedDoor` | bool | 否 | 是否是关门车，默认 `false`。 |
| `TargetLines` | `List<string>` | 是 | 目标股道全称。如有多个候选线，可传多个股道，例如：`["修1库内", "修2库内"]`。 |
| `ForceTargetPosition` | `List<int>` | 否 | 强制对位台位列表。除修1-修4库内允许多个目标股道外，其他线路启用强制对位时只能指定一条 `TargetLines`。 |

#### `TerminalLineDto` (库内股道配置)

| 字段名 | 类型 | 必填 | 描述 |
| --- | --- | --- | --- |
| `Line` | string | 是 | 修1-修4库内股道，例如：`"修1库内"`。该列表用于覆盖库内容量/迎检配置，不是可用线白名单；未列出的库内线沿用常规 5 位容量。 |
| `IsInspectionMode` | bool | 否 | 该库内线是否按迎检模式开放。请求中存在迎检目标时，库内台位使用迎检范围 7 位（否则使用常规 5 位）。 |

---

### 1.2 响应参数 (Response Body)

返回标准响应结构 `PlanGenerateResponseDto`，核心数据在 `Data` 字段中。

| 字段名 | 类型 | 描述 |
| --- | --- | --- |
| `Success` | bool | 业务处理是否成功。 |
| `Message` | string | 提示信息或异常信息。 |
| `StatusCode` | int | HTTP 状态码。 |
| `Data` | `PlanGenerateResultDto` | 处理成功时返回的数据载荷。 |

#### `PlanGenerateResultDto`

| 字段名 | 类型 | 描述 |
| --- | --- | --- |
| `Operations` | `List<PlanOperationDto>` | 调车机作业步骤。 |
| `GeneratedEndStatus` | `List<GeneratedEndCarDto>` | 计算后的最终对位结果。 |

#### `PlanOperationDto` (动作信息，提取关键字段)

| 字段名 | 类型 | 描述 |
| --- | --- | --- |
| `Index` | int | 动作序号。 |
| `Line` | string | 求解器规范化线路名（如 `"修1库内"`、`"存5线北"`、`"机北1"`），不保证转换为现场展示全称。 |
| `Action` | string | 操作动作：`Get` (取车), `Put` (放车), `Weigh` (称重)。 |
| `MoveCars` | `List<string>` | 参与本次移动的车号列表。 |
| `TrainCars` | `List<string>` | 操作后机车上承载的车号列表。 |
| `PassbyPath` | `List<string>` | 四阶段求解器生成的完整物理路径，保留规范化股道、联络线和内部路径节点，包含当前操作实际记录的起点和终点。例如：`["存5线北", "渡1", "联6", "渡2", "机北1", "机北2", "渡4", "机库线"]`。该字段也是内部回放校验使用的路径事实。 |
| `ByPassSwitch` | `List<string>` | 对外展示的经过道岔，只包含按行进顺序经过的 `L/Z` 道岔，不包含起点和终点股道。例如：`["L2", "L1", "L3", "L5", "L6", "L7"]`。同一实际道岔在内部图中可能对应连续两条边，此时只返回一次；库门、棚门或线路分界等不经过道岔的路径返回 `[]`。投影会结合 `locoNode.Line`、上一勾 `Line`、当前勾 `Line` 和完整路径补齐首尾，因此 `机南 -> 机走棚` 必定返回 `L8`。 |

`Action=Weigh` 时，`TrainCars` 是本勾挂着经过机库的整组车；`MoveCars` 只填本次实际标记为称重完成的车。若一勾挂了多辆称重车，只有编组最后一辆需要称重的车在本次 `Weigh` 中完成称重。

#### `GeneratedEndCarDto`

| 字段名 | 类型 | 描述 |
| --- | --- | --- |
| `No` | string | 车号。 |
| `Line` | string | 最终所在的求解器规范化线路名。 |
| `Position` | int | 最终台位号。 |

### 1.3 最小输入输出示例

下面示例只保留最小必要字段，用来说明请求和响应的基本结构；实际返回中 `PassbyPath` 和 `ByPassSwitch` 会随真实求解路径变化。

#### 最小输入示例

```json
{
  "case_id": "0101Z",
  "request": {
    "StartStatus": [
      {
        "Line": "存5线北",
        "Position": 1,
        "RepairProcess": "段",
        "Type": "C70",
        "No": "1000001",
        "Length": 14.3,
        "TargetLines": ["机库线"]
      }
    ],
    "TerminalLines": [
      {
        "Line": "修1库内",
        "IsInspectionMode": false
      }
    ],
    "locoNode": {
      "Line": "机库线",
      "End": "North"
    }
  }
}
```

#### 最小输出示例

```json
{
  "Success": true,
  "Message": "",
  "StatusCode": 200,
  "Data": {
    "Operations": [
      {
        "Index": 1,
        "Line": "存5线北",
        "Action": "Get",
        "MoveCars": ["1000001"],
        "TrainCars": ["1000001"],
        "PassbyPath": ["机库线", "渡4", "机北2", "机北1", "渡2", "联6", "渡1", "存5线北"],
        "ByPassSwitch": ["L7", "L6", "L5", "L3", "L1", "L2"]
      },
      {
        "Index": 2,
        "Line": "机库线",
        "Action": "Put",
        "MoveCars": ["1000001"],
        "TrainCars": [],
        "PassbyPath": ["存5线北", "渡1", "联6", "渡2", "机北1", "机北2", "渡4", "机库线"],

      }
    ],
    "GeneratedEndStatus": [
      {
        "No": "1000001",

        "Position": 1
      }
    ]
  }
```
