# 阶段1/阶段4边界拓扑门控策略

## 结论

第一阶段可以替第四阶段分担压力，但不能按“见空线就编组”的方式做。正确边界是：第一阶段先关闭浅层、可直接消化、会释放关键通路的前场业务；只有当某条临停线/前置线的下游债务已经关闭时，才把它降级为编组资源。第四阶段继续负责深层目标线重建、强制台位重排、大库库内/库外闭合和存4释放。

这次复核了 `data/truth2` 全量 113 个案例，并结合 `docs/物理约束.md` 的线路图重新做了路径门槛统计。核心修正是：空线不等于资源线。`机南`、`洗油北`、`机北1`、`机北2` 初始多为空线，但在下游债务未清前都是通路门槛。

## 阶段边界

### 第一阶段应该做

1. 清点每个目标的债务量、目标线洁净度、北端阻塞深度和路径门槛风险。
2. 对干净且路径清晰的目标线直接送达，不绕去存4北或编组线。
3. 对浅层阻塞目标先清北端前缀，再直接送达。
4. 优先关闭会释放通路的前场业务：抛丸、洗罐线北/洗罐站、油漆线。
5. 把从服务线释放出来、确实要去大库的车辆组织成第一批大库编组，但只能使用已解锁资源线。
6. 如果临时借用 `存4线`，必须在同一阶段内形成闭合动作：进 `存4线` 后马上转入编组资源，不能把存4北污染成长期缓存。

### 第四阶段应该保留

1. `调梁棚`、`预修线` 这类深层目标线重建。
2. 强制台位和 spotting 窗口重排。
3. 大库库内/库外闭合、修库摘解和存4释放。
4. `存5线南` 深层抽取、跨多源缓存重排。
5. 目标线本身很脏、北端阻塞很深、需要多次临停恢复的作业。

## 线路角色和解锁条件

| 线路 | 未解锁角色 | 解锁条件 | 解锁后可用方式 | 早期禁用 |
|---|---|---|---|---|
| `抛丸线` | 小容量作业目标 | 无外部去抛丸债务，抛丸现车已送出或消化 | 可释放 `机南` 参与编组 | 抛丸未闭合时把车放 `机南` |
| `机南` | 抛丸/机区/洗油侧通路 | 抛丸目标关闭，且无需要经 `机南` 的洗油/机区债务 | 第一批大库编组资源 | 作为抛丸去大库车默认缓存 |
| `洗油北` | 洗北/洗南/油漆硬门槛 | `洗罐线北`、`洗罐站`、`油漆线` 的即时债务关闭 | 高价值短缓存/编组资源 | 洗/油未闭合前甩车 |
| `洗罐线北` | 洗罐站前置门槛 | 洗罐站不再需要进出，或洗北自身已完成 | 可作为洗罐相关短缓存 | 洗罐站未闭合前压车 |
| `机走棚` | `机南`/`洗油北` 前置门槛，且自身有预修债务 | 洗油侧和机南侧债务关闭，且机棚目标不再紧急 | 第二批或短缓存 | 早期大库编组常驻 |
| `机走北` | 机棚前置门槛 | 机棚/调棚/机库方向近期债务关闭 | 短缓存 | 早期占用导致机区绕行 |
| `机北1` | 存1/机北2通路 | 存1、调棚、机库方向债务低风险 | 极短临停 | 当普通缓存 |
| `机北2` | 调棚/机库硬门槛 | 调棚、调北、机库方向即时债务关闭 | 极短临停 | 早期缓存 |
| `调梁线北` | 调棚硬门槛 | 调棚目标关闭或本轮不再进出调棚 | 第二批中服务调棚，不作为通用缓存 | 早期安全缓存 |
| `调梁棚` | 深层调梁/预修目标 | 油洗抛前场关闭后进入第二批 | 第二批目标重建/拆解 | 第一批中强行编组 |
| `预修线` | 深层预修目标和存2侧通路 | 预修目标可连续摘解，且不会阻断存2/渡7路径 | 目标处理 | 普通缓存 |
| `存4线` | 正式出段集结/释放口 | 出段车或闭合中转批次明确 | 存4释放、短闭合中转 | 长期存放去大库半成品 |
| `存4南` | 存4释放走廊 | 基本不作为资源 | 只允许计划内通过/短临停 | 普通缓存 |
| `存5线北` | 入段主队列 | 北端前缀按真实目标消化 | 源线整理 | 深挖南端导致多次翻线 |
| `存5线南` | 深层入段队列 | 存5北前缀清楚，南端目标成段 | 成段抽取 | 第一阶段盲目深挖 |
| `修1-4库外` | 对应库内硬门槛 | 对应库内出入闭合完成 | 大库闭合内部使用 | 缓存 |

## 推荐流程

### 0. 每案先算四个量

对每个候选目标线计算：

- `target_debt_count`: 还有多少车要去该目标。
- `target_clean`: 目标线为空，或既有车全部已满足且无强制台位冲突。
- `source_prefix_depth`: 从北端取到第一辆目标车要越过多少车。
- `route_gate_risk`: 候选动作会不会占用尚未解锁的门槛线。

只有 `target_clean=true` 且 `route_gate_risk=0` 时，才允许直接送达。目标线脏但阻塞浅时，先清目标线北端前缀；阻塞深或涉及门槛线时，转为阶段性解锁任务。

### 1. 先关闭抛丸

抛丸目标小、深度浅，适合作为第一批解锁对象。但规则是：

- 如果外部还有去 `抛丸线` 的车，禁止把抛丸线上去大库车默认放 `机南`。
- 抛丸线自身去大库车优先找不影响 `机南/机走棚/洗油北/调梁线北` 的短暂位置；如果找不到，则宁可先处理外部去抛丸车。
- 抛丸关闭后，`机南` 才从 gate 降级为可用编组资源。

### 2. 再关闭洗北/洗南/油漆

`洗油北` 是硬门槛。顺序不是固定洗罐先或油漆先，而是按洁净度和北端深度选：

- `洗罐站` 脏且 `洗罐线北` 有车时，先清 `洗罐线北`。
- `油漆线` 干净且路径清晰时，直接送油漆，不绕编组。
- 洗南、洗北、油漆都没有即时债务后，`洗油北` 才能作为高价值短缓存或编组资源。

### 3. 第一批大库编组

第一批只收“前场关闭过程中自然释放出来的去大库车”，不主动深挖所有大库车。候选来源包括：

- `抛丸线`、`洗罐站`、`洗罐线北`、`油漆线` 上已经不再服务本线的去大库车。
- `存5线北/存5线南/存2线/存3线` 北端正好挡住上述前场目标的去大库车。
- 已经成段、取出不破坏门槛线的其他去大库车。

编组资源选择顺序应动态决定：

1. 已解锁的 `机南`、`洗油北`、`机走棚/机走北`。
2. 非门槛、非目标脏线、容量足够的存车线。
3. `存4线` 只能作为闭合中转：进存4北后马上转入编组线，不能留下给后续存4释放制造口子污染。

### 4. 第二批处理调棚/调北/机库

洗油侧关闭后，再处理 `调梁棚`、`调梁线北`、`机库线`：

- `调梁线北` 未解锁前不能当缓存，因为它硬阻断调棚。
- `调梁棚` 初始 111/113 案例有车，目标脏案例 104 个，属于第二批目标重建，不适合第一阶段抢做。
- `机库线` 干净时可直送；若与调北/调棚互相卡口，放到第二批统一处理。

### 5. 预修和大库闭合留给后段

`预修线` 目标债务 651，源线债务 642，初始占用 111/113，通常不适合第一阶段深做。大库库内/库外和 `存4线` 释放是强业务闭合，继续由后段宏处理。

## 落地到 Stage1 的判定规则

建议在 `stage1_simple` 增加一个策略层，不直接改物理可达：

```text
LineRole:
  gate_lines: 机南, 洗油北, 机走棚, 机走北, 机北1, 机北2, 调梁线北, 洗罐线北, 存4南, 存5线北, 修库库外
  target_lines: 抛丸线, 洗罐站, 洗罐线北, 油漆线, 调梁棚, 调梁线北, 机库线, 预修线
  release_lines: 存4线
  assembly_lines: 只包含已解锁的临停/存车线
```

候选动作过滤：

1. 若 `candidate.target_line` 是未解锁 gate，且动作不是清这个 gate 的必要动作，拒绝或给极大惩罚。
2. 若目标线干净且路径清晰，直接送达优先级高于先编组。
3. 若目标线脏，优先清北端前缀；前缀深度超过阈值时，不在第一阶段强做。
4. 若动作会占用 `机南` 且仍有外部去抛丸债务，拒绝。
5. 若动作会占用 `洗油北` 且仍有洗北/洗南/油漆即时债务，拒绝。
6. 若动作会占用 `调梁线北/机北2` 且仍有调棚/机库即时债务，拒绝。
7. `存4线` 上非出段车只允许出现在同一阶段闭合链中，不能作为阶段结束状态。

候选动作评分：

```text
score =
  - gate_unlocked_reward
  - direct_delivery_reward
  + unsafe_gate_occupancy_penalty
  + target_dirty_depth_penalty
  + route_detour_delta
  + future_fragmentation_penalty
  + storage_pollution_penalty
```

## 可行性判断

这个策略是可行的，但前提是不要把“第一批”理解成固定线路列表。第一批应该是“前场服务闭合后自然释放的大库成段车”，第二批才是调棚/调北/机库/预修等深层目标。这样既能减轻 Stage4 的服务线清障压力，又不会提前占掉 `机南`、`洗油北`、`调梁线北` 这些关键通路。

下一步需要把上面的 `LineRole/ResourceGate` 编成 Stage1 候选剪枝和评分，然后做端到端 A/B：比较 Stage4 的 `put_route_blocked`、`active_blocked_by_fixed`、最终 unsatisfied、总钩数和 replay 违规。当前这份验证已经覆盖全量初始案例和拓扑路径风险，但还没有证明实现后的端到端收益。
