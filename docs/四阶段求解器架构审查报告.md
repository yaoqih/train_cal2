# 第1—4阶段求解器架构与正确性审查报告

## 1. 审查结论

当前四阶段求解器已经具备较强的物理约束校验、阶段性业务建模、独立回放和诊断能力，测试覆盖也明显高于一般原型系统；但从“达到或超过人工计划”的目标看，当前主要瓶颈不是缺少强化学习，而是以下四类基础问题尚未解决：

1. **状态等价和候选等价存在错误**：不同业务状态可能被视为相同状态，不同动作计划也可能得到同一候选 ID，搜索会错误剪枝或复用缓存。
2. **物理规则、业务完成规则和台位分配存在多个真源**：`physical.py`、`replay_validator.py`、Stage3 placement 对同一业务合同已经出现可复现的不同结论。
3. **主线优化不是统一的全局搜索**：Stage1 是不可回溯的单步承诺，Stage2 压缩局部 episode，Stage3 是重启式 portfolio，Stage4 先强制闭合 carry、再用第二层 episode 删除主线制造的冗余。
4. **Pipeline 只传一个阶段终态且没有全局目标反向反馈**：上游少一勾、下游多很多勾的方案无法被全局淘汰，这与人工进行全流程联合权衡的方式存在根本差异。

因此，本报告的总体判断是：

> 当前确实存在逻辑混杂、职责错位、隐式策略分层、重复规则和结构过载。没有发现“主 solver 失败后切换旧 solver”的传统 fallback，但 Stage1 的恢复候选池、Stage3 的模板×placement×预算 portfolio、Stage4 的主构造后二次优化，已经形成了类似 fallback 的诊断复杂度。

优先顺序应是：**先统一状态和合同，再优化 Stage1 主线，再建立 Stage2—3—4 的边界联合优化，最后才考虑学习方法和 LNS 增益。**

## 4. 跨阶段与共享层审查

### X-01 P0：API 最终门禁只做动作回放，不做业务完成校验

证据：

- `plan_api/pipeline.py:734-748` 的 `replay_gate()` 只调用 `rv.replay()`；
- `plan_api/pipeline.py:762-776` 的最终成功响应也只检查 replay violations；
- 真正检查目标股道、称重、强制台位、最终长度、关门车位置的 `business_errors()` 位于 `replay_validator.py:742-770`，没有进入 API gate。

最小复现：一辆关门车已经停在存4第1位，返回零操作：

```text
pipeline replay_gate: ok=True
business_errors: closed_door_cun4_position_violation
```

更一般地，一辆车仍在存1、目标是存2，零操作也可以通过 replay gate；只要阶段自报 `complete`，公共响应就可能返回 `Success=True`。

现有 `artifacts/fullflow_current` 中也有三例 Stage4 标为 complete，但当前 `business_errors()` 报最终线路超长：

|案例|冲突|
|---|---|
|truth2 `0114Z`|存5线南 `170.5 > 156.0`|
|truth2 `0129W`|存5线南 `167.2 > 156.0`|
|truth3 `0421W`|洗罐站 `91.3 > 88.7`|

这三例超长来自初始状态。业务上可以选择“拒绝输入”“要求求解器消债”或“允许初始既有超长保持”，但必须只有一个明确口径。当前同一系统同时给出 complete 和 business violation，是合同分裂。

整改：

1. 建立统一 `FinalBusinessValidator`；
2. Stage1—3 使用各自阶段边界 validator；
3. Stage4 和公共 API 必须执行完整最终业务 validator；
4. `Success=True` 必须由独立 gate 决定，不能只信任 solver 自报 status。

### X-02 P0：三套台位和业务真源已经出现实质分歧

当前至少存在：

1. `physical.build_depot_assignment()`：`scripts/solver_vnext/physical.py:984-1083`；
2. Stage3 placement 与 `terminal_depot_ok()`：`scripts/stage3_simple/placement.py`、`solve.py:2992`；
3. `replay_validator.build_slots()` 与 `business_errors()`：`replay_validator.py:680-770`。

已复现两类冲突。

#### 冲突 A：混合内/外目标长车

18m 车辆目标为 `修1库内/修1库外`，最终停在修1库外：

```text
physical.build_depot_assignment: no_feasible_depot_slot
physical.car_is_satisfied: False
replay_validator.business_errors: []
```

原因是 physical 先把存在内库目标的车辆纳入内库匹配并写入 failure，而 replay 对混合内/外目标不要求内库 matching certificate。

#### 冲突 B：段修位于厂修更深处

厂修在修1第4位、段修在第5位：

```text
physical.car_is_satisfied(段修): False
replay_validator.business_errors: []
```

Stage3 又单独实现了同类顺序规则。独立 replay 可以保留独立的转移执行器，但拓扑数据、容量、台位资格和最终业务合同不能由三份手写逻辑分别维护。

整改：

- 把线路、容量、台位域、修程顺序、强制位置和关门车规则抽成声明式 `BusinessRuleSet`；
- physical、Stage3 placement、replay 都读取同一份规则；
- 对所有规则增加 physical/replay/placement parity 测试；
- 明确 `locked slot`、`nominal assignment`、`actual terminal placement` 三种不同语义。

### X-03 P0：`locoNode.End` 被验证和文档承诺，但求解中被丢弃

- 接口文档 `docs/接口文档.md:33-40` 允许 `North/South`；
- `physical.LocoLocation` 只有 `line` 字段：`physical.py:685-692`；
- `replay_validator.replay()` 也只读取机车线路：`replay_validator.py:501-506`；
- Stage2、Stage3、Stage4 分别把交接请求强写为 North：
  - `stage2_simple/solve.py:421-440`
  - `stage3_simple/solve.py:3847-3854`
  - `stage4_simple/solve.py:245-254`

因此同一线路的 North 与 South 输入会得到相同初始物理状态。

如果现场只支持“调机固定在车列北侧”，API 应拒绝 South；如果 South 合法，状态、路径、取放端和回放必须真正建模 End。当前“接受后静默忽略”最危险。

### X-04 P0：全流程时间合同错误

业务文档 `docs/福州调车业务文档.md:291-293` 要求全流程 5 分钟。

API 默认值位于 `plan_api/pipeline.py:49-55`：

```text
Stage1 300s + Stage2 300s + Stage3 180s + Stage4 300s = 1080s
```

同时：

- `pipeline.py:35,118-127` 允许总预算 1800 秒；
- `plan_api/server.py:324-335` 的 watchdog 也按四阶段预算相加。

当前没有共享的 `global_deadline`。正确方式应是：

```text
global_deadline = request_start + 300s
stage_deadline = min(global_deadline, stage_start + stage_slice)
```

预处理、assignment、scope、回放和序列化也应计入 300 秒实际墙钟时间，而不是只统计某个 optimizer 内部时间。

### X-05 P1：Pipeline 是串行单终态，不是全局联合优化

`plan_api/pipeline.py:437-557` 每个阶段只传一个结果，任一阶段非 complete 就停止。

阶段按业务时间顺序执行本身没有问题；问题是边界只保留一个状态：

- Stage1 的服务收尾只能用自制 Stage3/4 proxy 估分，无法获得真实下游代价；
- Stage2 的卸轮/存4选择无法保留多个终态给 Stage3；
- Stage3 placement 无法按 Stage4 收口勾数反向选择；
- 上游少一勾、下游多十勾的方案无法被全局淘汰；
- partial 即使物理安全，也不能由下游接管继续完成。

这正是算法与人工差距中最核心的“全局联合优化”问题。建议保持阶段边界，但在关键边界传递小规模非支配 frontier，而不是只传一个终态。

### X-06 P1：Stage3 的真实 placement 没有形成正式交接合同

- Pipeline 在输入时预建原始 assignment：`pipeline.py:595-599`；
- Stage3 返回 `assignment_plan`：`stage3_simple/solve.py:3787-3794`；
- Pipeline 调 Stage4 时仍传原始 assignment：`pipeline.py:543-557,644-663`。

现有 truth3 产物中，Stage3 的 371 个内库台位决策有 63 个与原始 physical assignment 不同，其中 12 个目标修库不同、51 个位置不同。

当前 physical 对非 locked slot 的满足性较宽松，因此“仅名义 slot 不同”通常不会立即产生错误；但这是隐式行为，不是清晰合同。如果 Stage3 找到了 physical 原始 assignment 判 failure 的可行 placement，Stage4 仍可能继承旧 failure。

整改：Stage3 应输出不可变 `TerminalPlacement`，Pipeline 明确传给 Stage4；或者让 Stage4 完全只依赖 Stage3 实际终态和统一业务规则，不再依赖旧名义 assignment。

### X-07 P2：配置、统计和包装合同不统一

- API 默认 `stage4_max_labels=64`：`pipeline.py:55,105-109`；
- Stage4 CLI、optimizer、README 默认 128；
- Stage1—3 的 `business_hooks` 只计 Get/Put，Stage4 的 hooks 和 `business_hooks` 计 Get/Put/Weigh：`stage4_simple/solve.py:330-344`；
- Stage2/3/4 重复实现 case id、JSON I/O、机车终态、response 合并、aggregate 和 diagnostic summary；
- Stage1 异常 summary 与正常 summary 的 `stage1_debt` schema 不同，且 aggregate 把 error 的 0 勾计入平均值：`stage1_simple/solve.py:6196-6261`。

这些问题不一定直接改变计划，但会污染横向比较和管理决策。

## 5. Stage1 深度审查

### S1-01 P0：状态签名遗漏 `_Weighed`

`physical.state_signature()` 位于 `physical.py:3289-3299`，只包含：

```text
loco.line + (No, Line, Position)
```

但 `physical.apply_candidate()` 会修改 `_Weighed`：`physical.py:3212-3268`，而 `car_is_satisfied()` 在 `physical.py:1597-1600` 又明确依赖 `_Weighed`。

Stage1 将该不完整签名用于：

- 初始和已访问状态：`stage1_simple/solve.py:283,2090`；
- cycle 剪枝：`solve.py:714-717`；
- validation cache：`solve.py:2109-2123`；
- continuation cache：`solve.py:2136-2216`；
- rolling session cache：`solve.py:2931-2938`；
- downstream quality cache：`solve.py:5611-5625`。

最小复现已经确认：同一车辆称重前后 `state_signature` 完全相同。

影响不只是性能：

- 称重后的有效状态可能被判为 cycle；
- 过磅前的 validation、continuation、下游质量可能被复用；
- `downstream_quality()` 本身读取待称重状态，证明签名遗漏了真实依赖。

整改：定义不可变 `Stage1StateKey`，至少包含车辆号、线路、位置、`_Weighed`、机车线路和机车端位。各缓存再按自身依赖加入路径历史或 assignment 版本。

### S1-02 P0/P1：候选 ID 不完整，去重又采用“先到先得”

`physical.planlet_candidate_id()` 位于 `physical.py:820-832`，候选键只编码 Get/Put 的 action、line、MoveCars：

- 忽略 Weigh；
- 忽略 `planned_positions`；
- 忽略生成器、reason 和评分。

已经直接复现：

```text
Get A -> Put position 1
Get A -> Weigh A -> Put position 5
```

两者生成完全相同的 candidate ID。

随后 `generate_candidates()` 在 `stage1_simple/solve.py:2262-2302` 使用：

```python
unique.setdefault(candidate_id, item)
```

因此候选生成器注册顺序会改变最终保留的动作语义、位置方案、reason 和 score。相同物理计划由多个候选族生成时，后生成器的解释和评分也会静默丢失。

整改：

- 用完整 `PlanKey` 编码所有动作、顺序、位置和称重语义；
- generator/reason/score 作为 annotations 合并，不进入物理等价键；
- 相同 PlanKey 应统一重新评分或保留最佳 annotation，不能 `setdefault`；
- 增加“候选生成顺序不影响选择”的 metamorphic test。

### S1-03 P1：主线是不可回溯的单步永久承诺

`choose_and_accept_candidate()` 位于 `solve.py:440-548`：

- 无 contextual mode 时，遇到第一个排序后合法候选就立即接受：`solve.py:473-480`；
- 有 contextual mode 时，只在有限 examined/valid window 中比较，再永久提交一个候选；
- 后续没有阶段计划级回溯或统一 frontier。

`can_continue()` 只是有限深度的可继续性 probe，不等于全局搜索，也不能撤销主线已提交动作。

这会系统性产生：

- 局部清障正确、全局顺序错误；
- 为当前债务下降牺牲后续编组连续性；
- 先完成一个目标，之后为了门区/顺序再重复 Get；
- 高勾案例对候选排序和阈值非常敏感。

这是 Stage1 主线尚未优化好的核心原因之一。优先建议不是增加更多候选宏，而是将现有候选放入一个可控宽度的 beam/A*/best-first SearchController，至少保留多个非支配状态。

### S1-04 P1：候选体系已形成隐式恢复 portfolio

主池顺序位于 `solve.py:2291-2298`：

```text
stack closure
rolling session
direct assembly
blocker
route blocker
cleanup
release gate
direct delivery
```

内部还有：

- forced rebuild 的 direct word / staged word；
- direct delivery 的普通 Put / retained route；
- stack closure 的 blocker Get / spill；
- rolling session 的普通 lane / support lane；
- service closure 的 forced rebuild / spotting repack / prefix cleanup / target rebuild。

这不是失败后切旧 solver，因此不应叫传统 fallback；但策略变体嵌在各编译函数内部，行为上已经接近不可解释的候选 portfolio。每增加一个高勾案例补丁，都会进一步改变候选召回和排序。

整改：建立显式 `CandidateGeneratorRegistry`，每个生成器统一报告：

- 召回数量；
- 去重前后数量；
- 物理拒绝原因；
- 死端拒绝原因；
- 被评分窗口截断数量；
- 耗时和预算；
- generator/variant 标签。

### S1-05 P1：fork 共享路径相关缓存，纯状态与搜索上下文没有分开

`fork()` 位于 `solve.py:2218-2244`，probe 与 root 共享：

- `seen_states`；
- validation cache；
- continuation cache；
- downstream quality cache；
- rolling session cache；
- forced rebuild word cache。

共享纯函数缓存本身没有问题；真正危险的是 `continuation_cache`：它的值依赖可变 `seen_states`、`get_count_by_no`、候选顺序和共享 deadline，但 key 只有状态签名、hook index、depth 和 avoid 标记：`solve.py:2145-2151`。

同一布局由不同历史到达时，continuation 结果不一定相同。root 后续扩充共享 `seen_states` 后，旧缓存的语义也可能变化。

整改：

- `PhysicalCache` 只缓存纯状态函数；
- `SearchContext` 保存 seen/history/get counts/deadline；
- 路径相关缓存 key 必须包含完整上下文摘要，或只在一次不可变 probe 内使用。

### S1-06 P1：多个局部目标和下游代理互相竞争

Stage1 同时存在：

- 主债务下降；
- contextual score；
- rolling option/state/endpoint rank；
- route blocker pressure；
- service closure weighted surplus；
- Stage3 quality proxy；
- Stage4 quality proxy。

`downstream_quality()` 在 `solve.py:5611-5625` 直接把 Stage3、Stage4 和 service 代理合并。代理可以帮助局部决策，但它不是下游真实最优勾数；下游逻辑变化后，Stage1 代理还会陈旧。

Stage1 完成主编组后继续最多 12 个业务勾的 service closure，这符合“尽量提前完成其他线路”的业务方向，但当前没有用全流程实际节省量决定是否值得，因此可能出现 Stage1 多花 4 勾、Stage4 只省 1 勾。

整改：统一成一个阶段目标：

```text
硬可行性
-> 主债务/污染
-> 累计总勾的下界或真实短评估
-> 重复取车
-> 道岔动作
-> 路径代价
```

下游只暴露版本化 `HandoffEvaluator`，不要让 Stage1 内嵌 Stage3/4 规则副本。

### S1-07 P2：service closure 的停止原因不可观察

主债务完成后进入 closure：`solve.py:303-320`。但 closure 无候选、全部拒绝或超时，`step()` 在 `solve.py:421-422` 直接返回 False，不写 blocked trace；最终 status 仍只按主债务 complete 判定：`solve.py:5982-5985`。

无法区分：

- 已达到局部最优；
- 12 勾上限；
- 时间耗尽；
- 候选召回不足；
- 全部物理拒绝；
- lookahead 窗口截断。

应增加独立字段：`service_closure_status/stop_reason/generated/validated/accepted/budget_exhausted`。

## 6. Stage2 深度审查

### S2-01 P1：`exact/optimal` 标签没有成立条件

Stage2 README 和 `EpisodePlanner` 把局部模型描述为 exact，summary 在 `stage2_simple/solve.py:255-257` 只要 `result.state` 存在就写：

```text
optimal_in_monotone_episode_model
```

但至少有四类未证明剪枝。

#### 1. inversion 预剪枝不属于正式目标

`model.py:265-300` 和 `326-351` 只保留 inversion 最小的延期组合。正式目标 `solve.py:357-372` 不含 inversion，因此同延期数但 inversion 较高、全局勾数更少的组合会在搜索前被删除。

真实数据中已经发生组合压缩，例如：

- truth2 `0205W`：仅保留 3/6 个同级组合；
- truth2 `0226W`：2/7；
- truth3 `0415W`：2/5。

这些被删除组合是否更优，当前求解器没有验证机会。

#### 2. EpisodePlanner 只返回一个局部结果

`episodes.py:53-117` 遍历 scratch/absorb 变体后只返回一个 `EpisodePlan`。但不同结果会改变：

- yard 终态；
- collector；
- open unload；
- absorbed source；
- global completed mask。

这些未来状态不能仅按局部成本压成一个确定性 transducer。

#### 3. `<=3` 勾提前返回缺少全局支配证明

`episodes.py:89-97` 在无 scratch 且不超过 3 行操作时直接返回。注释只证明 scratch 在当前 episode 内较贵，没有证明“多一勾但顺带吸收另一个来源线”在全局不占优。

#### 4. 局部状态上限截断没有传播完整性

`episodes.py:12,261-268` 达到 12,000 个局部状态就返回 None，但最终 optimality 标签没有说明某些 episode 被截断。

整改：

- `EpisodeEnumerator` 返回按未来边界状态区分的非支配结果集；
- inversion 要么进入正式目标，要么不能删可行组合；
- 删除 `<=3` 提前返回，除非给出未来状态等价证明；
- 增加 `local_search_complete/global_search_complete/optimality_proven`。

### S2-02 P1：超时会丢掉已找到的完整 incumbent

`optimizer.py:71-76` 保存 `best_goal`，但 `optimizer.py:84-87` 遇到 deadline 或状态上限直接 `_failure()`。

因此可能出现：

```text
已找到完整、可 replay 方案
-> 继续搜索以证明更优
-> deadline 到达
-> 返回空 partial，而不是完整 incumbent
```

正确合同应把可行性与最优性分开：

- `complete_best_incumbent`；
- `optimality_proven=False`；
- 明确保留当前完整动作。

### S2-03 P1：局部最优 episode 被错误当成全局确定边

即使删除上述提前返回，当前 `plan()` 仍会把不同 scratch/absorb/终态压成一个边。全局 label-setting 只有在“相同输入边界必然产生未来等价终态”时才安全；当前没有这个性质。

建议全局状态保持清晰，局部层只做物理转移枚举，不承担跨来源吸收和未来状态删除。

### S2-04 P2：Yard 与公共物理层重复维护边界规则

`stage2_simple/yard.py` 自己实现 Get/Put、容量、关门车和机车状态，同时部分调用 physical 路由。`optimizer.py:191` 还硬编码 20，而不是统一常量。

Stage2 的模块分层是四阶段中相对最清晰的，但应让 Yard 成为公共原子转移内核的适配器，不再复制规则。

值得保留的结构：

- 明确的 `BoundaryState`；
- 全局 label-setting 框架；
- Stage2 独立 replay 与 Stage1+2 合并 replay；
- 没有传统备用 solver。

## 7. Stage3 深度审查

### S3-01 P1：最新 mixed-gate 宏绕过 deadline 和 expansion 预算

实现位于 `stage3_simple/solve.py:1405-1608`。它包含多层枚举：

- 4 个内库目标；
- held 全部后缀 × 暂存线；
- 内库全部前缀；
- 所有外库及其前缀；
- 每个变体重复路径、容量、台位和闭包计算。

但整个函数：

- 不调用 `deadline_reached()`；
- 不计 expansions；
- 没有局部节点上限；
- 在 `solve.py:1889` 被无条件加入 closed-form 候选；
- 即使 `search_budget_open=False`，仍会运行。

唯一累计 expansions 的主要是通用事务搜索，见 `solve.py:1906-1927`。因此 25/100/5000/10000 已不是实际工作量硬预算，一次 mixed-gate 调用也可能越过 deadline 后才返回。

这次新增宏有较完整的投影保护和专项测试，业务意图是合理的；问题在于它被放进了“无预算 closed-form”层。应把所有事务生成器统一接入 `SearchBudget`，统计 generated/probed/expanded，并在每层循环检查 deadline。

### S3-02 P1：预算耗尽后改变搜索图

`plan_transactions()` 位于 `solve.py:1790`。在 `solve.py:1882` 之后，预算前后动作集合不同：

- 预算前：direct、capacity、gate、mixed、alignment、通用合法邻接；
- 预算后：最多再展开 2 个 frontier 状态，只保留 direct、capacity、gate、mixed；
- alignment 和通用邻接停止；
- 预算后的 partial 进展不更新 best partial，但 complete 可以被接受。

这不是普通的“搜索停止”，而是同一状态在不同时间拥有不同后继。结果依赖状态何时出队，难以解释，也不能把 expansion limit 解释为统一模型的搜索深度。

建议二选一：

1. 到预算后干净停止并返回 incumbent；
2. 保持同一邻域，把 frontier 持久化，下一档预算原地续搜。

不建议继续维护“预算后救援邻域”。

### S3-03 P1：四档 deepening 是从头重算的复杂 portfolio

配置：`solve.py:48-55`；调度：`solve.py:676-889`；每次求解入口：`solve.py:1015`。

当前结构为：

```text
模板 A/B
× 每模板最多 8 个 placement
× 25/100/5000/10000 四档预算
× screen/finalist/quorum/assignment signature 覆盖
× 多层 early stop
```

同一 `(template, layout)` 从 25 加深到 100 时，不是继续剩余 75，而是从 pickup 后初态重新跑 100。上一轮 frontier、best-cost 和事务搜索结果全部丢弃，prior 只用于构型排名和结果选择。

它不是失败后切换旧 solver，所以不是传统 fallback；但运营上确实是重启式 portfolio。README 的“只有一套主路径”容易掩盖其真实复杂度。

整改：每个构型保存可恢复 `SearchSession`，统一 scheduler 只追加预算，不重启求解。

### S3-04 P1：只保留局部最小事务，没有全局支配证明

`transactions.enumerate_minimal_transactions()` 只保留当前状态下最小成本的严格进展事务。不同的较贵事务可能产生不同：

- committed set；
- 牵出列顺序；
- 门区占用；
- 后续可达性。

没有 Bellman 支配证明时，删除所有较贵事务可能丢失全局更优甚至唯一可行路径。持续增加 capacity/gate/alignment/mixed-gate 宏，本质上是在补偿主事务抽象的表达不足。

建议返回按 `(end_state, committed_after)` 区分的非支配事务，或允许有限 cost band；稳定闭包适合做启发式和状态摘要，不宜成为删除全部局部次优路径的依据。

### S3-05 P2/P1：候选验证缓存缺少 assignment context

`solve.py:3272-3348` 的缓存键只有：

```python
(candidate.state, candidate.ops)
```

但 `terminal_depot_ok()` 读取当前可变的 `assigned_line_by_no/assigned_slot_by_no`。当前调用顺序会在验证前恢复 candidate 的 template/layout，而且只验证已通过 `complete()` 的候选，尚未构造实际误复用案例；因此应标为高风险合同，而不是确定 bug。

整改：将 assignment 变为不可变 context，并把 assignment signature 或 `(template, layout)` 加入依赖 assignment 的缓存键。

### S3-06 P2：遗留优化诊断字段永远不更新

`optimization_attempted/optimization_expansions/optimization_budget_exhausted` 只在 `solve.py:245-247` 初始化，在 `solve.py:3722-3736` 输出，但没有任何更新位置。

结果是 summary 永远报告 `0/0/false`，`bounded_improvement_exhausted` 分支不可达。这属于旧结构残留，应删除或真正接入统计。

### S3-07 P2：Stage3Solver 职责过载和规则重复

当前 `solve.py` 约 4164 行、98 个方法，Stage3 总规模约 5277 行。一个类同时承担：

- placement 编译；
- portfolio 调度；
- pickup 模板；
- 事务搜索；
- 原子 Get/Put 和路由；
- 台位终态验证；
- replay；
- 结果和 CLI。

`terminal_depot_ok()` 位于 `solve.py:2992`，复杂度 46，且与 physical、replay、placement 再次重复业务规则。复杂度专项在 Stage3 检出 45 项告警；最新 mixed-gate 函数复杂度 36。

建议拆分：

- `PlacementCompiler`
- `AssignmentContext`
- `PhysicalTransitionKernel`
- `TransactionGenerator`
- `SearchSession/PortfolioScheduler`

值得保留的结构：

- placement 有独立有限域；
- Hall 缺集和外库容量 witness 可复算；

- partial 不输出可执行操作，安全边界清晰；
- Stage3 当前没有虚假声明全局最优。

## 8. Stage4 深度审查

### S4-01 P0/P1：多目标股道被一次性贪心冻结，容量检查漏判

`stage4_simple/scope.py:43-61` 调用 `physical.planned_target_for_car()`；后者位于 `physical.py:1096-1118`：

```python
target = min(targets, key=lambda line: (loads[line], line))
```

这里的 `loads` 是车辆数 Counter，不是长度；逐车选择后也不更新。随后 `scope.py:66-94` 的 capacity holdout 只处理 `len(options)==1` 的唯一目标车辆。

已复现：10 辆 14.3m 车辆均可去存1/存2：

```text
target counts = {'存1线': 10}
存1 assigned length = 143.0m
存1 capacity = 113.0m
infeasible_nos = 0
```

存2有足够容量，但所有车辆仍被固定到存1。另一个两辆 30m、候选抛丸/油漆的例子也会全部选到 42.3m 的抛丸线而不报 overflow。

这会造成假不可行、额外换位或错误 holdout。目标选择是业务优化决策，不应放在物理层。

整改：对多目标车使用最小费用流、CP-SAT/MIP 或小规模 assignment frontier，约束真实长度容量，并把后续预计勾数纳入费用。

### S4-02 P1：checkpoint 签名没有包含完整未来语义

`optimizer.py:421-440` 的 `_signature()`：

- stack 只编码 `(line, owner, nos)`；
- lease 只编码 `(line, owner, nos)`；
- lease 的 `gate_footprint` 被遗漏；
- stack 的 segments/ranks/protected/分段边界也没有显式编码。

但：

- `domain.py:39-75` 的 rank/protected 决定能否 prepend；
- `planner.py:335-337`、`construct.py:979-981` 使用 lease gate footprint 排除暂存线。

不同 gate footprint 的 lease 可以得到同一 optimizer 签名，却拥有不同后继集合，违反标签去重的基本要求。

整改：签名必须覆盖所有影响未来合法边、成本和恢复义务的字段；或者把 stack/lease 规范化为可从物理状态唯一重建的 canonical form，并用不变量测试证明。

### S4-03 P1：PlanBuilder 的部分 stack 消费会留下幽灵 lease

两套实现行为不同：

- `planner.py:431-444`：只在整个 lease.nos 都被 move 覆盖时删除 lease；部分消费会保留原完整 nos；
- `construct.py:1482-1507`：部分消费后会把 lease.nos 正确裁剪为剩余车辆。

例如 lease 为 `(A,B)`，只取走 A：

- PlanBuilder 仍保留 `(A,B)`；
- SourceWindowGenerator 保留 `(B)`。

幽灵 A 会继续占用恢复义务和 gate footprint，影响后续 staging 候选和 complete 判定。这是策略内核重复后已经产生的实质漂移。

### S4-04 P1：主线强制闭合 carry，episode 再删除主线冗余

证据：

- 初始 rehook checkpoint 必须闭合：`optimizer.py:59-68`；
- source successor 留 carry 会被拒绝：`optimizer.py:120-123`；
- target planner 拒绝开放 carry：`planner.py:1110-1123`；
- 每个 complete 都执行一次 1-label episode：`optimizer.py:94-100,133-139`；
- 最佳 complete 再执行 8192-label episode：`optimizer.py:152-157`；

这不是 fallback，因为不是失败后切换备用 solver；它是每个 complete 都会进入的第二优化层。但它表明主构造器的“每个 session 必须闭合”与最终追求的 open-carry 最优结构不一致。

最小复现：一辆已经满足、停在存4的车，Stage4 仍产生：

```text
Get 存4
Put 存4
```

共 2 勾、零净迁移。

2464 -> 2225 勾
85 案改善

### S4-05 P1：PlanBuilder 与 SourceWindowGenerator 是两套重叠策略内核

两者分别实现：

- apply/restore；
- stage；
- staging_candidates；
- target_dirty；
- place_tail；
- stack/lease 管理。

主要位置：

- `stage4_simple/planner.py:219-526`
- `stage4_simple/construct.py:611-1029,1482-1535`

两套候选约束、排序和 lease 更新不同，source-window 与 target-window 对同一暂存决策采用不同政策。S4-03 已经证明这不是纯代码重复，而是规则漂移。

### S4-06 P1：Stage4 的时间和 expansion 不是统一硬边界

- `max_expansions` 主要由 `PlanBuilder.probe()` 检查；

### S4-07 P1：出库重挂职责放在 residual closure，阶段边界不自然

但这些车辆和门区动作属于 Stage2—3 的大库开放窗口。把它推到 Stage4 会导致：

- Stage2 自报完成，但关键重挂动作未真正闭合；
- Stage4 即使没有 residual debt，也可能被迫 Get/Put；
- 下游 episode 再删除不必要闭合。

应明确该重挂是 Stage2 终态动作、Stage3 起始合同，还是一个共享 depot-window session，只能由一个阶段负责一次。

### S4-08 P1：优化目标没有完整表达业务目标

业务文档 `docs/福州调车业务文档.md:283-289` 的优先级是：

1. 总勾数；
2. 道岔扳动次数；
3. 路径总代价。

Stage4 `SearchCost` 位于 `search.py:17-24`：

```text

```

repeated Get 和 target reopen 是有价值的代理，但没有真实建模道岔扳动次数；Pipeline 也没有统一累计全流程目标。人工计划对门区开放、封闭和连续作业的权衡，不能只由这些代理完全表达。

值得保留的结构：

- source/target window 的显式建模；
- admissible lower bound 与排序估计分开；
- 完整候选重新从初态执行物理转移和 replay；
- episode contraction 有可观测记录；
- 没有冻结响应或旧 solver 兜底。

## 9. fallback 判定

### 9.1 没有发现的传统 fallback

没有发现以下模式：

- 主 solver 失败后调用旧 solver；
- 失败后切换历史响应；
- 失败后切 profile/案例特判；
- replay 失败后忽略错误继续返回 complete。

`latest_safe_response` 是安全返回最后一个通过 replay 的部分结果，不属于求解 fallback。

### 9.2 实际存在的隐式策略分层

|阶段|结构|准确称呼|
|---|---|---|
|Stage1|多个重叠候选恢复家族，内部再分 direct/staged/retained/spill/support variants|隐式候选 portfolio|
|Stage2|局部 episode 只保留一个代表，并用提前返回压缩其他变体|隐式候选压缩|
|Stage3|模板×placement×四档预算，多轮从头重算；预算后换成缩减邻域|重启式 portfolio + budget-triggered degradation|
|Stage4|主构造完成后，每个 complete 必跑 episode，最佳解再跑大 episode|常规二次优化层，不是 fallback|

问题不在于这些策略一定错误，而在于它们没有统一 SearchSession、统一状态等价、统一预算和统一诊断，容易掩盖真正的主线弱点。

## 11. 为什么高勾数与人工仍有差距

本次审查支持用户的判断：**主要差距是全局联合优化和主线结构，而不是人工对边界或对位要求不严格。**

人工计划本身就是对位位置和严格边界的来源，算法不应通过放宽人工规则来获得优势。当前差距更可能来自：

1. **Stage1 只看有限候选窗口并永久提交单步**，不能同时考虑后续编组线顺序、称重、门区和 Stage4 收口；
2. **Stage2 把多个未来不同的 episode 压成一个局部最优代表**；
3. **Stage3 placement 与事务路径分开优化，且只保留局部最小事务**；
4. **Stage4 主线要求频繁闭合 carry，后处理只能在既有访问 skeleton 上删动作**；
5. **阶段之间只传一个终态，没有累计勾数和下游成本反哺**；
6. **各阶段目标函数使用不同代理，未统一建模道岔动作和全流程总成本**；
7. **状态、候选和业务合同不一致会让搜索在错误空间上优化**。

因此，“LNS 应在主线全部优化之后进行”这个方向是正确的。当前应先把 mainline 变成可靠、统一、可持续加深的搜索；否则 LNS/episode 只是在删除主线制造的冗余，无法修复错误阶段边界、错误目标分配和被状态键剪掉的方案。

## 12. 建议目标架构

```text
API Request
   |
   v
Canonical Input Contract
   - line aliases
   - loco line + End
   - immutable car state
   - global deadline
   |
   +--------------------+
   | Shared Rule Specs  |
   | topology/capacity  |
   | depot/position     |
   | closed-door/weigh  |
   +--------------------+
   |
   v
Shared Atomic Transition Kernel <----> Independent Replay Executor
   |
   v
Stage1 SearchSession
   - complete StateKey
   - generator registry
   - unified objective
   - bounded beam/frontier
   |
   v  small Pareto handoff
Depot Window Joint Optimizer (Stage2 + Stage3 contracts)
   - unwheel/store4 decisions
   - terminal placement
   - resumable transaction frontier
   |
   v  small Pareto handoff
Stage4 Open-Carry SearchSession
   - one SessionKernel
   - source/target intents
   - exact target assignment
   |
   v
Final Physical Replay + Final Business Gate
   |
   v
LNS / learning-based ranking polish
```

这里不要求立刻把四阶段合成一个巨型搜索。更现实的做法是：

- 保留业务阶段；
- 每个阶段输出 4—16 个非支配边界状态；
- 用累计 `(总勾数, 道岔动作, 路径代价, 重复取车)` 排序；
- 下游快速 lower bound 或短时真实求解反向评价上游终态；
- 最终只对少量候选做完整求解。

## 13. 整改顺序

### P0：先保证搜索空间和完成判定正确

1. **建立完整 CanonicalState/StateKey**：包含 `_Weighed`、loco End、carry、业务位置和必要 assignment 版本。
2. **修复 Stage1 PlanKey 和去重**：包含 Weigh 与 planned positions，消除生成顺序依赖。
3. **统一业务规则真源**：physical、Stage3、replay 使用同一规则规格并做 parity test。
4. **建立最终业务 gate**：公共 API complete 必须通过独立 final validator。
5. **处理 loco End 合同**：真正建模，或在输入处拒绝 South。
6. **建立 300 秒 global deadline**：所有阶段、宏、预处理和回放共用。
7. **把多目标线路分配移出 physical**：使用容量感知的联合 assignment。

### P1：优化 mainline，而不是继续增加补丁宏

8. **Stage1 改为有 frontier 的搜索**：先用现有候选做 beam/best-first，不必立即重写全部生成器。
9. **Stage2 返回非支配 episode 集并保留 timeout incumbent**，撤销虚假 optimal 标签。
10. **Stage3 改为可恢复 SearchSession**：四档预算续搜；所有宏统一 budget；取消预算后缩减邻域。
11. **Stage3 放宽“只保留局部最小事务”**：保留不同终态/committed set 的非支配事务或有限 cost band。
12. **Stage4 合并两套 SessionKernel**，修复 lease canonical update 和完整签名。
13. **Stage4 把 open carry 纳入主搜索**，让 episode/LNS 只做小幅 polish。
14. **Pipeline 传递小规模 Pareto handoff**，建立跨阶段累计目标和下游成本反馈。

### P2：清理结构与诊断

15. 统一 summary schema、hook 口径、默认参数和 aggregate；
16. 删除 Stage3 永远为 0/false 的遗留优化字段；
17. 抽取 case I/O、response 合并、机车终态和诊断公共层；
18. 对每个生成器、事务族、标签剪枝和预算建立可消融统计；
19. 拆分 Stage1/Stage3 God Object，但避免只做机械文件切割，必须同时明确状态、合同和搜索职责。

## 14. 是否需要强化学习或更强求解方法

### 14.1 当前不建议优先使用强化学习直接控制动作

原因：

- 状态键当前不完整；
- 完成判定有多个真源；
- reward 没有统一表达全流程目标；
- 候选空间受生成顺序和局部压缩影响；
- 训练数据中的人工计划是一个可行解，不一定是每个局部决策的唯一最优标签。

在这种基础上直接上 RL，模型可能学会利用判定缺口，或把当前启发式偏差固化成策略。

### 14.2 更适合先使用的方法

|子问题|优先方法|
|---|---|
|多目标股道/台位分配|最小费用流、CP-SAT、MIP、matching|
|Stage1 主线顺序|bounded beam、A*/best-first、多标签动态规划|
|Stage2 episode|小规模精确枚举 + 非支配 transducer|
|Stage3 placement+事务|CP-SAT placement + resumable multi-label search|
|Stage4 block flow|open-carry label search + 统一 SessionKernel|
|跨阶段联合|Pareto frontier、下游 lower bound、Benders/Lagrangian 风格分解|
|主线完成后的改善|LNS、destroy/repair、局部重排|

### 14.3 学习方法适合放在后面做什么

当状态、合同、目标和 oracle 都稳定后，可以使用：

- imitation / learning-to-rank：学习人工更偏好的候选顺序；
- supervised cost prediction：预测某个 Stage1/3 handoff 的下游真实勾数；
- contextual bandit：分配不同构型和邻域的搜索预算；
- GNN/Transformer heuristic：为 beam/A* 提供排序，不直接决定合法性；
- RL：只在严格物理环境和独立 final gate 内做高层 session 决策。

学习模型应只影响排序和预算，物理合法性、状态转移和最终业务 gate 必须保持确定性。

## 15. 验收指标

### 15.1 正确性

- 称重前后 StateKey 必须不同；
- North/South 要么产生不同合法状态，要么 South 在入口被拒绝；
- 同 PlanKey 的候选选择与生成器注册顺序无关；
- physical/placement/replay 对所有台位规则 parity 100%；
- API `Success=True` 的结果 final business errors 必须为 0；
- 多目标分配不允许生成容量超限的冻结 assignment；
- stack/lease 同签名状态必须拥有相同后继集合；
- 所有超时返回保留完整 incumbent，并单独报告最优性未证明。

### 15.2 主线质量

按 truth2、truth3 和人工计划做配对统计：

- 四阶段 complete 率；
- 总勾数 mean/median/P90/P95/max；
- 算法相对人工的逐案差值；
- 高勾案例中重复 Get、临时 Put/Get、target reopen、门区重开占比；
- 强制对位和业务边界 100% 通过；
- 道岔扳动次数和路径代价；
- Stage1 handoff、Stage3 placement 对 Stage4 勾数的边际影响。

主线优化后，episode/LNS 的节省量应明显下降。可以设置管理指标，例如：polish 平均节省低于主线总勾数的 3%，且不能承担可解性修复。

### 15.3 预算和证明

- 全流程实际墙钟时间不超过 300 秒；
- 所有宏、assignment、scope、replay 都纳入同一 deadline；
- 每个结果区分 `feasible_complete`、`search_complete`、`optimality_proven`；
- 每个剪枝都有可验证的支配条件；
- 小规模案例与穷举 oracle 的勾数完全一致；
- iterative deepening 追加预算时不得重复展开已完成工作。

### 15.4 可诊断性

每个阶段至少输出：

- 候选/边 generated、deduplicated、validated、accepted；
- 各拒绝原因和耗时；
- incumbent 变化；
- budget/deadline 使用；
- frontier 大小；
- generator/transaction/episode 消融统计；
- mainline 与 polish 各自节省勾数。

## 17. 最终判断

当前系统不是“只差一个更强搜索器”，而是需要先完成一次主线架构收敛：

1. 把状态、物理、业务和完成判定统一；
2. 把 Stage1 从单步贪心改成真正保留替代路径的主搜索；
3. 把 Stage2/3 的局部压缩和重启 portfolio 改成非支配、可恢复的 SearchSession；
4. 把 Stage4 的 open carry 放回主线，并合并两套策略内核；
5. 让 Pipeline 在阶段边界保留少量 Pareto 终态，实现全流程联合优化；
6. 最后再用 LNS 和学习模型做排序、预算分配和小幅 polish。

如果按这个顺序推进，现有大量业务规则、测试、placement witness、stable closure、物理 transition 和 episode 审计都可以保留并复用；不需要推倒重来，也不建议继续针对单个高勾案例增加新的专用宏。
