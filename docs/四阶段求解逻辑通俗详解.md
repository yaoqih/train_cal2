# 四阶段调车求解逻辑通俗详解

> 本文以当前仓库中的 `scripts/stage1_simple`、`stage2_simple`、
> `stage3_simple`、`stage4_simple` 和 `plan_api/pipeline.py` 为准。
> 它解释的是代码实际执行的模型，不是对旧版 README 或人工调车流程的转述。

## 1. 先用一句话理解四个阶段

四阶段不是四个互不相关的求解器，而是一条逐步收缩问题的流水线：

1. **Stage1：先分类、预编组、打开前场通路。**
2. **Stage2：处理卸轮翻库和大库出库，把待释放车辆一次性收集到存4线。**
3. **Stage3：把去修库的车辆分配到具体库线和台位，并完成真实入库。**
4. **Stage4：处理前三阶段留下或明确延后的合法前场尾项、强制位置、脏目标线和多源重建。**

可以把它想成整理一间堆满箱子的仓库：Stage1 先按去向分堆，Stage2 把仓库里要出的箱子清出来，Stage3 把要入库的箱子放到精确货位，Stage4 再解决那些需要倒堆、借通道、恢复挡物才能完成的复杂尾项。

```text
原始请求
   |
   v
输入归一化、线路图和修库位预分配
   |
   v
Stage1 预编组 ---- replay 安全门禁
   |
   v
Stage2 卸轮/出库 ---- replay 安全门禁
   |
   v
Stage3 修库入库 ---- replay 安全门禁
   |
   v
Stage4 前场尾项闭合 ---- replay 安全门禁
   |
   v
根据累计 Operations 重新回放，生成最终对外响应
```

同一个案例的四阶段**不能并行执行**。Stage2 的起点是 Stage1 的真实回放终态，Stage3 又依赖 Stage2 结束后的车序、线路占用和机车位置，Stage4 的 active/protected 划分更是从 Stage3 终态动态计算。多个独立案例可以由 API 工作进程并行，但单案例内部必须严格串行。

## 2. 阅读本文前必须理解的几个概念

### 2.1 车辆状态不是只有“在哪条线”

每辆车至少有以下求解信息：

- `No`：唯一车号；
- `Line`：当前线路；
- `Position`：在线路上的次序或业务台位；
- `Length`：车长；
- `TargetLines`：允许的最终目标线路集合；
- `RepairProcess`：厂修、段修等修程；
- `IsHeavy`：是否重车；
- `IsWeigh`：是否要求过磅；
- `IsClosedDoor`：是否关门车；
- `ForceTargetPosition`：业务强制位置或允许位置窗口。

求解器不是把车号从一个列表搬到另一个列表。它必须同时保证线路、次序、长度、台位、路径、牵引和业务属性都合法。

### 2.2 “债务”表示本阶段还欠哪些工作

本文中的“债务”不是财务概念，而是尚未满足的调车责任。例如：

- Stage1 债务：还没有进入正确预编组线的车，以及污染预编组边界的错误车辆；
- Stage2 阻断债务：尚未完成的 required 卸轮任务和已选择的存4收集任务；允许延期项属于 handoff 诊断，不阻止 complete；
- Stage3 债务：尚未进入正确修库线路/台位的车，以及必须恢复的上下文车；
- Stage4 债务：所有仍在本阶段范围内、尚未满足最终目标的 active 车。

每个阶段都有自己的完成边界。因此 Stage1 complete 不表示全案已经完成，只表示“交给 Stage2 的边界已经合格”。

### 2.3 原子动作、planlet、episode 和 session

- **原子动作**：对外 Operations 中的一行，即 `Get`、`Put` 或 `Weigh`。
- **planlet**：由若干原子动作组成、可以整体做物理试演的局部计划。
- **episode**：Stage2 处理一条源线时的局部闭合搜索过程。
- **session**：Stage1 rolling 或 Stage4 target/source window 中的一段复合调车过程。
- **checkpoint**：一段 session 结束后的可保存搜索状态。

这些词的共同点是：求解器不会只看下一步能不能走，还会考虑一组动作结束后是否留下可解释、可恢复的状态。

### 2.4 active、protected、restoration

- **active**：当前阶段真正允许并需要处理的车辆；
- **protected**：阶段开始时已经满足，最终不能被破坏的车辆；
- **restoration**：本身不是本阶段业务目标，但为了够到后面的车或打开库门不得不临时移动，结束时需要恢复的车辆。

Stage3 主要使用 `task + restoration`，Stage4 主要使用 `active + protected`。两种表达方式都在解决同一个根本问题：不能只移动目标车而无视挡在它前面的车。

## 3. 四阶段共同遵守的物理模型

共同规则契约由 [physical.py](../scripts/solver_vnext/physical.py)、各阶段自己的 transition/yard 状态机和 [replay_validator.py](../replay_validator.py) 共同落实。Stage1/Stage4 直接使用公共 planlet 转移，Stage2 有自己的 Yard 转移，Stage3 复用公共拓扑但维护自己的执行状态；最终都要接受 replay 验收。任何启发式排序都不能绕开本阶段转移和累计回放。

### 3.1 线路是有方向的栈，不是任意数组

在当前模型中，`Position` 越小越靠可达端。假设某线从可达端向里依次是：

```text
Position:   1    2    3    4
车辆:       A    B    C    D
```

合法 Get 可以是 `[A]`、`[A,B]`、`[A,B,C]`，但不能直接取 `[C]`，也不能取 `[A,C]`。这就是“只能取可达端前缀”。

如果机车先 Get `[A,B]`，再去另一条线 Get `[E,F]`，挂车序列变成：

```text
[A, B, E, F]
```

后一次 Get 的车追加到挂车尾部。Put 时只能摘挂车尾部的精确后缀，所以可以先 Put `[E,F]`，然后再 Put `[A,B]`；不能在仍挂着 `[E,F]` 时先摘 `[A,B]`。

这个栈式顺序是四阶段求解难度的核心来源。很多看起来“把车送过去就行”的目标，实际上需要先取挡车、临时放尾组、再取目标车、最后按反向顺序恢复。

### 3.2 `TrainCars` 是动作后的完整挂车序列

每个对外操作行都包含：

```json
{
  "Index": 1,
  "Line": "某线路",
  "Action": "Get",
  "MoveCars": ["A", "B"],
  "TrainCars": ["A", "B"],
  "PassbyPath": ["...", "某线路"]
}
```

`MoveCars` 是本动作涉及的车，`TrainCars` 是本动作完成后机车后方的**全部车辆**。回放器会逐行重新计算挂车序列，并要求它与 `TrainCars` 完全一致。

### 3.3 普通 Put 与业务位置 Put

普通线路 Put 后，新放车辆成为可达端前缀。例如原线路是 `[C,D]`，Put `[A,B]` 后通常得到 `[A,B,C,D]`。

修库、修库外和有强制作业位置的线路不能总是只按连续下标处理。在支持显式业务位置的动作中，主要是 Stage3/Stage4 的修库、修库外或 spotting Put，Operations 会带 `Positions`；Stage2 的库外 Put 仍使用有序栈和长度模型，不输出该字段。显式位置动作必须证明：

- 每辆 MoveCars 都有位置；
- 位置不重复；spotting 线按放车顺序严格递增，其他显式位置 Put 通常还要求相邻位置连续 `+1`；
- 不与既有车冲突；
- 不能把新车凭空插到既有车的不可达深处；
- 强制位置、修库容量和修程规则都满足。

因此“车列顺序”和“业务台位”相关但不总是同一个概念。

### 3.4 牵引上限

牵引使用等效量而不是简单车数：

```text
普通车 = 1
重车   = 4
总上限 = 20
```

每次 Get 都检查“原有挂车 + 新挂车辆”的总当量。18 辆普通车再挂 1 辆重车会达到 22，当场拒绝；不是等到最终输出时才发现。

### 3.5 路径与占线约束

线路通过真实拓扑图连接，机车不能在图上瞬移。路径搜索至少考虑：

- 当前机车位置；
- 源线和目标线允许的进出端；
- 有静止车辆的线路通常不能作为中间通路；
- `联6`、`联7`、`渡1-13` 是运行线，能通过但不能作为 Get/Put 停车目标；
- `联6` 通行长度包含 15 米机车，限制为 192 米；
- 列明的折返三元组要检查机车、挂车和部分阻挡车的总长度；
- 目标线剩余长度必须容纳放入车辆，允许 0.5 米容差。

例如“图上连通”不等于“当前可达”。一条最短路径如果穿过停有静止车辆的门槛线，必须换路；没有替代路时，只能先把挡车搬开并留下恢复责任。

### 3.6 过磅规则

`Weigh` 不是任意服务动作。当前实现要求：

- 只能在线路 `机库线`；
- 一次恰好一辆；
- 被称车辆必须是当前挂车尾车；
- 车辆确实要求过磅且尚未过磅；
- 机库线不能有不属于当前挂车的静止车辆。

过磅成功后内部 `_Weighed` 状态变为 true。Stage1 和 Stage4 可以生成 Weigh；Stage2 不生成 Weigh；Stage3 遇到自己负责且仍待过磅的车会提前给出 partial，而不是在入库过程中偷偷补称。

### 3.7 修库与关门车业务规则

公共业务层还会检查：

- 修库内容量默认 5 个台位，检查模式为 7；
- 长度 `>= 17.6m` 的长车只能进入修3或修4；
- 厂修车只能位于 Position 4/5；
- 段修车不能排在厂修车更深处；
- 已合法停在修库中的 locked stayer 不能被破坏；
- 存4线上的关门车最终 Position 必须大于 3；
- 其他线路放车时，若整列第一辆为关门车，则大于 10 辆或含重车的整列禁止放入；
- 强制位置和 spotting 连续窗口必须满足。

## 4. 阶段之间如何传递状态

### 4.1 增量 response 与 combined response

Stage1 的 `response` 就是当前累计计划。Stage2、Stage3、Stage4 都同时维护两种响应：

- `response`：只包含本阶段新增 Operations；
- `combined_response`：从 Stage1 到当前阶段的全部 Operations，重新连续编号。

API 管线向下一阶段传的是累计链路：Stage2 接 Stage1 response，Stage3 接 Stage2 combined response，Stage4 接 Stage3 的 request/response/combined 三类产物。

### 4.2 API 链路不盲信 `GeneratedEndStatus`

API 管线以及 Stage2/Stage3 的交接都会从上一阶段 Operations 重新 replay。它关注的不只是上一阶段自报的终态，还会重算：

- 每辆车最终线路和位置；
- 挂车是否清空；
- 机车最终站位；
- 每步路径、牵引、容量和栈序；
- GeneratedEndStatus 是否和操作回放一致。

这相当于把上一阶段当成一个需要验收的承包方，而不是相信它口头说“已经完成”。有一个独立调用细节需要说明：Stage4 构造器初始化时只把 Stage3 replay 的 schema/physical 当作直接异常，随后会采用上游终态；正常 API 场景中，Stage3 累计 response 已先经过 pipeline 的 state 一致性门禁，所以该前提成立。绕过 API 单独调用 Stage4 时，调用方必须自己保证上游产物已完整验收。

### 4.3 API 的累计 replay 门禁

[pipeline.py](../plan_api/pipeline.py) 在每个阶段结束后，都用**原始请求 + 当前累计 response**重新回放：

1. response 缺失：failed；
2. operation replay 返回任何 schema/physical/state 等违规：failed；
3. replay 通过：更新 `latest_safe_response` 和 `last_safe_stage`；
4. summary.status 不是 complete：停止下游；
5. 只有 complete 才更新 `completed_stage` 并启动下一阶段。

所以 `completed_stage` 和 `last_safe_stage` 含义不同：

- `completed_stage`：最近一个业务边界 complete 的阶段；
- `last_safe_stage`：最近一个累计 Operations 已通过 replay 的阶段。

例如 Stage4 输出一个未清完债务但物理闭合的 partial checkpoint，管线 replay 通过后，`last_safe_stage` 可以是 4，而 `completed_stage` 仍是 3。

这里的 pipeline gate 只调用 operation replay，并不单独调用完整的 `business_errors()`。目标是否完成、修库/阶段合同是否满足，仍由各阶段自身验证和 `summary.status` 决定；累计 gate 负责证明“返回的操作链能按声明状态安全重放”，不能替代阶段业务完成判定。

## 5. Stage1：预编组和前场解锁

代码入口：[stage1_simple/solve.py](../scripts/stage1_simple/solve.py)。

### 5.1 Stage1 真正负责什么

Stage1 的硬边界只有两类合同：

- 去卸轮线的车辆，先编入 `存4线`；
- 去修1-4库内/库外的车辆，先编入 `机南/洗油北/机走棚/机走北` 四条大库编组线之一。

Stage1 不负责把大库车直接送到具体修库，修库内外和运行线是禁止操作边界。初始已经位于卸轮线的车不再形成 Stage1 卸轮预编组主债务，但卸轮线本身并非绝对禁止来源，服务或 rolling 上下文仍可能触及。待过磅的 Stage1 合同车必须先完成过磅，才算本阶段完成。

还有一个容易忽略的纯度要求：

- 存4线只能留下 Stage1 的卸轮合同车；
- 四条大库编组线只能留下 Stage1 的修库合同车。

如果目标车都进了编组线，但某条编组线上混入不属于该合同的车，Stage1 仍然不能 complete。

### 5.2 Stage1 债务模型

`stage1_debt()` 返回四个核心信息：

```text
pending_stage1_nos       尚未进入正确预编组边界的合同车
pollution_nos            五条边界线上不该出现的污染车
blocked_g_count          因前缀挡车而暂不可直接取得的合同车数
lines_with_pending...    仍有 Stage1 债务的来源线
```

完成条件可以写成：

```text
Stage1 complete
  = pending_stage1_nos 为空
  AND pollution_nos 为空
```

`blocked_g_count` 不直接进入 complete 公式，但它是判断候选是否真正打开局面的重要进度指标。

### 5.3 候选动作从哪里来

每轮求解器会同时生成多类候选，而不是只找“最近的一辆目标车”：

| 候选族 | 通俗解释 |
|---|---|
| 直接编组 | 从源线可达前缀取同类车，优先送存4或四条编组线；必要时从初始源线先落安全缓存，但禁止反复临线横移 |
| 前场直送 | 不损害主合同的情况下，把干净可达的服务车直接送真实目标 |
| rolling session | 连续多次 Get/Put，保留部分挂车跨来源合并同一目标流 |
| 前缀清障 | 搬走压在合同车前面的非合同车 |
| 路径清障 | 显式清理取车进路占线；放车阻挡主要由 put-unlock、release-gate 等候选间接处理 |
| 边界清污 | 把错误停在存4或四编组线上的车清走 |
| gate release | 编组车占住后续通路时，暂时释放门槛线 |
| 强制位置/目标重建 | 主合同完成后，对前场服务目标做受限重排 |
| 缓存与恢复 | 闭合宏中的临时尾组会当场恢复；普通 blocker relocation 也可形成安全 handoff 缓存，留给后续阶段继续处理 |

临时线不是随便挑空线。求解器会优先考虑真实目标、同一 Stage4 目标的聚集效果、是否污染干净服务线、是否挡住尚未处理的 Stage1 源线，以及是否至少仍有后续推进路径。普通安全缓存不承诺一定在 Stage1 内取回；rolling/目标重建等闭合 planlet 才要求宏内完成恢复。

### 5.4 rolling session 为什么比普通 Get-Put 强

普通候选大致是：

```text
Get(source A) -> Put(target)
```

rolling session 可以做：

```text
Get(source A 的一段)
Put(当前挂车尾部不属于主流的段)
保留主流车辆
Get(source B 的同目标段)
Put(合并后的目标段)
```

它内部使用受限束搜索，状态保存当前车辆布局、挂车顺序、每段来源/目标、已执行步骤、机车位置、重复取车数和各类收益。当前主要边界包括最多 12 步、beam 24、有限 Get/Put 分支、最多一次 re-get，并只输出少量结构上最好的闭合候选。

无论内部走了多少步，最终必须：

- 挂车清空；
- 每个 Put 都是尾部精确后缀；
- 没有破坏 Stage1 边界；
- 服务质量和强制位置不能无理由倒退；
- 整个 planlet 能通过公共 physical 验证。

### 5.5 候选不是生成后就直接提交

候选选择大致经过四层：

```text
生成候选
  -> 公共 physical.validate_planlet 硬过滤
  -> fork 一个求解器状态进行完整试演
  -> 检查债务、循环、路线阻塞和有限前瞻
  -> 按上下文评分选择一个
  -> 才提交真实 Operations
```

试演会拒绝：

- 任何物理违规；
- 回到已见过的车辆/机车状态；
- 复合动作既不降债也不解锁；
- 没有上下文收益的横向搬运；
- Stage1 债务大幅反弹；
- 走完候选后 1-2 层有限前瞻没有可继续动作；
- rolling endpoint 违反其明确的服务/强制位置单调边界。

主阶段中，一般的服务质量或 Stage3/4 下游布局退化主要进入评分并被降权，不是通用硬拒绝。service finish 才对指定服务指标实施硬单调保护。待称重车也不会进入 rolling carry；普通或其他 planlet 会在待称尾车 Put 前由物理构造器自动插入 Weigh。

### 5.6 Stage1 的评分不是单一距离

Stage1 不是全局最短路搜索，也没有一个简单的“距离最小”分数。它先判断当前属于哪种上下文：

- `get_unlock`：重点是降低取车进路的占线压力；
- `put_unlock`：重点是打开目标或路径；
- `debt_window`：重点是直接减少阶段债务。

然后按字典序比较完成、解锁、债务变化、服务收益、路径质量和稳定 tie-break。大意上：

1. 一步完成 Stage1 最好；
2. 高价值路径解锁、强制位置改善次之；
3. 能显著降低阻挡或债务的正常推进再次；
4. 只有解锁但暂不降债的动作需要更强证明；
5. 无收益或债务反弹的动作排在最后并常被拒绝。

它还计算下游质量，但这只是**选优项**，不是 Stage1 complete 的硬条件：

- Stage3：大库目标分组碎片、run 数、前缀阻挡；
- Stage4：可达债务、弱下界、尾债、目标碎片、已稳定南端尾组；
- 前场服务：强制位置满足数、南端连续数、已满足数。

这就是为什么两个都能完成 Stage1 的候选中，求解器会偏好让 Stage3/4 更容易接手的布局。

为限制单轮试演成本，上下文窗口默认最多检查 140 个候选，收集到 20 个有效候选或 8 个可直接完成的有效候选后可以提前停止。主债务期间的前场直送还最多使用 5 个 planlet。这些是搜索宽度限制，不是业务规则。

### 5.7 主合同完成后的 service finish

当 pending 和 pollution 都清零后，Stage1 可以再做少量前场收尾，例如整线重建、同线 repack、坏后缀抽取后恢复、service-only rolling 或干净目标直送。

这部分受到严格单调保护：

- 每一步之后 Stage1 主合同仍必须 complete；
- 强制位置、已满足车辆和南端连续性不能倒退；
- 收益必须足以覆盖新增 Get/Put；
- 当前实现最多 4 个 planlet、12 行 Get/Put，并要求 `2 * weighted_gain >= business_hooks`。

因此 service finish 是“主任务完成后的有收益顺手优化”，不是为了追求服务目标而重新打乱编组边界。

### 5.8 Stage1 停止与 partial

默认边界是 80 个内部 planlet、300 秒。还会因以下情况停止：

- 编组总长度证明容量不足；
- 连续 30 步债务和阻挡均无改善；
- 连续 35 步没有刷新历史最好窗口；
- 没有合法候选；
- 到达时间或 planlet 上限。

此时只要 debt 尚未清零，summary 就是 partial。已经提交的每个 planlet 都经过物理验证，API 还会对累计结果再 replay，所以 Stage1 partial 可以携带一段安全但未完成的 Operations。

需要注意统计口径：Stage1 summary 中 `hooks/move_batches` 是接受的 planlet 数，`business_hooks` 才是 Get/Put 操作行数，两者不能直接比较。

## 6. Stage2：卸轮翻库和一次性存4收集

代码分布在 [stage2_simple](../scripts/stage2_simple/) 的 `model.py`、`yard.py`、`episodes.py`、`optimizer.py` 和 `solve.py`。

### 6.1 Stage2 解决的两个问题

Stage2 从 Stage1 回放终态接管：

1. 把必须卸轮的车送入 `卸轮线`，容量允许时可顺带处理目标为油漆线的 optional 卸轮车；
2. 从修库内/外、卸轮线、存4线抽取待释放车辆，形成一个 collector，最终一次性放入存4线。

它的重点不是修库入库，而是把“大库内部出车”和“卸轮翻库”闭合后，为 Stage3 准备清晰的存4出发列。

### 6.2 车辆任务分类

`Stage2Problem` 将车编译成以下职责：

| Duty | 含义 |
|---|---|
| `FINAL4 / OFF` | 非 C4 的出库收集车，必须进入存4 collector |
| `FINAL4 / C4` | 明确以存4为目标的收集车 |
| `UNWHEEL` | 必须进入卸轮线 |
| `OPTIONAL_UNWHEEL` | 油漆车，容量允许时先进入卸轮线 |
| `DEFER` | 合法留给后续阶段，例如某些库外目标或已在卸轮的油漆车 |
| `RESTORE` | 普通 episode 中只是挡车，临时移动后需恢复；显式走廊搬迁属于另一个已建模分支 |

OFF/C4 的区别不仅用于统计，还决定最终 collector 的结构。

### 6.3 Stage2 的边界状态

全局 `BoundaryState` 只保留四类关键信息：

```text
completed_mask    已闭合源线的 bitmask
yard              全场线路、挂车和机车状态
collector         挂车前部持续收集的存4列
open_unload       挂车尾部尚未落入卸轮线的卸轮组
```

每个 episode 的边界要求：

```text
held == collector + open_unload
```

这里的“闭合 episode”不一定表示挂车为空，而是表示挂车被规范地分成“前部 collector + 尾部 open_unload”，普通临时挡车和 scratch 副作用已经恢复，或者变化已经被显式的 corridor relocation / absorbed source 决策吸收进新边界，能够安全进入下一条源线的 episode。

### 6.4 局部 episode 怎么处理一条源线

对某条源线，局部搜索可以：

1. 取到最深业务车之前的可达前缀；
2. 把尾部卸轮组放入卸轮线；
3. 把尾部 DEFER 组送到共有的合法库外目标；
4. 普通情况下把纯阻挡车按原顺序恢复源线；
5. 借其他修库外线作为 scratch，再精确取回；
6. 也可走显式 corridor relocation，把整条库外线永久搬到另一合法库外线；
7. 对满足条件的 scratch 源线，可把整组吸收进 collector 并直接标记该源线完成；
8. 最终验证所有变化符合 source 边界以及 collector/open_unload 分区。

局部层不是贪心规则表，而是以完整 `YardState` 做标签搜索。同一个状态只保留更小的字典序代价。每个无 scratch、某条 scratch、可选 absorbed 等局部搜索变体最多扩展 12000 个状态；一次 `plan()` 可依次尝试多个变体，所以累计 `episode_expansions` 可以超过 12000。

### 6.5 为什么全局是“单调 DAG”

一条业务源线完成 episode 后，其 bit 从 0 变 1，之后不再回退到未完成。这使全局搜索形成单调有向无环结构：

```text
未处理任何源线
  -> 闭合源线 A
     -> 闭合源线 B
        -> ...
           -> 所有源线闭合
```

每条全局边内部调用 EpisodePlanner 生成一个规范后继。全局最多扩展 100000 个状态/标签，并与局部搜索共享同一个 wall-clock deadline；已发现但尚未扩展的 queue/best 项理论上可以更多。

这里的“optimal”有更窄的明确边界：EpisodePlanner 会在无 scratch、不同 scratch 和 absorbed 变体中只返回一个按自身规则选出的代表终态，全局层并不会接收该 choice 的全部合法 episode 终态。因此它表示在 **EpisodePlanner 实际产生的单一规范后继图** 内最优，不是对现实中所有可能的任意调车序列证明全局最优。

### 6.6 存4最终提交合同

Stage2 有三条求解层专属硬规则：

1. 存4线最多 Put 一次；
2. 这次 Put 必须是 Stage2 最后一条操作；
3. 新增存4段必须符合 `OFF*C4*`。

`OFF*C4*` 表示先是零辆或多辆 OFF，之后才是零辆或多辆 C4。一旦 collector 中已经出现 C4，后面不能再追加 OFF。

此外，C4 子序列前三辆不能是关门车。最后一次 Put 还必须按存4南侧合同进入。Stage2 当前调用的 `rv.replay()` 不执行完整最终 `business_errors()`，而存4逐步关门车校验也有意留到终态；所以该前三辆规则由 Stage2 的 `collector_closed_door_ok()` 和终端合同直接保证。“一次且最后一次 Put”“OFF*C4*”同样是 Stage2 自己额外保证的策略边界。

### 6.7 延期不是求解失败

如果所有 FINAL4 车辆牵引当量超过 20，OFF 不能延期，只允许从 C4 中选择最少数量的 holdout：

1. 先最小化延期 C4 数；
2. 再减少原始源线顺序的 inversion；
3. 还必须让剩余 C4 保留满足关门车前部规则的可能性，最终 collector 顺序再做一次终态检查。

optional 卸轮车也按卸轮线剩余长度选择最大可容纳集合，等价于先最小化延期数量，再最小化顺序破坏。

这些延期是模型明确允许的 handoff，因此存在延期车不自动导致 Stage2 partial。真正的 required 任务、已选择 optional 任务和最终 collector 都完成，仍可是 complete。

### 6.8 Stage2 的七维目标

全局按以下元组严格字典序最小化：

```text
(
  延期 C4 数量,
  延期 optional 卸轮数量,
  操作行数,
  恢复时重复搬动 DEFER 车辆次数,
  经过联7的操作数,
  总路径节点数,
  OFF 相邻目标键切换次数
)
```

这意味着“少延期”比“少几勾”优先级更高。只有前面的维度相同，才比较后面的路径和碎片质量。

### 6.9 Stage2 complete 与 partial

complete 必须同时满足：

- 全局搜索到达所有源线闭合状态；
- 所选 collector 全部在存4；
- required 和已选 optional 卸轮车全部在卸轮线；
- 单次最终存4 Put 合同成立；
- Stage2 独立 replay 通过；
- Stage1+2 combined replay 通过；
- 两条 replay 的线路顺序一致；
- 内部 Yard 终态与 replay 终态一致。

搜索无解、超时、状态上限、回放违规或状态投影不一致都会得到 partial。Stage2 不会把一个尚未闭合的局部 episode 暴露给外部；没有完整全局结果时，增量 response 通常为空，combined response 退回到安全的 Stage1 累计链路。

## 7. Stage3：修库目标构型与物理入库

代码入口：[stage3_simple/solve.py](../scripts/stage3_simple/solve.py)。

### 7.1 Stage3 不是简单地“把四条编组线清空”

Stage3 要同时回答两个问题：

1. **目标构型问题**：每辆车最终去修1、修2、修3还是修4，进入库内还是库外，放在哪个台位？
2. **物理执行问题**：按照当前挂车尾部顺序、库门占用和线路可达性，怎样真正把这个构型摆出来？

如果只做第一个问题，可能得到纸面合法但无法按尾部顺序卸下的布局；如果只做第二个问题，搜索空间又会被大量等价台位组合撑爆。因此 Stage3 明确分成“先分配、后对齐”的两层。

### 7.2 Stage3 的输入门禁和起机合同

Stage3 首先对 Stage2 combined response 做操作 replay。它会阻断 replay 返回的 schema、physical、逐步 business 或 state 违规；`rv.replay()` 本身不运行通用最终 `business_errors()`，Stage2 的业务边界主要依赖上游 `summary.status=complete` 和 Stage2 自身终态检查。

Stage2 最后一项业务通常是把出库 collector 放到存4线。当前 replay 对 `存4南 -> 存4线` 的特殊 Put 会把机车位置记为 `存4线`。Stage3 仍无条件把自己的起点设为内部线路键 `存4线`，用它表达阶段边界上的北侧重新连挂假设；这不是 response 中额外生成的一行重挂 Operation。

因此 summary 同时记录：

- `stage2_final_loco`：Stage2 操作真实推导出的终点；
- `stage3_start_loco`：Stage3 合同规定的重新连挂起点。

当前常见产物中两项通常同为 `存4线`。保留两个 summary 字段是为了区分“上游操作推导值”和“本阶段合同起点”，而不是声称 replay 一定先得到存4南。

### 7.3 task、active 和 restoration 的区别

`task_nos` 是 Stage3 的业务任务车，主要包括：

- 四条编组线上的修库目标车；
- 存4线上只有修库外目标的车；
- 需要从库内出到库外的车；
- 当前在卸轮线或某库外、需要换到另一库外的车；
- 位于库内/库外/卸轮线、最终要去油漆线或存4线的 Stage4 延后车；
- 已在目标修库但台位不合法的车；
- 已在目标库外但强制位置不合法的车。

但是物理上不能只把这些车拿出来。求解器对 task 做“可达前缀和库门闭包”，形成 `active_nos`：

1. 某任务车前面的可达端前缀全部纳入；
2. 某车要进库内，对应库外门线上的车辆全部纳入；
3. 某个库外目标被触碰，其上下文车辆纳入；
4. 一旦某条 Stage3 来源线被触及，该线所有车辆都保守纳入 active 求解范围；
5. 只要本阶段有任务，卸轮线车辆也纳入完整上下文。

然后：

```text
restoration_nos = active_nos - task_nos
```

restoration 车只是为了够到目标车或打开库门而移动。它们最终必须回到原线路；真正位于被取前缀中的 restoration 还要求恢复原始次序/位置。active 是保守求解闭包，不表示其中每辆车都会被移动，实际 pickup 仍只取到最深 task 所需前缀。未纳入 active 的车辆成为 fixed，仍然占用线路长度、影响路径并保护既有稀疏台位。

### 7.4 第一层：库内最小费用 slot assignment

对于要进修库内的车，Stage3 构造“车辆 -> 合法台位”的二分图：

```text
车辆节点 --合法边--> (修库线路, Position) 节点
```

只有同时满足以下条件才有边：

- 线路在车辆 TargetLines 中；
- 位置在该库容量内；
- 满足 ForceTargetPosition；
- 长车只去修3/4；
- 厂修只去 4/5 位；
- 不撞 fixed 台位；
- 不越过无法移动的深位 fixed 车辆；
- 对应库外门线不存在不可移动的 fixed 门车；active 门车能否真正搬开留给执行层证明。

然后使用精确最小费用流完成一车一槽匹配。费用表达的是业务偏好而不是最终勾数，例如：

- 厂修偏深位；
- 长车偏修3并兼顾浅深位置；
- 普通车偏修1/2；
- A 模板第二趟暴露的洗油北车辆会避免形成不利分配；
- slot 排名只作为确定性 tie-break。

如果车辆数大于所有可达 slot 数，会输出明确的容量不可行证书。这个证书能证明当前明显容量不足，但它不是对任意二分图子集都做完整 Hall 定理证明。

`assigned_slot_by_no` 需要谨慎解读：对单一强制位置，它是执行层必须绑定的硬 slot；对没有强制位置或存在多个允许位置的车，它主要是匹配可行性和深浅对齐用的名义 slot，实际 Put 可以选择另一个仍然合法且能从门口插入的位置。最终位置事实应以 Operations replay 和 `GeneratedEndStatus` 为准。

### 7.5 cost 布局与 cohesive 布局

Stage3 不只尝试一种目标构型。

**cost 布局**主要最小化每辆车到 slot 的独立业务偏好成本。它擅长找到“每辆车单看都放得合理”的精确匹配。

**cohesive 布局**沿模板实际暴露顺序做动态规划，优先：

1. 减少目标线路连续段数，也就是减少理论上必须拆成多少个 Put 段；
2. 少占用后面还要打开的库门；
3. 再比较 slot 偏好成本。

举例：暴露顺序是 `[A,B,C,D]`，如果目标分配为 `[修1,修2,修1,修2]`，至少形成四个交替段；分配为 `[修1,修1,修2,修2]`，通常只需两个尾部段。两种都可能台位合法，但后者更容易真实卸车。

cohesive 还会禁止形成“厂修车在浅处、段修车随后被迫放到更深处”的暴露顺序。它若报告 `cohesive_direct_unload_order_infeasible`，只表示这类凝聚布局不可行，不代表 cost 匹配一定无解。

### 7.6 库外目标和 Stage4 延后车如何分配

库外线路没有修库内那样固定 5/7 位，但有 49.3 米长度、门线冲突和强制位置。Stage3 用带容量的回溯分配：

- 尽量保留已经合法停放的车辆；
- 避免把新车塞到已有 stayer 的线路；
- 避免占用本阶段即将使用的对应库内门线；
- 强制库外位置必须做唯一匹配；
- 线路子集的需求长度超过总容量时输出证书。

最终目标为油漆线或存4线的 Stage4 延后车，在 Stage3 不会假装已经最终满足。Stage3 只负责清出库门并把它们受控地 staging 到 `卸轮线` 或修库外，必要时按最终 Stage4 目标减少临时块数。真正的目标重建留给 Stage4。

### 7.7 第二层：A/B 取车模板

目标布局确定后，还要决定四条编组线如何形成挂车暴露顺序。

**模板 B：单批统一暴露**

```text
先取存4线合同前缀
再依次取：机走北 -> 机走棚 -> 洗油北 -> 机南
随后统一做目标对齐和卸车
```

**模板 A：洗油北分成第二批**

```text
先取存4线合同前缀
第一批：机走北 -> 机走棚 -> 机南
挂车清空
第二批：单独取洗油北
```

Put 只能从挂车尾部开始，所以“目标被处理的暴露顺序”是取车拼接顺序的反向。模板 A 可以降低单次携带量并改变深位车的暴露时机，但同一修库目标跨两批出现时，至少需要两个 Put epoch。

每条来源并不是机械整线全取，而是取到最深 task 所需的可达前缀。前缀中的 restoration 车辆随同移动，并在终态恢复。

### 7.8 Greedy 规格不是一条固定规则链

Stage3 会组合评估以下规格：

- 模板/布局：`B-cohesive`、`A-cohesive`、`B-cost`、`A-cost`；
- 有 Stage4 deferred 车时：提前清理或不提前清理；
- terminal compatible 尾组：允许合并或不合并；
- 清库门策略：`eager` 或 `just_in_time`。

没有 deferred 车时基础为 16 个 greedy 规格，有 deferred 车时为 32 个。每个规格都在同一状态和硬约束模型内运行，不是某个失败后切换到一套不相干的旧求解器。

Greedy 最多推进 120 个状态，大致按以下次序寻找闭合宏：

1. eager 模式先清可能阻挡的库内/库外门线，JIT 模式只清当前尾组需要的门；
2. 尝试完整库门事务；
3. 把已经满足深浅顺序的尾组直接放入修库；
4. 放置不会制造新库门债务的终态组；
5. 把遮住下一辆关键目标车的尾组放到合法 buffer；
6. 可选合并 assigned target 不同、但能合法共享同一实际终态或 staging 线路的混合尾组；
7. 最后再尝试普通 Put、取库外挡车或取库内挡车。

### 7.9 原子库门宏为什么重要

假设目标车要进修2库内，但修2库外停着门车 X。一个不完整策略可能先把 X 搬走，却发现 X 无处恢复，导致状态比开始更糟。

`greedy_clear_gate_macro` 会先在不可变副本上试演一个原子“清门并提交目标”的事务：

```text
Get 修2库外门车 X
Put X 到合法暂存或永久位置
Put 当前目标尾组到修2库内
```

只有“清门 + 当前目标入库”整组都可行才提交；第一扇门车的剩余恢复债务可以继续保存在状态中，但必须在 Stage3 complete 前闭合。两扇库门互相依赖时，还可以做嵌套 swap：清第二门、换出、完成一组入库、取回换出车、按原稀疏位置恢复，再完成第一门目标；只有这种 nested 路径显式核验第二门块恢复原稀疏位置。这类宏减少了“局部每步合法、整体却走入死路”的问题。

### 7.10 小状态精确搜索和大状态有界改进

当 `active_count <= 6` 时，Stage3 会在固定模板/布局/clear 起点后运行代码所称的 exact 小状态显式搜索。队列带 `remaining_put_lower_bound` 和多项 tie-break，更接近受限 best-first/A*-like 搜索，而不是纯 Dijkstra，也不表示已穷举现实全局空间。邻居仍然只有合法的：

- 挂车尾部 Put；
- 库内/库外/卸轮线的可达前缀 Get；
- 阻挡库门所需的前缀动作；
- A 模板 phase 切换。

默认上限 300000 expansions，并受全局 180 秒预算切片限制。

大案例先依靠 block greedy 得到 complete incumbent。只有当某个下界距离 incumbent 不超过 2 勾时，才额外投入总计约 0.05 秒、600 expansions 做有界改进。这样把精确搜索用在可能真正省勾的邻域，而不是让大状态空间无限膨胀。

### 7.11 Stage3 的四维代价

每个非 Phase 动作增加：

```text
(
  1,
  路径是否经过联7,
  PassbyPath 节点数,
  本次 Get/Put 的作业线路是否为修库外
)
```

累计后严格字典序最小化，所以优先级是：

1. 最少 Get/Put 操作；
2. 最少经过联7；
3. 最少总路径节点；
4. 最少以修库外为作业线路的 Get/Put。

Stage3 不接手待称重业务，因此第一维实际就是业务 Get/Put 数。同成本候选再偏好 cohesive、deferred clear、terminal merge、eager 和模板 B。

### 7.12 下界和“最优”该怎样解读

固定模板/布局下界由以下部分组成：

- 必须访问的来源线数；
- 目标库内线及 A/B epoch 导致的必要 inner Put；
- 按允许终点和长度计算的最少非库内 Put 线路数；
- 某些必然产生的前沿重搬债务。

精确模式还使用一个更松、与具体 assignment 无关的下界；搜索队列用 `已用操作 + remaining_put_lower_bound` 排序。

summary 的 `search_space_lower_bound_reached`、`best_known_with_gap` 等，只针对已经声明并评估的模板/布局/策略空间。即使 gap 为 0，也不能扩张成“对所有现实调车序列证明全局最优”。

### 7.13 Stage3 complete 的多重验证

内部 complete 只是第一关，要求：

- 挂车为空；
- A 模板已经进入第二 phase；
- 四条编组线上不再有 task；
- 所有 active 车位于其类别允许的合法终态线路；inner/outer 可接受任一合法目标，deferred 可接受任一 Stage4 staging 线，不要求再次严格等于名义 assigned line；
- restoration 回到原线，必要者恢复原位；
- fixed/locked 未破坏；
- 库内/库外位置、长车、厂修/段修和库门关系全部合法。

完整候选随后再做：

1. 构造独立 Stage3 request；
2. replay 无 GeneratedEndStatus 的操作；
3. 由 replay 结果生成 GeneratedEndStatus；
4. 再 replay 并检查 Generated 状态；
5. 比较内部状态投影与 replay 投影；
6. 将操作追加到 Stage2 combined 后对原始请求 replay；
7. 两条链都追加 Stage3 自己拥有的最终业务检查。

任何 schema、physical、business、state 或 terminal 违规都会把 complete 候选降为 partial。

### 7.14 Stage3 partial 为什么不输出尝试前缀

Stage3 可能在 trace 中已经尝试很多动作，甚至某些动作物理上闭合，但只要整个 Stage3 边界没有 complete，对外增量：

```text
Operations = []
GeneratedEndStatus = []
```

combined response 只保留 Stage1+2 的安全累计计划。`attempted_operations` 和 trace 仍用于诊断，但不会把半个入库布局交给 Stage4 或外部执行。

## 8. Stage4：闭合 session 上的 block-flow 标签搜索

当前 Stage4 已经是多模块实现，不再是旧版单文件启发式。主要代码位于 [stage4_simple](../scripts/stage4_simple/)：

| 模块 | 职责 |
|---|---|
| `scope.py` | 冻结 active、protected、范围外和容量留置 |
| `domain.py` | OwnedStack、RecoveryLease、合同状态 |
| `topology.py` | 目标线路依赖的 gate 资源闭包 |
| `model.py` | 动态 block、访问挡块和 target window 诊断 |
| `search.py` | 目标秩、真正物理转移和四维代价 |
| `contracts.py` | 存4/卸轮/油漆重挂合同与目标合同图 |
| `construct.py` | source-window 闭合边生成 |
| `planner.py` | target-window 和复杂拼列合同 |
| `episode.py` | complete 方案的开放 carry 线路访问骨架最短投影 |
| `optimizer.py` | checkpoint 级统一标签搜索 |
| `solve.py` | 整体物理证书、独立 replay 和 combined replay |

### 8.1 Stage4 先冻结求解范围

Stage4 从 Stage3 终态调用 `physical.unsatisfied_cars()`，把车辆分成：

| 集合 | 含义 |
|---|---|
| `active_nos` | 本阶段范围内、可行动且要解决的最终债务 |
| `protected_nos` | Stage4 开始时已满足，最终不得破坏 |
| `out_of_scope_nos` | 目标为空、目标为修库/卸轮或目标不在当前物理表 |
| `excluded_source_nos` | 来源还停在联/渡运行线上，不由 Stage4 直接取 |
| `infeasible_nos` | 唯一目标线发生硬长度溢出后计算出的最少留置车 |

每辆 active 车会通过 `planned_target_for_car()` 选择并冻结一个目标。后续不能因为搜索困难就随意切到另一条目标线；protected 车则保留原本合法目标集合，用于判断临时移动后是否恢复满足。

### 8.2 容量 holdout 的证明范围

对于只有一个 Stage4 合法目标线的车辆，若全体总长度超过线路长度，Stage4 枚举最少 holdout：

1. 留置车辆数最少；
2. 同样数量时，移除长度超过缺口的浪费最小；
3. 再按原 Position 总和和车号稳定排序。

这些车从 active 中移除，但会在 summary 的 infeasible debt 中继续报告。

这个证书只针对“唯一目标线”的硬容量冲突。多目标车辆是否能通过重新分配解决，不由这段 holdout 逻辑证明，因此不能把它解释成全场通用容量最优器。如果当前 active 候选的总长度本身不足以覆盖溢出，函数会退化为留置全部 active；这时 holdout 数也不是对整条线恢复可行的完整证明。

### 8.3 最终秩与 owner 顺序

spotting 目标不仅要求车在线路上，还要求相对位置和强制窗口合法。Stage4 预计算：

- `final_rank_by_no`：通过公共目标位置规划得到的目标物理秩基准，可能包含空号，不保证连续；
- `owner_order_by_target`：某个 spotting 目标最终希望看到的车辆 owner 顺序。

source-window 在建立 spotting 临编栈时，会再按 owner 顺序把相关车辆压缩成 `1..N` 的**逻辑连续栈秩**。这样物理台位中的合法空号不会被误判成“两个段不相邻”，同时不同搜索标签各自维护 rank ledger，不会相互污染。

必须区分：物理 Position 用于最终业务校验，逻辑连续秩用于证明临时栈能按正确顺序拼接。

### 8.4 FlowModel 是可重建的诊断投影，不是真实搜索状态

`FlowModel` 可以从给定真实车辆状态重建连续 Block。当前 `solve.py` 只在输出 summary 时针对 **Stage4 初始状态** 构建一次 `flow_relaxation`，optimizer、planner 和 construct 不消费这些估计。只有以下属性全部相同的相邻车辆才合并：

- 目标 owner；
- 当前是否仍为 active pending；
- 是否 protected；
- 强制位置集合；
- 是否重车；
- 是否仍待过磅。

每个 Block 还记录总牵引当量和长度。模型本身支持从新状态重新构建，不会把 block 当成可修改的真实车辆状态；但当前主调用链没有在每次 frontier 转移后重算它。

对每个 active block，`AccessAlternative` 记录它前面的挡块。对每个目标，`TargetWindow` 记录：

```text
existing           目标线上现有车序
inbound            需要从其他线进入的车
outbound           目标线上需要出去的车
source_runs        inbound 按来源线形成的有序段
source_inversions  来源暴露顺序与目标位置顺序的相邻逆序
capacity_rounds    按 20 当量粗算的批次数
estimated_hooks    来源/目标/逆序的解释性工作量
```

其中一个 target window 的估计式是：

```text
estimated_hooks
  = inbound 所在来源线数
  + (目标线已有车且有 inbound ? 1 : 0)
  + (存在 inbound 或 outbound ? 1 : 0)
  + source_inversions
```

`capacity_rounds` 虽被单独记录，却没有加进这个 estimated_hooks。整体 `relaxed_hook_estimate` 则是不同来源线数、不同目标线数、dirty window 数和 inversion 数之和。这也进一步说明它们是解释性统计，不是严密操作下界。

### 8.5 两种“下界”绝不能混淆

FlowModel 的 `estimated_hooks` 和 `relaxed_hook_estimate` 当前只用于 summary 解释和诊断，不参与 frontier 排序。源码明确说明它们**不是可采纳最优性下界**，因为不同 target window 可能共享一次 Get 或保留同一挂车，简单相加会重复计数。

真正标为 admissible 的 `hook_lower_bound` 是：

```text
剩余债务所在的不同来源线数
+ 剩余债务涉及的不同目标线数
+ 尚未过磅车辆数
```

如果没有剩余债务但仍挂着车，则至少还需一次 Put。这个下界很安全，但比较弱；`实际 hooks - lower_bound` 只能表示相对弱下界的 gap，不能单独证明全局最优。Stage4 summary 写 `feasible_unproved` 或 `not_proved`，不会声称 optimal。

### 8.6 OperationTransitions 是唯一真实动作系统

Stage4 所有构造器最终都必须通过 `OperationTransitions`。每一个 Get/Put/Weigh 都立即调用 `physical.transition_plan_step()`：

```text
构造器提出 PlanStep
  -> 公共物理层计算路径和状态转移
  -> accepted 才产生 successor
  -> rejected 则该搜索边不存在
```

所以可达前缀、挂车尾部、20 当量、线路容量、进出端、占线绕行、折返长度、Positions、关门车和过磅条件都是**实时硬过滤**，不是先生成错误计划、最后才靠 replay 补救。

### 8.7 Stage4 精确四维目标

`SearchCost` 按以下元组严格字典序比较：

```text
(
  hooks,
  repeated_gets,
  target_reopens,
  route_cost
)
```

含义是：

- `hooks`：每一行 Get/Put/Weigh 都加 1；
- `repeated_gets`：本次 Get 中，每辆以前已经 Get 过的车各加 1，是车辆次数，不是操作次数；
- `target_reopens`：某目标已在完整债务、恢复和 gate 义务下被证明 `SEALED`，后来又从该线
  Get；普通目标 Put 只把窗口置为 `OPEN`，不计语义重开；
- `route_cost`：每步增加 `max(0, PassbyPath 节点数 - 1)`，路径中每出现一个 `渡10/联7/渡9/渡8/渡7/渡4` 热点节点再加 2。

最少 hooks 永远优先于其他三项。只有 hooks 相同，才依次少重复搬车、少重开目标、少占热点和走短路。

### 8.8 OwnedStack：带 owner 的可恢复临编栈

当尾组暂时不能去真实目标时，Stage4 不只记录“它被放到了某条空线”，还建立 `OwnedStack`：

```text
OwnedStack(
  line = 临编线,
  owner = 最终目标或恢复责任,
  segments = 按可达端排列的车辆段
)
```

带有效逻辑秩的 owner 栈只允许严格相邻的更小秩段前插。例如已有 `[rank 4,5]`，可以在可达端前插 `[2,3]`，形成 `[2,3,4,5]`；不能插 `[1]`，因为中间缺 rank 2/3 的恢复证明。

从 OwnedStack 取车也只能消费可达前缀，可以在一个 segment 中间切开，但不能越过前部直接拿深处车辆。

这个模型避免把临时线当作“可以任意排序的内存数组”。

### 8.9 RecoveryLease：临时移动的退出凭证

为了打开目标路径，求解器可能临时移动 protected 车或 gate 阻挡车。`RecoveryLease` 记录：

- 当前 staging 线路；
- owner/恢复目标；
- 涉及车辆；
- 占用或保护的 gate footprint。

其中 `lease:<target>:<resource>` 专门表示“为了打开 target，搬走 resource 线路车辆”。其他 lease 也可表示普通临时 staging 的恢复责任。

标签允许 protected 车辆暂时不满足，但这种 damage 必须被某个 stack 或 lease 显式覆盖。若 checkpoint 中出现没有退出凭证的 protected damage，整个 successor 立即拒绝。

### 8.10 “闭合 checkpoint”到底闭合什么

Stage4 的 session closed 首先要求：

```text
当前挂车为空
```

它**允许**仍存在 OwnedStack 或 RecoveryLease，但必须作为标签状态的一部分完整保存，不能成为构造器内部忘记记录的副作用。

真正全局 complete 才进一步要求：

```text
挂车为空
AND active 全满足
AND protected 无损坏
AND stacks 为空
AND leases 为空
```

因此“闭合”不是“所有临时动作都已经恢复”，而是“没有匿名的半截挂车或未知副作用，剩余临编责任已经显式建模”。

### 8.11 合同图与存4/卸轮/油漆重挂

合同图首先建立 `DEPOT_OUTBOUND_REHOOK`，然后每个 active 目标建立 `TARGET_WINDOW:<line>`，所有目标合同都依赖 rehook 完成。

rehook 专门处理：

- 存4线已有骨干；
- 卸轮线中最终去油漆线的尾组；
- 为够到最后一辆油漆车必须随取的卸轮前缀挡车；
- 油漆线上已有车和需要出线的车；
- 合并后是否超过 20 当量。

据此分为：

```text
not_required
c4_only
direct
target_rebuild
outbound_dependency
prefix_dependency
batched
```

优化器建立正式 frontier 之前先完整 resolve 该合同，而且 rehook 后的首个 checkpoint 不允许留下挂车、stack 或 lease。episode 只冻结其中强制的获取前缀：先 Get 存4骨干，再 Get 卸轮线上必须进入油漆的前缀；窗口内后续临时 Put/Get 仍可重排。这样既保留业务顺序，又不把整个构造实现误当成不可优化合同。

### 8.12 target-window edge 做什么

每个标签会尝试一条 target edge。`ContractPlanner` 从 checkpoint 只推进一段闭合 session；复杂目标通常由 `ConsistAssembler` 完成：

1. 根据 owner 最终顺序确定想要的 consist；
2. 从多个来源取到目标车所需的可达前缀；
3. 当前挂车尾部若不是下一段目标车，就放到有恢复证明的 staging；
4. 目标线脏时先取出 existing，分离 outbound 和保留成员；
5. 在 20 当量内尽量让多个 target segment 共用一列；
6. 按挂车尾部反向 fan-out 到各自目标；
7. 关闭已满足的 target contract，留下无挂车 checkpoint。

若当前没有 complex target，target edge 还可能先执行一次 terminal `sweep_source()`，所以并非每条 target edge 都一定进入 ConsistAssembler。目标优先级精确考虑强制位置数量、目标是否脏、是否服务线、批量大小、来源数和目标名；拓扑依赖在选中后的资源释放/构造阶段处理。它不是固定按线路名称从小到大扫一遍。

### 8.13 source-window edge 做什么

同一个标签还会并列尝试若干 source edge。主要流程是：

1. 先尝试无需 staging 的 exposed spotting 目标直接闭合；
2. 若无直接闭合，选择一条高价值来源线；
3. 找到该线最深 active 车，确定必须消化的前缀；
4. 按 20 当量截断，并优先在 owner/rank 边界切块；
5. Get 可达前缀；
6. 从挂车尾部识别下一段意图；
7. 能直接送目标就送，能与另一来源形成同 owner 列就续挂；
8. 否则放到合法 staging，建立 OwnedStack/lease；
9. 待过磅尾车在合法时插入 Weigh；
10. 尝试关闭已经 ready 的 owner stacks，并在终局恢复临时栈。

每次 staging Put 还会预探测反向 Get，至少证明这个临编栈在当前局部条件下可以重新取回，避免把车放到一个立即不可恢复的死角。

### 8.14 gate 资源怎样释放和恢复

`topology.py` 描述目标线依赖的 gate，例如某些服务线必须经过洗油北、机走棚等资源。依赖会做传递闭包。

真正打开目标前，构造器会：

1. 找出当前被占用的候选 gate；
2. 枚举能打开目标的最小占用子集；
3. 逐线 Get 挡车；
4. 若挡车恰好属于同一个 active owner，可直接转换成持续 owner stack；
5. 否则建立 lease 并 staging；
6. 目标操作结束后按 lease 回收并恢复资源。

这让“清障”成为带进入和退出证明的事务，而不是只要当前路径能开就把挡车遗忘在别处。

### 8.15 target 和 source 不是 fallback 关系

对每个 checkpoint，优化器同时生成：

```text
一条 target-window 候选边
+ 若干 source-window 候选边
```

二者是同一标签图中的并列分支。不能描述成“target 失败后换 source 策略”，也不能说存在独立的 retry/profile/portfolio 求解器。

source 内部的选择也被编码成 choice vector。默认先走全零向量，然后从每个决策点派生替代向量，形成结构化分支，而不是随机重试。

### 8.16 顶层标签包含哪些状态

一个完整 checkpoint 保存：

- 全部车辆的 Line、Position、Weighed；
- 机车位置和当前挂车；
- 每条 OwnedStack 的 line、owner、车序；
- 每个完整 RecoveryLease，包括 gate footprint；
- 各合同的 pending/active/closed 状态；
- 临时 goal owner。

用于 dominance 去重的签名比 checkpoint 本身更简化：OwnedStack 只编码 `(line, owner, nos)`，不编码 segment ranks/protected；RecoveryLease 只编码 `(line, owner, nos)`，不编码 gate footprint。相同简化签名只保留更小的 SearchCost。因此 line/owner/nos 不同的 lease 一定不会合并，但单纯 gate footprint 或 segment 元数据不同，当前实现会被视为同一 dominance 状态。这是实现边界，不应表述成所有恢复元数据都参与去重。

此外，签名显式包含 `handled_nos`、目标 `OPEN/SEALED` 状态、carry 来源、合同状态和临时
goal owner。相同物理站场但重复摘挂历史、窗口风险或未来责任不同的标签不会互相支配。

### 8.17 frontier 优先级不是纯 A*

标签队列的精确优先级是：

```text
(
  已用 hooks + admissible hook_lower_bound,
  已用 hooks + ordered_block_estimate,
  lease 数,
  完整 SearchCost,
  剩余 active 数
)
```

`hook_lower_bound` 只有来源线数、目标线数和待过磅数，保证 admissible，可用于安全排序和
剪枝。`ordered_block_estimate` 进一步计算来源 owner word 的最少 run，只用于同层标签排序；
来源 word 可以拆分，因此它不能作为 admissible 下界。剩余 active 数只作最后 tie-break。

best partial 使用：

```text
(
  剩余 active 数,
  protected damage 数,
  lease 数,
  SearchCost
)
```

best complete 则只按完整 SearchCost 选最小。找到第一个 complete 后搜索仍可继续，直到预算或 frontier 边界，以寻找同模型内成本更低的 complete。

### 8.18 Stage4 的搜索边界

默认参数：

- 300 秒；
- 最多评估 128 个非 stale 外层标签；
- target planner 上限 30000 expansions。

source choice 使用结构状态边界，而不是失败驱动的自适应重试：

- 普通 checkpoint 最多评估 16 个 choice vector；
- 剩余 pending 非空且已全部记录在显式 stacks 时，使用外层 `max_labels` 做终局恢复；
- rehook 后初始状态若只有一条来源线且最大 owner-run 至少为 7，也使用外层预算，避免
  单来源深度换序被 16 标签截断；
- 多来源碎片状态仍保持 16，防止一个 source-window 吃掉整个全局预算；
- 不再固定只提交 4 条 source successor，所有非支配闭合后继都回到外层 frontier。

每个 complete 候选先做 1-label 的廉价成对事件规范化。主 frontier 结束后，只对最佳
incumbent 做一次 8192-label `OpenCarryEpisodeOptimizer`：每个既有 hook 是一个可跳过的
线路访问槽，可执行合法前缀 Get、尾部 Put 或尾车 Weigh；临时租约取消、source suffix
延迟和最高连续 final-rank 后缀直接提交都在同一图中比较。标签预算或时间耗尽会显式
记录，算法不会切换 fallback/profile。

外层停止原因只有：时间耗尽、label 预算耗尽或 frontier 耗尽。

`max_expansions=30000` 主要由 target-window 的 PlanBuilder probe 检查。source-window 的动作会累计进 checkpoint expansions，但当前实现并非对所有 source 原子动作逐步执行同一个统一硬上限。文档或监控中不能把它误写成“Stage4 所有动作展开严格最多 30000 次”。

### 8.19 optimizer complete、summary complete 与 actionable complete

优化器内部 complete 要求 active/protected/stack/lease 全闭合。对外 summary 使用另一套门槛，要求：

- `physical.unsatisfied_cars()` 最终为空；
- 因为全体 `final_unsatisfied` 必须为空，out-of-scope、excluded 和 infeasible 也不能遗留；
- 独立 Stage4 replay 没有硬违规；
- Stage1-4 combined replay 没有硬违规。

当前 `summary.complete` 布尔式没有显式把 `optimization.plan.complete`、stack 为空或 lease 为空再次并入，只会在 plan 未 complete 时追加 blocking reason，并另报 `unrecovered_lease_count`。正常 complete 搜索路径会同时满足两套门槛，但从纯代码定义看，它们是不同检查面，不能简单说 summary 门槛是 optimizer 门槛的严格超集。

所以常见的另一种结果是：

```text
stage4_debt.actionable_complete = true
summary.status = partial
```

典型原因是 active 都解决、protected 也没损坏，但唯一目标线容量不足的 holdout 仍然未满足。summary 会把这种容量受限尾项和真实搜索残债分开报告。

### 8.20 Stage4 partial 会保留最佳闭合 checkpoint

如果没有 complete，Stage4 不是一律返回空操作，而是选择 best partial checkpoint。它已经满足“挂车为空、所有临时副作用有显式状态”的 checkpoint 边界，然后：

1. 若 steps 非空，把全部 steps 重新合成一个完整 planlet；空步骤直接跳过该证书构造；
2. 再跑一次 `physical.validate_planlet()` 独立物理证书；
3. 生成对外 Get/Put/Weigh Operations；
4. 做 Stage4 独立 replay；
5. 追加 Stage3 combined 后对原始请求做全链 replay；
6. 重新生成 GeneratedEndStatus 和债务统计。

因此 Stage4 partial 可以包含一段安全、已闭合、但未清完所有目标的实际 Operations。API 管线先对这个 combined response 做 replay，若通过就会把它更新为 `latest_safe_response`，随后因 Stage4 summary 非 complete 停止。换言之，partial 仍可能成为最终对外返回的最近安全累计方案。

### 8.21 当前全量验证结果

冻结 truth2/truth3 共 140 个 Stage4 起点，原响应合计 2464 勾。对同一起点和同一完整
响应执行 8192-label episode 投影后：

| 指标 | 结果 |
|---|---:|
| 优化后总勾数 | 2225 |
| 平均勾数 | 15.893 |
| 减少勾数 | 239 |
| 改善案例 | 85 |
| 回退案例 | 0 |
| 标签耗尽案例 | 3 |
| 语义 SEALED 重开 | 0 |

140/140 均重新通过 planlet、Stage4 独立 replay、Stage1-4 combined replay 和强制重挂
前缀检查。把 3 个标签耗尽案例单独提高到 16384 后，`0104W` 仍为 30 勾，`0109W`
仍为 16 勾，`0225W` 仍为 27 勾，没有获得额外 incumbent。

必须区分两种审计：`audit_stage4_episode_optimizer.py` 固定已有完整响应，只证明访问骨架

## 9. 用一个简化例子串起四个阶段

下面是概念示例，重点展示责任边界，不对应某个固定案例路径。

假设有六辆车：

| 车辆 | 起点 | 最终目标 | 特殊属性 |
|---|---|---|---|
| A | 存5线北 Position 1 | 卸轮线 | 待过磅 |
| B | 存5线北 Position 2 | 修2库内 | 普通 |
| C | 存5线北 Position 3 | 修3库内 | 长车 |
| D | 修1库内 | 存4线 | 出库车 |
| E | 卸轮线 | 油漆线 | 普通 |
| X | 修2库外 | 修2库外 | 已满足挡车 |

### 9.1 Stage1

- A 的 Stage1 目标不是直接卸轮，而是先进入存4线；且 A 待过磅，需在合法挂车尾部到机库线完成 Weigh。
- B、C 不直接进修库，而是进入四条大库编组线之一。
- 如果 A/B/C 前面有非合同挡车，Stage1 先做前缀或路径清障。
- Stage1 complete 时，存4只留下卸轮合同 A，四条编组线留下修库合同 B/C，边界无污染。

### 9.2 Stage2

- A 从存4被送到卸轮线，完成 required unwheel。
- D 从修1库内被收集为 OFF 或 C4 类 collector 成员。
- E 若卸轮容量和模型选择允许，可以留作 optional/handoff；否则明确延期。
- collector 最后一次 Put 到存4，而且这是 Stage2 最后一条操作。

### 9.3 Stage3

- B/C 从编组线进入目标构型层。
- C 是长车，只能匹配修3/4的合法 slot，因此选择修3。
- B 分配到修2合法 slot，修2库外的 X 因库门闭包进入 Stage3 active，身份是 restoration，不是本阶段 protected。
- 执行层先原子保证清门与 B 入库，X 的恢复债务可以留在状态里继续推进，但 Stage3 complete 前必须回到合法原线/原位。
- D 已在存4且目标就是存4，本例中不是 Stage3 task，会保持不动。只有另有更深的存4库外任务车触发前缀闭包时，D 才可能作为 restoration 被重新挂取。

### 9.4 Stage4

- E 的最终油漆目标仍未满足，进入 active。
- Stage3 complete 后，已满足的 A/B/C/D/X 等在 Stage4 scope 中成为 protected。
- 如果油漆线已有要出车，先形成 dirty target window；如果洗油北/机走棚被占，还需 gate lease。
- source-window 取卸轮线到 E 所需前缀，非 E 尾组进入有 owner 的临编栈；target-window 重建油漆线。
- 最终恢复 gate、清空 stack/lease、挂车为空，E 到达合法油漆位置，全案 complete。

这个例子说明：同一辆车可能经历“预编组 -> 翻库/收集 -> 入库或 staging -> 最终闭合”，每阶段只承担自己能够证明安全的边界。

## 10. complete、partial、failed、unavailable、error 的区别

### 10.1 求解器阶段状态

| 状态 | 含义 |
|---|---|
| `complete` | 本阶段业务边界、内部终态和 replay 全部通过 |
| `partial` | 求解正常运行，但预算、无解或剩余债务使边界未完成 |
| `unavailable` | CLI 文件模式下，上游产物缺失或上游不是 complete |
| `error` | CLI 阶段包装捕获到程序异常 |

API 管线把阶段结果再归并成：

| API solve_status | 含义 |
|---|---|
| `complete` | Stage1-4 全部 complete |
| `partial` | 某阶段正常返回 partial，累计 response 仍通过 replay |
| `failed` | 阶段异常、response 缺失、replay gate 失败或非法阶段状态 |

### 10.2 各阶段 partial 的对外差异

| 阶段 | partial 时对外 Operations |
|---|---|
| Stage1 | 保留已提交且经验证的安全 planlet |
| Stage2 | 不暴露未闭合 episode；通常为空增量，combined 回到 Stage1 |
| Stage3 | 明确为空增量，combined 只保留 Stage1+2 |
| Stage4 | 可保留 best closed checkpoint，并再次做整体物理证书和 replay |

这四种行为不同，是各阶段状态边界不同导致的，不应该用一条“partial 一律清空”规则概括。

### 10.3 最终 GeneratedEndStatus 从哪里来

API 最终不会简单转发某阶段自报的终态。它取 `latest_safe_response` 的累计 Operations，从原始请求重新 replay，再生成：

```text
No + Line + Position
```

因此最终对外状态与实际返回的 Operations 保持同源。complete 时 `Success=true`；partial 时 HTTP 业务 StatusCode 仍为 200，但 `Success=false` 并说明停在哪一阶段；failed 时为 500 语义。

## 11. 硬约束、求解选择和诊断估计对照

这是阅读 summary 和调试结果时最重要的分类。

| 类型 | 例子 | 违反后的结果 |
|---|---|---|
| 物理硬约束 | Get 前缀、Put 尾部、20 当量、路径、长度 | 候选/转移直接拒绝，replay 也会失败 |
| 业务硬约束 | 修库台位、locked、关门车、强制位置 | 候选拒绝或终态不能 complete |
| 阶段合同 | Stage2 一次最终存4 Put、Stage1 边界纯度 | 阶段不能 complete，即使通用物理动作合法 |
| 搜索选择 | Stage1 上下文评分、Stage3 cohesive 偏好 | 只影响先选哪个合法方案 |
| 搜索剪枝 | label/expansion/time 上限、单调 DAG | 可能更快，也可能导致 partial 或未证明最优 |
| 诊断估计 | Stage4 relaxed_hook_estimate | 用于解释工作量，不能作为最优性证书 |
| 可采纳弱下界 | Stage4 source+target+weigh 数 | 可用于排序/gap，但到达与否不代表覆盖所有现实策略 |

## 12. 默认参数与性能边界

API 当前默认参数来自 [PipelineOptions](../plan_api/pipeline.py)：

| 阶段 | 默认参数 |
|---|---|
| Stage1 | `max_hooks=80`，`time_budget_seconds=300` |
| Stage2 | `time_budget_seconds=300` |
| Stage3 | `time_budget_seconds=180` |
| Stage4 | `time_budget_seconds=300`，`max_labels=128`，`max_expansions=30000` |

这些参数主要控制“愿意搜索多久和多宽”，不是放宽物理规则。提高预算可能让 partial 变 complete，或找到更少勾的方案；它不会允许超牵引、错误栈序或穿越占用线。

总预算还受 API 的全局上限约束。多个 API job 可由线程池监管并各自启动独立 worker 子进程并行，但每个 job 内的四阶段顺序不变。

## 14. 常见误解

### 误解 1：四阶段可以同时跑，最后拼起来

错误。后阶段依赖前阶段动态终态、机车位置、线路占用和车辆顺序。只能并行不同案例，不能并行同一案例的四阶段。

### 误解 2：Stage1 complete 表示所有车都到最终目标

错误。Stage1 只证明卸轮/修库预编组边界完成并且边界纯净。

### 误解 3：路径图连通就一定能走

错误。占线、进出端、机车和挂车长度、折返空间都会让图上路径当前不可用。

### 误解 4：临时线是任意缓存空间

错误。临时 Put 必须有容量、路径、栈序、反向可取和恢复责任；Stage4 还要求 owner stack/lease 显式入标签。

### 误解 5：Stage2 有延期就不是 complete

错误。最少 C4 或 optional 延期是模型允许的明确 handoff；required 合同都完成仍可 complete。

### 误解 6：Stage3 的 min-cost flow 已经解决整个调车问题

错误。它只解决目标 slot assignment。真实 Get/Put、库门事务和挂车暴露顺序由第二层完成。

### 误解 7：Stage4 relaxed_hook_estimate 是最优下界

错误。它会重复计算可共享动作，只用于诊断。真正的 admissible bound 是更弱的来源线数、目标线数和过磅数之和。

### 误解 8：找到 complete 就证明全局最优

错误。Stage1 是有限前瞻启发式；Stage2 只在单调 episode 模型内最优；Stage3 只在声明的模板/布局空间内给出下界和改进状态；Stage4 明确报告 feasible unproved。

### 误解 9：partial 就一定没有可执行价值

错误。Stage1 和 Stage4 可以返回 replay-safe 的闭合部分解；Stage2/3 则选择不暴露未完成内部计划。最终仍应依据 `Success/solve_status/completed_stage/last_safe_stage` 判断业务是否完整。

## 16. 最后总结

四阶段设计的核心不是把一个大问题机械切成四份，而是为每个阶段建立一个**可验证的交接边界**：

- Stage1 用预编组和边界纯度降低后续混杂度；
- Stage2 用单调 episode 和一次最终存4提交闭合出库/卸轮；
- Stage3 把目标 slot assignment 与真实库门/栈序执行分开，再用双 replay 合并证明；
- Stage4 用显式 owner stack、recovery lease 和闭合 checkpoint 搜索复杂前场尾项；
- API 在每次交接后从原始请求重放累计 Operations，保证对外返回永远来自最近一次通过 replay 的实际操作链。

真正的安全性来自“每步物理硬过滤 + 阶段终态业务检查 + 独立/累计 replay”三层共同作用；启发式、下界和搜索预算只决定能多快找到多好的合法方案，不能替代这三层证明。
