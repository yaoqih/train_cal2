# 第四阶段：剩余调车收口设计方案

## 0. 阶段定位

第四阶段只做一件事：在第一、第二、第三阶段已经完成大库相关车辆处理之后，完成所有剩余非大库车辆的归位，并以操作行数最少为第一目标。

本阶段不是新的全场调车器，也不重新处理大库入库、出库、台位分配问题。它是一个明确的收口阶段：

```text
阶段3终态
-> 计算剩余未满足车辆
-> 策略空间排除大库相关线
-> 对前场剩余车辆做最短路收口
-> replay + 全车满意度验收
```

本方案吸收前面数学化讨论里的结论，但不采用过强的闭式最优公式作为最终证明。第四阶段的数学核心是：

```text
带机后车列的多栈重排最短路
```

依赖图、段归约、反馈集下界用于压缩搜索和解释最优性；最终“勾数最少”由 Dijkstra/A* 在真实状态图上证明。

## 1. 已确认业务口径

| 事项 | 第四阶段口径 |
|---|---|
| 勾数定义 | `Get` 和 `Put` 各算一勾；`Weigh` 如果出现也算操作行，但第四阶段默认不应再出现称重债务 |
| 连续携带 | 允许。可以连续执行多次 `Get` 后再 `Put`，机后车列跨操作行保留 |
| 大库相关线 | 策略生成不包含修库内/外、卸轮线等大库相关线；不额外写复杂残债分支，最终是否完成交给验收函数 |
| 存4线释放 | 允许释放 `存4线` 上目标不是 `存4线` 的车；已经满足存4线目标的出段集结车不主动移动 |
| 联7 | 不封锁，可作为路径通过；但 `联7` 仍是走行线，不能作为 `Get` / `Put` 作业股道 |
| 称重残债 | 可顺路处理。只有带称重属性且未称重车辆在机后尾部时才允许 `Weigh(机库线)`；无法处理时 partial |
| 物理可达 | 必须通过图搜索和现有物理约束，不用纯拓扑假设替代 |
| 操作方向 | 所有停车/作业线仍按北端栈处理，不能从南端自由取放 |
| 验收标准 | replay 无硬物理错误，且 `physical.unsatisfied_cars` 为空 |

## 2. 阶段输入与输出

### 2.1 输入

第四阶段从第三阶段的阶段内输出继续，用第三阶段自己的 request/response replay 得到阶段3终态：

```text
artifacts/stage3_simple/<case_id>_stage3_request.json
artifacts/stage3_simple/<case_id>_response.json
```

读取原始 case 后，仍保留第三阶段 combined response 用于最终拼接：

```text

```

阶段3终态 replay：

```python
replayed, replay_bad = rv.replay(stage3_request, stage3_response)
final_cars = rv.final_cars(stage3_response, replayed)
```

如果第三阶段输出了 `GeneratedEndStatus`，需要按既有阶段3口径把它叠加到 `final_cars` 上。

第四阶段起始机车位置必须使用 `stage3_response` 的真实最终机车位置，不能凭空重连到前场。若业务允许阶段切换时重连，必须以显式配置和 summary 字段表达。

## 3. 债务定义

阶段4债务直接来自统一业务验收函数：

```python
unsatisfied = physical.unsatisfied_cars(cars, depot_assignment)
```

对每辆未满足车，用：

```python
target_line, target_position, reason = physical.planned_target_for_car(
    car,
    cars,
    depot_assignment,
    physical.line_loads(cars),
)
```

得到本阶段目标。

### 3.1 大库相关线排除

如果未满足车辆的目标属于：

```text
修1库内 / 修2库内 / 修3库内 / 修4库内
修1库外 / 修2库外 / 修3库外 / 修4库外
```

则第四阶段策略不为它生成 `Get` / `Put` / `cache` 候选。原因是这些线属于第二、三阶段使用的大库作业股道，第四阶段只做前场剩余收口；但路径仍可经过 `联7` 等走行资源。

处理方式：

```text
不进入 active strategy set
summary.out_of_scope_count 记录数量
最终 replay + unsatisfied_cars 自然决定 complete / partial
```

如果后续确实需要第四阶段兜底大库相关线，应另开配置开关或专项分支，不要污染主模型。

### 3.2 存4线释放债

若车辆当前在 `存4线`，但目标不是 `存4线`，则属于第四阶段债务。  
这些车允许被取出并送往真实目标线。

若车辆当前在 `存4线` 且目标就是 `存4线`，并且已经满足关门车终态规则，则视为固定车，不主动移动。

## 4. 数学模型

### 4.1 多栈状态

每条相关线路是一条北端栈：

```text
S_l = [c1, c2, ..., ck]
```

其中 `c1` 是北端可先取车辆。

机后车列：

```text
H = [h1, h2, ..., hm]
```

`H` 允许跨操作行保留。连续携带使第四阶段不能退化成简单 `move(i,j,k)` 闭合动作；状态必须显式包含 `held`。

完整搜索状态：

```python
State = (
    lines,       # 相关线路上的 active 车辆顺序
    held,        # 当前机后车列
    loco,        # 调机当前位置集合
    closed,      # 已完成且不再允许作为缓存的目标线
)
```

固定车不进入 `lines` 的 active 部分，但在容量和路径占用计算时仍然存在。

### 4.2 合法操作

状态图边对应一条 API 操作行。

#### Get

```text
Get(line, prefix)
```

要求：

1. `prefix` 是该线北端连续前缀。
2. `held + prefix` 的牵引当量不超过 20。
3. 关门车过程顺位合法。
4. 从当前 `loco` 到 `line` 的取车路径可达。
5. 取车后不违反北端剩车离开限制。

状态变化：

```text
S_line = prefix + rest
H      = H + prefix

=> S_line = rest
   H      = H + prefix
```

#### Put

```text
Put(line, suffix)
```

要求：

1. `suffix` 是 `held` 的尾部连续后缀。
2. 目标线容量足够。
3. 放入后不破坏强制位置和业务终态规则。
4. 从当前 `loco` 到 `line` 的送车路径可达。
5. 若 `line` 已经 `closed`，禁止 Put。

状态变化：

```text
H      = prefix + suffix
S_line = old

=> H      = prefix
   S_line = suffix + old
```

### 4.3 代价函数

第四阶段代价采用字典序：

```text
cost = (
  operation_count,
  hot_throat_count,
  route_length_cost,
  buffer_pollution_penalty
)
```

其中：

1. `operation_count`：每条 `Get`/`Put` 操作行 +1。
2. `hot_throat_count`：经过 `联7`、`渡10` 等核心咽喉加权；它们可以作为路径资源，但不能作为作业股道。
3. `route_length_cost`：路径节点数或静态路径代价。
4. `buffer_pollution_penalty`：把车放入非目标线、未 ready 目标线、或高风险缓存线的惩罚。

最优性只看第一维。后面三维只在操作行数相同的解之间排序。

## 5. 段归约

直接按单车枚举会产生过多无意义候选。第四阶段先做约束安全段划分。

### 5.1 初始段定义

同一线路上，北端顺序相邻，且满足以下条件的车合并为一个段：

1. 目标线相同。
2. 强制位置需求兼容。
3. 关门车/重车过程约束不会要求中间切断。
4. 累计牵引当量不超过 20。
5. 累计长度不会超过所有可能目标线容量上限。

段记录：

```python
Segment = (
    segment_id,
    car_nos,
    source_line,
    target_line,
    length_m,
    pull_equivalent,
    flags,
)
```

### 5.2 强制切分

以下情况必须切分，不能为了段数少而合并：

1. 牵引当量超过 20。
2. 目标线容量只可能接收部分车辆。
3. 称重未完成车辆位于段内部，不能作为机后尾车顺路称重。
4. 存4线关门车终态顺位需要单独控制。
5. 同目标但强制位置不连续或相互冲突。
6. 一段内既有已经满意车又有未满意车。

### 5.3 段归约的作用边界

段归约只用于生成候选和下界估计，不改变最终 replay 的车级真实性。  
输出操作仍然写具体车号列表。

需要避免过强承诺：

```text
“整段移动一定最优”
```

更准确的口径是：

```text
在约束不打断的最大同质段上，整段移动支配任意无目的拆分。
```

因此搜索可以优先枚举整段，也保留被容量、关门车、路径可达打断后的必要子段。

## 6. 依赖图与快速路径

### 6.1 线路依赖图

对每个未完成段建立有向边：

```text
source_line -> target_line
```

如果目标线自身还有出车债务，则该目标线暂时不是稳定接收线。

线路依赖图用于三件事：

1. 判断哪些目标线 ready。
2. 识别互换环和强连通分量。
3. 给搜索一个可采纳下界和可行上界。

### 6.2 Ready 目标

目标线 `T` ready 的条件：

```text
T 上不存在需要离开 T 的 active 段
且 T 容量足够接收当前候选段
且 T 不是 closed line
```

若 `T` not ready，Put 到 `T` 不是绝对禁止，但要高惩罚；除非搜索证明必须这样做，否则优先先清 `T` 的出车债。

### 6.3 无环快速路径

如果依赖图无环，且所有 ready Put 的物理可达和容量校验都通过，则按逆拓扑顺序搬运：

```text
先清没有外出依赖的目标线
再逐层把源线车辆送入 ready 目标
```

该路径通常能给出最优解或很紧的上界。  
但只有在以下条件同时满足时，才能直接声明最优：

1. 每个段只需一次 Get 和一次 Put。
2. 没有连续携带带来的跨源合并收益。
3. 没有容量导致的额外缓存。
4. 每条操作路径均可达。

否则无环快速路径只作为上界，仍由 Dijkstra/A* 证明。

### 6.4 环与反馈集下界

依赖图存在环时，至少需要缓存破环。  
令 `K` 为未就位段数，`fvs(D)` 为依赖图最小反馈顶点集大小，可得理想化下界：

```text
LB_fvs = 2K + 2 * fvs(D)
```

这个下界不直接等于真实最优值，因为真实问题还有路径、容量、连续携带和关门车限制。  
它的用途是：

1. 作为 A*/Dijkstra 的可采纳下界候选之一。
2. 作为 trace 中解释“为什么至少要这么多操作行”的证书字段。
3. 作为环处理构造的优先级。

## 7. 精确搜索

### 7.1 为什么仍需搜索

第四阶段允许连续携带，例如：

```text
Get 存5线北 [去预修]
Get 存3线   [去预修]
Put 预修线 [两批合并]
```

这比两个独立 `Get+Put` 少一条 `Put`。因此闭式 `2K + 2fvs` 只能做下界，不能直接作为最终最优公式。

同时，机车位置会影响下一条 `Get` 是否可达，不能完全从状态中删除。

### 7.2 Dijkstra / A*

主算法：

```text
1. 用 greedy/依赖图构造一个可行上界 U。
2. 在 State 图上跑 A*；Dijkstra 只作为小 case 对拍基准。
3. 第一个弹出的 complete 状态即为允许动作集内的字典序最优。
4. 若所有剩余状态 lower_bound >= U，则证明 U 在允许动作集内最优。
```

优先级：

```text
priority = (
  used_operations + admissible_lower_bound(state),
  unsatisfied_segment_count,
  -len(held),
  used_hot_throat_count,
  used_route_cost,
  used_buffer_penalty
)
```

如果启发式有任何不可采纳风险，必须退回 Dijkstra：

```text
priority = cost
```

### 7.3 可采纳下界

下界只允许低估，不能高估。

建议组合：

```text
LB_sum(state) =
  source_line_get_lb
+ target_group_put_lb

LB(state) = max(
  LB_sum(state),
  held_put_lb,
  fvs_lb_remaining
)
```

其中：

```text
source_line_get_lb = 仍有 active 车且不在 held 的源线数
target_group_put_lb = 仍需最终接收车辆的可合并目标组数量下界
held_put_lb    = held 非空时至少 1
fvs_lb_remaining = 理想依赖图破环下界
```

`source_line_get_lb` 数的是未来至少需要的 Get 行，`target_group_put_lb` 数的是未来至少需要的 Put 行，两者操作类型不同，可以相加，仍然是低估。  
注意：如果允许一次 Put 把多个来源的同目标车合并，则 `target_group_put_lb` 应按目标线或可合并目标组计算，不能按段数计算。

### 7.4 候选生成

#### Get 候选

对每条含 active 车的线路，取北端可取前缀。切点只放在有意义的位置：

1. 目标线变化。
2. 强制位置变化。
3. 关门车/重车可能改变过程合法性。
4. 牵引当量达到上限。
5. 段边界。

优先枚举长前缀，再枚举必要短前缀。

#### Put 候选

对 `held` 的尾部后缀生成 Put：

1. 同目标尾段优先放目标线。
2. 若目标线 ready，低惩罚。
3. 若目标线 not ready，高惩罚，但不绝对禁止。
4. 缓存 Put 只允许放到安全缓存线。
5. 禁止把已经完成且 closed 的目标线作为缓存。

#### 缓存线

候选缓存线按风险排序：

```text
临停线/阶段允许缓存线
一般存车线空余
已清空且后续不再作为源线的线路
```

本阶段默认不使用：

```text
修1-4库内 / 修1-4库外
联线 / 渡线
已经 closed 的目标线
```

如后续需要借用库外线处理极端前场债务，必须单独打开配置开关，并在 summary 中记录。

### 7.5 支配剪枝

同一抽象状态下，如果存在两条到达路径：

```text
cost_a <= cost_b
且 held、lines、loco、closed 完全相同
```

则丢弃 `b`。

另外可加安全支配：

1. 同样 `lines` 和 `closed`，`held` 更长且 cost 不高，不一定支配，因为 held 顺序影响 Put，不能随意剪。
2. 同样 `lines` 和 `held`，`loco` 可达集合更大且 cost 不高，可以支配；第一版可不做，避免证明复杂。

## 8. 物理校验

第四阶段所有路径必须调用现有图搜索口径：

```python
graph.route_avoiding_occupied(
    source,
    target,
    physical.occupied_lines_for_route(cars, moving_nos),
    source_departure_lines=physical.route_departure_lines_for_source(...),
    target_approach_lines=physical.route_approach_lines_for_get/put(...),
    cars=cars,
    moving_nos=moving_nos,
    train_length_m=train_length,
)
```

### 8.0 联7与大库股道口径

第四阶段不封锁 `联7`。`联7` 可以作为路径通过，尤其用于从第三阶段真实机车终点回到前场；但它仍是走行线，禁止作为 `Get` / `Put` 作业线。

第四阶段策略生成不包含大库相关作业股道：

```text
修1-4库内
修1-4库外
卸轮线
```

这些线不作为 source、target 或 cache 候选；若车辆仍需要这些线，最终会由 `unsatisfied_cars` 自然导致 partial，而不是额外写复杂阶段残债规则。

### 8.1 Get 路径

`Get(line, prefix)` 时：

```text
moving_nos = held ∪ prefix
train_length = length(held)
target_approach_lines = route_approach_lines_for_get(line)
```

取车路径只需要调机带当前 held 到达源线；被取 prefix 尚未挂上，因此倒车长度按当前 held 计算。  
挂上后的关门车、牵引当量和后续状态另行校验。

### 8.2 Put 路径

`Put(line, suffix)` 时：

```text
moving_nos = held
train_length = length(held)
target_approach_lines = route_approach_lines_for_put(line, cars, moving_nos)
```

Put 后调机位置按：

```python
rv.put_loco_positions(path, line)
```

更新为可达集合。

### 8.3 终端 replay

搜索内部的物理校验只能保证每一步按当前模型合法。最终仍以 replay 为裁判：

```python
replayed, replay_bad = rv.replay(stage4_request, stage4_response)
hard_bad = [v for v in replay_bad if v.kind in {"schema", "physical"}]
```

只有 `hard_bad` 为空且最终 `unsatisfied_cars` 为空，才输出 `complete`。

此外，`combined_response` 必须对原始 request 做端到端 replay：

```python
combined_replayed, combined_bad = rv.replay(original_request, combined_response)
```

`combined_bad` 中的 schema/physical/business 硬错误会使本阶段结果降级为 partial。这样可以暴露阶段间机车位置不连续等拼接问题。

## 9. 终态规则

`complete(state)` 必须满足：

1. `held` 为空。
2. 所有 active 车都满足 `car_is_satisfied`。
3. 所有固定车仍满足或未被错误移动。
4. 不存在未完成的 out-of-scope 车辆，或这些车辆本来已经满足。
5. replay 无硬物理错误。

终态不要只检查“车在目标线”。原因是：

1. 强制位置可能不满足。
2. 存4线关门车前三位规则可能不满足。
3. 大库台位虽然本阶段不处理，但仍可能暴露阶段3残债。

## 10. 最优性证书

每个 complete case 应输出最优性字段：

```json
{
  "lower_bound_initial": 10,
  "best_solution_operations": 12,
  "optimality": "proved_within_move_model",
  "proof": {
    "method": "astar",
    "expanded_states": 1842,
    "remaining_queue_bound": 12
  },
  "move_model_restrictions": [
    "fixed_satisfied_cars_are_not_moved",
    "cache_lines_are_whitelisted",
    "get_cuts_are_segment_boundaries",
    "depot_related_lines_are_excluded_from_get_put_cache_candidates"
  ]
}
```

如果通过搜索证明：

```text
queue_min_lower_bound >= best_solution_operations
```

则 `optimality = proved_within_move_model`。

不能声称无条件全局最优，除非后续版本取消缓存白名单、固定车硬冻结、段切点裁剪等所有会砍掉合法动作的限制。

如果时间耗尽但已有可行解：

```json
{
  "optimality": "feasible_unproved",
  "lower_bound_initial": 10,
  "best_solution_operations": 13,
  "gap": 3
}
```

这里的 gap 是严格下界 gap，不是经验差距。

## 11. 实现步骤

### 11.1 MVP

第一版只做主闭环：

1. 新建 `scripts/stage4_simple/solve.py`。
2. 从阶段3合并响应 replay 出终态。
3. 计算 `unsatisfied_cars`。
4. 大库相关线不进入 active strategy set。
5. 构建 active/fixed 状态。
6. 生成 Get/Put/Weigh 候选。
7. 用 greedy 构造上界，用 A* 求允许动作集内最少操作行。
8. 输出 response、combined_response、summary、trace。
9. 用 replay + `unsatisfied_cars` 验收。

### 11.2 第二步优化

加入：

1. 段归约。
2. 依赖图 ready 目标排序。
3. greedy 上界。
4. A* 下界。
5. FVS 下界与证书字段。

### 11.3 第三步增强

加入：

1. 可达集合支配剪枝。
2. 缓存线配置。
3. 存4线释放专项统计。
4. 极端 case 的 bounded beam 上界生成。
5. `feasible_unproved` 输出模式。

## 12. 风险与边界

| 风险 | 处理 |
|---|---|
| 阶段3仍有大库相关未完成车 | 不生成大库相关策略；最终验收自然 partial |
| 连续携带导致状态增多 | 用段归约、切点控制、强上界和 A* 下界限制候选 |
| 机车位置影响可达 | `loco` 进入状态，不能用纯组合状态替代 |
| 目标线 not ready 但容量够 | 允许搜索使用，但加高惩罚；最终由最短路比较 |
| replay 与内部状态不一致 | replay 是最终裁判，hard_bad 则 partial |
| A* 下界写错 | 默认可退回 Dijkstra；只有可采纳下界才能参与证明 |
| 大库相关股道误入策略 | source/target/cache 候选生成统一排除修库内外和卸轮线 |
| 硬剪枝导致最优性名不副实 | summary 标记 `proved_within_move_model`，并输出 `move_model_restrictions` |

## 13. 推荐默认配置


第一版主路径使用 A*。Dijkstra 只保留为小 case 对拍和启发式审计工具，避免连续携带导致单位代价分层搜索爆炸。

## 14. 一句话总结

第四阶段的落地模型是：

```text
用 physical.unsatisfied_cars 定义债务；
策略空间排除大库相关线；
把剩余车辆建成带 held/loco 的多栈状态图；
允许路径经过 联7，但不把走行线/大库相关股道作为作业候选；
用 Get/Put/Weigh 操作行级 A* 求允许动作集内字典序最短路；
用 replay 和全车满意度验收；
用依赖图、段归约和下界证书解释最优性。
```

这样既保留了数学上的清晰性，也不牺牲现场物理约束和连续携带带来的真实最优空间。
