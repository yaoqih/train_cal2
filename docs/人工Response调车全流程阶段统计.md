# 人工 Response 调车全流程阶段统计

## 数据口径

统计目录：

```text
artifacts/manual_restored_interface/bundles/
```

共有 109 个 bundle JSON，每个文件包含：

- `Request`: 初始车辆、目标线、机车位置。
- `Response.Data.Operations`: 已还原的调车动作。
- `Summary`: 人工钩数、还原动作数、阻塞原因。
- `Trace`: 人工原始钩与还原状态的逐钩诊断。

本分析以 `Response.Data.Operations` 为主。109 个 Response 中，84 个成功完整还原，25 个失败或较早阻塞。成功样本共有 2740 条还原动作。

整体分布：

| 指标 | 数值 |
|---|---:|
| bundle 文件 | 109 |
| Response 成功 | 84 |
| Response 失败 | 25 |
| 人工钩数中位数 | 36 |
| 还原动作数中位数 | 31 |
| 还原动作数 75 分位 | 35 |
| 还原动作数最大 | 49 |
| noop 中位数 | 2 |
| Trace 中 restored 动作 | 2916 |
| Trace 中 noop | 242 |
| Trace 中 blocked / blocked_after_prior_gap | 590 |

失败样本多数是还原器在早期报 `no_candidate_from_current_identity_state`，说明这 25 份不适合直接抽取完整流程，但仍能用于识别接口还原能力边界。

## 总体人工流程

人工 Response 可以抽象成 6 个阶段：

```text
P0 起机/定位/无车钩
P1 前场功能线整理
P2 存车分流塑形
P3 修库出库到存4
P4 大库入库重放
P5 尾项回收
```

其中 P0 主要体现在 `Trace` 的 noop 中，`Response.Data.Operations` 通常从第一个有效 `Get` 开始。

成功样本的主流程顺序：

| 阶段顺序 | 案例数 |
|---|---:|
| P1 -> P2 -> P3 -> P4 -> P5 | 47 |
| P1 -> P3 -> P4 -> P5 | 17 |
| P1 -> P2 -> P4 -> P5 | 4 |
| P1 -> P2 -> P3 -> P4 | 4 |
| P1 -> P4 -> P5 | 4 |
| 其他变体 | 8 |

结论：人工不是固定五段全齐，而是围绕关键边界跳过空阶段。完整骨架是“前场整理 -> 存车塑形 -> 存4释放 -> 大库入库 -> 尾项”，但某些案例没有显式存车分流或没有完整修库释放段。

## 阶段边界识别

可以用以下规则还原每个 Response 的过程阶段：

1. `P0 起机/定位/无车钩`
   - 来自 `Trace.status=noop`。
   - 不进入 `Response.Data.Operations`。

2. `P1 前场功能线整理`
   - 从第一个有效 `Get` 开始。
   - 结束于第一次从存车线取车，或第一次进入修库出库段。
   - 存车线包括 `存1线/存2线/存3线/存5线北/存5线南`。

3. `P2 存车分流塑形`
   - 从第一次 `Get 存车线` 开始。
   - 结束于第一次 `Get 修1-4库内`。

4. `P3 修库出库到存4`
   - 从第一次 `Get 修1-4库内` 开始。
   - 到第一次 `Put 存4线` 结束。

5. `P4 大库入库重放`
   - 从第一次 `Put 存4线` 后开始。
   - 到最后一次 `Put 修1-4库内` 结束。

6. `P5 尾项回收`
   - 最后一次 `Put 修1-4库内` 之后的所有动作。
   - 如果没有大库入库段，则用最后一个明确主闭合动作之后的剩余服务线/存车线动作标记。

边界位置统计：

| 边界 | 出现样本 | 相对位置中位数 |
|---|---:|---:|
| 第一次 `Get 存车线` | 82 | 0.32 |
| 第一次 `Get 修库内` | 83 | 0.60 |
| 第一次 `Put 存4线` | 81 | 0.73 |
| `Put 存4线` 后第一次 `Put 修库内` | 77 | 0.82 |
| 最后一次 `Put 修库内` | 84 | 0.88 |

这说明人工计划通常前 30% 左右处理前场和存车开口，60% 左右进入修库出库，70% 以后完成存4释放，后 20% 才是入库重放和尾项。

## P0 起机/定位/无车钩

P0 在 Response 里不表现为 `Operation`，但在 `Trace` 中常见。典型含义：

- 机车从初始位置出发。
- 人工单中写了线路或方向，但没有真实车辆位移。
- 首钩/空钩用于到达可操作端。

统计上，noop 中位数为 2，最多 4。成功 Response 的第一条有效动作 84/84 都是 `Get`。

第一条有效 `Get` 的线路分布：

| 首个取车线路 | 次数 |
|---|---:|
| `油漆线` | 20 |
| `抛丸线` | 19 |
| `调梁棚` | 13 |
| `洗罐站` | 11 |
| `机走棚` | 9 |
| `预修线` | 6 |
| `机库线` | 3 |

人工开局明显偏向先动前场功能线，而不是直接从存5或大库开始。

## P1 前场功能线整理

P1 是人工最像“打开局面”的阶段。目标不是马上完成所有终点，而是先处理洗、油、抛、调棚、预修、机棚这些会影响后续通路和编组的车。

统计：

| 指标 | 数值 |
|---|---:|
| 非零案例 | 82/84 |
| 每案动作中位数 | 6 |
| 每案动作 75 分位 | 11 |
| 每案动作最大 | 17 |

高频动作：

| 动作 | 次数 |
|---|---:|
| `Put 机库线` | 158 |
| `Get 预修线` | 81 |
| `Get 调梁棚` | 78 |
| `Get 油漆线` | 56 |
| `Get 抛丸线` | 36 |
| `Get 洗罐站` | 34 |
| `Put 预修线` | 34 |
| `Put 洗罐站` | 20 |
| `Put 油漆线` | 17 |
| `Put 抛丸线` | 12 |
| `Put 调梁棚` | 12 |

关键规律：

- P1 经常把前场取出的车送到 `机库线` 承接，`Put 机库线` 是第一阶段最高频动作。
- 抛丸、油漆、洗罐的动作批量较小，抛丸 `Get/Put` 批量中位数都是 2，油漆和洗罐通常 2 到 3 辆。
- `机南/洗油北` 在 Response 中更多是路径节点，不是大量长期缓存节点。
- P1 不是单纯“把去大库车编走”，而是同时处理去大库、去洗油抛、去调棚/调北、去预修/机棚的混合债务。

## P2 存车分流塑形

P2 是从 `存5线北/存5线南/存1/存2/存3` 这些存车线抽取车辆，按后续目标做分流和车列塑形。

统计：

|---|---:|
| 非零案例 | 61/84 |
| 每案动作中位数 | 9 |
| 每案动作 75 分位 | 17 |
| 每案动作最大 | 29 |

高频动作：

| 动作 | 次数 |
|---|---:|
| `Get 存5线北` | 152 |
| `Put 机库线` | 132 |
| `Put 调梁棚` | 75 |
| `Put 预修线` | 67 |
| `Get 调梁棚` | 62 |
| `Get 预修线` | 53 |
| `Put 存2线` | 33 |
| `Put 洗罐站` | 23 |
| `Get 存2线` | 23 |
| `Put 抛丸线` | 20 |
| `Get 存3线` | 20 |

关键规律：

- `存5线北` 是 P2 的主源线，但人工不是盲目清空存5，而是按目标成段抽取。
- P2 的主要去向是 `机库线/调梁棚/预修线/存2线/洗罐站/抛丸线`。
- 这阶段的本质是“后续可摘解性塑形”：把目标相近、后续可连续摘解的小段先形成。
- P2 是第一阶段和大库阶段之间的缓冲层。没有它，后面大库入库和尾项会变复杂。

## 机库线的核心作用

成功 Response 中：

| 动作 | 次数 | 出现案例 |
|---|---:|---:|
| `Put 机库线` | 381 | 81/84 |
| `Get 机库线` | 126 | 80/84 |
| 最后一钩落 `机库线` | 58 | 58/84 |

`Put 机库线` 的车辆目标构成：

| 目标族 | 车辆数 |
|---|---:|
| 去大库 | 892 |
| 去预修/机棚 | 246 |
| 去调棚/调北 | 117 |
| 去洗油抛 | 95 |
| 去机库 | 46 |
| 去存4 | 34 |

解释：

- 人工把 `机库线` 当成核心承接/编组线。
- 它承接的不只是最终去机库车辆，更大量是去大库和后续需要再分解的车辆。
- 这与“直接把车塞到 `机南/洗油北`”不同；`机南/洗油北` 更像通路门槛和短时通过点。

## 多取后甩的人工模式

从 Response 序列看，人工经常连续多个 `Get` 后再 `Put`：

| 连续 Get 源数 | 次数 |
|---|---:|
| 1 个源后 Put | 793 |
| 2 个源后 Put | 458 |
| 3 个源后 Put | 105 |
| 4 个源后 Put | 32 |
| 5 个源后 Put | 8 |
| 6 个源后 Put | 1 |

约 43% 的 Put 前面有两个及以上取车源。这说明人工计划不是简单的“一取一送”，而是先把同一后续意图的小列挂成形，再按尾端可摘顺序甩到目标或承接线。

## 可抽象出的调车全流程

人工全流程可以还原成下面的决策框架：

1. 先开前场，不先碰大库主闭合。
   - 优先处理 `油漆线/抛丸线/洗罐站/调梁棚/预修线/机走棚`。
   - 目标是释放通路、形成机库承接段、消化浅层服务线债务。

2. 再从存车线塑形。
   - 以 `存5线北` 为主源线。
   - 只抽取能形成后续目标小段的车辆。
   - 去向主要是 `机库线/调梁棚/预修线/存2/存3/洗罐站/抛丸线`。

3. 中后段做修库出库到存4。
   - 从修1至修4库内集中取出去存4或出段相关车辆。
   - 中间可顺手处理卸轮和库外。
   - 形成 14 至 16 辆左右的大批量 `Put 存4线`。

4. 存4释放后再做大库入库。
   - 从 `机库线` 或其他承接线取已编好的去大库车辆。
   - 按修库目标连续摘解入库。
   - 这一段是强闭合，不能提前零散做。

5. 最后清尾项。
   - 回到 `存5线北/机库线/卸轮线/调梁棚/预修线/油漆线`。
   - 处理未闭合的小批量目标。
   - 最终常落在 `机库线` 或 `油漆线`。

## 对阶段1/阶段4边界的启发

这批 Response 支持一个更清晰的阶段划分：

| 人工阶段 | 求解器应承担位置 |
|---|---|
| P1 前场功能线整理 | Stage1 应显式吸收一部分 |
| P2 存车分流塑形 | Stage1/Stage3 之间需要结构层 |
| P3 修库出库到存4 | Stage4 或专门大库出库宏 |
| P4 大库入库重放 | Stage4 或专门大库入库宏 |
| P5 尾项回收 | Stage4 末段或补充宏 |

具体策略：

- Stage1 不应只做“卸轮去存4 + 去大库编组”，还应做 P1 的前场整理和一部分 P2 的存车塑形。
- Stage1 做 P1/P2 时必须保护 `机南/洗油北/调梁线北/机北2/存4南` 这些门槛线。
- `机库线` 应被建模为人工高频承接资源，但要受机库目标、称重、调棚/机区通路约束。
- Stage4 应保留 P3/P4 的闭合强宏，即修库出库到存4和大库入库重放。
- P5 不应完全靠搜索碰运气，应按“尾项目标族”做小宏：存5尾项、机库尾项、卸轮到油漆、调棚/预修尾项。

## 还原可行性

可以还原调车的每个过程阶段。当前最稳定的还原方式不是按人工钩号硬切，而是按 Response 操作中的线路事件切：

```text
first_valid_Get
first_storage_Get
first_depot_inner_Get
first_store4_Put
first_depot_inner_Put_after_store4
last_depot_inner_Put
tail_ops
```

这套边界能覆盖 84 个成功 Response 的主流程，并能解释大部分跳段变体。后续可以把阶段标签写成脚本，给每个 `Operation` 增加 `manual_phase`，用于和 Stage1/Stage4 输出逐阶段对比。
