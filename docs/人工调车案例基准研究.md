# 人工调车案例基准研究

## 1. 研究目标

本文用于把 `data/人工调车数据` 中的人工调车作业单转成可比较、可验收、可超越的基准。

核心问题不是“复刻人工钩序”，而是回答三件事：

1. 人工调车实际在优化什么结构。
2. 什么程度算达到人工。
3. 算法要在哪些指标上超过人工，而不是只在局部少一两钩。

本文结论基于当前仓库内可读取资料：

- 人工调车作业单：`data/人工调车数据/*-人工调车作业单/调车作业通知单*.xlsx`
- 取送车计划转换数据：`data/truth2/*.json`
- 业务约束文档：`docs/福州调车业务文档.md`
- 方案文档：`docs/flow_edge_structure_model_design.md`

## 2. 数据口径

### 2.1 人工单解析口径

人工 Excel 的有效调车钩序主要位于每张 `Sheet1` 的两个块：

| 块 | 列 |
|---|---|
| 前 30 钩 | `顺序 / 股道 / 作业方法 / 辆数 / 备注` |
| 后续钩 | 第二组 `顺序 / 股道 / 作业方法 / 辆数 / 备注` |

右侧还有一套空模板列，解析时不计入有效钩序。

本次解析得到：

| 指标 | 数值 |
|---|---:|
| 人工调车作业单文件数 | 118 |
| 唯一短案号数 | 116 |
| 有效钩序总数 | 4076 |
| 单案最少钩数 | 5 |
| 单案平均钩数 | 34.54 |
| 单案中位数钩数 | 35 |
| 单案 75 分位钩数 | 40 |
| 单案 90 分位钩数 | 45 |
| 单案最多钩数 | 58 |

说明：

- `0130Z`、`0302W` 在人工单中各出现 2 个文件，后续做逐案对比时必须按“文件级”或“场景源文件级”匹配，不能只按短案号去重。
- 人工单有 109 个文件可与当前 `truth2` 短案号匹配；匹配集人工中位数为 36 钩。
- 未匹配人工单多为库内消化、直接处理或特殊短链案例，不能随意删除；它们反而能补充 `truth2` 中较少出现的异常业务形态。

### 2.2 人工动作简称

人工单使用的是现场简称：

| 简称 | 业务解释 |
|---|---|
| `机` | 机库/机走相关承接与缓冲 |
| `库` | 大库侧总称或库回动作 |
| `修1` 至 `修4` | 修1至修4库内/库外相关动作 |
| `存4` | 存4北释放口/集结口 |
| `存1/存2/存3/存5` | 存车区组织、腾位、归并 |
| `调` | 调棚/调梁相关队列 |
| `洗/油/抛/轮` | 功能线与运营线 |
| `注意存4/注意机/注意库` | 人工强调关键端别或安全注意，不应当被当作普通股道名 |

动作方法中：

| 写法 | 解释 |
|---|---|
| `+` | 挂/取/接入当前车列 |
| `-` | 摘/送/甩出当前车列 |
| `称` | 称重相关动作 |
| `顶` | 顶送/顶位 |
| `代` | 代位或临时代替位置 |
| `回` | 回库/回机等收尾语义 |

备注中的高频结构词：

| 备注 | 出现次数 | 含义 |
|---|---:|---|
| `北头` | 630 | 北端取送、靠口、整理 |
| `叉线` | 329 | 使用交叉线/中间缓冲进行重排 |
| `接` | 206 | 承接、连挂、机接或功能线接入 |
| `摘` | 119 | 摘解，常见于修库消化末端 |
| `北头摘` | 105 | 存4北释放关键动作 |
| `南` | 68 | 南端/南头相关限制 |
| `库外2` | 62 | 修库库外缓冲或摘解位置 |

这些词不是装饰备注，而是人工计划的结构信号。

## 3. 人工基准的核心结构

### 3.1 人工不是简单按目标线搬车

人工计划中最重要的结构不是“车从 A 到 B”，而是：

```text
外场/功能线/预修/调棚整理
  -> 存4北形成释放口
  -> 存4北大摘
  -> 机接承接
  -> 修1-修4分线摘解
  -> 轮/油/洗/抛等尾部功能线处理
  -> 库回或收束
```

在 118 个人工文件中，按结构信号粗分类：

| 人工案例族 | 文件数 | 含义 |
|---|---:|---|
| 标准存4释放-机接-修库消化 | 97 | 有 `存4 - 北头摘`、`机 + 接`、修库摘解 |
| 库内消化/直接处理 | 10 | 缺少标准机接，但有修库摘解和库回 |
| 混合/一般整理 | 7 | 以外场、功能线、调棚、机区整理为主 |
| 有存4释放无标准机接 | 2 | 有存4释放，但机接信号不完整 |
| 短链直接入库 | 2 | 已接近库侧，少量动作即可完成 |

匹配 `truth2` 的 109 个文件中，标准链仍有 97 个，说明 `truth2` 的主问题确实以“大库周转 + 存4释放 + 机接 + 修库消化”为主。

### 3.2 高频动作族

| 动作族 | 覆盖文件数 | 基准意义 |
|---|---:|---|
| 连接/接入信号 | 118 | 每案都存在某种承接动作 |
| 摘解信号 | 118 | 每案都存在某种摘解动作 |
| 特殊微动作 | 118 | `顶/代/称/对位/回/停` 等不可忽略 |
| 功能线服务 | 117 | 洗、油、抛、轮几乎每案参与 |
| 机区动作 | 117 | 机车位置和机区缓冲是少钩关键 |
| 北头操作 | 116 | 端别和北头靠口是人工核心操作面 |
| 修库摘解/送入 | 116 | 大库消化不是可选尾项 |
| 存车区组织 | 113 | 存1/2/3/5 的整理是主任务 |
| 调棚队列 | 107 | 调棚不是 residual |
| 存4释放 | 104 | 存4北是主释放口 |
| 机接 | 102 | 标准链的硬边界 |
| 叉线缓冲 | 104 | 人工大量利用中间缓冲减少返工 |
| 库回 | 76 | 收束信号，不能提前触发 |

这些覆盖率说明：人工调车的真实基准是全站编排，不是单一修库路径。

### 3.3 人工钩序的典型时间位置

按有效钩序位置中位数估算：

| 动作族 | 中位出现位置 | 解释 |
|---|---:|---|
| 功能线服务 | 第 8 钩 | 早期先处理洗/油/抛/轮等可消化项 |
| 机区动作 | 第 12 钩 | 机车位置与机区缓冲早早参与 |
| 调棚队列 | 第 14 钩 | 调棚/预修相关队列在中前段整理 |
| 叉线缓冲 | 第 14 钩 | 中前段大量使用叉线重排 |
| 存车区组织 | 第 17 钩 | 外场组流持续到中段 |
| 北头操作 | 第 20 钩 | 逐步靠存4北或北头组织 |
| 修库送入/接入 | 第 23 钩 | 进入库侧前的主列组织 |
| 存4释放 | 第 25.5 钩 | 标准链关键转折点 |
| 机接 | 第 27 钩 | 机接通常紧跟存4释放 |
| 修库摘解 | 第 28 钩 | 进入大库消化阶段 |
| 库回 | 第 36 钩 | 多数情况下是后段收束 |

这与业务文档中的大库作业顺序吻合：前段先完成除大库区域外的大部分调车，再集中处理大库出入和库内分散。

## 4. 代表案例拆解

### 4.1 `0117Z`：标准完整链

人工钩数：40。

关键链条：

```text
01-22: 抛/油/机/调/存2/存5/存3 组织和功能线处理
23:    存4 - 北头
24-31: 轮/修1/修2/修3/修4 组合形成大库目标群
32:    存4 - 16 北头摘
33:    机 + 18 接
34-37: 修4/修3/修2/修1 摘解
38-40: 轮/油/库回收束
```

人工价值：

- 前 22 钩不是杂活，是在为存4释放口和机接主列塑形。
- 第 32-33 钩是硬边界：`存4大摘 -> 机接`。
- 第 34-37 钩是大库消化，不允许被提前当尾项。

算法达到人工的要求：

- 能识别 `APPROACHING_CUN4 -> PORT_READY -> MACHINE_ACCEPTED -> DEPOT_DIGEST`。
- 机接后不能为了少钩拆散主列。
- 大库摘解完成前不能 closeout。

算法超越空间：

- 前段存2/存5/存3多次往返，可用更精确的目标分组和端别可摘性减少重复整理。
- 轮/油尾项可作为 support contract 提前或并行评估，但不能破坏主列。

### 4.2 `0310W`：late 存4链

人工钩数：46。

关键链条：

```text
01-28: 抛/洗/油/调/存1/存2/存5 大量整理
29:    注意存4 - 北头代2
30-35: 抛/修2/轮/修1/修3/修4 形成库侧接入群
36:    存4 - 14 北头摘
37:    机 + 14 接
38-41: 修2/修4/修3/修1 摘解
42-46: 轮/油/库/机/库回
```

人工价值：

- 这个案例前段已经在为存4释放服务，不能把它误判成还需要重新外场补链。
- 第 36-37 钩是标准 `release -> accept`。

算法达到人工的要求：

- 一旦识别 `PORT_READY` 或接近 `PORT_READY`，禁止回外场重新拼长链。
- 目标合同应优先生成 `存4释放 / 机接 / 入库摘解` 候选。

算法超越空间：

- 人工前 28 钩外场组织较长，算法可通过同合同内局部搜索减少调/存5/存2之间的反复。
- 但少钩候选必须证明不会污染存4释放口。

### 4.3 `0103W`：库内消化与缺标准机接

人工钩数：49。

关键链条：

```text
01-18: 库/抛/油/洗/机/存1/存2 复杂组织，含称重/顶送
19-22: 轮、修2、修3、修4 摘解
23-29: 注意机、洗接、存2/存3、修4/修3/修1 摘解
30-48: 存5/存1/存2/洗/调/机 大量尾部组织
49:    库回
```

人工价值：

- 没有标准 `存4释放 -> 机接`，但存在明确修库摘解层。
- 这类案例不能被算法归为“尾项”或“普通整理”。
- 称重、顶送、对位等微动作会影响合法性。

算法达到人工的要求：

- 能从当前物理状态识别 `DEPOT_DIGEST_ONLY` 或 `DIRECT_REPAIR_ENTRY`。
- 不要求补一个不存在的标准存4长链。
- 修库摘解债务完成前不允许库回 closeout。

算法超越空间：

- 该类人工钩数高，常由功能线、称重、库内摘解和外场整理交错造成。
- 算法要用 `SPECIAL_REPAIR_PROCESS` 把称重/顶送/对位独立建模，否则少钩结果可能非法。

### 4.4 `0213W`：短链直接入库

人工钩数：5。

关键链条：

```text
01: 注意机
02: 修1 + 5 接
03: 修2 + 5
04: 注意存4 - 10 北头摘
05: 库回
```

人工价值：

- 这是极短直接处理，不需要完整外场组织。
- 算法若强行套用标准大链，会明显劣于人工。

算法达到人工的要求：

- 识别 `DIRECT_REPAIR_ENTRY`。
- 允许跳过外场集货和标准机接链。

算法超越空间：

- 这种案例人工已接近下界，算法不要追求“更少钩”而破坏结构；目标应是同钩数、更完整安全校验和可解释 trace。

### 4.5 `0306W`：短链但信号不完整

人工钩数：10。

关键链条：

```text
01-02: 机、存4北头
03-05: 修3/修1/修2 接入
06:    存4 - 15 北头摘
07-08: 机 + 2 / 机 + 1 南头
09-10: 油、库回
```

人工价值：

- 有存4释放，但机接信号不标准。
- 这类案例应被识别为低信号或混合直接处理，而不是强行生成高置信机接。

算法达到人工的要求：

- 允许 `MIXED_SIGNAL_REPAIR`。
- 低置信下禁止主动跨 `ACCEPTED`。
- 必须输出矛盾原因和保守动作边界。

## 5. 人工基准定义

### 5.1 第一层：结构基准

算法达到人工，首先不是钩数更少，而是结构不能错。

必须满足：

| 基准项 | 达标标准 |
|---|---|
| 主体流向识别 | 大库、存4、预修、调棚、功能线、机区、存车整理都有合同 |
| 存4释放口 | 能识别 `存4北头/北头摘` 的释放意义 |
| 机接硬边界 | `机 + 接` 后主列不能随意拆散 |
| 修库消化 | 修1-修4摘解完成前不能 closeout |
| 直接入库 | 短链/近库侧案例不能强套标准长链 |
| 低信号案例 | 能输出 confidence / contradiction，不能低置信主动机接 |
| 预修/调棚 | 不能当作 residual 或尾项 |
| 功能线 | 洗、油、抛、轮必须有服务完成条件 |
| 特殊微动作 | 称、顶、代、对位、停等必须作为约束，不是备注噪声 |

### 5.2 第二层：人工动作族覆盖基准

每个算法案例都应输出类似人工的结构 trace：

```json
{
  "case": "0117Z",
  "contract_groups": {
    "REPAIR_INBOUND": 16,
    "DEPOT_OUTBOUND": 16,
    "PRE_REPAIR_STAGING": 11,
    "DISPATCH_SHED_QUEUE": 8,
    "FUNCTION_LINE_SERVICE": 5,
    "YARD_REBALANCE": 18
  },
  "critical_transitions": [
    "FORM_RELEASE_GROUP",
    "CUN4_RELEASE",
    "MACHINE_ACCEPT",
    "DEPOT_DETACH",
    "TAIL_CLOSEOUT"
  ],
  "resource_claims": [
    "CUN4_NORTH_BUFFER",
    "LINK7_GATE",
    "DEPOT_SLOT_RESOURCE",
    "LOCO_CARRY_ORDER"
  ]
}
```

如果算法给不出这类 trace，就算生成了钩序，也无法证明达到人工。

### 5.3 第三层：数值基准

以人工文件为基准：

| 指标 | 人工基准 |
|---|---:|
| 全量人工单中位数 | 35 钩 |
| 匹配 `truth2` 人工单中位数 | 36 钩 |
| 标准链人工文件数 | 97 |
| 标准链覆盖率 | 82.2% |
| 匹配集标准链文件数 | 97 |
| 匹配集标准链覆盖率 | 89.0% |

建议验收时分案例族比较：

| 案例族 | 达到人工 | 超越人工 |
|---|---|---|
| 标准存4释放-机接-修库消化 | 无硬违规，钩数不高于人工 5%，关键转折一致 | 钩数低于人工，且不增加存4污染/库位冲突/机车空驶 |
| 库内消化/直接处理 | 不强套标准长链，先完成库内摘解 | 同钩或少钩，并输出更清楚的库位 swap |
| 短链直接入库 | 不明显多于人工，不额外绕行 | 保持人工钩数或更少，安全 trace 更完整 |
| 混合/低信号 | 不低置信跨硬边界 | 用补证据动作减少返工 |
| 功能线/调棚重案例 | 功能线不被当尾项 | 减少功能线与外场间反复取送 |

## 6. 如何达到人工

### 6.1 不要从“全局搜索”开始

人工计划显示，人的强项是先确定结构边界：

```text
哪些车先在外场/功能线消化
哪些车应逐步靠存4
什么时候存4释放口成形
什么时候机接后不能再拆
哪些库内车必须先出让位
哪些尾项只能最后处理
```

所以算法达到人工的首要条件是：

```text
FlowClassify
  -> TargetContract
  -> StructuralIntent
  -> ResourceRequest
  -> ContractDelta
  -> ResourceDelta
```

而不是：

```text
生成大量动作
  -> 搜索
  -> 靠分数碰运气
```

### 6.2 必须先实现人工结构识别

最低实现要求：

1. 识别 `存4 - 北头 / 北头摘`。
2. 识别 `机 + 接`。
3. 识别 `修1-修4 - n / 摘 / 库外n`。
4. 识别 `库 - 回` 只能作为收束，不是万能完成信号。
5. 识别 `调`、`洗`、`油`、`抛`、`轮` 都是独立服务项。
6. 识别 `叉线`、`北头` 是端别/缓冲结构，不是普通备注。
7. 识别 `称/顶/代/对/停` 是硬约束候选。

### 6.3 必须先通过人工案例族回放

建议先建立 9 个金标案例：

| 案例 | 类型 | 必验点 |
|---|---|---|
| `0117Z` | 标准完整链 | 存4释放、机接、修库摘解完整推进 |
| `0310W` | late 存4链 | 不回外场补长链 |
| `0103W` | 库内消化 | 无标准机接但不能 closeout |
| `0223W` | 库内消化 | 修库摘解优先于尾项 |
| `0213W` | 短链直接入库 | 不强套标准链 |
| `0306W` | 短链低信号 | 不低置信主动机接 |
| `0130Z` | 混合信号 | 允许反推但必须保守 |
| `0201W` | 机接前信号缺口 | 不能半成品机接 |
| `0128W` | 标准链高钩数 | 评估少钩优化空间 |

每案输出：

```text
人工关键结构序列
算法识别出的合同序列
算法候选动作族
被拒绝动作及原因
硬约束检查
算法钩数 vs 人工钩数
```

## 7. 如何超越人工

### 7.1 超越人工不是全面模仿

人工的价值是结构判断，不是每一钩都最优。

算法应继承：

- 存4释放口保护。
- 机接后主列保护。
- 大库先出后进。
- 预修/调棚/功能线独立服务。
- 尾项必须在主合同完成后收束。

算法可以超越：

- 外场反复取送。
- 存2/存5/存3 多次交叉线重排。
- 功能线和大库主列混合携带导致的重复摘挂。
- 机车空驶和端别切换。
- 库位 swap 依赖靠经验处理。

### 7.2 超越人工的可量化指标

建议将“超越人工”定义为多指标支配，而不是单看钩数。

硬性前提：

```text
hard_violation_count = 0
所有车辆最终目标满足
大库台位规则满足
关门车/重车/称重规则满足
运行干涉和临停容量满足
求解时间 <= 5 分钟
```

在硬性前提满足后比较：

| 指标 | 超越标准 |
|---|---|
| 总钩数 | 标准链案例低于人工；短链案例不高于人工 |
| 大库进出次数 | 不多于人工，优先接近“一进一出” |
| 存4污染次数 | 少于人工或等于人工 |
| 叉线/临停使用 | 少于人工，或证明减少后续返工 |
| 机车空驶距离 | 少于人工 |
| 道岔扳动次数 | 少于人工 |
| 修库 swap 等待 | 明确先出后进，减少冲突 |
| 计划可解释性 | 每个关键动作都有 ContractDelta / ResourceDelta |

### 7.3 超越策略

#### 策略一：把人工“中段靠存4”提前变成资源目标

人工通常在中段形成存4释放口。算法可以更早判断：

```text
哪些外场车最终服务 REPAIR_INBOUND
哪些车只是 YARD_REBALANCE
哪些车会污染 CUN4_NORTH_BUFFER
```

提前禁止随机占用存4北，可减少后段清口钩数。

#### 策略二：用 DepotSlotGraph 替代经验判断

人工能大致判断哪些库位要先让，但不一定全局最优。

算法应显式输出：

```text
inbound_vehicle -> target slot/band
blocking outbound vehicle
released slot
reserved slot
swap dependency
```

只要 slot/band 依赖正确，算法有机会减少“先进错位再调整”。

#### 策略三：保留 ordered carry sequence

人工少钩依赖车列顺序和尾端可摘性。

算法不能只用 `CARRYING_CONTRACT_GROUP` 这种枚举压缩，必须保留：

```text
ordered_contract_segments
accessible_end
detachable_from_tail
```

否则看似少钩，实际会产生非法摘挂或需要返工。

#### 策略四：同结构等级内做少钩搜索

少钩搜索只能在这些前提后启动：

```text
同一个 TargetContract
同一个 StructuralIntent
无 ContractDelta 硬违反
无 ResourceDelta 硬违反
不破坏更高优先级合同
```

这样才能避免“少一钩但错主结构”。

#### 策略五：功能线和调棚独立成合同

人工计划里功能线几乎每案出现，调棚覆盖 107 个文件。

算法若把它们当 residual，会在后段产生大量补丁。

必须有：

```text
FUNCTION_LINE_SERVICE
DISPATCH_SHED_QUEUE
PRE_REPAIR_STAGING
```

并且能判断服务完成、退出服务线、是否可转大库。

## 8. 评估矩阵建议

每个案例输出一行：

| 字段 | 含义 |
|---|---|
| `case_id` | 案例编号 |
| `manual_file` | 人工作业单路径 |
| `truth2_file` | 匹配的输入 JSON |
| `case_family` | 人工案例族 |
| `manual_hook_count` | 人工钩数 |
| `solver_hook_count` | 算法钩数 |
| `hook_delta` | 算法 - 人工 |
| `hard_violation_count` | 硬违规数量 |
| `contract_coverage` | 需动车合同覆盖率 |
| `residual_vehicle_ratio` | residual 比例 |
| `cun4_release_detected` | 是否识别存4释放 |
| `machine_accept_detected` | 是否识别机接 |
| `depot_digest_complete` | 修库消化是否完成 |
| `depot_slot_swap_valid` | 大库 slot/band swap 是否合法 |
| `function_service_complete` | 功能线是否完成 |
| `pre_repair_stable` | 预修是否稳定归组 |
| `loco_order_valid` | 携带顺序是否合法 |
| `runtime_seconds` | 求解耗时 |
| `result_level` | failed / reached_human / exceeded_human |

`result_level` 建议定义：

```text
failed:
  任一硬约束违规，或最终目标不满足。

reached_human:
  无硬违规；
  关键结构不弱于人工；
  标准链钩数 <= 人工 * 1.05；
  短链/直接处理钩数不明显多于人工。

exceeded_human:
  无硬违规；
  关键结构不弱于人工；
  钩数低于人工，或钩数相同但道岔/空驶/存4污染/库位冲突更少；
  输出完整 ContractDelta + ResourceDelta trace。
```

## 9. 结论

人工调车的基准不是一条固定 workflow，而是一套稳定结构经验：

```text
前段消化功能线和外场整理；
中段保护并形成存4北释放口；
关键时刻做存4大摘和机接；
后段连续完成修库摘解；
最后才处理库回和尾项。
```

要达到人工，算法必须先学会这些结构边界，并且能解释每一步为什么是履约。

要超越人工，算法不能盲目复刻人工钩序，而应在同样结构价值下利用：

- 更精确的合同分类。
- 更严格的资源仲裁。
- 更细的大库 slot/band swap。
- 更可靠的机车携带顺序。
- 同结构等级内的小范围少钩搜索。

本研究的输出是人工结构基准：存4释放、机接、修库摘解、库回、大库 swap、机车携带顺序、功能线和调棚服务等。这些基准用于后续评价算法动作是否保留人工结构价值。

## 10. 结构化 rollout 对人工基准的校准

本轮新增了 `scripts/generate_rollout_trace.py`，用于把 P5-P9 串成多步结构化 rollout，并与人工计划软上界对照。

当前结果：

| 指标 | 结果 |
|---|---:|
| truth2 案例数 | 113 |
| 可匹配人工计划案例数 | 107 |
| 匹配案例终态完成数 | 107 |
| 缺人工计划基线案例数 | 6 |
| 总勾数不超过人工计划软上界 | 105 / 107 |
| H 阶段勾数不超过人工计划阶段软上界 | 97 / 107 |
| 硬违规 | 0 |
| 状态循环 | 0 |

这个结果说明：

```text
结构可解性已经有正向证据：
  匹配案例全部可完成终态。

人工级钩数还没有完全证明：
  0306W、0327W 总勾数超过人工计划软上界。
  10 个匹配案例存在阶段级超界。
```

剩余 gap 分布：

| gap | 数量 | 人工基准含义 |
|---|---:|---|
| 缺人工计划基线 | 6 | 不能做严格 reached/exceeded human 判断 |
| 低信号/短链阶段合同过压缩 | 9 | 人工短链不是“无阶段”，需要单独的 H2/H4/H5 保守边界 |
| H1 前段服务合批弱于人工 | 4 | 人工会混合携带调棚、功能线、预修、存车整理 |
| H4 大库 swap 合批弱于人工 | 2 | 人工能把库线摘解、库外衔接和入库分散压成更少钩 |
| 总勾数超过软上界 | 2 | 不能宣称全面达到人工 |

因此，人工基准要继续保持三层判定：

```text
1. final_target_satisfied 只能证明可解。
2. total_hook_count <= manual_soft_hook_upper_bound 才能证明整案达到人工。
3. phase_hook_count <= manual_phase_soft_upper_bound 才能证明 H1-H5 每个阶段都不弱于人工。
```

下一步不应只比较最终案例结果，而应逐项关闭 `artifacts/rollout_audit/rollout_gap_audit.csv` 中的结构 gap。
