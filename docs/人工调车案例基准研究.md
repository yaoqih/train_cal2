# 人工调车案例基准研究

## 1. 研究目标

本文用于把 `data/人工调车数据` 中的人工调车作业单转成可比较、可验收、可超越的基准。

核心问题不是“复刻人工钩序”，而是回答三件事：

1. 人工调车实际在优化什么结构。
2. 什么程度算达到人工。
3. 算法要在哪些指标上超过人工，而不是只在局部少一两钩。

本文结论基于当前仓库内可读取资料：

- 人工调车作业单：`data/人工调车数据/*-人工调车作业单/调车作业通知单*.xlsx`
- 取送车计划转换数据：`data/truth2/*.json`
- 业务约束文档：`docs/福州调车业务文档.md`
- 方案文档：`docs/flow_edge_structure_model_design.md`

## 2. 数据口径

### 2.1 人工单解析口径

| 后续钩 | 第二组 `顺序 / 股道 / 作业方法 / 辆数 / 备注` |

右侧还有一套空模板列，解析时不计入有效钩序。

本次解析得到：

| 人工调车作业单文件数 | 118 |
| 唯一短案号数 | 116 |
| 有效钩序总数 | 4076 |
| 单案最少钩数 | 5 |
| 单案平均钩数 | 34.54 |
| 单案中位数钩数 | 35 |
| 单案 75 分位钩数 | 40 |
| 单案 90 分位钩数 | 45 |
| 单案最多钩数 | 58 |

说明：

- `0130Z`、`0302W` 在人工单中各出现 2 个文件，后续做逐案对比时必须按“文件级”或“场景源文件级”匹配，不能只按短案号去重。
- 人工单有 109 个文件可与当前 `truth2` 短案号匹配；匹配集人工中位数为 36 钩。
- 未匹配人工单多为库内消化、直接处理或特殊短链案例，不能随意删除；它们反而能补充 `truth2` 中较少出现的异常业务形态。

### 2.2 人工动作简称

人工单使用的是现场简称：

| 简称 | 业务解释 |
|---|---|
| `机` | 机库/机走相关承接与缓冲 |
| `库` | 大库侧总称或库回动作 |
| `修1` 至 `修4` | 修1至修4库内/库外相关动作 |
| `存4` | 存4北释放口/集结口 |
| `存1/存2/存3/存5` | 存车区组织、腾位、归并 |
| `调` | 调棚/调梁相关队列 |
| `洗/油/抛/轮` | 功能线与运营线 |
| `注意存4/注意机/注意库` | 人工强调关键端别或安全注意，不应当被当作普通股道名 |

动作方法中：

| 写法 | 解释 |
|---|---|
| `+` | 挂/取/接入当前车列 |
| `-` | 摘/送/甩出当前车列 |
| `称` | 称重相关动作 |
| `顶` | 顶送/顶位 |
| `代` | 代位或临时代替位置 |
| `回` | 回库/回机等收尾语义 |

备注中的高频结构词：

| 备注 | 出现次数 | 含义 |
|---|---:|---|
| `北头` | 630 | 北端取送、靠口、整理 |
| `叉线` | 329 | 使用交叉线/中间缓冲进行重排 |
| `接` | 206 | 承接、连挂、机接或功能线接入 |
| `摘` | 119 | 摘解，常见于修库消化末端 |
| `北头摘` | 105 | 存4北释放关键动作 |
| `南` | 68 | 南端/南头相关限制 |
| `库外2` | 62 | 修库库外缓冲或摘解位置 |

这些词不是装饰备注，而是人工计划的结构信号。

## 3. 人工基准的核心结构

### 3.1 人工不是简单按目标线搬车

人工计划中最重要的结构不是“车从 A 到 B”，而是：

```text
外场/功能线/预修/调棚整理
  -> 存4北形成释放口
  -> 存4北大摘
  -> 机接承接
  -> 修1-修4分线摘解
  -> 轮/油/洗/抛等尾部功能线处理
  -> 库回或收束
```

在 118 个人工文件中，按结构信号粗分类：

| 人工案例族 | 文件数 | 含义 |
|---|---:|---|
| 标准存4释放-机接-修库消化 | 97 | 有 `存4 - 北头摘`、`机 + 接`、修库摘解 |
| 库内消化/直接处理 | 10 | 缺少标准机接，但有修库摘解和库回 |
| 混合/一般整理 | 7 | 以外场、功能线、调棚、机区整理为主 |
| 有存4释放无标准机接 | 2 | 有存4释放，但机接信号不完整 |
| 短链直接入库 | 2 | 已接近库侧，少量动作即可完成 |

匹配 `truth2` 的 109 个文件中，标准链仍有 97 个，说明 `truth2` 的主问题确实以“大库周转 + 存4释放 + 机接 + 修库消化”为主。

### 3.2 高频动作族

| 动作族 | 覆盖文件数 | 基准意义 |
|---|---:|---|
| 连接/接入信号 | 118 | 每案都存在某种承接动作 |
| 摘解信号 | 118 | 每案都存在某种摘解动作 |
| 特殊微动作 | 118 | `顶/代/称/对位/回/停` 等不可忽略 |
| 功能线服务 | 117 | 洗、油、抛、轮几乎每案参与 |
| 机区动作 | 117 | 机车位置和机区缓冲是少钩关键 |
| 北头操作 | 116 | 端别和北头靠口是人工核心操作面 |
| 修库摘解/送入 | 116 | 大库消化不是可选尾项 |
| 存车区组织 | 113 | 存1/2/3/5 的整理是主任务 |
| 调棚队列 | 107 | 调棚不是 residual |
| 存4释放 | 104 | 存4北是主释放口 |
| 机接 | 102 | 标准链的硬边界 |
| 叉线缓冲 | 104 | 人工大量利用中间缓冲减少返工 |
| 库回 | 76 | 收束信号，不能提前触发 |

这些覆盖率说明：人工调车的真实基准是全站编排，不是单一修库路径。

### 3.3 人工钩序的典型时间位置

按有效钩序位置中位数估算：

| 动作族 | 中位出现位置 | 解释 |
|---|---:|---|
| 功能线服务 | 第 8 钩 | 早期先处理洗/油/抛/轮等可消化项 |
| 机区动作 | 第 12 钩 | 机车位置与机区缓冲早早参与 |
| 调棚队列 | 第 14 钩 | 调棚/预修相关队列在中前段整理 |
| 叉线缓冲 | 第 14 钩 | 中前段大量使用叉线重排 |
| 存车区组织 | 第 17 钩 | 外场组流持续到中段 |
| 北头操作 | 第 20 钩 | 逐步靠存4北或北头组织 |
| 修库送入/接入 | 第 23 钩 | 进入库侧前的主列组织 |
| 存4释放 | 第 25.5 钩 | 标准链关键转折点 |
| 机接 | 第 27 钩 | 机接通常紧跟存4释放 |
| 修库摘解 | 第 28 钩 | 进入大库消化阶段 |
| 库回 | 第 36 钩 | 多数情况下是后段收束 |

这与业务文档中的大库作业顺序吻合：前段先完成除大库区域外的大部分调车，再集中处理大库出入和库内分散。

### 3.4 当前文档深度审查结论

原文已经足够回答“人工调车在优化什么结构”和“算法如何评价是否达到人工”，但还不足以让一个没有调车经验的人只靠本文就编出优秀计划。

不足主要有五类：

| 缺口 | 表现 | 对新人或算法的风险 |
|---|---|---|
| 决策入口不足 | 说明了 `H1-H5`，但没有讲拿到初始状态后第一眼看什么 | 容易从局部搬车开始，错过主结构 |
| 阶段退出条件不足 | 有阶段名称和案例，但缺少“什么时候可以进入下一阶段”的可操作条件 | 容易过早机接、过早库回或反复回外场 |
| 物理端别不足 | 原文强调北头，但没有把取放顺序、被阻塞线路和必须清空的前置车辆整理成手册 | 容易生成看似少钩、实际不可取放的计划 |
| 取舍规则不足 | 说明了人工不是每钩最优，但没有给出少钩与保结构冲突时的优先级 | 容易为了少一钩污染存4或打散主列 |
| 新手复核清单不足 | 有评价矩阵，但没有面向编制过程的检查项 | 不能稳定复现人工经验 |

因此，本文后续新增第 11 节，定位不是评价算法，而是把人工经验改写成“可执行调车手册”。最低目标是：不懂现场的人能按这个手册先做出结构正确、物理可行、不会犯大错的计划；优秀目标是：在结构正确的前提下，用更少的重复取送、空驶和临停完成同样任务。

## 4. 代表案例拆解

### 4.1 `0117Z`：标准完整链

人工钩数：40。

关键链条：

```text
01-22: 抛/油/机/调/存2/存5/存3 组织和功能线处理
23:    存4 - 北头
24-31: 轮/修1/修2/修3/修4 组合形成大库目标群
32:    存4 - 16 北头摘
33:    机 + 18 接
34-37: 修4/修3/修2/修1 摘解
38-40: 轮/油/库回收束
```

人工价值：

- 前 22 钩不是杂活，是在为存4释放口和机接主列塑形。
- 第 32-33 钩是硬边界：`存4大摘 -> 机接`。
- 第 34-37 钩是大库消化，不允许被提前当尾项。

算法达到人工的要求：

- 能识别 `APPROACHING_CUN4 -> PORT_READY -> MACHINE_ACCEPTED -> DEPOT_DIGEST`。
- 机接后不能为了少钩拆散主列。
- 大库摘解完成前不能 closeout。

算法超越空间：

- 前段存2/存5/存3多次往返，可用更精确的目标分组和端别可摘性减少重复整理。
- 轮/油尾项可作为 support contract 提前或并行评估，但不能破坏主列。

### 4.2 `0310W`：late 存4链

人工钩数：46。

关键链条：

```text
01-28: 抛/洗/油/调/存1/存2/存5 大量整理
29:    注意存4 - 北头代2
30-35: 抛/修2/轮/修1/修3/修4 形成库侧接入群
36:    存4 - 14 北头摘
37:    机 + 14 接
38-41: 修2/修4/修3/修1 摘解
42-46: 轮/油/库/机/库回
```

人工价值：

- 这个案例前段已经在为存4释放服务，不能把它误判成还需要重新外场补链。
- 第 36-37 钩是标准 `release -> accept`。

算法达到人工的要求：

- 一旦识别 `PORT_READY` 或接近 `PORT_READY`，禁止回外场重新拼长链。
- 目标合同应优先生成 `存4释放 / 机接 / 入库摘解` 候选。

算法超越空间：

- 人工前 28 钩外场组织较长，算法可通过同合同内局部搜索减少调/存5/存2之间的反复。
- 但少钩候选必须证明不会污染存4释放口。

### 4.3 `0103W`：库内消化与缺标准机接

人工钩数：49。

关键链条：

```text
01-18: 库/抛/油/洗/机/存1/存2 复杂组织，含称重/顶送
19-22: 轮、修2、修3、修4 摘解
23-29: 注意机、洗接、存2/存3、修4/修3/修1 摘解
30-48: 存5/存1/存2/洗/调/机 大量尾部组织
49:    库回
```

人工价值：

- 没有标准 `存4释放 -> 机接`，但存在明确修库摘解层。
- 这类案例不能被算法归为“尾项”或“普通整理”。
- 称重、顶送、对位等微动作会影响合法性。

算法达到人工的要求：

- 能从当前物理状态识别 `DEPOT_DIGEST_ONLY` 或 `DIRECT_REPAIR_ENTRY`。
- 不要求补一个不存在的标准存4长链。
- 修库摘解债务完成前不允许库回 closeout。

算法超越空间：

- 该类人工钩数高，常由功能线、称重、库内摘解和外场整理交错造成。
- 算法要用 `SPECIAL_REPAIR_PROCESS` 把称重/顶送/对位独立建模，否则少钩结果可能非法。

### 4.4 `0213W`：短链直接入库

人工钩数：5。

关键链条：

```text
01: 注意机
02: 修1 + 5 接
03: 修2 + 5
04: 注意存4 - 10 北头摘
05: 库回
```

人工价值：

- 这是极短直接处理，不需要完整外场组织。
- 算法若强行套用标准大链，会明显劣于人工。

算法达到人工的要求：

- 识别 `DIRECT_REPAIR_ENTRY`。
- 允许跳过外场集货和标准机接链。

算法超越空间：

- 这种案例人工已接近下界，算法不要追求“更少钩”而破坏结构；目标应是同钩数、更完整安全校验和可解释 trace。

### 4.5 `0306W`：短链但信号不完整

人工钩数：10。

关键链条：

```text
01-02: 机、存4北头
03-05: 修3/修1/修2 接入
06:    存4 - 15 北头摘
07-08: 机 + 2 / 机 + 1 南头
09-10: 油、库回
```

人工价值：

- 有存4释放，但机接信号不标准。
- 这类案例应被识别为低信号或混合直接处理，而不是强行生成高置信机接。

算法达到人工的要求：

- 允许 `MIXED_SIGNAL_REPAIR`。
- 低置信下禁止主动跨 `ACCEPTED`。
- 必须输出矛盾原因和保守动作边界。

## 5. 人工基准定义

### 5.1 第一层：结构基准

算法达到人工，首先不是钩数更少，而是结构不能错。

必须满足：

| 基准项 | 达标标准 |
|---|---|
| 主体流向识别 | 大库、存4、预修、调棚、功能线、机区、存车整理都有合同 |
| 存4释放口 | 能识别 `存4北头/北头摘` 的释放意义 |
| 机接硬边界 | `机 + 接` 后主列不能随意拆散 |
| 修库消化 | 修1-修4摘解完成前不能 closeout |
| 直接入库 | 短链/近库侧案例不能强套标准长链 |
| 低信号案例 | 能输出 confidence / contradiction，不能低置信主动机接 |
| 预修/调棚 | 不能当作 residual 或尾项 |
| 功能线 | 洗、油、抛、轮必须有服务完成条件 |
| 特殊微动作 | 称、顶、代、对位、停等必须作为约束，不是备注噪声 |

### 5.2 第二层：人工动作族覆盖基准

每个算法案例都应输出类似人工的结构 trace：

```json
{
  "case": "0117Z",
  "contract_groups": {
    "REPAIR_INBOUND": 16,
    "DEPOT_OUTBOUND": 16,
    "PRE_REPAIR_STAGING": 11,
    "DISPATCH_SHED_QUEUE": 8,
    "FUNCTION_LINE_SERVICE": 5,
    "YARD_REBALANCE": 18
  },
  "critical_transitions": [
    "FORM_RELEASE_GROUP",
    "CUN4_RELEASE",
    "MACHINE_ACCEPT",
    "DEPOT_DETACH",
    "TAIL_CLOSEOUT"
  ],
  "resource_claims": [
    "CUN4_NORTH_BUFFER",
    "LINK7_GATE",
    "DEPOT_SLOT_RESOURCE",
    "LOCO_CARRY_ORDER"
  ]
}
```

如果算法给不出这类 trace，就算生成了钩序，也无法证明达到人工。

### 5.3 第三层：数值基准

以人工文件为基准：

| 指标 | 人工基准 |
|---|---:|
| 全量人工单中位数 | 35 钩 |
| 匹配 `truth2` 人工单中位数 | 36 钩 |
| 标准链人工文件数 | 97 |
| 标准链覆盖率 | 82.2% |
| 匹配集标准链文件数 | 97 |
| 匹配集标准链覆盖率 | 89.0% |

建议验收时分案例族比较：

| 案例族 | 达到人工 | 超越人工 |
|---|---|---|
| 标准存4释放-机接-修库消化 | 无硬违规，钩数不高于人工 5%，关键转折一致 | 钩数低于人工，且不增加存4污染/库位冲突/机车空驶 |
| 库内消化/直接处理 | 不强套标准长链，先完成库内摘解 | 同钩或少钩，并输出更清楚的库位 swap |
| 短链直接入库 | 不明显多于人工，不额外绕行 | 保持人工钩数或更少，安全 trace 更完整 |
| 混合/低信号 | 不低置信跨硬边界 | 用补证据动作减少返工 |
| 功能线/调棚重案例 | 功能线不被当尾项 | 减少功能线与外场间反复取送 |

## 6. 如何达到人工

### 6.1 不要从“全局搜索”开始

人工计划显示，人的强项是先确定结构边界：

```text
哪些车先在外场/功能线消化
哪些车应逐步靠存4
什么时候存4释放口成形
什么时候机接后不能再拆
哪些库内车必须先出让位
哪些尾项只能最后处理
```

所以算法达到人工的首要条件是：

```text
FlowClassify
  -> TargetContract
  -> StructuralIntent
  -> ResourceRequest
  -> ContractDelta
  -> ResourceDelta
```

而不是：

```text
生成大量动作
  -> 搜索
  -> 靠分数碰运气
```

### 6.2 必须先实现人工结构识别

最低实现要求：

1. 识别 `存4 - 北头 / 北头摘`。
2. 识别 `机 + 接`。
3. 识别 `修1-修4 - n / 摘 / 库外n`。
4. 识别 `库 - 回` 只能作为收束，不是万能完成信号。
5. 识别 `调`、`洗`、`油`、`抛`、`轮` 都是独立服务项。
6. 识别 `叉线`、`北头` 是端别/缓冲结构，不是普通备注。
7. 识别 `称/顶/代/对/停` 是硬约束候选。

### 6.3 必须先通过人工案例族回放

建议先建立 9 个金标案例：

| 案例 | 类型 | 必验点 |
|---|---|---|

## 7. 如何超越人工

### 7.1 超越人工不是全面模仿

人工的价值是结构判断，不是每一钩都最优。

算法应继承：

- 存4释放口保护。
- 机接后主列保护。
- 大库先出后进。
- 预修/调棚/功能线独立服务。
- 尾项必须在主合同完成后收束。

算法可以超越：

- 外场反复取送。
- 存2/存5/存3 多次交叉线重排。
- 功能线和大库主列混合携带导致的重复摘挂。
- 机车空驶和端别切换。
- 库位 swap 依赖靠经验处理。

### 7.2 超越人工的可量化指标

建议将“超越人工”定义为多指标支配，而不是单看钩数。

硬性前提：

```text
hard_violation_count = 0
所有车辆最终目标满足
大库台位规则满足
关门车/重车/称重规则满足
运行干涉和临停容量满足
求解时间 <= 5 分钟
```

在硬性前提满足后比较：

| 指标 | 超越标准 |
|---|---|
| 总钩数 | 标准链案例低于人工；短链案例不高于人工 |
| 大库进出次数 | 不多于人工，优先接近“一进一出” |
| 存4污染次数 | 少于人工或等于人工 |
| 叉线/临停使用 | 少于人工，或证明减少后续返工 |
| 机车空驶距离 | 少于人工 |
| 道岔扳动次数 | 少于人工 |
| 修库 swap 等待 | 明确先出后进，减少冲突 |
| 计划可解释性 | 每个关键动作都有 ContractDelta / ResourceDelta |

### 7.3 超越策略

#### 策略一：把人工“中段靠存4”提前变成资源目标

人工通常在中段形成存4释放口。算法可以更早判断：

```text
哪些外场车最终服务 REPAIR_INBOUND
哪些车只是 YARD_REBALANCE
哪些车会污染 CUN4_NORTH_BUFFER
```

提前禁止随机占用存4北，可减少后段清口钩数。

#### 策略二：用 DepotSlotGraph 替代经验判断

人工能大致判断哪些库位要先让，但不一定全局最优。

算法应显式输出：

```text
inbound_vehicle -> target slot/band
blocking outbound vehicle
released slot
reserved slot
swap dependency
```

只要 slot/band 依赖正确，算法有机会减少“先进错位再调整”。

#### 策略三：保留 ordered carry sequence

人工少钩依赖车列顺序和尾端可摘性。

算法不能只用 `CARRYING_CONTRACT_GROUP` 这种枚举压缩，必须保留：

```text
ordered_contract_segments
accessible_end
detachable_from_tail
```

否则看似少钩，实际会产生非法摘挂或需要返工。

#### 策略四：同结构等级内做少钩搜索

少钩搜索只能在这些前提后启动：

```text
同一个 TargetContract
同一个 StructuralIntent
无 ContractDelta 硬违反
无 ResourceDelta 硬违反
不破坏更高优先级合同
```

这样才能避免“少一钩但错主结构”。

#### 策略五：功能线和调棚独立成合同

人工计划里功能线几乎每案出现，调棚覆盖 107 个文件。

算法若把它们当 residual，会在后段产生大量补丁。

必须有：

```text
FUNCTION_LINE_SERVICE
DISPATCH_SHED_QUEUE
PRE_REPAIR_STAGING
```

并且能判断服务完成、退出服务线、是否可转大库。


## 9. 结论

人工调车的基准不是一条固定 workflow，而是一套稳定结构经验：

```text
前段消化功能线和外场整理；
中段保护并形成存4北释放口；
关键时刻做存4大摘和机接；
后段连续完成修库摘解；
最后才处理库回和尾项。
```

要达到人工，算法必须先学会这些结构边界，并且能解释每一步为什么是履约。

要超越人工，算法不能盲目复刻人工钩序，而应在同样结构价值下利用：

- 更精确的合同分类。
- 更严格的资源仲裁。
- 更细的大库 slot/band swap。
- 更可靠的机车携带顺序。
- 同结构等级内的小范围少钩搜索。

本研究的输出是人工结构基准：存4释放、机接、修库摘解、库回、大库 swap、机车携带顺序、功能线和调棚服务等。这些基准用于后续评价算法动作是否保留人工结构价值。

## 10. 人工基准如何进入当前 P10 评估

人工基准不再用旧的结构化 rollout 数字表达。当前统一看物理 runtime 产物：

```text
artifacts/current_truth2_eval/
docs/P10_人工差距结构诊断.md
```

人工基准在当前评估中承担三件事：

| 基准 | 当前字段 | 判定 |
|---|---|---|
| 整案勾数 | `manual_vs_solver_case_compare.csv` 的 `manual_hook_count`、`solver_business_hook_count` | `solver <= manual_soft_hook_upper_bound` 才能说整案达到人工 |
| 远端消化 | `manual_remote_hook_count`、`solver_remote_business_hook_count`、`solver_remote_session_count` | 远端勾数和 session 不能明显高于人工 |
| 远端/非远端来回切换 | `case_summary.csv` 的 `remote_business_transition_count` | 当前先用全量 P50/P90 控制；后续补人工切换解析后改为逐案 `solver <= manual + 1` |

因此，人工基准要继续保持三层判定：

```text
1. final_target_satisfied 只能证明可解。
2. total_hook_count <= manual_soft_hook_upper_bound 才能证明整案达到人工。
3. remote hook / remote session / remote business transition 同时受控，才能证明远端组织达到人工。
```

下一步不应只比较最终案例结果，而应逐项关闭 `structure_work_audit.csv` 和 `structural_repair_acceptance.csv` 中的结构 gap。

## 11. 面向无经验编制者的人工调车决策手册

本节把前面的统计结论改写成一套可执行流程。它不要求编制者知道所有现场经验，但要求每一步都回答清楚三个问题：

```text
当前最重要的业务债务是什么？
这一步会不会破坏后面更重要的结构？
做完这一勾以后，下一阶段的入口是否更清楚？
```

一页式流程：

```text
盘点车辆债务
  -> 判断案例族
  -> 查阻塞和北端可取放
  -> 先做 H1 前段服务与外场整理
  -> H2 只围绕存4释放口塑形
  -> H3 连续完成 存4北头摘 + 机接
  -> H4 连续完成修库摘解/让位
  -> H5 清尾项和库回
  -> 用复核清单逐项验收
```

如果某一步无法判断，就不要跳到下一步；回到“盘点车辆债务”和“查阻塞和北端可取放”。

### 11.1 先读状态，不要先想动作

拿到一个案例后，先做 5 分钟状态盘点。不要先问“哪辆车从 A 到 B”，而要先给车辆分组：

本文第 11 节在规则描述中优先使用股道全称，括号内给出现场简称，避免把人工单简称误当成不同线路：

| 全称 | 现场简称/规范简称 | 新手记忆 |
|---|---|---|
| `机走北1线` | 机北1 | 临停线，挡 `存1线` 和 `机走北2线` 通路 |
| `机走北2线` | 机北2 | 临停线，挡渡4/渡5方向和机区通路 |
| `机走北` | 机北3 | `机走棚` 北侧前置线 |
| `机走棚` | 机棚 | 预修/存车复合线，挡 `机走线南` 和 `洗罐油漆北` |
| `洗罐油漆北` | 洗油北 | `洗罐线北` 与 `油漆线` 的前置门槛 |
| `洗罐线北` | 洗北 | `洗罐站` 北侧缓存 |
| `洗罐站` | 洗南 | 洗罐作业线 |
| `调梁线北` | 调北 | `调梁棚` 北侧前置线 |
| `调梁棚` | 调棚 | 调梁和南端预修作业线 |
| `存4线` | 存4北 | 正式出段集结和释放口 |
| `存4线南` | 存4南 | 过程临停，不是正式终点 |

| 分组 | 判断方法 | 人工含义 | 首要动作 |
|---|---|---|---|
| 大库债务车 | 目标含 `修1库内` 至 `修4库内`，或已经在修库内/库外但未完成摘解 | 主合同，决定整案骨架 | 规划入库、出让位、摘解顺序 |
| 出段/存4债务车 | 目标是 `存4线`，或已经完成工序等待集结 | 释放口和交站目标 | 保护 `存4线` 北端，不随便污染 |
| 预修/调梁债务车 | 目标或当前位置在 `预修线`、`机走棚`、`调梁棚` | 前段服务合同 | 先完成可独立服务，再判断是否转大库 |
| 功能线债务车 | 涉及 `洗罐站`、`油漆线`、`抛丸线`、`卸轮线`、`机库线`称重 | 不可当尾项 | 尽早处理能直接消化的车 |
| 存车整理车 | 在 `存1线`、`存2线`、`存3线`、`存5线北`、`存5线南` 等待分流 | 队列塑形材料 | 按后续目标归并，少做无目的换线 |
| 阻塞车 | 停在某条串行通道的北段或库外段，挡住南段/库内/作业线 | 物理前置条件 | 先清阻塞，再取放被阻塞线路 |

盘点后先写出本案属于哪一种：

| 案例族 | 判断信号 | 推荐主流程 |
|---|---|---|
| 标准存4释放-机接-修库消化 | 有大量大库债务，人工常见 `存4 - 北头摘`、`机 + 接` | H1 外场服务 -> H2 存4塑形 -> H3 释放/机接 -> H4 修库摘解 -> H5 收尾 |
| 库内消化/直接处理 | 修库内外已有可摘解债务，缺少标准 `存4释放 -> 机接` | 直接进入 H4，先消化库内/库外，再收尾 |
| 短链直接入库 | 总钩数很少，车辆已靠近库侧或存4口 | 不补长链，完成近端接入/摘解即可 |
| 混合低信号 | 有存4或机区信号，但缺标准机接或摘解链 | 保守推进，只在证据充分时跨硬边界 |

人工样本给出的强信号是：118 个有效人工文件中，97 个是标准完整链，14 个是库内消化，6 个是混合低信号，1 个是短链直接入库。也就是说，默认可以先怀疑是标准链，但一旦发现“已经在库侧能直接消化”，就不要强行补一条标准长链。

### 11.2 五阶段人工流程

人工优秀计划不是平均用力，而是把全案压成五个阶段。

| 阶段 | 现场语言 | 主要目标 | 退出条件 | 禁止事项 |
|---|---|---|---|---|
| H1 前段服务与外场整理 | 洗、油、抛、轮、调、存1/2/3/5、机区整理 | 先处理独立服务项，把大库债务车逐步归组 | 存4释放口方向明确，主列候选开始成形 | 不许过早占死 `存4线` 北端；不许把功能线当作最后补丁 |
| H2 存4释放口塑形 | `存4 - 北头`、`北头代`、修库接入群成形 | 把要进库/出库的主列组织到可释放状态 | 可以执行 `存4 - n 北头摘`，且机车能立即承接 | 不许释放后还回外场反复拼车 |
| H3 严格释放与机接 | `存4 - n 北头摘` 后紧跟 `机 + n 接` | 完成主列从存4释放到机区承接 | 主列已经被机接，顺序可用于修库摘解 | 不许把 H3 拆散；不许机接后打散主列做无关整理 |
| H4 修库消化 | `修1/2/3/4 - n 摘`、库外摘解 | 依次送入/摘解修库车辆，处理库位冲突 | 修库债务基本清完，剩余只是不影响主结构的尾项 | 不许在修库摘解未完成时库回 |
| H5 收尾与库回 | 轮/油/机/调/存车尾项，`库回` | 清尾项，机车回库或状态收束 | 全目标满足且无阻塞尾债 | 不许把 `库回` 当作修库完成的替代信号 |

统计上，标准完整链的阶段长度很稳定：

| 阶段 | 标准链中位钩数 | 75 分位 | 说明 |
|---|---:|---:|---|
| H1 | 18 | 22 | 大量功能线、存车、调棚组织都在这里完成 |
| H2 | 6 | 7 | 存4释放口塑形通常很短，不应无限拖长 |
| H3 | 2 | 2 | `存4北头摘 -> 机接` 基本是连续两钩 |
| H4 | 3 | 4 | 修库摘解是短而硬的集中消化 |
| H5 | 1 | 4 | 多数只是库回，复杂尾项才会拉长 |

这给新手一个非常实用的判断：如果 H3/H4 被拉成很多钩，通常说明前段没有把主列塑好，或者机接后又被错误打散。

### 11.3 标准链的硬顺序

标准链的核心不是“先进哪个库”，而是三段硬边界：

```text
形成存4释放口
  -> 存4线 北头摘
  -> 机库线/机区 接
  -> 修1/修2/修3/修4 连续摘解
```

人工数据中，97 个标准链里：

| 转折 | 统计事实 | 调车含义 |
|---|---:|---|
| `存4北头摘 -> 机接` | 96/97 是下一钩完成，只有 1 例间隔 3 钩 | 释放和承接是一组动作，不应拆开 |
| `机接 -> 修库摘解` | 97/97 是下一钩进入摘解 | 机接不是停车成果，而是进入 H4 的入口 |
| H4 修库摘解 | 中位 3 钩，75 分位 4 钩 | 大库消化应集中完成，不应夹杂大量外场整理 |

因此，编制计划时要把 H3 当作原子结构：

```text
如果不能保证释放后马上机接，就不要释放；
如果机接后不能马上开始修库摘解，就不要机接；
如果摘解顺序会被车列尾端限制，就回到 H2 先重排。
```

### 11.4 每阶段怎么编

#### H1：前段服务与外场整理

目标不是把所有车都“搬到目标”，而是把后续债务变简单。

优先顺序：

1. 先处理已经在功能线或调棚、预修的车，能完成服务并退出的先退出。
2. 对存车线车辆按最终债务归并：大库债务归大库主列，功能线债务归服务线，存4债务归出段集结。
3. 使用 `存2线`、`存3线`、`存5线北/南` 时，优先保持北端可取，不要把后面急用车压到南端。
4. 对 `机库线` 称重、机走棚预修等特殊动作，按业务硬约束处理，不要当作普通缓存。
5. H1 结束前要能说清：哪些车会组成存4释放主列，哪些车暂时不参与主列。

常见好动作：

| 动作 | 好在哪里 |
|---|---|
| 功能线先接再送出 | 减少 H5 尾项 |
| 调棚/预修先消化一批 | 避免后段再跨区回头 |
| 存2/存5 与调棚交替整理 | 用前段时间把顺序塑好 |

常见坏动作：

| 动作 | 问题 |
|---|---|
| 没有目标地把车塞进 `存4线` | 污染释放口，H2 要额外清口 |
| 把要先进库的车压在不可取端 | H3/H4 会因为尾端不可摘返工 |
| 为少一钩把功能线车混入大库主列 | 后面必须拆出来，实际更慢 |

#### H2：存4释放口塑形

H2 的任务是把“能释放的一组车”组织到 `存4线` 北端或等价释放位置。这里要保护三个资源：

| 资源 | 要求 |
|---|---|
| `存4线` 北端 | 只能放即将释放或正式出段集结的车 |
| 机车承接通路 | 释放后机车必须能接，不被 `机走北1线`、`机走北2线`、`机走棚`、`洗罐油漆北` 等阻塞打断 |
| 车列顺序 | 进入 H4 时，修库摘解应能从可摘端连续完成 |

进入 H2 的条件：

```text
大库债务车已经基本明确；
主列中不应夹入大量无关功能线车；
存4北端或接近存4北端的位置可整理出释放口。
```

退出 H2 的条件：

```text
能执行一勾明确的 存4线 - n 北头摘；
下一勾能执行机区接入；
接入后至少能连续完成一批修库摘解。
```

H2 可以使用 `北头代`、`北头接`、`叉线` 等动作，但这些动作都必须服务于释放口，不是普通换线。

#### H3：释放与机接

H3 是全案最硬的边界。优秀人工几乎总是：

```text
存4线 - n 北头摘
机库线/机区 + n 接
```

执行前检查：

| 检查项 | 必须满足 |
|---|---|
| 释放数量 | `n` 对得上要进入主列的车数 |
| 机车位置 | 能从当前机车位置到达承接点 |
| 通路 | `机走北1线`、`机走北2线`、`机走北`、`机走棚`、`洗罐油漆北` 等不会阻塞必要路径 |
| 车列顺序 | 接后修库摘解顺序可行 |
| 后续库位 | 修1至修4至少有可执行摘解/入库方案 |

H3 禁忌：

```text
释放后先做无关洗/油/调/存车整理；
机接后又把主列送回外场；
只因为机区空着就提前机接；
低信号案例强行构造标准机接。
```

#### H4：修库消化

H4 的目标是把主列按修库目标拆掉，同时处理库外/库内前后阻塞。

基本顺序：

1. 先看四条修库线各自的库外是否挡住库内。
2. 先处理会释放关键库位的出库/摘解动作。
3. 再把主列按可摘端顺序送入 `修1库内` 至 `修4库内`。
4. 如果修库目标车辆在主列中顺序不适合连续摘解，不要硬送，回到临时 staging 重新分段。
5. 修库消化完成前，不允许用 `库回` 结束。

修库动作的人工写法常见：

| 人工写法 | 含义 |
|---|---|
| `修1 - 4 摘` | 向修1库内摘解 4 辆或从库侧摘出 |
| `修4 - 5 库外2` | 通过修4库外缓冲处理第 2 类库外位置/占位 |
| `修2 + 7 接` | 向修2方向接入一组，可能是 H2/H4 交界动作 |

文档和求解器都不能只看 `+/-`，还要结合阶段判断它是接入、摘解、让位还是恢复。

#### H5：尾项和库回

H5 不是“剩下什么做什么”，而是主结构完成后的收束。

可以进入 H5 的条件：

```text
修库摘解债务完成或不适用；
存4正式集结没有被临停车污染；
功能线关键服务完成；
被阻塞线路没有仍待取放的车；
机车回库不会打断未完成合同。
```

H5 常见内容：

| 内容 | 说明 |
|---|---|
| 轮/油/洗/抛尾项 | 处理不会反过来影响大库主链的功能线车 |
| 存车线收束 | 把暂存车归到最终存车线或可交付位置 |
| 调棚/预修尾项 | 清理前段留下的服务线尾债 |
| `库回` | 机车/大库收束信号，不能替代业务完成 |

### 11.5 物理端别和阻塞手册

人工计划中 `北头` 出现 628 次，`北头摘` 出现 105 次，说明端别不是备注，而是决定能不能少钩的核心约束。

当前物理口径：

```text
取车和放车都按北端顺序操作；
被北段或库外车辆阻塞的线路，不能直接取放车；
必须先清空阻塞车辆，再操作被阻塞线路；
存5线南只能从存5线北一侧操作，不能从 L12 侧直接操作。
```

必须记住的阻塞关系：

| 阻塞车所在股道 | 被阻塞股道/作业 | 编制规则 |
|---|---|---|
| `修1库外` 至 `修4库外` | 对应 `修1库内` 至 `修4库内` | 库外不清，库内不能直接取放 |
| `机走棚` | `机走线南`、`洗罐油漆北` | 先清 `机走棚`，再去 `机走线南` 或 `洗罐油漆北` |
| `机走北` | `机走棚` | 先清 `机走北`，才能操作 `机走棚` |
| `机走北2线` | 渡4/渡5方向、`机走北2线`后续路径 | `机走北2线` 有车会挡关键机区通路 |
| `机走北1线` | `存1线`、`机走北2线` | `机走北1线` 不能随意当缓存 |
| `洗罐油漆北` | `洗罐线北`、`油漆线` | `洗罐油漆北` 是 `洗罐线北` 和 `油漆线` 的前置门槛 |
| `洗罐线北` | `洗罐站` | `洗罐线北` 不清，`洗罐站` 不能直接取放 |
| `调梁线北` | `调梁棚` | `调梁线北` 挡 `调梁棚` |
| `存4线`、`存3线` | `存4线南` | `存4线南` 只是临停，不能无视北侧阻塞 |
| `存5线北` | `存5线南` | `存5线南` 只能从北侧经 `存5线北` 操作 |

新手最容易犯的错是“图上看有路，实际北端被车挡住”。处理方法是：每计划一勾前，先问目标线北侧有没有前置车；有的话，这一勾不是取放目标车，而是清前置车。

### 11.6 车列顺序和摘挂原则

少钩计划的核心是“挂上以后能按尾端摘掉”。人工不是随便多挂，而是在维护可摘顺序。

编制时对每个计划候选都要写一行：

```text
挂车顺序：A-B-C-D
下一步摘解顺序：D/C/B/A 中哪些会先摘
是否与目标库线顺序一致：是/否
如果否，在哪里重排：存2/存3/存5/调棚/临停
```

原则：

| 原则 | 说明 |
|---|---|
| 先考虑尾端可摘 | 不能只看目标线相同，要看车在列中的位置 |
| 同一库线尽量成段 | 修1、修2、修3、修4目标车最好形成可连续摘解的小段 |
| 不把尾项夹进主列中间 | 洗/油/轮尾项夹在修库车中间会增加摘挂 |
| 临停只服务重排 | `机走北1线`、`机走北2线`、`机走线南`、`洗罐油漆北`、`存4线南` 不能成为最终停车方案 |
| 称重车要保留称重顺序 | 经过 `机库线` 称重时，整列长度和称重车位置都要合法 |

如果发现主列顺序不支持连续摘解，不要在 H4 硬送；应该回到 H1/H2 通过 `叉线` 或临时 staging 做一次有目的的重排。

### 11.7 少钩优化的优先级

当“少一钩”和“保结构”冲突时，按下面优先级取舍：

```text
硬物理可达
  > 端别可取放
  > 不污染存4释放口
  > 机接后主列不拆散
  > 修库消化连续
  > 功能线服务不遗漏
  > 少钩
  > 少空驶/少道岔
```

可以接受多一钩的情况：

| 情况 | 为什么 |
|---|---|
| 为了清阻塞车 | 否则目标线不可取放 |
| 为了保护存4北端 | 存4被污染会导致后段多次清口 |
| 为了让修库目标成段 | H4 连续摘解比前段少一钩更重要 |
| 为了完成称重/对位 | 特殊业务不能被普通搬车覆盖 |

不应接受少一钩的情况：

| 情况 | 后果 |
|---|---|
| 机接后又回外场整理 | 破坏 H3/H4 硬边界 |
| 临停线作为最终目标 | 物理过程合法但业务非法 |
| 存5线南从 L12 侧直接取放 | 物理操作口错误 |
| 库外有车仍直接操作库内 | 大库阻塞未解除 |
| H4 未完成就库回 | 计划看似结束，业务未完成 |

### 11.8 人工高频结构的解释

人工单里一些词看似备注，实际是决策提示。

| 词 | 人工意图 | 新人解释 |
|---|---|---|
| `北头` | 从北端组织可取放顺序 | 先看北端第一辆是不是你要动的车 |
| `叉线` | 利用交叉线或中间通路重排 | 不是无目的绕行，是为后面连续摘解塑形 |
| `接` | 承接、连挂、形成主列 | 接后通常进入下一阶段，不是普通停车 |
| `摘` | 从列中摘出或向目标线甩出 | 要检查尾端可摘 |
| `北头摘` | 存4释放硬信号 | 下一钩通常应机接 |
| `库外1/2/3` | 大库外缓冲/占位提示 | 先处理库外阻塞和库位让位 |
| `代` | 临时代位/替代位置 | 说明该钩服务于后续结构，不应当作最终目标 |
| `顶` | 顶送或顶位 | 需要检查前方容量和端别 |
| `称` | 称重 | 必须经过机库称重能力，不能用普通到达替代 |
| `回` | 库回/收束 | 只在主债务完成后有效 |
| `停` | 停留或临时状态 | 要确认不是未完成尾债 |

### 11.9 手工编制工作表

建议每个案例先填这个表，再写钩计划：

| 问题 | 填写内容 |
|---|---|
| 本案案例族 | 标准链 / 库内消化 / 短链直接 / 混合低信号 |
| 大库债务车数量和目标 | 修1、修2、修3、修4分别多少辆，是否已有库外阻塞 |
| 功能线债务 | 洗、油、抛、轮、称重分别哪些车 |
| 预修/调棚债务 | 预修、机走棚、调梁棚哪些车未完成 |
| 存4释放口状态 | 存4北端是否可用，是否有无关车污染 |
| 存5状态 | 存5线北是否挡住存5线南 |
| 机区通路状态 | `机走北1线`、`机走北2线`、`机走北`、`机走棚`、`洗罐油漆北` 是否有阻塞车 |
| 初始主列候选 | 哪些车准备进入 H3/H4 主列 |
| 主列摘解顺序 | 接后按什么顺序送修1/修2/修3/修4 |
| 临停/重排点 | 如果顺序不对，在哪条线重排 |
| 预计 H3 钩 | `存4线 - n 北头摘` 和 `机库线/机区 + n 接` 是否连续 |
| H4 完成标准 | 哪些修库动作完成后才允许收尾 |
| H5 尾项 | 剩余功能线、存车线、库回动作 |

如果这张表填不出来，说明还没有进入写钩计划的阶段。

### 11.10 新人复核清单

计划写完后逐项检查：

1. 是否所有取放都按北端顺序可达。
2. 是否所有被阻塞线路都先清阻塞车。
3. `存5线南` 是否只从 `存5线北` 一侧操作。
4. `存4线` 北端是否只服务释放口或正式集结。
5. 是否存在 `存4北头摘` 后没有立即机接。
6. 是否存在机接后没有立即进入修库消化。
7. 修库摘解前，修库库外是否清楚。
8. 修库目标车在车列中是否尾端可摘。
9. 功能线车辆是否被夹在大库主列中间。
10. 称重、顶送、对位、关门车、重车等特殊约束是否被显式处理。
11. 临停线是否只作过程用途，没有成为最终目标。
12. `库回` 是否只在主债务完成后出现。

只要有一项回答“不确定”，这不是优秀计划，必须回到相应阶段重排。

### 11.11 给求解器的实现含义

如果把本手册落到算法，不能只实现一个全局最短路。至少要有以下层次：

| 层 | 必须输出 | 用途 |
|---|---|---|
| CaseClassifier | 标准链/库内消化/短链/混合低信号 | 决定是否强制 H1-H5 |
| DebtLedger | 大库、存4、功能线、预修、调棚、存车整理债务 | 知道每勾在还什么债 |
| PhaseGate | H1-H5 入口、退出、跳过理由 | 防止过早机接或库回 |
| ResourceGate | 存4口、修库库外、机区通路、功能线容量、临停线 | 防止物理可达但业务破坏 |
| CarryOrderModel | 有序车列、尾端可摘、整列长度 | 防止少钩假象 |
| DeltaAudit | 每勾前后债务变化 | 证明动作不是乱动 |

求解器如果只输出“车辆最终到位”，还没有达到人工；如果能解释每勾处于哪个阶段、还了什么债、占用了什么资源、为什么没有破坏后续结构，才接近人工水平。

## 12. 边界和操作哲学分层

第 11 节解决“怎么写计划”，本节解决“为什么要这样写”。人工调车不是把车从一个点搬到另一个点，而是在连续占用、释放和转换边界。每一钩都会同时改变车辆位置、车列顺序、可达关系、线路容量、阶段承诺和剩余债务。优秀计划的核心不是局部动作漂亮，而是始终知道当前动作跨过了什么边界、消耗了什么资源、保留了什么选择、创造了什么后续出口。

### 12.1 总原则：先定边界，再定动作

写计划前先判定边界，不先选择动作。边界是动作前必须被确认的条件，也是动作后最难无代价恢复的结构。一个动作只有同时说清以下五件事，才可以进入计划：

1. 当前动作跨过或占用了哪类边界。
2. 当前动作偿还、隔离或推迟了哪类债务。
3. 当前动作消耗了哪些线路、端侧、折返长度、临停空间或车列顺序资源。
4. 当前动作让哪些后续动作变得可行，或者让哪些后续动作变得不可行。
5. 当前动作的退出信号是什么，什么时候必须离开当前状态。

无法回答这些问题时，不应执行高承诺动作。此时只能做低承诺的盘点、清阻塞、重排、补证据或释放资源动作。

### 12.2 七类边界

| 边界类别 | 核心问题 | 常见错误倾向 | 判定口径 |
|---|---|---|---|
| 物理连通边界 | 通路、岔区、平交道、尽头、阻塞关系、折返长度是否真实可行 | 只按拓扑连通判断可达 | 取放前必须同时确认通路、可操作端、串行阻塞、整列长度和目标容量 |
| 操作口边界 | 线路实际允许从哪一侧取放 | 把一条股道当成两端都能自由操作 | 以规定操作侧建立可取顺序，另一侧不能默认可用 |
| 车列顺序边界 | 目标车是否位于可取端，摘挂后顺序是否仍可服务后续动作 | 忽略夹车和尾端可摘条件 | 任何摘放都按车列方向和北端顺序审查，不能只看车号是否在线上 |
| 资源占用边界 | 临停线、存车线、库外、功能线、牵出空间、机车携带能力是否被占用 | 把过程资源当成无限缓存 | 每次占用都必须有退出条件，过程资源不得变成无解释的最终状态 |
| 业务债务边界 | 车辆是否完成应完成的检修、洗罐、油漆、抛丸、称重、装卸、集结或存放目标 | 把物理到达当成业务完成 | 没有完成证据时债务仍在，不能用位置变化掩盖业务未完成 |
| 阶段承诺边界 | 当前是否已经进入 H1-H5 中更高承诺阶段 | 提前机接、提前库回、机接后回外场整理 | 阶段入口、阶段退出和跳阶段理由必须明确，高承诺阶段不能被低承诺动作反向打散 |
| 证据置信边界 | 当前信息是否足以支持跨越硬边界 | 低置信信息下主动构造主链 | 证据不足时选择保守推进，先补状态、清阻塞、保留可逆性 |

这七类边界共同决定动作是否合法、是否合时、是否值得。任一高位边界未通过，低位优化都没有意义。

### 12.3 三层判断：能不能做、该不该现在做、值不值得做

调车判断必须自上而下分三层。

第一层是安全合法层，回答“能不能做”。它包含物理可达、端侧可操作、阻塞已解除、容量足够、整列长度足够、临停线用途合法、最终位置合法。此层不通过，动作直接无效。

第二层是结构正确层，回答“该不该现在做”。它包含阶段是否合适、债务是否优先、资源是否应当先释放、车列顺序是否正在为主链塑形、是否会制造更难处理的夹车或阻塞。此层不通过，动作即使物理可行，也不应进入计划。

第三层是效率优化层，回答“值不值得这样做”。它包含总钩数、空走、重复取放、临停次数、重排次数、主列连续性、尾项收束速度。此层只能在前两层均通过后比较方案，不能反过来为违反安全或结构的动作辩护。

三层之间的关系是单向约束：效率服从结构，结构服从安全。短钩数不能覆盖硬边界错误，局部顺手不能覆盖阶段错误，物理可行不能覆盖业务未完成。

### 12.4 硬边界、软边界、策略边界

所有规则不应混在同一强度下处理，应分成三类。

硬边界是不可跨越边界。违反硬边界时，计划必须被判无效。硬边界包括物理不可达、操作端错误、阻塞未解除、容量或长度不足、车列顺序不可摘、最终目标非法、业务债务未完成却被关闭、阶段承诺被反向破坏。硬边界不接受钩数补偿，也不接受事后解释。

软边界是可临时跨越但必须带补偿的边界。软边界通常来自非推荐但物理可行的通路、短时间临停、临时占用过程资源、低置信状态下的保守中间动作。软边界一旦被使用，必须同时记录使用理由、占用范围、退出时点、补偿动作和不得继续扩大的限制。软边界不能沉淀成最终状态，也不能被连续使用到掩盖主债务。

策略边界是可比较的选择边界。策略边界决定先处理哪组债务、先释放哪个资源、采用哪条重排路线、何时合并车列、何时分拆车列。策略边界允许按钩数、稳定性、余量和人工习惯取舍，但取舍前必须证明硬边界全部通过、软边界全部受控。

这三类边界的价值不同。硬边界保证计划成立，软边界保证计划可恢复，策略边界保证计划更优。

### 12.5 阶段推进哲学

阶段不是标签，而是承诺强度的变化。

H1 的目标是打开可能性。此时应优先盘点状态、识别债务、清除显性阻塞、保留多个后续出口。H1 不追求立即形成主列，不应过早消耗关键释放口。

H2 的目标是收缩可能性。此时应把分散债务整理成可连续处理的结构，释放关键通路，修正车列顺序，隔离无关车辆。H2 的好坏决定 H3 是否干净。

H3 的目标是正式承诺。进入 H3 后，计划已经把主要债务绑定到主列和核心资源上。此时不再把主列当成普通临停车列，也不再回到外场做无关整理。

H4 的目标是兑现承诺。此时动作应围绕主债务连续消化，不能让功能线尾项、存车整理或临停占用打断主链。H4 的完成标准不是车移动过，而是目标债务被证据关闭。

H5 的目标是关闭残余。此时处理尾项、恢复资源、消除临停、完成库回和存车状态收束。H5 不能重新打开 H3/H4 已经关闭的主债务。

阶段推进的基本哲学是：早期保留选择，中期塑造结构，承诺后快速兑现，收尾时关闭残债。越靠后，动作越应当确定；越靠前，动作越应当可逆。

### 12.6 债务哲学

每一辆未完成目标的车、每一个阻塞位置、每一段错误顺序、每一次临时占用、每一条缺失证据，都是债务。计划不是动作列表，而是债务清偿顺序。

债务至少分为五类：

| 债务类型 | 含义 | 优先口径 |
|---|---|---|
| 目标债务 | 车辆还没有到达业务最终状态 | 最终必须关闭，但不一定最先处理 |
| 阻塞债务 | 某车挡住另一条线路、作业位、操作端或通路 | 通常先于被阻塞目标处理 |
| 顺序债务 | 车列顺序导致目标车不可摘、不可送或会夹带无关车 | 在主链承诺前尽量消化 |
| 资源债务 | 临停线、库外、功能线、牵出空间、关键通路被占用 | 必须有退出信号，不能无期限滚动 |
| 证据债务 | 状态不足以判断动作是否合法或合时 | 未补清前不得跨硬边界 |

债务的优先级不只由目标重要性决定，也由它对后续动作的封锁能力决定。阻塞债务和证据债务经常比目标债务更急，因为它们会让正确动作不可判定或不可执行。优秀计划会先处理会关闭未来选择的债务，再处理只影响局部位置的债务。

### 12.7 可逆性和承诺

人工调车的高水平不在于每一步都最短，而在于该可逆时保持可逆，该承诺时坚决承诺。

可逆动作用于盘点、清阻塞、重排和释放资源。它的价值是增加后续可操作性，而不是立即关闭最终目标。可逆动作不应占用关键资源太久，也不应引入新的顺序债务。

承诺动作用于形成主列、进入核心通道、开始连续摘解或关闭重要业务债务。承诺动作一旦发生，后续动作就应围绕该承诺快速收束。承诺后频繁回到前序整理，说明承诺过早或前序塑形不足。

判断一个动作是否过早承诺，要看它是否减少了可操作口、绑定了关键资源、改变了车列主序、迫使后续必须连续执行。如果答案为是，而当前债务和资源尚未准备好，就应推迟该动作。

### 12.8 反向审查法

每写一钩前，从未来反查当前动作，而不是只从当前位置向前搜索。反向审查必须回答：

1. 这钩之后，下一类必须动作是什么。
2. 这钩之后，哪个边界被打开，哪个边界被关闭。
3. 这钩之后，哪项债务减少，哪项债务增加。
4. 这钩之后，关键资源还能否退出。
5. 这钩错了之后，是否还能不跨硬边界恢复。

如果只能说明“车能到那里”，不能说明“下一步为什么更好”，这钩就不是成熟动作。成熟动作必须带有方向性：要么打开关键出口，要么消除关键阻塞，要么修正主序，要么关闭明确债务。

### 12.9 人工和求解器的共同输出标准

无论人工还是求解器，每一钩都应能输出同一套解释：

| 字段 | 必须说明 |
|---|---|
| 阶段 | 当前处于 H1-H5 哪一阶段，是否发生阶段切换 |
| 债务 | 本钩偿还、隔离、推迟或新增了哪类债务 |
| 边界 | 本钩通过了哪些硬边界，使用了哪些软边界 |
| 资源 | 本钩占用或释放了哪些线路、端侧、空间、车列顺序资源 |
| 顺序 | 本钩前后车列方向、可摘端和夹车关系是否变化 |
| 退出 | 本钩形成的临时状态什么时候必须关闭 |
| 置信 | 支持本钩的状态证据是否足够，缺失证据如何保守处理 |

只输出起点、终点和车辆编号，不足以证明计划正确。没有边界解释的动作，很容易把物理可达误判为业务可行；没有债务解释的动作，很容易把移动误判为进展；没有退出解释的动作，很容易把临停误判为收束。

### 12.10 “优秀计划”的判定

优秀计划不是单纯钩数最少，而是在硬边界零违反、软边界可追踪、策略选择可解释的前提下，用尽量少的动作完成全部债务。

判断计划是否优秀，应看以下结果：

1. 所有取放都符合真实可达、真实端侧、真实顺序、真实长度和真实容量。
2. 所有阻塞都在被阻塞线路动作前解除或显式受控。
3. 所有临时占用都有退出信号，没有过程状态被偷换成最终状态。
4. 所有阶段切换都有入口条件和退出条件。
5. 主链承诺前完成必要塑形，主链承诺后连续兑现。
6. 功能线、库内、库外、存车线、临停线的业务性质没有混用。
7. 每一钩都能说明它对债务、边界、资源、顺序和后续动作的影响。
8. 低置信状态下不主动跨硬边界，高置信状态下不反复做低价值整理。

达到这个标准，计划才不是靠经验记忆堆出来的动作序列，而是一套可复核、可迁移、可训练、可由求解器学习的调车决策结构。
