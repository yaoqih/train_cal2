# 人工调车案例基准研究

这份文档的目的，是把人工调车作业单从“现场师傅写的钩计划”整理成一套可以理解、可以复核、可以让求解器学习的基准。

核心不是逐钩照抄人工。人工计划真正有价值的地方，是它在复杂线路、端别、库位、功能线、临停线和车列顺序之间做结构判断：什么时候整理，什么时候承诺，什么时候释放，什么时候机接，什么时候连续消化大库，什么时候才允许库回。

本文按五条线组织：

1. 数据从哪里来，哪些可信，哪些只能做结构参考。
2. 人工计划到底在优化什么，不只是把车从 A 搬到 B。
3. 标准人工计划的骨架是什么，非标准案例又该怎么识别。
4. 无经验编制者如何按阶段写出结构正确的计划。
5. 求解器如何学习、验收并超过人工，而不是靠局部少一两钩冒充进步。

## 1. 先给结论

人工调车的主结构可以概括为一句话：

```text
前段消化功能线和外场债务；
中段保护并形成存4北释放口；
关键时刻连续完成存4北头摘和机接；
后段集中完成修库摘解和库位让位；
最后才处理尾项、临停退出和库回。
```

人工计划不是固定模板，而是一套稳定的组织方式。它在每一钩都同时考虑五件事：

| 维度 | 人工真正关心的问题 |
|---|---|
| 债务 | 哪些车还没完成大库、功能线、存4集结、调棚、预修、称重等业务 |
| 边界 | 这一钩会不会跨过存4释放、机接、修库消化、库回等阶段边界 |
| 资源 | 存4北、机区通路、修库库外、功能线、临停线、牵出长度是否被占用 |
| 顺序 | 车列接上以后，后续能不能从可摘端连续摘解 |
| 证据 | 当前信息是否足够支持进入更高承诺阶段 |

所以，达到人工不是“最终所有车到位”这么简单。最终到位只能证明可解；只有关键结构、钩数、远端连续性、物理端别、车列顺序和业务债务都受控，才算接近人工。

### 2.4 最重要的数据结论

这次清洗的核心结论是：人工数据可以放心用于结构学习，但不能全量当作车号级逐钩真值。

| 口径 | 数值 | 正确用法 |
|---|---:|---|
| 清洗动作 | 4052 | 后续质量、重放、召回统一以这个为分母 |
| 严格 clean 动作 | 1724 | 车号级、动作级精确样例 |
| 结构 clean 动作 | 3414 | 阶段、线路、目标、意图、episode 结构学习 |
| 结构 review 动作 | 638 | 保留问题，不能硬塞进规则 |
| invalid 动作 | 0 | 没有发现表格层面的硬错误 |
| 学习参考动作 | 4003 | `0103W` 的 49 钩按现场确认排除，不作为学习参考 |

严格 clean 只有 42.55%，主要不是人工计划差，而是人工单不是为机器逐车号回放写的。最大问题是车号备注缺失：按当前规则，挂车动作都应标注末辆车号，摘 7 辆及以上应标注近机首辆车号。清洗发现 2108 钩需要车号，但只有 337 钩实际写了 7 位车号，因此出现 1793 个 `car_no_annotation_missing`。

这会挡住“动车到底是哪几辆”的精确回放，但不一定影响“这钩属于哪个阶段、哪个股道、哪类业务”的结构学习。

正确分工是：

```text
strict clean:
  用来做车号级监督、精确样例、候选召回小分母。

structural clean:
  用来学阶段、线路作用域、业务意图、主链结构。

review/gap:
  保留原文和 issue_code，先审计原因，不靠猜测强洗。
```

### 2.5 已确认和未确认的现场语义

线路解析结果比车号可靠：

| 线路解析状态 | 数量 | 解释 |
|---|---:|---|
| `alias_exact` | 2533 | 现场简称能直接映射到运行线路 |
| `inferred` | 1223 | 需要结合备注、聚合状态或现场语义推断 |
| `ambiguous` | 296 | 主要是 `库/注意库`，只能先按机库走廊看 |
| `unknown` | 0 | 没有完全无法识别的线路名 |

已经可以固化的语义：

| 现场写法 | 审计结论 |
|---|---|
| `存2叉` | 就是预修线/存2叉，不是普通道岔备注 |
| `存2 + 叉线` | 可按预修线来源建模，车序还要靠后续证据验证 |
| `存2 - 叉线` | 可按预修线目标作用域建模，端点和顺序不能无证据补全 |
| `留道口洗` | 可进入洗罐相关作用域，当前清洗落到 `洗罐站` |
| `修N/注意修N` | 需在 `修N库内/修N库外` 间判断 |
| `库外N/库外N摘` | 可高置信推出 `effective_count=N` |
| `存5 北头` | 是端别提示，不等于存4阶段 |
| `库/注意库` | 当前只能按机库走廊聚合语义处理 |
| `库/机 +1`、`称`、`回/停` | 多数是机车、称重或收束 no-op，不应凭字面生成货车位移 |

不能强推断的语义：

| 现场写法 | 不能怎么用 |
|---|---|
| `北头代N` | 不能把 N 当辆数，保留 structural gap |
| `代N/南代N/机代N` | 不能当作车辆数、端点或普通 `+/-` |
| `存5北头顶N` | 不能学成一条普通挂/摘钩，需要单独的存5内部重排 episode |
| 省略辆数 | 不能靠猜测补数量，除非属于已确认的 `库外N` 模式 |
| `库/注意库` | 不能直接拆成精确机南、机北、机棚或机库内外车辆 |

## 3. 人工计划到底在解什么问题

### 3.1 不是搬车，是管理债务

如果只看表面，人工单像是一串“股道 + 方法 + 辆数”。但人工真正解决的是债务排序问题。

每一辆车至少可能带着几类债务：

| 债务 | 含义 | 常见线路 |
|---|---|---|
| 大库债务 | 需要进入修1至修4，或从库内/库外消化出来 | `修1库内` 至 `修4库内`、`修1库外` 至 `修4库外` |
| 存4债务 | 需要在存4北形成出段/释放/集结状态 | `存4线`、`存4线南` |
| 功能线债务 | 洗、油、抛、轮、称重等业务未完成 | `洗罐站`、`油漆线`、`抛丸线`、`卸轮线`、`机库线` |
| 预修/调棚债务 | 预修、调梁、机走棚作业未完成 | `预修线`、`调梁棚`、`机走棚` |
| 存车整理债务 | 车辆已经在存车线，但顺序或分组还不服务后续动作 | `存1线`、`存2线`、`存3线`、`存5线北/南` |
| 阻塞债务 | 某车挡住更深层线路或操作口 | 修库库外、机区北段、洗油北、调梁线北等 |
| 顺序债务 | 车在列中位置不支持后续尾端摘解 | 主列、临停列、存车线 |
| 证据债务 | 状态不足以确认动作是否合法或合时 | 低信号案例、缺车号动作、复合钩 |

优秀计划不是先还最显眼的目标债务，而是先处理会挡住后续选择的债务。很多时候，清阻塞、保端别、调顺序，比直接把某辆车送到目标线更重要。

### 3.2 人工的主链不是一条线，而是一组边界

标准人工计划通常围绕五个阶段展开：

```text
H1 前段服务与外场整理
  -> H2 存4释放口塑形
  -> H3 存4北头摘 + 机接
  -> H4 修库连续消化
  -> H5 尾项和库回
```

这五个阶段不是标签，而是承诺强度变化：

| 阶段 | 承诺强度 | 主要动作 |
|---|---|---|
| H1 | 低承诺 | 盘点、服务、清阻塞、重排、归组 |
| H2 | 中承诺 | 围绕存4北释放口塑形，准备主列 |
| H3 | 高承诺 | 存4释放和机接，主列正式成形 |
| H4 | 高兑现 | 修库摘解、库外让位、库位消化 |
| H5 | 收束 | 尾项、临停退出、库回 |

越靠前，动作越应可逆；越靠后，动作越应确定。H3 之后还频繁回外场整理，通常说明 H2 没塑好，或者 H3 过早承诺。

### 3.3 案例族

在 118 个人工文件中，按结构信号粗分：

| 人工案例族 | 文件数 | 含义 |
|---|---:|---|
| 标准存4释放-机接-修库消化 | 97 | 有 `存4 - 北头摘`、`机 + 接`、修库摘解 |
| 库内消化/直接处理 | 10 | 缺少标准机接，但有修库摘解和库回 |
| 混合/一般整理 | 7 | 以外场、功能线、调棚、机区整理为主 |
| 有存4释放无标准机接 | 2 | 有存4释放，但机接信号不完整 |
| 短链直接入库 | 2 | 已接近库侧，少量动作即可完成 |

匹配 `truth2` 的 109 个文件中，标准链仍有 97 个，说明当前主问题确实是“大库周转 + 存4释放 + 机接 + 修库消化”。但非标准案例不能忽略，它们决定求解器是否会强行套错模板。

### 3.5 人工钩序的典型时间位置

按有效钩序位置中位数估算：

| 动作族 | 中位出现位置 | 解释 |
|---|---:|---|
| 功能线服务 | 第 8 钩 | 早期先处理洗/油/抛/轮等可消化项 |
| 机区动作 | 第 12 钩 | 机车位置与机区缓冲早早参与 |
| 调棚队列 | 第 14 钩 | 调棚/预修相关队列在中前段整理 |
| 叉线缓冲 | 第 14 钩 | 中前段大量使用叉线重排 |
| 存车区组织 | 第 17 钩 | 外场组流持续到中段 |
| 北头操作 | 第 20 钩 | 逐步靠存4北或北头组织 |
| 修库送入/接入 | 第 23 钩 | 进入库侧前的主列组织 |
| 存4释放 | 第 25.5 钩 | 标准链关键转折点 |
| 机接 | 第 27 钩 | 机接通常紧跟存4释放 |
| 修库摘解 | 第 28 钩 | 进入大库消化阶段 |
| 库回 | 第 36 钩 | 多数情况下是后段收束 |

人工前半段看起来“杂”，其实是在为后半段压缩复杂度。前段如果不做功能线、外场和端别整理，后段就会变成反复清口、反复回外场、机接后拆主列。

## 4. 五阶段人工计划骨架

### 4.1 阶段总表

| 阶段 | 现场语言 | 主要目标 | 退出条件 | 禁止事项 |
|---|---|---|---|---|
| H1 前段服务与外场整理 | 洗、油、抛、轮、调、存1/2/3/5、机区整理 | 处理独立服务项，把大库债务车逐步归组 | 存4释放口方向明确，主列候选开始成形 | 不许过早占死存4北；不许把功能线当最后补丁 |
| H2 存4释放口塑形 | `存4 - 北头`、`北头代`、修库接入群成形 | 组织可释放主列，保护存4北和机接通路 | 可以执行 `存4 - n 北头摘`，下一钩能机接 | 不许释放后还回外场反复拼车 |
| H3 释放与机接 | `存4 - n 北头摘` 后紧跟 `机 + n 接` | 主列从存4释放到机区承接 | 主列已机接，顺序可用于修库摘解 | 不许拆散 H3；不许机接后做无关整理 |
| H4 修库消化 | `修1/2/3/4 - n 摘`、库外摘解 | 依次送入或摘解修库车辆，处理库位冲突 | 修库债务基本清完 | 不许修库未完成就库回 |
| H5 尾项和库回 | 轮/油/机/调/存车尾项、`库回` | 清尾项，退出临停，机车/大库收束 | 全目标满足且无阻塞尾债 | 不许把库回当作业务完成替代 |

标准完整链的阶段长度也很稳定：

| 阶段 | 标准链中位钩数 | 75 分位 | 说明 |
|---|---:|---:|---|
| H1 | 18 | 22 | 大量功能线、存车、调棚组织都在这里完成 |
| H2 | 6 | 7 | 存4释放口塑形通常很短，不应无限拖长 |
| H3 | 2 | 2 | `存4北头摘 -> 机接` 基本是连续两钩 |
| H4 | 3 | 4 | 修库摘解是短而硬的集中消化 |
| H5 | 1 | 4 | 多数只是库回，复杂尾项才会拉长 |

如果 H3/H4 被拉得很长，通常不是业务本身复杂，而是前段没有把主列塑好，或者机接后又把主列错误打散。

### 4.2 H1：前段服务与外场整理

H1 的目标不是“能送哪辆就送哪辆”，而是让后续债务变简单。

优先处理：

1. 已经在功能线、调棚、预修、机区附近的车，能完成服务并退出的先退出。
2. 存车线车辆按最终债务归并：大库债务归大库主列，功能线债务归服务线，存4债务归出段集结。
3. 使用 `存2线`、`存3线`、`存5线北/南` 时，优先保持北端可取，不把急用车压到不可取端。
4. `机库线` 称重、`机走棚` 预修等特殊动作按业务硬约束处理，不当作普通缓存。
5. H1 结束前必须能说清：哪些车会组成存4释放主列，哪些车暂时不参与主列。

好动作通常长这样：

| 动作 | 好在哪里 |
|---|---|
| 功能线先接再送出 | 减少 H5 尾项 |
| 调棚/预修先消化一批 | 避免后段再跨区回头 |
| 存2/存5 与调棚交替整理 | 用前段时间把顺序塑好 |
| 先清机区或洗油北阻塞 | 打开后续路径，而不是盲目移动 |

坏动作通常长这样：

| 动作 | 问题 |
|---|---|
| 没有目标地把车塞进存4北 | 污染释放口，H2 要额外清口 |
| 把要先进库的车压在不可取端 | H3/H4 会因为尾端不可摘返工 |
| 为少一钩把功能线车夹进大库主列 | 后面必须拆出来，实际更慢 |
| 反复在存2/存5 换线但没有主列目标 | 看似忙，债务没有减少 |

### 4.3 H2：存4释放口塑形

H2 的任务是把能释放的一组车组织到 `存4线` 北端或等价释放位置。

它保护三个资源：

| 资源 | 要求 |
|---|---|
| `存4线` 北端 | 只放即将释放或正式出段集结的车 |
| 机车承接通路 | 释放后机车必须能接，不被机区北段、机棚、洗油北等阻塞 |
| 车列顺序 | 进入 H4 时，修库摘解应能从可摘端连续完成 |

进入 H2 的条件：

```text
大库债务车基本明确；
主列中不应夹入大量无关功能线车；
存4北端或接近存4北端的位置可整理出释放口。
```

退出 H2 的条件：

```text
能执行一勾明确的 存4线 - n 北头摘；
下一勾能执行机区接入；
接入后至少能连续完成一批修库摘解。
```

H2 可以用 `北头代`、`北头接`、`叉线` 等动作，但它们必须服务于释放口，不是普通换线。

### 4.4 H3：释放与机接

H3 是全案最硬的边界。标准人工几乎总是：

```text
存4线 - n 北头摘
机库线/机区 + n 接
```

人工数据中，97 个标准链里：

| 转折 | 统计事实 | 调车含义 |
|---|---:|---|
| `存4北头摘 -> 机接` | 96/97 是下一钩完成，只有 1 例间隔 3 钩 | 释放和承接是一组动作，不应拆开 |
| `机接 -> 修库摘解` | 97/97 是下一钩进入摘解 | 机接不是停车成果，而是进入 H4 的入口 |
| H4 修库摘解 | 中位 3 钩，75 分位 4 钩 | 大库消化应集中完成，不应夹杂大量外场整理 |

执行前必须检查：

| 检查项 | 必须满足 |
|---|---|
| 释放数量 | `n` 对得上要进入主列的车数 |
| 机车位置 | 能从当前机车位置到达承接点 |
| 通路 | 机区北段、机棚、洗油北等不会阻塞必要路径 |
| 车列顺序 | 接后修库摘解顺序可行 |
| 后续库位 | 修1至修4至少有可执行摘解/入库方案 |

H3 禁忌：

```text
释放后先做无关洗/油/调/存车整理；
机接后又把主列送回外场；
只因为机区空着就提前机接；
低信号案例强行构造标准机接。
```

一句话：如果不能保证释放后马上机接，就不要释放；如果机接后不能马上开始修库消化，就不要机接。

### 4.5 H4：修库消化

H4 的目标是把主列按修库目标拆掉，同时处理库外/库内阻塞。

基本顺序：

1. 先看四条修库线各自的库外是否挡住库内。
2. 先处理会释放关键库位的出库或摘解动作。
3. 再把主列按可摘端顺序送入 `修1库内` 至 `修4库内`。
4. 如果修库目标车辆在主列中顺序不适合连续摘解，不要硬送，回到临时 staging 重新分段。
5. 修库消化完成前，不允许用 `库回` 结束。

人工写法要结合阶段理解：

| 人工写法 | 可能含义 |
|---|---|
| `修1 - 4 摘` | 向修1库内摘解 4 辆，或从库侧摘出，需要结合上下文 |
| `修4 - 5 库外2` | 通过修4库外缓冲处理库外位置或占位 |
| `修2 + 7 接` | 向修2方向接入一组，可能是 H2/H4 交界动作 |

不能只看 `+/-`。同一个符号，在 H1、H2、H4 中可能代表不同业务意图。

### 4.6 H5：尾项和库回

H5 不是“剩下什么做什么”，而是主结构完成后的收束。

进入 H5 的条件：

```text
修库摘解债务完成或不适用；
存4正式集结没有被临停车污染；
功能线关键服务完成；
被阻塞线路没有仍待取放的车；
机车回库不会打断未完成合同。
```

H5 常见内容：

| 内容 | 说明 |
|---|---|
| 轮/油/洗/抛尾项 | 处理不会反过来影响大库主链的功能线车 |
| 存车线收束 | 把暂存车归到最终存车线或可交付位置 |
| 调棚/预修尾项 | 清理前段留下的服务线尾债 |
| `库回` | 机车或大库收束信号，不能替代业务完成 |

## 5. 物理端别、阻塞和车列顺序

### 5.1 先统一现场简称

人工单大量使用简称。编制和求解器都应转成规范线路，再谈规则。

| 全称 | 现场简称/规范简称 | 新手记忆 |
|---|---|---|
| `机走北1线` | 机北1 | 临停线，挡 `存1线` 和 `机走北2线` 通路 |
| `机走北2线` | 机北2 | 临停线，挡渡4/渡5方向和机区通路 |
| `机走北` | 机北3 | `机走棚` 北侧前置线 |
| `机走棚` | 机棚 | 预修/存车复合线，挡 `机走线南` 和 `洗罐油漆北` |
| `洗罐油漆北` | 洗油北 | `洗罐线北` 与 `油漆线` 的前置门槛 |
| `洗罐线北` | 洗北 | `洗罐站` 北侧缓存 |
| `洗罐站` | 洗南 | 洗罐作业线 |
| `调梁线北` | 调北 | `调梁棚` 北侧前置线 |
| `调梁棚` | 调棚 | 调梁和南端预修作业线 |
| `存4线` | 存4北 | 正式出段集结和释放口 |
| `存4线南` | 存4南 | 过程临停，不是正式终点 |

### 5.2 北头不是备注，是操作口

人工计划中 `北头` 出现约 630 次，`北头摘` 出现 105 次。它不是随手写的备注，而是在提醒：

```text
从北端组织可取放顺序；
目标车必须在可操作端；
后续摘解必须从尾端连续完成；
存4北头摘通常是 H3 释放信号。
```

当前物理口径：

```text
取车和放车都按北端顺序操作；
被北段或库外车辆阻塞的线路，不能直接取放车；
必须先清空阻塞车辆，再操作被阻塞线路；
存5线南只能从存5线北一侧操作，不能从 L12 侧直接操作。
```

新手最容易犯的错是“图上看有路，实际北端被车挡住”。每计划一钩前先问：目标线北侧有没有前置车？有的话，这一钩不是取放目标车，而是先清前置车。

### 5.3 关键阻塞关系

| 阻塞车所在股道 | 被阻塞股道/作业 | 编制规则 |
|---|---|---|
| `修1库外` 至 `修4库外` | 对应 `修1库内` 至 `修4库内` | 库外不清，库内不能直接取放 |
| `机走棚` | `机走线南`、`洗罐油漆北` | 先清 `机走棚`，再去 `机走线南` 或 `洗罐油漆北` |
| `机走北` | `机走棚` | 先清 `机走北`，才能操作 `机走棚` |
| `机走北2线` | 渡4/渡5方向、机区后续路径 | 有车会挡关键机区通路 |
| `机走北1线` | `存1线`、`机走北2线` | 不能随意当缓存 |
| `洗罐油漆北` | `洗罐线北`、`油漆线` | 是洗罐北和油漆线的前置门槛 |
| `洗罐线北` | `洗罐站` | 不清洗北，洗罐站不能直接取放 |
| `调梁线北` | `调梁棚` | 调北挡调棚 |
| `存4线`、`存3线` | `存4线南` | 存4南只是临停，不能无视北侧阻塞 |
| `存5线北` | `存5线南` | 存5南只能从北侧经存5北操作 |

### 5.4 车列顺序决定少钩是否真实

少钩计划的核心不是“多挂几辆”，而是“挂上以后能按尾端摘掉”。

每个候选动作都应能回答：

```text
挂车顺序是什么？
下一步要从哪一端摘？
目标车是否形成连续小段？
有没有把功能线车夹进大库主列？
如果顺序不对，在哪里重排？
```

基本原则：

| 原则 | 说明 |
|---|---|
| 先考虑尾端可摘 | 不能只看目标线相同，要看车在列中的位置 |
| 同一库线尽量成段 | 修1、修2、修3、修4目标车最好形成可连续摘解小段 |
| 不把尾项夹进主列中间 | 洗/油/轮尾项夹在修库车中间会增加摘挂 |
| 临停只服务重排 | 机北、机棚、洗油北、存4南不能成为无解释的最终状态 |
| 称重车保留称重顺序 | 经过机库称重时，整列长度和称重车位置都要合法 |

如果主列顺序不支持连续摘解，不要在 H4 硬送。应回到 H1/H2，用 `叉线` 或临时 staging 做一次有目的的重排。

### 5.5 高频现场词怎么读

| 词 | 人工意图 | 新人解释 |
|---|---|---|
| `北头` | 从北端组织可取放顺序 | 先看北端第一辆是不是要动的车 |
| `叉线` | 利用交叉线或中间通路重排 | 不是无目的绕行，是为后面连续摘解塑形 |
| `接` | 承接、连挂、形成主列 | 接后通常进入下一阶段 |
| `摘` | 从列中摘出或向目标线甩出 | 要检查尾端可摘 |
| `北头摘` | 存4释放硬信号 | 下一钩通常应机接 |
| `库外1/2/3` | 大库外缓冲/占位提示 | 先处理库外阻塞和库位让位 |
| `代` | 临时代位/替代位置 | 服务后续结构，不应当作最终目标 |
| `顶` | 顶送或顶位 | 需要检查前方容量和端别 |
| `称` | 称重 | 必须经过机库称重能力，不能用普通到达替代 |
| `回` | 库回/收束 | 只在主债务完成后有效 |
| `停` | 停留或临时状态 | 要确认不是未完成尾债 |

## 6. 代表案例

### 6.1 `0117Z`：标准完整链

人工钩数：40。

关键链条：

```text
01-22: 抛/油/机/调/存2/存5/存3 组织和功能线处理
23:    存4 - 北头
24-31: 轮/修1/修2/修3/修4 组合形成大库目标群
32:    存4 - 16 北头摘
33:    机 + 18 接
34-37: 修4/修3/修2/修1 摘解
38-40: 轮/油/库回收束
```

读法：

- 前 22 钩不是杂活，是在为存4释放口和机接主列塑形。
- 第 32-33 钩是硬边界：`存4大摘 -> 机接`。
- 第 34-37 钩是大库消化，不允许被提前当尾项。

求解器要求：

- 能识别 `APPROACHING_CUN4 -> PORT_READY -> MACHINE_ACCEPTED -> DEPOT_DIGEST`。
- 机接后不能为了少钩拆散主列。
- 大库摘解完成前不能 closeout。

### 6.2 `0310W`：late 存4链

人工钩数：46。

关键链条：

```text
01-28: 抛/洗/油/调/存1/存2/存5 大量整理
29:    注意存4 - 北头代2
30-35: 抛/修2/轮/修1/修3/修4 形成库侧接入群
36:    存4 - 14 北头摘
37:    机 + 14 接
38-41: 修2/修4/修3/修1 摘解
42-46: 轮/油/库/机/库回
```

读法：

- 这个案例前段很长，但一直在为后段存4释放服务。
- 第 36-37 钩仍是标准 `release -> accept`。
- 一旦接近 `PORT_READY`，禁止回外场重新拼长链。

超越空间在前段：算法可以减少调、存5、存2之间的反复，但必须证明不会污染存4释放口。

### 6.3 `0103W`：库内消化与缺标准机接

人工钩数：49。

关键链条：

```text
01-18: 库/抛/油/洗/机/存1/存2 复杂组织，含称重/顶送
19-22: 轮、修2、修3、修4 摘解
23-29: 注意机、洗接、存2/存3、修4/修3/修1 摘解
30-48: 存5/存1/存2/洗/调/机 大量尾部组织
49:    库回
```

读法：

- 没有标准 `存4释放 -> 机接`，但有明确修库摘解层。
- 这类案例不能被归为“尾项”或“普通整理”。
- 称重、顶送、对位等微动作会影响合法性。

注意：清洗审计中 `0103W` 的 49 钩已按现场确认排除出算法学习参考集，因为它是节假日合并钩计划。它仍可作为业务复杂性观察样例，但不能当普通金标学习。

### 6.4 `0213W`：短链直接入库

人工钩数：5。

关键链条：

```text
01: 注意机
02: 修1 + 5 接
03: 修2 + 5
04: 注意存4 - 10 北头摘
05: 库回
```

读法：

- 这是极短直接处理，不需要完整外场组织。
- 如果求解器强套标准 H1-H5 长链，会明显劣于人工。
- 目标不是追求更少钩，而是同钩数、更完整安全校验和可解释 trace。

### 6.5 `0306W`：短链但信号不完整

人工钩数：10。

关键链条：

```text
01-02: 机、存4北头
03-05: 修3/修1/修2 接入
06:    存4 - 15 北头摘
07-08: 机 + 2 / 机 + 1 南头
09-10: 油、库回
```

读法：

- 有存4释放，但机接信号不标准。
- 应识别为低信号或混合直接处理，不要强行生成高置信机接。
- 低置信下禁止主动跨 `ACCEPTED`，必须输出矛盾原因和保守动作边界。

## 7. 手工编制方法

### 7.1 先读状态，不要先想动作

拿到一个案例后，先盘点状态，不要先问“哪辆车从 A 到 B”。先给车辆分组：

| 分组 | 判断方法 | 人工含义 | 首要动作 |
|---|---|---|---|
| 大库债务车 | 目标含 `修1库内` 至 `修4库内`，或已经在修库内/库外但未完成摘解 | 主合同，决定整案骨架 | 规划入库、出让位、摘解顺序 |
| 出段/存4债务车 | 目标是 `存4线`，或已经完成工序等待集结 | 释放口和交站目标 | 保护存4北端，不随便污染 |
| 预修/调梁债务车 | 目标或当前位置在 `预修线`、`机走棚`、`调梁棚` | 前段服务合同 | 先完成可独立服务，再判断是否转大库 |
| 功能线债务车 | 涉及洗罐、油漆、抛丸、卸轮、称重 | 不可当尾项 | 尽早处理能直接消化的车 |
| 存车整理车 | 在存1、存2、存3、存5 等等待分流 | 队列塑形材料 | 按后续目标归并，少做无目的换线 |
| 阻塞车 | 停在串行通道北段或库外段，挡住南段/库内/作业线 | 物理前置条件 | 先清阻塞，再取放被阻塞线路 |

然后判断案例族：

| 案例族 | 判断信号 | 推荐主流程 |
|---|---|---|
| 标准存4释放-机接-修库消化 | 大量大库债务，常见 `存4 - 北头摘`、`机 + 接` | H1 -> H2 -> H3 -> H4 -> H5 |
| 库内消化/直接处理 | 修库内外已有可摘解债务，缺少标准机接 | 直接进入 H4 或 H1/H4 混合 |
| 短链直接入库 | 车辆已靠近库侧或存4口，总钩数很少 | 不补长链，完成近端接入/摘解即可 |
| 混合低信号 | 有存4或机区信号，但缺标准机接或摘解链 | 保守推进，只在证据充分时跨硬边界 |

### 7.2 一页式流程

```text
盘点车辆债务
  -> 判断案例族
  -> 查阻塞和北端可取放
  -> H1 处理前段服务与外场整理
  -> H2 围绕存4释放口塑形
  -> H3 连续完成存4北头摘和机接
  -> H4 连续完成修库摘解和让位
  -> H5 清尾项和库回
  -> 用复核清单逐项验收
```

如果某一步无法判断，不要跳到下一步。回到“盘点车辆债务”和“查阻塞与端别”。

### 7.3 少钩优化的优先级

当“少一钩”和“保结构”冲突时，按下面顺序取舍：

```text
硬物理可达
  > 端别可取放
  > 不污染存4释放口
  > 机接后主列不拆散
  > 修库消化连续
  > 功能线服务不遗漏
  > 少钩
  > 少空驶/少道岔
```

可以接受多一钩的情况：

| 情况 | 为什么 |
|---|---|
| 为了清阻塞车 | 否则目标线不可取放 |
| 为了保护存4北端 | 存4被污染会导致后段多次清口 |
| 为了让修库目标成段 | H4 连续摘解比前段少一钩更重要 |
| 为了完成称重/对位 | 特殊业务不能被普通搬车覆盖 |

不应接受少一钩的情况：

| 情况 | 后果 |
|---|---|
| 机接后又回外场整理 | 破坏 H3/H4 硬边界 |
| 临停线作为最终目标 | 物理过程合法但业务非法 |
| 存5线南从 L12 侧直接取放 | 操作口错误 |
| 库外有车仍直接操作库内 | 大库阻塞未解除 |
| H4 未完成就库回 | 计划看似结束，业务未完成 |

### 7.4 手工编制工作表

每个案例先填表，再写钩计划：

| 问题 | 填写内容 |
|---|---|
| 本案案例族 | 标准链 / 库内消化 / 短链直接 / 混合低信号 |
| 大库债务车数量和目标 | 修1、修2、修3、修4分别多少辆，是否已有库外阻塞 |
| 功能线债务 | 洗、油、抛、轮、称重分别哪些车 |
| 预修/调棚债务 | 预修、机走棚、调梁棚哪些车未完成 |
| 存4释放口状态 | 存4北端是否可用，是否有无关车污染 |
| 存5状态 | 存5线北是否挡住存5线南 |
| 机区通路状态 | 机北1、机北2、机北3、机棚、洗油北是否有阻塞车 |
| 初始主列候选 | 哪些车准备进入 H3/H4 主列 |
| 主列摘解顺序 | 接后按什么顺序送修1/修2/修3/修4 |
| 临停/重排点 | 如果顺序不对，在哪里重排 |
| 预计 H3 钩 | `存4线 - n 北头摘` 和机区接入是否连续 |
| H4 完成标准 | 哪些修库动作完成后才允许收尾 |
| H5 尾项 | 剩余功能线、存车线、库回动作 |

如果这张表填不出来，说明还没有进入写钩计划的阶段。

### 7.5 新人复核清单

计划写完后逐项检查：

1. 所有取放是否按北端顺序可达。
2. 被阻塞线路是否先清阻塞车。
3. `存5线南` 是否只从 `存5线北` 一侧操作。
4. `存4线` 北端是否只服务释放口或正式集结。
5. 是否存在 `存4北头摘` 后没有立即机接。
6. 是否存在机接后没有立即进入修库消化。
7. 修库摘解前，修库库外是否已处理清楚。
8. 修库目标车在车列中是否尾端可摘。
9. 功能线车辆是否被夹在大库主列中间。
10. 称重、顶送、对位、关门车、重车等特殊约束是否显式处理。
11. 临停线是否只作过程用途，没有成为最终目标。
12. `库回` 是否只在主债务完成后出现。

只要有一项回答“不确定”，这不是优秀计划，必须回到相应阶段重排。

## 8. 边界和操作哲学

### 8.1 先定边界，再定动作

人工调车不是把车从一个点搬到另一个点，而是在连续占用、释放和转换边界。每一钩都会同时改变车辆位置、车列顺序、可达关系、线路容量、阶段承诺和剩余债务。

一个动作只有能说明以下五件事，才可以进入计划：

1. 它跨过或占用了哪类边界。
2. 它偿还、隔离或推迟了哪类债务。
3. 它消耗了哪些线路、端侧、折返长度、临停空间或车列顺序资源。
4. 它让哪些后续动作变得可行，或者让哪些后续动作变得不可行。
5. 它的退出信号是什么，什么时候必须离开当前状态。

无法回答这些问题时，不应执行高承诺动作。此时只能做低承诺的盘点、清阻塞、重排、补证据或释放资源动作。

### 8.2 七类边界

| 边界类别 | 核心问题 | 常见错误倾向 | 判定口径 |
|---|---|---|---|
| 物理连通边界 | 通路、岔区、阻塞关系、折返长度是否可行 | 只按拓扑连通判断可达 | 取放前同时确认通路、可操作端、串行阻塞、整列长度和目标容量 |
| 操作口边界 | 线路实际允许从哪一侧取放 | 把股道当成两端自由操作 | 以规定操作侧建立可取顺序，另一侧不能默认可用 |
| 车列顺序边界 | 目标车是否位于可取端，摘挂后是否服务后续动作 | 忽略夹车和尾端可摘 | 任何摘放都按车列方向和北端顺序审查 |
| 资源占用边界 | 临停线、存车线、库外、功能线、牵出空间是否被占用 | 把过程资源当无限缓存 | 每次占用都必须有退出条件 |
| 业务债务边界 | 车辆是否完成检修、洗罐、油漆、抛丸、称重、集结等目标 | 把物理到达当业务完成 | 没有完成证据时债务仍在 |
| 阶段承诺边界 | 当前是否进入 H1-H5 中更高承诺阶段 | 提前机接、提前库回、机接后回外场 | 阶段入口、退出和跳阶段理由必须明确 |
| 证据置信边界 | 信息是否足以支持跨硬边界 | 低置信信息下主动构造主链 | 证据不足时保守推进 |

### 8.3 三层判断

调车判断必须自上而下分三层：

| 层次 | 回答的问题 | 不通过时 |
|---|---|---|
| 安全合法层 | 能不能做 | 动作直接无效 |
| 结构正确层 | 该不该现在做 | 即使物理可行，也不进计划 |
| 效率优化层 | 值不值得这样做 | 只在前两层通过后比较 |

三层关系是单向约束：效率服从结构，结构服从安全。短钩数不能覆盖硬边界错误，局部顺手不能覆盖阶段错误，物理可行不能覆盖业务未完成。

### 8.4 可逆性和承诺

人工高水平不在于每一步都最短，而在于该可逆时保持可逆，该承诺时坚决承诺。

可逆动作用于盘点、清阻塞、重排和释放资源。它的价值是增加后续可操作性，而不是立即关闭最终目标。

承诺动作用于形成主列、进入核心通道、开始连续摘解或关闭重要业务债务。承诺动作一旦发生，后续动作就应围绕该承诺快速收束。承诺后频繁回到前序整理，说明承诺过早或前序塑形不足。

判断一个动作是否过早承诺，要看它是否减少了可操作口、绑定了关键资源、改变了车列主序、迫使后续必须连续执行。如果答案为是，而当前债务和资源尚未准备好，就应推迟。

### 8.5 反向审查法

每写一钩前，从未来反查当前动作，而不是只从当前位置向前搜索：

1. 这钩之后，下一类必须动作是什么。
2. 这钩之后，哪个边界被打开，哪个边界被关闭。
3. 这钩之后，哪项债务减少，哪项债务增加。
4. 这钩之后，关键资源还能否退出。
5. 这钩错了之后，是否还能不跨硬边界恢复。

如果只能说明“车能到那里”，不能说明“下一步为什么更好”，这钩就不是成熟动作。

## 9. 求解器如何学习人工计划

### 9.1 不要从全局搜索开始

人工计划显示，人的强项是先确定结构边界：

```text
哪些车先在外场/功能线消化
哪些车应逐步靠存4
什么时候存4释放口成形
什么时候机接后不能再拆
哪些库内车必须先出让位
哪些尾项只能最后处理
```

所以求解器达到人工的首要条件是：

```text
FlowClassify
  -> TargetContract
  -> StructuralIntent
  -> ResourceRequest
  -> ContractDelta
  -> ResourceDelta
```

而不是：

```text
生成大量动作
  -> 搜索
  -> 靠分数碰运气
```

### 9.2 最低结构识别要求

求解器至少要识别：

1. `存4 - 北头 / 北头摘` 的释放意义。
2. `机 + 接` 的机接硬边界。
3. `修1-修4 - n / 摘 / 库外n` 的修库消化语义。
4. `库 - 回` 只能作为收束，不是万能完成信号。
5. `调`、`洗`、`油`、`抛`、`轮` 都是独立服务项。
6. `叉线`、`北头` 是端别/缓冲结构，不是普通备注。
7. `称/顶/代/对/停` 是特殊业务约束或待审计复合钩。

### 9.3 应有的实现层

| 层 | 必须输出 | 用途 |
|---|---|---|
| CaseClassifier | 标准链/库内消化/短链/混合低信号 | 决定是否强制 H1-H5 |
| DebtLedger | 大库、存4、功能线、预修、调棚、存车整理债务 | 知道每勾在还什么债 |
| PhaseGate | H1-H5 入口、退出、跳过理由 | 防止过早机接或库回 |
| ResourceGate | 存4口、修库库外、机区通路、功能线容量、临停线 | 防止物理可达但业务破坏 |
| CarryOrderModel | 有序车列、尾端可摘、整列长度 | 防止少钩假象 |
| DeltaAudit | 每勾前后债务变化 | 证明动作不是乱动 |

求解器如果只输出“车辆最终到位”，还没有达到人工。它必须解释每勾处于哪个阶段、还了什么债、占用了什么资源、为什么没有破坏后续结构。

### 9.4 人工标签怎么用

| 数据层 | 用法 | 禁止事项 |
|---|---|---|
| strict clean | 精确车号回放、小样本候选召回、金标动作 | 不能拿它代表全部人工计划 |
| structural clean | 训练阶段识别、业务意图、episode 生成 | 不能推断缺失车号 |
| review/gap | 审计不确定语义，建立人工复核清单 | 不能强洗成确定动作 |
| identity replay clean | owner/blocker 和精确候选比较 | 不能扩大到未闭合状态 |
| uncertainty classes | 固化已确认现场语义，隔离低置信词 | 不能把低置信类当规则 |

身份重放当前结果很克制：

| 审计项 | 数值 | 解释 |
|---|---:|---|
| 可重放钩 | 1118/4052 | 有足够状态候选能推进 |
| 唯一确定动车身份钩 | 563/4052 | 真正能精确知道“动车是谁”的比例还低 |
| 完整闭合案例 | 6 | 多数案例会被前序 gap 阻断 |
| owner/blocker clean | 85 | 只有很少钩能干净标出 owner/blocker |
| owner/blocker partial | 35 | 有部分可用，但不能当硬真值 |

候选召回审计中 exact 只有 17、structural 只有 3。这不能简单理解成“算法完全召不回人工计划”。主要缺口在人工标签还没有足够身份化，例如 `owner_label_gap:no_unique_moved_nos`、`no_unique_replay`、`truth_missing`、`identity_noop`。因此，修求解器之前，先要把人工状态身份层继续补齐。

### 9.5 建议金标案例

建议先建立 9 个金标案例：

| 案例 | 类型 | 必验点 |
|---|---|---|
| `0117Z` | 标准完整链 | 存4释放、机接、修库摘解完整推进 |
| `0310W` | late 存4链 | 不回外场补长链 |
| `0103W` | 库内消化观察样例 | 不作为普通学习参考，但用于理解复杂复合钩 |
| `0223W` | 库内消化 | 修库摘解优先于尾项 |
| `0213W` | 短链直接入库 | 不强套标准链 |
| `0306W` | 短链低信号 | 不低置信主动机接 |
| `0130Z` | 混合信号 | 允许反推但必须保守 |

每案输出：

## 10. 达到人工和超越人工

### 10.1 达到人工的三层基准

第一层是结构基准。算法达到人工，首先不是钩数更少，而是结构不能错：

| 基准项 | 达标标准 |
|---|---|
| 主体流向识别 | 大库、存4、预修、调棚、功能线、机区、存车整理都有合同 |
| 存4释放口 | 能识别 `存4北头/北头摘` 的释放意义 |
| 机接硬边界 | `机 + 接` 后主列不能随意拆散 |
| 修库消化 | 修1至修4摘解完成前不能 closeout |
| 直接入库 | 短链/近库侧案例不能强套标准长链 |
| 低信号案例 | 能输出 confidence/contradiction，不能低置信主动机接 |
| 预修/调棚 | 不能当作 residual 或尾项 |
| 功能线 | 洗、油、抛、轮必须有服务完成条件 |
| 特殊微动作 | 称、顶、代、对位、停等必须作为约束或 gap |

第二层是动作族覆盖。每个算法案例都应输出类似人工的结构 trace：

```json
{
  "case": "0117Z",
  "contract_groups": {
    "REPAIR_INBOUND": 16,
    "DEPOT_OUTBOUND": 16,
    "PRE_REPAIR_STAGING": 11,
    "DISPATCH_SHED_QUEUE": 8,
    "FUNCTION_LINE_SERVICE": 5,
    "YARD_REBALANCE": 18
  },
  "critical_transitions": [
    "FORM_RELEASE_GROUP",
    "CUN4_RELEASE",
    "MACHINE_ACCEPT",
    "DEPOT_DETACH",
    "TAIL_CLOSEOUT"
  ],
  "resource_claims": [
    "CUN4_NORTH_BUFFER",
    "DEPOT_SLOT_RESOURCE",
    "LOCO_CARRY_ORDER"
  ]
}
```

第三层是数值基准：

| 指标 | 人工基准 |
|---|---:|
| 全量人工单中位数 | 35 钩 |
| 匹配 `truth2` 人工单中位数 | 36 钩 |
| 标准链人工文件数 | 97 |
| 标准链覆盖率 | 82.2% |
| 匹配集标准链文件数 | 97 |
| 匹配集标准链覆盖率 | 89.0% |

### 10.2 分案例族验收

| 案例族 | 达到人工 | 超越人工 |
|---|---|---|
| 标准存4释放-机接-修库消化 | 无硬违规，钩数不高于人工 5%，关键转折一致 | 钩数低于人工，且不增加存4污染、库位冲突、机车空驶 |
| 库内消化/直接处理 | 不强套标准长链，先完成库内摘解 | 同钩或少钩，并输出更清楚的库位 swap |
| 短链直接入库 | 不明显多于人工，不额外绕行 | 保持人工钩数或更少，安全 trace 更完整 |
| 混合/低信号 | 不低置信跨硬边界 | 用补证据动作减少返工 |
| 功能线/调棚重案例 | 功能线不被当尾项 | 减少功能线与外场间反复取送 |

### 10.3 超越人工不是全面模仿

人工的价值是结构判断，不是每一钩都最优。

算法应继承：

- 存4释放口保护。
- 机接后主列保护。
- 大库先出后进。
- 预修/调棚/功能线独立服务。
- 尾项在主合同完成后收束。

算法可以超越：

- 外场反复取送。
- 存2/存5/存3 多次交叉线重排。
- 功能线和大库主列混合携带导致的重复摘挂。
- 机车空驶和端别切换。
- 库位 swap 依赖人工经验处理。

超越人工必须满足硬前提：

```text
hard_violation_count = 0
所有车辆最终目标满足
大库台位规则满足
关门车/重车/称重规则满足
运行干涉和临停容量满足
求解时间在业务可接受范围内
```

再比较：

| 指标 | 超越标准 |
|---|---|
| 总钩数 | 标准链案例低于人工；短链案例不高于人工 |
| 大库进出次数 | 不多于人工，优先接近“一进一出” |
| 存4污染次数 | 少于人工或等于人工 |
| 叉线/临停使用 | 少于人工，或证明减少后续返工 |
| 机车空驶距离 | 少于人工 |
| 道岔扳动次数 | 少于人工 |
| 修库 swap 等待 | 明确先出后进，减少冲突 |
| 计划可解释性 | 每个关键动作都有 ContractDelta / ResourceDelta |

### 10.4 评估字段建议

每个案例至少输出：

| 字段 | 含义 |
|---|---|
| `case_id` | 案例编号 |
| `manual_file` | 人工作业单路径 |
| `truth2_file` | 匹配的输入 JSON |
| `case_family` | 人工案例族 |
| `manual_hook_count` | 人工钩数 |
| `solver_hook_count` | 算法钩数 |
| `hook_delta` | 算法 - 人工 |
| `hard_violation_count` | 硬违规数量 |
| `contract_coverage` | 需动车合同覆盖率 |
| `residual_vehicle_ratio` | residual 比例 |
| `cun4_release_detected` | 是否识别存4释放 |
| `machine_accept_detected` | 是否识别机接 |
| `depot_digest_complete` | 修库消化是否完成 |
| `depot_slot_swap_valid` | 大库 slot/band swap 是否合法 |
| `function_service_complete` | 功能线是否完成 |
| `pre_repair_stable` | 预修是否稳定归组 |
| `loco_order_valid` | 携带顺序是否合法 |
| `remote_business_transition_count` | 远端/非远端业务切换次数 |
| `runtime_seconds` | 求解耗时 |
| `result_level` | failed / reached_human / exceeded_human |

`result_level` 建议定义：

```text
failed:
  任一硬约束违规，或最终目标不满足。

reached_human:
  无硬违规；
  关键结构不弱于人工；
  标准链钩数 <= 人工 * 1.05；
  短链/直接处理钩数不明显多于人工。

exceeded_human:
  无硬违规；
  关键结构不弱于人工；
  钩数低于人工，或钩数相同但道岔/空驶/存4污染/库位冲突更少；
  输出完整 ContractDelta + ResourceDelta trace。
```

## 11. 最终判断标准

优秀计划不是单纯钩数最少，而是在硬边界零违反、软边界可追踪、策略选择可解释的前提下，用尽量少的动作完成全部债务。

判断计划是否优秀，看八件事：

1. 所有取放都符合真实可达、真实端侧、真实顺序、真实长度和真实容量。
2. 所有阻塞都在被阻塞线路动作前解除或显式受控。
3. 所有临时占用都有退出信号，没有过程状态被偷换成最终状态。
4. 所有阶段切换都有入口条件和退出条件。
5. 主链承诺前完成必要塑形，主链承诺后连续兑现。
6. 功能线、库内、库外、存车线、临停线的业务性质没有混用。
7. 每一钩都能说明它对债务、边界、资源、顺序和后续动作的影响。
8. 低置信状态下不主动跨硬边界，高置信状态下不反复做低价值整理。

达到这个标准，计划才不是靠经验堆出来的动作序列，而是一套可复核、可迁移、可训练、可由求解器学习的调车决策结构。
