# Stage4 相比人工调车缺失能力审核

## 1. 审核结论

本轮重构已经解决旧 Stage4 在当前 `truth2` 数据集上的主要结构性缺口，但没有把 Stage4 扩大成一个独立的人工计划全流程求解器。

当前准确定位是：

> Stage3 完成大库主流程后，Stage4 使用一套确定性的结构策略，对前场残债执行拓扑可达、业务可验、全局无持车泄漏的闭合 session 求解。

全量结果从旧基线的 `88/109` 个 Stage3 可用案例完成，提高到 `102/109`。剩余 7 例都存在由原始 Request 单目标车长直接证明的线路容量赤字，理论至少留下 8 辆，算法实际也只留下 8 辆。换言之，当前 102 个容量可行的 Stage3 可用案例全部完成，没有剩余拓扑或搜索 partial。

| 指标 | 旧基线 | 当前结果 |
|---|---:|---:|
| truth2 案例 | 113 | 113 |
| Stage3 可用 | 109 | 109 |
| 完成 | 88 | 102 |
| Stage3 可用 partial | 21 | 7 |
| Stage3 可用残车 | 75 | 8 |
| 非容量搜索 partial | 21 | 0 |
| complete 平均钩数 | 26.830 | 25.951 |
| complete 中位数 / P95 / 最大值 | 26 / 51 / 61 | 25 / 46 / 54 |
| Stage4 / combined replay 硬违反 | 0 / 0 | 0 / 0 |

这次提升不是通过多策略重跑取得。代码、CLI 和 API 现在只有一个 `structural` 求解入口，每步候选在同一个池中验证和排序；不存在另一套策略、失败后重试或结果择优。

## 2. 证据和验收口径

审核使用五类相互独立的证据：

1. `scripts/stage4_simple/solve.py` 的候选生成、拓扑保护、容量证明、评分和结果口径。
2. `scripts/solver_vnext/spotting.py` 的强台位布局、同线分块和跨线重排实现。
3. `docs/物理约束.md` 与 `solver_vnext.physical.TrackGraph` 的实际线路连接、北端操作、占线和折返约束。
4. `artifacts/manual_restored_interface/bundles` 中 84 个成功人工 Response、2740 条还原动作及人工阶段统计。
5. `artifacts/stage4_refactor_single_full_final` 的 113 例输出，以及对原始 Request 的独立容量复算和 replay 复验。

完成案例必须同时满足：

- Stage4 response 从 Stage4 request 重放无 schema、physical、business、state 硬违反。
- combined response 从原始 Request 重放同样四类硬违反为 0。
- 最终不存在业务未满足车辆。
- 每个闭合 session 结束无 carry，摘挂车辆集合和顺序一致。
- 初始已满足车辆没有被破坏。

partial 不能只看求解器给出的原因。7 个容量案例另行使用原始 Request 中“唯一目标线”的车辆总长、`TRACK_SPECS` 线路长度和 0.5m 容差计算最少留置辆数，因此不是用算法结果反向证明算法正确。

## 3. 人工能力与实现状态

人工计划的关键不是逐辆直送，而是维护线路角色、门槛解锁、机后列形、目标窗口和阶段出口。旧 Stage4 只能做单源前缀和有限目标重放；当前实现状态如下。

| 人工能力 | 旧问题 | 当前结构 |
|---|---|---|
| 目标干净时直接送 | 缓存动作可能抢在直送前 | 真实残债下降候选优先，零残债准备动作只在结构收益成立时保留 |
| 线路作为动态 gate | 静态缓存白名单会占住预修、机走棚、调梁线北 | 对每个持久 Put 比较前后服务路径与机车接近路径，新增锁即拒绝 |
| 临存后可继续取 | 只证明 Put 可达，没有证明后续 Get | 缓存选择同时证明 Put 路径、回取路径和真实目标方向 |
| 存5按目标成段塑形 | 普通前缀截断，清到存2后可能被预修锁死 | 显式存5北/南整段 session，并把容量留置车作为独立组 |
| 清障后完成主服务 | blocker 只能形成局部死循环 | Get/Put route session 在一个闭合边界内清障、服务、恢复 |
| 多来源目标线重建 | 单一 source group 无法形成完整窗口 | layout rebuild 联合来源线、目标既有车和路径 blocker，先求布局再搬运 |
| 同线台位重排 | 车辆已在目标线但顺序错误时无动作 | same-line repack 将连续块分线临停，再按最终顺序逆向重建 |
| 对位线防埋车 | 新到目标车可能挡住待迁出车 | 多迁出目标一律拒绝；单一迁出目标也必须可达、可清且不超牵引深度 |
| 容量不可行诊断 | 把物理无解混入 `no_valid_closed_macro` | 精确选择最少留置辆数，再完成其余可行车辆 |
| 结构进度 | 只统计源线前缀 blocker | 增加路径不可用、目标污染数量/深度、台位重建和 hot gate 阻塞指标 |
| 低钩数优先 | 多策略结果选优，错误来源难定位 | 单一候选池按每解决一辆的钩数、结构成本、路径成本确定性排序 |

仍应注意：当前 session 可以在内部多次 Get/Put 并联合多个来源，但全局状态仍保持闭合。这保留了 replay 安全边界，也意味着它没有完全复制人工跨较长阶段持续持车的自由度。

## 4. 拓扑决策边界

### 4.1 统一规则

每次准备持久 Put 时，求解器执行以下判断：

1. 目标线是车辆真实目标且当前可接收时，优先直接送达。
2. 目标线不干净时，只允许普通非通道终端线上的可恢复堆叠；对位线、业务前置线和仍被其他债务使用的通道不允许。
3. 临存线若仍有外部车辆需要进入，整条线预约保留。
4. 临存线若位于该组从源线到真实目标的中间路径，拒绝使用。
5. 模拟 Put 后，若任何其他残债从“可服务”变成“不可服务”，拒绝。
6. 模拟 Put 后，若机车对其他残债源线新增接近阻塞，拒绝；同一真实目标在目标线汇聚、且后续能统一重建时才允许。
7. 临存车辆本身必须存在后续 Get 路径，否则拒绝。
8. 目标 Put 若新埋住待迁出车辆，必须证明迁出目标唯一、目标干净、路线可用、整段清理不超牵引上限。

这里的“干净”不是简单空线：既有车辆必须满足本线目标和业务台位，并且本线不承担尚未关闭债务的中间通路。

### 4.2 抛丸、机南和机走棚

实际连接包含：

```text
抛丸线 - 渡10 - 机南 - 机走棚
```

因此“先处理抛丸后可以使用机南”有严格边界：

- 抛丸线干净、外部去抛丸车辆可达时，直接送抛丸，不先放缓存。
- 外部仍有去抛丸债务时，把其他车辆持久放在机南或相关通路上，若导致抛丸服务路径从可用变不可用，会被统一 route-lock guard 拒绝。
- 抛丸目标族关闭后，机南才可能在闭合布局 session 内短时承接；它当前不属于持久缓存候选，仍需通过机南的洗油侧或机区债务会继续约束它。
- 抛丸线自身有去大库或其他目标车辆时，可先取出，但落点仍必须通过同一可达性证明，不能默认送机南。

这直接覆盖了“抛丸车放机南会挡住去抛丸路线”的问题，判断依据是当前图和占线，不是固定地永远禁用机南。

### 4.3 洗南、洗北、油漆和洗油北

实际串联关系是：

```text
洗罐站 - 洗罐线北 - 洗油北 - 机走棚
油漆线 ----------- 洗油北 - 机走棚
```

策略边界：

- 洗罐站为空或现有布局可直接接收时，去洗罐车辆直送。
- 洗罐站有强台位冲突时，先求完整布局；浅层可恢复才允许局部处理，深层使用 layout/same-line rebuild。
- 洗罐站仍需进出时，洗罐线北和洗油北是 gate，不是普通缓存。
- 油漆线仍有入线债务时保留接车能力；卸轮或其他来源到油漆的路径会同时检查机南、机走棚、洗油北的当前占线。
- 洗南、洗北和油漆相关债务都关闭后，洗油北才可在闭合布局 session 内短时使用；它当前不属于持久缓存候选。

旧基线中 `0204W/0206W/0206Z/0309W` 的油漆路径和 `0210W/0321Z` 的洗罐路径均在此区域失败；当前这些案例全部完成。

### 4.4 调梁棚和调梁线北

```text
调梁棚 - 调梁线北 - 渡4
```

策略边界：

- 调梁棚仍有入线或重建债务时，调梁线北保持 gate 角色。
- 目标线已有车不等于可直接 Put；先求强制位置、业务窗口和全线顺序。
- 外部来车能作为一个连续窗口插入时，用 target rebuild。
- 需要跨既有窗口或联合多条来源线时，用 layout rebuild。
- 车辆已经都在调梁棚但窗口错误时，用 same-line chunk repack，不制造虚假的外部来源。
- 任一完整 session 都必须通过调梁线北的 Get/Put 接近端和占线回放。

因此 `0116W/0204Z/0205W/0303Z/0305Z/0306W` 等原强台位失败案例不再依赖“首车能否插入”的偶然条件。

### 4.5 预修、存2和存5南

关键连接包含：

```text
存2 - 预修线 - 渡7
渡9 - 预修线
渡8 - 存5线南 - 存5线北
```

旧算法把存5北车辆清到存2后，只看到源线阻塞深度下降，却没有检查存2到存5南会被预修线封住。当前处理为：

- 从存5北抽取前，显式找到所有仍需去存5南的业务段和容量留置段。
- 目标为存5南的车辆尽量在一次 sweep 中处理，不依赖普通前缀数量上限。
- 任何中间缓存 Put 都比较前后 `source -> real target` 服务路径；预修线新增占线导致路径损失时直接拒绝。
- 存5南容量不足时，只留下数学上不可避免的最小车辆子集，其余车辆仍按完整段完成。

结果上，`0306Z` 从旧基线残 9 辆降为容量下界 1 辆；`0109W/0113W/0129Z/0330Z` 也只保留各自容量必需留置量。

### 4.6 机库、机走北和普通终端线

人工成功 Response 中机库线是高频承接和列形资源。当前实现把机库纳入 assembly cache 候选，但不是无条件缓存：

- 有真实机库入线债务时预约保留。
- 占用机库若新增其他车辆的渡4/机区接近锁则拒绝。
- 缓存车辆必须证明可以再次从机库取出。
- 布局 session 可以短时使用更多普通线路，但所有临停都在同一 session 内恢复或进入最终目标。

预修线、机库线等普通终端目标如果已有待迁出车，可以允许目标车形成可恢复堆叠，前提是该线不是当前其他债务的通道、迁出前缀在牵引上限内、且不会形成多目标埋车。这个规则解决了“目标必须完全清空才可送车”造成的不必要等待，同时保留拓扑安全。

## 5. Session 模型

当前不是逐动作贪心，而是以闭合 session 为最小可提交单位。

| session | 入口 | 核心动作 | 退出条件 |
|---|---|---|---|
| `service_sweep` | 北端前缀可牵 | 一次 Get，按尾端目标段连续 Put | 持车清空，真实残债或结构指标改善 |
| `source_corridor_release` | 满足车尾段挡住串联目标 | 整段 Get、尾段临停、主组送达、尾段恢复 | 源线结构恢复，主组完成 |
| `route_clearing` | Get/Put 路径被 1-2 条占线阻断 | 取 blocker、主服务、逆序恢复 | 路径占线恢复且主债务下降 |
| `target_rebuild` | 外部同目标组可连续插入 | 取目标前后段和来源组，一次按最终位置 Put | 目标全线业务合法 |
| `layout_rebuild` | 多来源或跨窗口布局 | 求最终布局、拆连续块、分线临停、统一重建 | 所有参与线闭合、目标布局合法 |
| `same_line_repack` | 车已在目标线但台位错误 | 整线取出、按连续块临停、逆序回取 | 同线最终位置和组窗合法 |
| `cun5_segment_transfer` | 存5北含去存5南完整段 | 整段挂出，目标组和容量留置组分别处理 | 可行车辆完成，留置量等于容量下界 |

session 内可以连续多次 Get，也可以清路径 blocker，但不会把未闭合持车状态交给下一轮搜索。这比旧单一闭合宏更接近人工计划的意图单位，同时避免全局 held 状态使错误难以重放。

## 6. 容量下界

有效容量为 `TRACK_SPECS.length_m + 0.5m`。下表只统计目标集合恰好为该线路的车辆，因此不能通过选择其他等价目标消除赤字。

| 案例 | 单一目标线 | 车辆数 | 总长 m | 有效容量 m | 赤字 m | 理论最少留置 | 实际残车 |
|---|---|---:|---:|---:|---:|---:|---:|
| `0109W` | 存5线南 | 12 | 179.3 | 156.5 | 22.8 | 2 | 2 |
| `0113W` | 存5线南 | 12 | 170.5 | 156.5 | 14.0 | 1 | 1 |
| `0128W` | 洗罐站 | 6 | 93.5 | 89.2 | 4.3 | 1 | 1 |
| `0129Z` | 存5线南 | 12 | 167.2 | 156.5 | 10.7 | 1 | 1 |
| `0203W` | 调梁棚 | 12 | 179.3 | 174.8 | 4.5 | 1 | 1 |
| `0306Z` | 存5线南 | 12 | 169.4 | 156.5 | 12.9 | 1 | 1 |
| `0330Z` | 存5线南 | 12 | 162.8 | 156.5 | 6.3 | 1 | 1 |
| 合计 |  |  |  |  |  | 8 | 8 |

`0128W` 和 `0203W` 是强台位线，留置哪一辆会影响窗口可行性，因此不在搜索前武断指定车号；求解器保留线路容量不可行证明，让布局 session 自己确定最终未满足车辆。只有留置辆数不超过理论最少值、且留置总长足以覆盖全部赤字时才按容量闭合。达到该边界后，两例和其他 5 例一样以 `hard_capacity_infeasible` 显式结束，不再报无有效候选。其余普通线先精确求“最少辆数、最小长度浪费、优先留置易取车辆”的子集。

## 8. 已清理的多策略机制

本轮删除了以下机制和接口：

- `focused/legacy/full` 排名模式。
- `defer/critical` 重排策略分支。
- completion/all/off 多策略运行模式。
- CLI 中的策略选择参数。
- API 中的策略选择参数和多次 `run_solver` 调用。
- 按 replay、残债、钩数从多份求解结果中择优。

当前只保留一个候选生成器和一个排序池。直接 session、结构准备和布局重建都在同一状态上接受相同验证；布局重建通过排序键体现更高结构成本，不是在失败后启用另一套算法。

超时、容量不足、Stage3 不完整或无有效候选都会显式输出对应原因，不会切换算法掩盖根因。批处理对单例异常只记录 `solver_exception` 以继续审计其他案例，不生成替代方案。

## 9. 相比人工仍缺失的能力

### 9.1 不能接管 Stage3 未闭合的大库残局

人工 P3/P4 包含修库出库、存4释放、机库主列接续和大库入库重放。当前系统把这些责任放在 Stage2/Stage3，Stage4 将修 1-4 库目标和卸轮目标视为上游边界。4 个 Stage3 partial 因而不会被 Stage4 接管。

这不是当前 102 个前场案例的缺陷，但意味着 Stage4 不能单独声称与人工计划 P4 等价。若以后要求跨阶段恢复，应先定义明确的 Stage3 -> Stage4 session 契约，而不是让 Stage4 猜测未完成的大库意图。

### 9.2 全局仍不保存跨 session 持车

人工 Response 中约 43% 的 Put 前出现两个及以上 Get 来源。当前 layout、route clearing 和 corridor release 已能在单个 session 内表达多来源 Get，但 session 退出必须清空持车。超出当前牵引和临停上限的更长人工计划仍可能需要新的通用 session，而不是放开任意全局 held。

### 9.3 不证明最少钩

求解器按局部结构成本和残债向量选择，不做全局组合优化。当前总体比旧基线少钩，但 25 个配对案例增加。应优先研究：

1. 相邻 target rebuild 能否合并成一次布局 session。
2. 同一目标线是否只需抽最小冲突窗口，而不是整线重放。
3. blocker 能否在清障时直接完成真实目标，而不是机械恢复。
4. 多个缓存 sweep 能否在保持 gate 开放的条件下合并取送。

这些方向必须继续用单一策略和全量回归验证，不能用多结果选优代替结构优化。

### 9.4 线路业务角色仍有少量静态先验

所有持久 Put 都经过动态图可达性检查，但 `LINE_READY_PREREQUISITES`、持久缓存候选、目标 tier 和源线优先级仍编码了调梁、洗罐、存5等稳定业务语义。它们是全站线路规则，不是案例特判；不过新增线路或改变作业制度时仍需更新领域配置。

当前对已释放线路采取保守策略：机南、洗油北可以在完整布局 session 中短时使用，却不会自动升级为跨 session 持久缓存；调梁线北、洗罐线北等明确候选也必须先满足前置目标关闭条件。这降低了死锁风险，但可能损失人工计划利用已释放 gate 合并钩序的机会。

更长期的方向是从当前债务族和图割点自动推导更多解锁顺序，让静态先验只负责同收益候选的 tie-break。

### 9.5 容量不可行需要业务层决策

7 个容量案例数学上不能让所有单目标车辆同时满足。当前正确行为是给出最小 partial 和明确诊断。生产系统还需要业务规则决定：延期哪辆、是否允许改目标线、是否拆批次或修改计划。求解器不能自行把唯一目标改成其他线路来制造“complete”。

## 10. Stage1 与 Stage4 的边界

本轮 Stage4 补强后，Stage1 不需要为了害怕 Stage4 无解而提前清空所有功能线。更准确的责任划分是：

| 情形 | Stage1 | Stage4 |
|---|---|---|
| 目标线干净、路径清楚 | 直接完成 | 仅处理上游未做的残余 |
| 功能线北端浅阻塞，清后可直送 | 可以清理并完成 | 可作为闭合 sweep 接管 |
| 抛丸/洗罐完成可释放机南/洗油北 | Stage1 优先取得解锁收益 | 动态 route guard 继续保证不误占 |
| 存5北有连续可服务小段 | Stage1 可做浅层塑形 | 处理完整南北段、容量和深层阻塞 |
| 目标线脏且多来源、强台位、跨窗口 | 不应为此打乱大库编组 | layout/same-line rebuild |
| 修库出库、存4释放、大库入库 | Stage1 只负责准备主列 | 由 Stage2/Stage3 主流程负责，不属于当前 Stage4 |
| 容量数学不可行 | 不靠搬运掩盖 | 输出最小留置与明确下界 |

核心边界仍是：干净则直送；脏但浅且能证明清理收益时提前做；深层、多来源、强台位和完整路径清障交给闭合 session。任何阶段都不能为了暂存占住尚未关闭目标族的必要通路。

## 11. 最终判断

当前 Stage4 已从“有限单源闭合宏 + 多策略重跑”重构为“一套拓扑保护下的结构 session 求解器”。它补齐了当前数据集中真正造成失败的能力：动态 gate 保护、缓存可回取、存5成段塑形、多来源布局重建、同线台位重排、可恢复脏终端堆叠和容量下界处理。

验证结果支持三个结论：

1. 可解性已经达到当前 Stage3 可用且容量可行案例的 `102/102`。
2. 剩余 8 辆全部由独立容量下界解释，不再有拓扑死锁残车。
3. 在新增 14 个完成案例的同时，旧完成集合平均每例减少 1.58 钩，且 replay 硬违反保持为 0。

仍不能越界声称人工计划全流程等价或全局最少钩。下一阶段真正有价值的工作是减少布局 session 的搬运窗口、让清障车顺手完成真实目标，并定义 Stage3 未闭合时的显式跨阶段接管契约。
