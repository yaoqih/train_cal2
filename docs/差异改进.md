  放哪一批、线路路径是否可达”开始，而是先生成业务上想做的动作，再让物理校

  不建议继续补丁
  不建议继续调这些东西：

  remote_session_contract_violation_reason 的阈值
  source_clear_restore 的放行条件
  route_clear_session 的排序
  manual + 2 budget 的例外
  各种 p7_reject/p8_reject 分支

  这些只能让某几个案子好一点，但会继续把结构变复杂，而且无法从 31/113 稳定
  回到人工计划水平。

  建议重构路径
  做一个不向后兼容的 vNext，保留硬物理校验，替换求解结构：

  1. 先建 PhysicalState / AccessFrontier
     每一轮先算：调车机在哪、每条线从当前可达端能取到的前缀、路径是否被占
     用、接上车列后的尾部放车顺序。不可达就不生成候选。

  2. 把 route_clear/source_clear 从独立候选降级为“合同内部步骤”
     清障不能自己成为目标，必须绑定后续主动作，并且记录 corridor lease，禁
     止刚清开又马上回填。

  3. 建人工式 Episode
     用三个大块替代大候选池：
     FrontPrepEpisode 前场整理；
     RemoteEpisode 修库/卸轮线整块处理；
     TailCloseoutEpisode 尾部收束。

  4. RemoteEpisode 必须一次声明远端债务
     先决定本次远端 session 要完成哪些 outbound/inbound/internal 车辆，再
     生成物理可达步骤。不能像现在这样每轮临时挑一个修库动作。

  5. 短链单独模板
     0306W、0327W 不能套完整大链。人工计划就是短链/库内消化，目标应是 1 个
     远端块，接近 10-12 勾，而不是 30-50 勾。

  6. 优化目标改成字典序
     第一层硬物理可行；
     第二层远端切换；
     第三层完成合同债务；
     第四层勾数；
     第五层临停污染、存4污染、重复清障。

  结构应该做成什么样
  目标结构应该是这条硬链：

  StartStatus
    -> FlowClassify
    -> FlowGraph / FlowEdge
    -> EdgeContract
    -> TargetContractSelector
    -> StructuralIntent
    -> WorkPatternTemplate
    -> EdgeBoundedCandidateGenerator
    -> ResourceRequest
    -> StationResourceGraph
    -> ContractDelta + ResourceDelta
    -> AcceptRejectGate
    -> LocalTieBreakSearch
    -> State Update / FlowGraph Rebuild
